03Gaps identified (not fixed this pass)

Honest findings surfaced by driving the real loop. Ordered by impact. None blocked the E2E, but each is a rough edge worth a ticket.

1 · Task worktrees aren't wired to the harness — parallel isolation doesn't hold architectural

Fan-out creates a per-task git worktree (.arke/worktrees/<hash> on a task branch), but the implementer session is created on the root-scoped adapter. The observed result: the Foundation task's files landed in the project root, not its worktree, which stayed an untouched checkout. With the default concurrency of 10, ten implementers would write into one directory. On Windows this is compounded by OpenCode's inability to serve a non-primary directory (the same class of issue the per-root harness ports were built for).
mitigation used Ran the E2E with ARKE_MAX_CONCURRENT_TASKS=1 so one task built cleanly. Real fix a per-worktree harness server (reuse perRootHarnessPort) so each task session is scoped to its own worktree directory.

2 · Fan-out queue never auto-advances against OpenCode high

The queue drains on session.status === "done", but the OpenCode adapter only ever emits idle for a finished turn — it has no done. So a completed task stays running and the next queued task never dispatches. The Foundation task finished and produced code, but tasks 2…29 remained queued.
fix Treat a task session's idle-after-dispatch as terminal for a single-turn implementer (it does exactly one turn), or have the adapter emit a synthetic done when a task session goes idle with no pending fan-out follow-up.

3 · Client review panel shows "Review complete" while reviewers are still streaming medium

During the run the panel rendered "Review complete · 0 issues" for ~2 minutes while both reviewers were still mid-analysis. It was partly a symptom of the parse bugs (§2), but the UI also lacks a per-reviewer "reviewing…" state gated on session.idle.
fix Keep each reviewer column in a running state until its panel.issue/idle arrives; only show "complete" after panel.complete.

4 · Project scaffold doesn't gitignore its own .arke/ state medium

After the run, git status in inner-siege lists .arke/sessions.ndjson, .arke/trace.ndjson, .arke/fanout.ndjson and .arke/worktrees/ as changes. The implementer added a node .gitignore, but the scaffold itself never writes a root .gitignore covering Arke's own runtime state, so it leaks into the user's working tree.
fix Scaffold a root .gitignore with .arke/ (state) while keeping grounding rules intact.

5 · Cold project open is slow (> 90 s) medium

First open of a managed-harness project can exceed 90 s (spawn + health + model-catalog read). The client timeout was raised to 180 s so it no longer reports a false failure, but the latency itself is a poor first impression.
fix Stream readiness progress to the UI, and adopt an already-healthy server faster (the adopt path is in place; the catalog read dominates).

6 · Harness version drift is unguarded medium

An unattended opencode auto-update (1.17.11 → 1.17.13) changed the ?directory= wire contract overnight and wedged the coordinator until diagnosed. Arke trusts whatever opencode is on PATH.
fix Record the harness version at connect, warn on an untested range, and document a pinned/vendored harness option.

7 · Reviewer model independence is registry-level only phase-2

The panel enforces distinct capable model strings, but both reviewers currently run through the same adapter instance. True multi-instance / multi-vendor physical routing (so a blind spot in one provider can't mask another) is Phase-2.
fix Route each reviewer to its own connected instance per the pinned instanceId.

8 · The second-human approval gate needs a UI affordance low

Advancing to approved fails closed unless the PR approver differs from the spec owner (correct security posture — no self-approval). But a spec with no owner, or with no way to name a distinct approver, can never be approved. In this run the transition had to be driven by a webhook payload.
fix Surface owner/approver in the cockpit and make the approve action carry a non-owner reviewer identity.
