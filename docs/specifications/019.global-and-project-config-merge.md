# Global + project configuration merge and first-run harness setup

## Why
A harness is a property of *your machine*, not of every repository â€” yet today the only place
to configure one is each project's `.arke/config.json`, so a freshly-opened folder shows "no
coding agent is running" until the engineer hand-writes a registry. Configure the harness once,
globally (the way OpenCode uses `~/.config/opencode/`), let projects inherit and override it, and
present a guided first-run setup when no harness exists yet â€” and a new project just works.

## What changes
- ADDED config-layering â€” the system gains a **global config** at `ARKE_HOME/config.json` and resolves each project context's **effective configuration** by deep-merging the project's `.arke/config.json` over the global file (project wins on conflicts), then applying `ARKE_*` env overrides. This resolves the SPEC-005 open question that deferred a user-level config merged over the project file. (breaking: no)
- MODIFIED config-layering â€” process-wide settings are now read from the **named** global config `settings` block (or `ARKE_*` env), and a project `settings` block stays advisory; the deep-merge SHALL exclude process-wide keys from project override. This makes SPEC-005's "global/launch source" concrete without weakening its boundary. (breaking: no)
- ADDED config-layering â€” an explicit **credential precedence** rule (project `credentialsRef` wins for a same-id instance; else global), closing the second SPEC-005 open question. (breaking: no)
- ADDED harness-onboarding â€” when the effective config has no harness, the launch screen's Harness step presents a guided **quick setup**: choose a supported coding agent (OpenCode Â· Claude Code Â· Codex) with its install + start commands, or point at an existing host and Connect; **Omnigent** is offered as a substrate choice that prompts for an Omnigent URL Arke validates (no host auto-detection). Connecting **persists the harness to the global config**, so every later project inherits it. (breaking: no)

> **Layering stance.** The grain of the split is *machine-level vs. project-level*. `registry.instances` (your harnesses, their `credentialsRef` and `serves`) are machine-level â€” their canonical home is the global config. `registry.roster` (role â†’ tier bindings) and `integrations` are project-level â€” their canonical home is the project file. A project MAY override or extend either; the merge result is the contract. This mirrors OpenCode's "configs are merged together, not replaced" with project precedence over global.
>
> **What does *not* change (SPEC-005 boundaries kept).** Vendor model ids and `credentialsRef` values stay host-side and never reach the client (NFR-1). Process-wide settings (coordinator `port`, `maxProjects`, idle-TTL, OTLP endpoint, global query limits) remain launch-time-only â€” they live in the global config or `ARKE_*` env, never in a project file. Substrate exclusivity holds after the merge: a project resolves to *either* leaf instances *or* an Omnigent substrate, never both ([ADR-0004](../decisions/0004-agent-model-vs-omnigent-and-the-substrate-mapping.md)).

## Requirements

### Requirement: Resolve effective configuration by deep-merging global under project
`capability: config-layering` Â· `delta: ADDED (feat/global-and-project-config-merge)`

The system SHALL maintain an optional global configuration file at `ARKE_HOME/config.json` (the same `arkeHome()` location the project registry already uses â€” `ARKE_HOME` ?? OS default) and SHALL resolve each project context's **effective configuration** by deep-merging that project's `.arke/config.json` **over** the global file, with the project value winning on every conflicting key. The merge SHALL key `registry.instances` by `id` and `registry.roster` by role name: a project entry with the same key replaces the global entry of that key, and non-conflicting entries from both files are retained in the result. After the merge, individual resolved keys MAY be overridden by `ARKE_*` environment variables (env wins; `ARKE_*` are overrides, not the source of truth). The absence of either file SHALL NOT be an error â€” a missing global file yields the project config alone, a missing project file yields the global config alone, and neither present yields an empty-but-valid config.

#### Scenario: Global instance inherited when the project omits it
- **WHEN** the global config defines instance `opencode-local` and the project's `.arke/config.json` defines no instances
- **THEN** the effective config for that project contains `opencode-local`
- **AND** the project routes against it with no per-project instance entry

#### Scenario: Project instance with the same id overrides the global one
- **WHEN** the global config defines `opencode-local` on `localhost:4096` and the project defines `opencode-local` on `localhost:5000`
- **THEN** the effective instance `opencode-local` resolves to `localhost:5000`
- **AND** no duplicate `opencode-local` entry appears in the effective config

#### Scenario: Project roster merges with global instances
- **WHEN** the global config defines the instances and the project defines only a `roster`
- **THEN** the effective config carries the global instances **and** the project roster
- **AND** `RegistryResolver.resolve(role)` resolves each role against the merged result

#### Scenario: ARKE_* env overrides a resolved key
- **WHEN** an `ARKE_*` environment variable is set for a key also present in the merged config
- **THEN** the env value wins over both the project and global file values

#### Scenario: Neither file present is valid, not an error
- **WHEN** neither `ARKE_HOME/config.json` nor the project's `.arke/config.json` exists
- **THEN** the loader returns an empty-but-valid effective config (no instances, no roster)
- **AND** no exception is thrown; the launch screen treats this as "no harness configured" (see harness-onboarding)

### Requirement: Machine-level harness config lives globally; project-level config stays per project
`capability: config-layering` Â· `delta: ADDED (feat/global-and-project-config-merge)`

The system SHALL treat `registry.instances` (harness instances, including their `credentialsRef` and `serves`) as **machine-level** configuration whose canonical home is the global config, and `registry.roster` and `integrations` as **project-level** configuration whose canonical home is the project's `.arke/config.json`. The roster is **strictly project-level**: the global config SHALL NOT carry a roster, so no roleâ†’tier default leaks across repositories. A project MAY override a global instance (by `id`) or add project-only instances; the effective config is always the merge. A first-run write of a newly configured harness (see harness-onboarding) SHALL target the **global** config, never any project's file.

#### Scenario: One global harness serves every project
- **WHEN** a harness instance is configured in the global config and two different projects are opened
- **THEN** both project contexts resolve the same instance from the merge
- **AND** neither project required its own `.arke/config.json` to obtain a harness

#### Scenario: A project pins a role without touching the global harness
- **WHEN** a project's `.arke/config.json` adds `roster.reviewer-b: { tier: capable, instance: opencode-local }`
- **THEN** the effective roster binds `reviewer-b` to `opencode-local`
- **AND** the global config's instances are unchanged for other projects

### Requirement: Process-wide settings come only from the global/launch source
`capability: config-layering` Â· `delta: MODIFIED (feat/global-and-project-config-merge)`

Process-wide settings â€” the coordinator `port`, the `maxProjects` concurrency cap, the project idle-TTL, the OTLP endpoint, and global query limits â€” SHALL be read **only** from the global config's `settings` block or from `ARKE_*` environment variables (env wins). A `settings` block in a project's `.arke/config.json` SHALL be advisory for that project context only (e.g. its own permission/generation timeouts) and SHALL NOT set any process-wide value. The deep-merge SHALL exclude process-wide keys from project override: even when a project `settings` value would win the merge, the process-wide value SHALL come from the global/launch source. A single coordinator cannot bind two ports or apply two concurrency caps, so this boundary is structural, not stylistic.

#### Scenario: A project settings block cannot change the coordinator port
- **WHEN** the global config sets `settings.coordinatorPort: 4319` and a project sets `settings.coordinatorPort: 9999`
- **THEN** the coordinator binds `4319`
- **AND** the project's `9999` is ignored for the process-wide port (it may still inform that context's advisory settings)

#### Scenario: Global settings plus env set process-wide values; env wins
- **WHEN** the global config sets `settings.maxProjects: 5` and `ARKE_MAX_PROJECTS=8` is set in the environment
- **THEN** the effective `maxProjects` is `8`

### Requirement: Credential precedence is explicit â€” project wins, else global
`capability: config-layering` Â· `delta: ADDED (feat/global-and-project-config-merge)`

When both the global and the project config define a `credentialsRef` for the same instance `id`, the **project** value SHALL win; when only one of them defines it, that one SHALL be used. The `credentialsRef` SHALL continue to be resolved on the coordinator host at session-create time and SHALL NOT appear â€” neither the value nor the ref string â€” in any WebSocket frame, snapshot, event, or log entry visible to the client (SPEC-005, NFR-1). This rule closes the SPEC-005 open question on credential resolution order.

#### Scenario: Same instance id defined in both â€” project credentialsRef wins
- **WHEN** the global config sets `opencode-local.credentialsRef: opencode/gateway` and the project sets `opencode-local.credentialsRef: opencode/project-key`
- **THEN** session-create on `opencode-local` resolves the credential via `opencode/project-key`

#### Scenario: Only the global config defines the ref â€” global is used
- **WHEN** an instance's `credentialsRef` is set only in the global config
- **THEN** the project inherits and resolves it from the global value

#### Scenario: No credentialsRef reaches the client after the merge
- **WHEN** a client connects to a project resolved from a global+project merge
- **THEN** no `credentialsRef` value or ref string appears in the snapshot or any event

### Requirement: First-run quick setup when no harness is configured
`capability: harness-onboarding` Â· `delta: ADDED (feat/global-and-project-config-merge)`

When the effective configuration resolves no harness (the global config is absent or its registry has no instances, and no host answers the probe), the launch screen's **Harness** step SHALL present a guided **quick setup** rather than only a re-probe affordance. The quick setup SHALL offer the supported harnesses â€” **OpenCode** (recommended) and an **Omnigent** substrate. For OpenCode it SHALL show the **install** and **start** commands, a primary **"Start OpenCode"** action (Arke spawns the harness for you â€” managed, SPEC-016), and a secondary "or point at an existing host" field with a **Connect** action that attaches to a running server. The panel SHALL state that authentication happens in the harness and that **Arke never collects credentials**. The screen SHALL follow the canonical `arke-design` quick-setup design (`SO_HarnessSetup`), not a re-derived layout. Harnesses without a runnable adapter in this build (Claude Code, Codex) SHALL NOT be offered.

#### Scenario: No global harness shows quick setup, not just re-probe
- **WHEN** the launch screen loads and the effective config has no harness instance
- **THEN** the Harness step renders the quick-setup chooser (OpenCode + Omnigent)
- **AND** the "no coding agent is running" notice explains the host is the trust boundary

#### Scenario: OpenCode offers a managed start and an attach fallback
- **WHEN** the engineer selects OpenCode in the chooser
- **THEN** the panel shows `curl -fsSL https://opencode.ai/install | sh` and `opencode serve --port 4096`
- **AND** a primary **Start OpenCode** action is shown, plus a secondary "or point at an existing host" field with a Connect action

#### Scenario: Start OpenCode spawns the harness and verifies it came live
- **WHEN** the engineer clicks **Start OpenCode**
- **THEN** the coordinator writes a managed OpenCode instance and reloads, whereupon the adapter spawns `opencode serve` (SPEC-016)
- **AND** readiness only becomes ready after the spawned server passes the probe; a spawn failure (e.g. OpenCode not installed) rolls the write back and surfaces the reason rather than a false-ready state

#### Scenario: Attaching to an existing host probes it before proceeding
- **WHEN** the engineer enters a host and clicks Connect
- **THEN** the coordinator probes that endpoint (SPEC-002 capability probe) before persisting
- **AND** readiness only becomes ready after the probe passes; a failed probe surfaces a reason, not a false-ready state

### Requirement: Selecting a harness persists it to the global config
`capability: harness-onboarding` Â· `delta: ADDED (feat/global-and-project-config-merge)`

When a harness is connected through quick setup â€” a host that Arke started or attached to passes the probe â€” the coordinator SHALL write that instance's descriptor (a stable `id`, the `driver`, the `host`/endpoint, a `serves` placeholder for the tiers it will map, and a `credentialsRef` **pointer**) to the **global** config at `ARKE_HOME/config.json`, creating the file (and the directory) if absent. The credential **secret itself SHALL NOT be written** â€” only a reference resolved later on the host. A project opened after this write SHALL inherit the harness through the deep-merge with no per-project config required.

#### Scenario: A successful connect writes the instance to the global config
- **WHEN** quick setup connects a harness and the probe passes
- **THEN** an instance descriptor is appended/updated in `ARKE_HOME/config.json` under `registry.instances`
- **AND** the file is created with that single instance if it did not previously exist

#### Scenario: A newly opened project inherits the harness with no project file
- **WHEN** a harness has been persisted globally and the engineer opens a folder that has no `.arke/config.json`
- **THEN** the project's effective config resolves the global harness
- **AND** the Harness step shows "Reachable" without any project-level configuration

#### Scenario: Only a credentialsRef pointer is written, never a secret
- **WHEN** the global config is written by quick setup
- **THEN** the persisted instance carries a `credentialsRef` string only
- **AND** no API key, token, or credential value is present in the file

### Requirement: Omnigent is selected by URL, validated, and mutually exclusive with leaf harnesses
`capability: harness-onboarding` Â· `delta: ADDED (feat/global-and-project-config-merge)`

The quick setup SHALL always offer **Omnigent** as a distinct substrate choice alongside the leaf coding agents â€” there is **no host auto-detection**. Selecting Omnigent SHALL prompt for an **Omnigent URL**, which the coordinator SHALL **validate** (probe the endpoint) before accepting; an unreachable or invalid URL SHALL be rejected with a reason and SHALL NOT be persisted. A validated Omnigent SHALL configure the registry as a **substrate** (not as leaf instances); per [ADR-0004](../decisions/0004-agent-model-vs-omnigent-and-the-substrate-mapping.md), a project's registry SHALL be *either* leaf instances *or* an Omnigent substrate, never both, and this exclusivity SHALL hold after the global+project merge. Selecting a substrate SHALL NOT disable agent authoring and SHALL NOT hand model choice to the substrate â€” the roster and tiers remain authored in Arke, with `listModels()` reading the substrate's catalog to resolve tiers (SPEC-005, SPEC-016).

#### Scenario: Omnigent is always offered as a choice
- **WHEN** the engineer opens quick setup
- **THEN** "Omnigent" appears as a selectable substrate option distinct from OpenCode / Claude Code / Codex
- **AND** no host detection gates its availability

#### Scenario: Selecting Omnigent prompts for a URL that is validated
- **WHEN** the engineer selects Omnigent and enters an Omnigent URL
- **THEN** the coordinator probes that URL and only accepts it after validation passes
- **AND** an unreachable or invalid URL is rejected with a reason and nothing is persisted

#### Scenario: A validated Omnigent configures a substrate registry, not leaf instances
- **WHEN** a valid Omnigent URL is connected
- **THEN** the persisted registry records a substrate configuration (the validated URL)
- **AND** no leaf `registry.instances` are written for that selection

#### Scenario: A project cannot resolve to both a substrate and leaf instances
- **WHEN** the effective (merged) config would contain both an Omnigent substrate and leaf instances
- **THEN** the coordinator rejects it as a configuration error (substrate exclusivity)
- **AND** the launch screen surfaces the conflict rather than silently picking one

## Design

### Architectural decision
Adopt OpenCode's layered-config model â€” **global defaults, project overrides, deep-merge not replace** â€” split along the machine-vs-project grain, and make the first-run experience write the harness to the **global** layer so it is configured once. The merge happens entirely on the coordinator host, inside each `ProjectContext` at project-open time (SPEC-018); the client continues to receive only a credential-free, model-id-free projection.

### Target architecture

```
ARKE_HOME/config.json  (global: registry.instances [+ default roster], process-wide settings)
        â”‚  deep-merge (project wins; instances keyed by id, roster by role)
.arke/config.json      (project: roster, integrations, project instances/overrides, advisory settings)
        â”‚  + ARKE_* env overrides (individual keys)
        â–Ľ
ProjectContext.effectiveConfig  â”€â”€â–¶ RegistryLoader / RegistryResolver (SPEC-005)
        â–˛                                   (process-wide keys taken from global/launch only)
        â”‚ first-run write (global only)
Coordinator: harness.connect  â—€â”€â”€ Client: quick setup (SO_HarnessSetup)
```

- **`GlobalConfigLoader`** (`packages/coordinator/src/global-config.ts`, new): reads `arkeHome()/config.json` with the same lenient parsing as `registry-config.ts` (drop malformed entries rather than throw); returns `null` when absent.
- **`resolveEffectiveConfig(global, project, env)`** (`packages/coordinator/src/config-resolve.ts`, new): deep-merges the two `RegistryConfig`s with project precedence (instances keyed by `id`, roster by role), applies `ARKE_*` overrides, and strips process-wide keys from the project side. This is the single function `ProjectContext` calls at open; `RegistryLoader` (SPEC-005) then consumes the merged result unchanged.
- **First-run write** (`harness.connect` handler): on a passing probe, upsert the instance into the global file (create dir+file if absent) via an atomic write (temp + rename), mirroring `ProjectRegistry`'s persistence pattern.
- **Client `Picker`** (`packages/client/src/screens/picker-init-library.tsx`): the Harness step's unconfigured/unreachable branch renders the `SO_HarnessSetup` quick-setup panel (ported from `arke-design`'s `util.jsx`) instead of the bare re-probe card; Connect sends `harness.connect`.

> **Per-project scoping (SPEC-018).** The merge and the resolved `effectiveConfig` are **per project context** â€” each `ProjectContext` resolves its own global+project merge at open and releases it on evict. The global file is read once per open (re-read on reload); two open projects each compute their own effective config over the same global base.

### Data model
The global file reuses the existing `RegistryConfig` shape (instances + roster) plus a process-wide `settings` block:

```jsonc
// ARKE_HOME/config.json  (global â€” machine-level)
{
  "registry": {
    "instances": [
      { "id": "opencode-local", "driver": "opencode", "host": "localhost",
        "cwd": ".", "credentialsRef": "opencode/gateway",
        "serves": [{ "tier": "capable", "model": "<model>" }, { "tier": "mid", "model": "<model>" }] }
    ]
    // no roster here â€” the roster is strictly project-level
  },
  "settings": { "coordinatorPort": 4319, "maxProjects": 10, "otlpEndpoint": null }
}
```

```jsonc
// .arke/config.json  (project â€” project-level; overrides/extends the global)
{
  "registry": { "roster": { "reviewer-a": { "tier": "capable", "instance": "claude-local" } } },
  "integrations": { "github": { "enabled": false } }
  // a "settings" block here is advisory for THIS context only â€” never process-wide
}
```

`resolveEffectiveConfig` returns the same `RegistryConfig` the rest of SPEC-005 already consumes â€” no downstream type changes. New types: `GlobalConfig` (registry + process-wide `settings`), `EffectiveConfig` (alias of `RegistryConfig` + a resolved `settings` view).

### Interfaces and contracts
- **New coordinator module** `config-resolve.ts`: `resolveEffectiveConfig(global: GlobalConfig | null, project: RegistryConfig | null, env: NodeJS.ProcessEnv): EffectiveConfig`. Pure and unit-testable; deep-merge keyed by id/role; process-wide keys sourced from `global.settings`/env only.
- **New coordinator module** `global-config.ts`: `loadGlobalConfig(): GlobalConfig | null` and `upsertGlobalInstance(descriptor): void` (atomic write under `arkeHome()`).
- **WebSocket clientâ†’coordinator** (new): `{ type: 'harness.connect', driver, endpoint }` â€” probe/validate the endpoint (for `driver: 'omnigent'`, `endpoint` is the Omnigent URL); on success, persist the instance/substrate to the global config and emit `registry.updated` + a fresh `harness.reachability`. An unreachable or invalid endpoint is rejected with a reason and nothing is persisted.
- **Snapshot extension**: the launch projection gains a `harnessSetup: { configured: boolean }` hint so the client knows whether to show quick setup vs. the reachable/re-probe state. The agent templates and the Omnigent choice are static client metadata, not a server detection. No credentials, refs, or model ids cross the wire (unchanged).
- The quick-setup **agent templates** (id, name, scheme, host, install, start, note, recommended) are the same set already defined as `HARNESS_SETUP` in `arke-design`'s `util.jsx`; they are static UI metadata, not config.

### Cross-cutting
- **Configuration (SPEC-005):** this spec makes SPEC-005's "global/launch source" concrete as `ARKE_HOME/config.json` and adds the deep-merge with project precedence. The per-project `.arke/config.json` remains the project's context-local source; the global file is the machine-level source. `ARKE_*` env still overrides individual keys. Runtime artefacts under `.arke/` remain state, not config.
- **Multi-project scoping (SPEC-018):** the merge is per `ProjectContext`, computed at project-open and released on evict; the global file lives beside `projects.json` under `arkeHome()`. Opening a project upserts it into the project registry as today; resolving its config now also reads the global base.
- **Security (NFR-1):** the global config is user-scoped (under `ARKE_HOME`/OS default). `credentialsRef` is a pointer, never the secret; quick setup never collects credentials â€” authentication happens in the harness. The client projection carries tier labels only â€” no `credentialsRef`, no vendor model id â€” after the merge exactly as before.
- **Harness lifecycle (SPEC-016):** a harness connected through quick setup may be *attached* (point-at-existing-host) or, where SPEC-016 managed mode is configured, *started*; this spec covers configuration + persistence, not the process supervisor, which stays SPEC-016's. Substrate selection ties to SPEC-016 agent materialisation and ADR-0004.
- **Design template:** the quick-setup UI follows the `arke-design` skill (`.claude/skills/arke-design/`) â€” the `SO_HarnessSetup` panel, shadcn neutral tokens, Geist, Lucide, the prototype launch screen; see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Global config + project config, **deep-merge with project precedence** | Configure a harness once per machine and let projects inherit/override; matches OpenCode's "merge not replace" with project winning, the proven layered model |
| 2 | Split by **machine-level vs. project-level**: instances â†’ global, roster + integrations â†’ project | The harness (and its credentials) is a property of the machine; the roleâ†’tier roster and integrations are properties of the repo. The split makes inheritance natural |
| 3 | First-run connect **writes to the global** config, not the project file | So the next project just works; persisting per-project would re-introduce the per-repo setup chore the spec removes |
| 4 | Process-wide settings stay **global/launch-only**; a project `settings` block is advisory | One coordinator cannot bind two ports or apply two caps; this boundary is structural and is preserved through the merge (project keys excluded from process-wide override) â€” extends SPEC-005 |
| 5 | Credential precedence: **project ref wins, else global** | Closes the SPEC-005 open question deterministically; a project can substitute its own credential without editing the machine default |
| 6 | Quick setup never collects credentials; only a `credentialsRef` **pointer** is written | Authentication happens in the harness (NFR-1); the client and the config file stay off the credential path |

| 8 | Reuse the canonical `SO_HarnessSetup` design and the existing `RegistryConfig`/`arkeHome()` plumbing | The quick-setup panel and the global-home resolver already exist; this spec wires them, it does not re-derive them |

| 11 | Connect **verifies the harness came live and rolls the global write back** if not | A server that answers `/global/health` but lacks the full capability surface would otherwise persist and immediately reload as configured-but-dead; verify-then-rollback keeps only genuinely-working connects (review round 1) |

| 13 | Quick setup offers only **OpenCode + Omnigent** (Claude Code / Codex dropped) | Only OpenCode has a runnable adapter and Omnigent is the one substrate; showing harnesses we can't run misleads. Trimmed to the supported set |
