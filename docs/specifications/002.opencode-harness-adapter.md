# OpenCode harness adapter — live integration

## Why
The foundation ships an OpenCode adapter skeleton and a coordinator that runs on a mock
event stream; nothing yet talks to a real harness. Until the adapter drives a live
`opencode serve`, none of the loop (cockpit, board, fan-out) can move real work. This makes
the first harness real, behind the backend-agnostic interface so no harness becomes
load-bearing.

## What changes
- ADDED harness-adapter — `@arke/adapter-opencode` fully implements the `HarnessAdapter`
  contract against a running OpenCode server, normalising its events into the canonical
  domain model, and the coordinator can run on it instead of the mock.
- MODIFIED opencode-integration — the skeleton in `packages/adapter-opencode` becomes a
  working driver: real HTTP calls, a resilient SSE subscription, and tier→model resolution.

## Requirements

### Requirement: Connect to a configured OpenCode server
`capability: opencode-integration` · `delta: ADDED (feat/opencode-harness-adapter)`

The adapter SHALL connect to an OpenCode server identified by a base URL, authenticate with
HTTP Basic credentials when a server password is configured, scope every request to the
project working directory, and report the set of capabilities it supports. Credentials SHALL
be read on the host and SHALL never be returned to the client.

#### Scenario: Authenticated request
- **WHEN** the adapter calls the server and a server password is configured
- **THEN** every request carries an `Authorization: Basic` header and the project `directory`
- **AND** the credentials are never included in any value sent to the browser client

#### Scenario: Capability report
- **WHEN** the coordinator asks the adapter what it supports
- **THEN** it returns the capability set `{ events, todos, diff, permissions, commands }`

### Requirement: Create spec and task sessions
`capability: harness-adapter` · `delta: ADDED (feat/opencode-harness-adapter)`

The adapter SHALL create a session for a specification, and SHALL create a task session as a
child of its spec session, preserving the parent relationship so the board can group a spec
with its tasks.

#### Scenario: Spec session
- **WHEN** a new specification session is requested
- **THEN** a session is created on the server and its id is returned

#### Scenario: Task session is a child of its spec
- **WHEN** a task session is created with a parent spec session
- **THEN** the server records it as a child of that parent
- **AND** the adapter can later resolve the task's owning spec from the session

### Requirement: Drive an agent role at a model tier
`capability: harness-adapter` · `delta: ADDED (feat/opencode-harness-adapter)`

The adapter SHALL send a message to a session naming the agent role and a logical model tier,
resolving the tier to a concrete provider/model via the injected resolver, and SHALL support
both a synchronous send and a non-blocking dispatch that returns without waiting for the run.

#### Scenario: Tier is resolved, not hardcoded
- **WHEN** a message is sent for the `capable` tier
- **THEN** the request carries the concrete provider/model the registry resolves for that tier
- **AND** no vendor model id is hardcoded in the adapter

#### Scenario: Non-blocking dispatch
- **WHEN** a task is dispatched asynchronously
- **THEN** the call returns immediately without waiting for the agent to finish
- **AND** progress is observed only through the event stream

### Requirement: Normalise the harness event stream
`capability: opencode-integration` · `delta: ADDED (feat/opencode-harness-adapter)`

The adapter SHALL subscribe to the server's event stream and translate each provider-native
event into the canonical `DomainEvent` model, emitting only events that conform to the schema
and dropping or rejecting any that do not.

#### Scenario: Session lifecycle maps to status
- **WHEN** the server emits a session idle / error / status event
- **THEN** the adapter emits a normalised `session.status` event carrying the owning spec id and kind

#### Scenario: Permission request maps through
- **WHEN** the server emits a permission-asked event
- **THEN** the adapter emits a normalised `permission.asked` event with the session and request ids

#### Scenario: Malformed provider event is contained
- **WHEN** the server emits an event the adapter cannot map to the schema
- **THEN** the malformed event is dropped at the boundary
- **AND** it never reaches the coordinator's read model or the client

### Requirement: Read per-session todos and diff
`capability: harness-adapter` · `delta: ADDED (feat/opencode-harness-adapter)`

The adapter SHALL read a session's todo list and its diff summary on demand, returning them
in the canonical shapes the board and the session-detail surfaces consume.

#### Scenario: Diff summary
- **WHEN** a session's diff is requested
- **THEN** the adapter returns the added/removed line counts and the changed-file count

### Requirement: Answer a gated action
`capability: harness-adapter` · `delta: ADDED (feat/opencode-harness-adapter)`

The adapter SHALL relay a human's approve/deny decision for a permission request back to the
server. Because the server may acknowledge a stale request, the adapter SHALL NOT treat a
successful HTTP status alone as proof the decision was applied, and SHALL confirm via the
subsequent `permission.replied` event.

#### Scenario: Approval is relayed and confirmed
- **WHEN** a human approves a pending permission
- **THEN** the adapter sends the approve decision to the server
- **AND** treats the action as confirmed only on the matching `permission.replied` event

### Requirement: Survive reconnects without losing state
`capability: opencode-integration` · `delta: ADDED (feat/opencode-harness-adapter)`

Because the server's event stream cannot replay missed events, the adapter SHALL, on
reconnect, re-fetch current session/todo/diff state via the REST API so the read model
re-converges rather than silently drifting.

#### Scenario: Reconnect re-fetches truth
- **WHEN** the event stream drops and is re-established
- **THEN** the adapter re-fetches current state over REST
- **AND** the read model reflects the true current state, not a gap

## Design

### Architectural decision
Implement the existing `OpenCodeAdapter` (in `packages/adapter-opencode`) against the
documented OpenCode HTTP + SSE surface using the global `fetch` (no extra dependency), keeping
all behaviour behind the `HarnessAdapter` interface in `@arke/contracts` so the coordinator
and client never learn an OpenCode-specific fact. Normalisation and validation happen in the
adapter; capability differences are absorbed here, not upstream.

### Target architecture
`@arke/adapter-opencode` (`OpenCodeAdapter implements HarnessAdapter`) ⇄ `opencode serve`
(HTTP for commands, SSE `GET /global/event` for state). The coordinator constructs the adapter
in place of `MockAdapter` and pumps its `streamEvents()` through validate → read model → trace
→ push. Endpoint/event mapping is the one in
[`docs/analysis/opencode-integration-guide.md`](../analysis/opencode-integration-guide.md);
the existing `src/sse.ts` parser is reused. A reconnect loop wraps `streamEvents()` and calls
a `resyncViaRest()` on (re)connect.

### Data model
No persistent store. In-memory `sessions: Map<sessionId, { specId, kind }>` so SSE events
(which carry only a session id) can be enriched into `DomainEvent`s. Tier resolution via an
injected `resolveModel(tier) -> { provider, name }`, defaulting to the `gateway` provider.

### Interfaces and contracts
- Consumes `HarnessAdapter`, `DomainEvent`, `Capability`, `ModelTier` from `@arke/contracts`.
- OpenCode endpoints used: `POST /session` (+`parentID`), `POST /session/:id/message` and
  `/prompt_async`, `GET /session/:id/todo` and `/diff`, `POST /permission/:id/reply`,
  `POST /session/:id/command`, `GET /global/event` (SSE). Always passes `?directory=`.
- Adds `resolveModel` to `OpenCodeConfig` (present) and a `resyncViaRest()` internal.

### Cross-cutting
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.
- **Security (NFR-1/5):** credentials stay on the host; the adapter runs inside the
  coordinator, never in the browser; provider points at the internal gateway.
- **Reliability (NFR-8):** reconnecting subscription; REST resync on reconnect; schema
  validation at the boundary. **Caveats** from the integration guide are honoured: no
  `Last-Event-ID`; stale-permission HTTP 200; the v1.14.42+ message-event regression —
  validate shapes against the live `GET /doc` for the pinned version.

  malformed-event rejection; tier→model resolution; auth/header construction.
- Integration: a stub HTTP/SSE server exercising create→message→todo→diff→permission and a
  forced disconnect proving REST resync.
- Manual: run a real `opencode serve`, drive one session from the coordinator, watch the trace.

### Definition of done

live adapter with a real OpenCode server; a reviewer has signed off.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | `fetch` over `@opencode-ai/sdk` | Zero extra dependency; the surface used is small and stable |
| 2 | REST resync on reconnect | OpenCode has no `Last-Event-ID`; re-fetch is the only correct recovery |
| 3 | Confirm permissions via event, not HTTP status | The reply endpoint returns 200 for stale ids (issue #15386) |
