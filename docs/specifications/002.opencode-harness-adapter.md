# OpenCode harness adapter — live integration

## Why
The foundation ships an OpenCode adapter skeleton and a coordinator that runs on a mock
event stream; nothing yet talks to a real harness. Until the adapter drives a live
`opencode serve`, none of the loop (cockpit, board, fan-out) can move real work. This makes
the first harness real, behind the backend-agnostic interface so no harness becomes
load-bearing.

## What changes
- ADDED harness-adapter — `@arke/adapter-opencode` fully implements the `HarnessAdapter`
  contract against a running OpenCode server, normalising its events into the canonical
  domain model, and the coordinator can run on it instead of the mock.
- MODIFIED opencode-integration — the skeleton in `packages/adapter-opencode` becomes a
  working driver: real HTTP calls, a resilient SSE subscription, and tier→model resolution.

## Requirements

### Requirement: Connect to a configured OpenCode server
`capability: opencode-integration` · `delta: ADDED (feat/opencode-harness-adapter)`

The adapter SHALL connect to an OpenCode server identified by a base URL, authenticate with
HTTP Basic credentials when a server password is configured, and scope every request to the
configured **project root** — a path the adapter canonicalises and validates as the configured
workspace, never a value taken from client or user input. Credentials SHALL be read on the host
and SHALL never be returned to the client.

#### Scenario: Authenticated, directory-scoped request
- **WHEN** the adapter calls the server and a server password is configured
- **THEN** every request carries an `Authorization: Basic` header and a `directory` equal to the
  canonicalised configured project root
- **AND** the credentials are never included in any value sent to the browser client

#### Scenario: A non-workspace directory is refused
- **WHEN** a directory value would resolve outside the configured project root (e.g. via `..` or a
  client-supplied override)
- **THEN** the adapter refuses the request rather than scoping to an arbitrary path

### Requirement: Report capabilities from a startup probe
`capability: opencode-integration` · `delta: ADDED (feat/opencode-harness-adapter)`

The adapter SHALL determine the capabilities it advertises by probing the connected server at
startup (health/version and the availability of the endpoints each capability needs), rather than
hard-coding the set. If an endpoint a declared capability depends on is unavailable, the adapter
SHALL omit that capability so the board and cockpit degrade to the server's real surface — or, when
the capability is required for readiness, SHALL fail readiness with a clear reason. It SHALL never
silently assume support.

#### Scenario: Capability set reflects the live server
- **WHEN** the coordinator asks the adapter what it supports after connecting
- **THEN** the returned set reflects what the probed server actually exposes (a missing endpoint is
  not advertised)

#### Scenario: A required endpoint is unavailable
- **WHEN** an endpoint a required capability depends on is absent or errors at probe time
- **THEN** the adapter fails readiness with a reason naming the missing capability
- **AND** does not present itself as ready with a capability it cannot honour

### Requirement: Create spec and task sessions with resolvable identity
`capability: harness-adapter` · `delta: ADDED (feat/opencode-harness-adapter)`

The adapter SHALL create a harness session for a specification and a child harness session for
each task (via `parentID`), and SHALL record, for every session it creates, the canonical identity
later events need (see Design → Identifiers): the harness `sessionId`, the parent `sessionId` (for
tasks), the `kind` (`spec` | `task`), and the owning specification's frontmatter `spec_id`. This
mapping SHALL be written to the durable session graph, not held only in adapter memory.

#### Scenario: Spec session records its spec_id
- **WHEN** a session is created for specification `SPEC-2026-06-28-x`
- **THEN** the server returns a harness `sessionId`
- **AND** the durable session graph records `sessionId → { kind: spec, spec_id }`

#### Scenario: Task session is a child that resolves to its spec
- **WHEN** a task session is created with its spec session as `parentID`
- **THEN** the server records it as a child of that parent
- **AND** the session graph records `sessionId → { kind: task, parentSessionId, spec_id }` so the
  task resolves to its owning specification

### Requirement: Drive an agent role at a model tier, with correlation
`capability: harness-adapter` · `delta: ADDED (feat/opencode-harness-adapter)`

The adapter SHALL send a message to a session naming the agent role and a logical model tier,
resolving the tier to a concrete provider/model via the injected resolver. Every send SHALL carry
a caller-supplied **correlation id** (the OpenCode `messageID`) so the coordinator can attribute
later events to the originating request. The adapter SHALL offer a synchronous send (returns the
assistant message when the turn completes) and a non-blocking dispatch (returns immediately); for
the non-blocking path, turn completion SHALL be signalled by `session.idle` / the turn-quiescence
receipt defined in SPEC-003, never inferred by the caller.

#### Scenario: Tier is resolved, not hardcoded
- **WHEN** a message is sent for the `capable` tier
- **THEN** the request carries the concrete provider/model the registry resolves for that tier
- **AND** no vendor model id is hardcoded in the adapter

#### Scenario: Events correlate to their request
- **WHEN** a message is sent with correlation id `m1`
- **THEN** the events produced by that turn carry the `sessionId` and `m1` so the coordinator can
  attribute them to the request

#### Scenario: Non-blocking dispatch completes on a signal, not a guess
- **WHEN** a task is dispatched asynchronously
- **THEN** the call returns immediately without waiting for the agent to finish
- **AND** the turn is considered complete only on `session.idle` / the turn-quiescence receipt

### Requirement: Normalise the harness event stream onto canonical identity
`capability: opencode-integration` · `delta: ADDED (feat/opencode-harness-adapter)`

The adapter SHALL subscribe to the server's event stream and translate each provider-native event
into the canonical `DomainEvent` model, attaching the canonical identity for that event (the owning
`spec_id`, harness `sessionId`, and `kind`) resolved from the session graph, and emitting only
events that conform to the schema.

#### Scenario: Session lifecycle maps to status with identity
- **WHEN** the server emits a session idle / error / status event
- **THEN** the adapter emits a normalised `session.status` event carrying the harness `sessionId`,
  the resolved owning `spec_id`, and the `kind`

#### Scenario: Permission request maps through
- **WHEN** the server emits a permission-asked event
- **THEN** the adapter emits a normalised `permission.asked` event with the session and request ids

#### Scenario: An event for an unknown session resolves before emission
- **WHEN** an event arrives for a `sessionId` absent from the session graph
- **THEN** the adapter re-fetches that session (and its parent) from REST to resolve ownership
- **AND** emits the event with correct identity, rather than guessing or emitting unknown ownership

#### Scenario: Malformed or unmappable event is dead-lettered, not silently dropped
- **WHEN** the server emits an event the adapter cannot map or validate
- **THEN** the event is excluded from the read model and the client
- **AND** it is written to the audit trace as a dead-letter record (raw payload, reason, sequence)
  and counted, so the gap is actionable rather than invisible

### Requirement: Read per-session todos and diff
`capability: harness-adapter` · `delta: ADDED (feat/opencode-harness-adapter)`

The adapter SHALL read a session's todo list and its diff summary on demand, returning them
in the canonical shapes the board and the session-detail surfaces consume.

#### Scenario: Diff summary
- **WHEN** a session's diff is requested
- **THEN** the adapter returns the added/removed line counts and the changed-file count

### Requirement: Answer a gated action, with explicit failure handling
`capability: harness-adapter` · `delta: ADDED (feat/opencode-harness-adapter)`

The adapter SHALL relay a human's approve/deny decision for a permission request back to the
server and confirm it via the matching `permission.replied` event — not by HTTP status, since the
reply endpoint returns 200 even for stale ids. The adapter SHALL define what happens when
confirmation does not arrive, and SHALL be idempotent under duplicate decisions and reconnects.

#### Scenario: Approval is relayed and confirmed
- **WHEN** a human approves a pending permission
- **THEN** the adapter sends the approve decision to the server
- **AND** treats the action as confirmed only on the matching `permission.replied` event

#### Scenario: No matching reply within the timeout
- **WHEN** no matching `permission.replied` arrives within a bounded timeout
- **THEN** the adapter marks the decision unconfirmed, re-fetches `GET /permission/`, and surfaces a
  user-visible "could not confirm — retry" state rather than reporting success

#### Scenario: Stale or already-resolved permission id
- **WHEN** a decision targets a permission id the server no longer lists as pending
- **THEN** the adapter reports the request as stale/expired and offers to refresh, instead of
  treating the 200 as success

#### Scenario: Duplicate decision is idempotent
- **WHEN** the same decision for the same permission id is submitted twice
- **THEN** the second submission has no additional effect

#### Scenario: Reconnect while a decision is pending
- **WHEN** the event stream drops while a decision awaits confirmation
- **THEN** on reconnect the adapter re-fetches pending permissions and reconciles the decision's
  state rather than leaving it hung

### Requirement: Recover identity and state on connect and reconnect
`capability: opencode-integration` · `delta: ADDED (feat/opencode-harness-adapter)`

Because the server's event stream cannot replay missed events, the adapter SHALL, on every connect
and reconnect, rebuild the **session ownership graph** and current state from the REST API rather
than trusting in-memory state alone: list sessions, follow `parentID` to recover parent/child
(spec/task) relationships, resolve each session's owning `spec_id`, and re-fetch
session/todo/diff/pending-permission state. The session graph SHALL be durable so a coordinator
restart recovers ownership without re-deriving it from chance event order.

#### Scenario: Reconnect rebuilds ownership and state
- **WHEN** the event stream drops and is re-established
- **THEN** the adapter rebuilds the session graph from REST (sessions + `parentID` + `spec_id`) and
  re-fetches current session/todo/diff/permission state
- **AND** subsequent events are enriched with correct ownership and the read model re-converges

#### Scenario: Coordinator restart recovers ownership
- **WHEN** the coordinator restarts with sessions already running on the harness
- **THEN** the adapter reconstructs `sessionId → { kind, parent, spec_id }` from the durable graph
  and/or a REST resync
- **AND** no event is emitted with unknown or guessed ownership

## Design

### Architectural decision
Implement the existing `OpenCodeAdapter` (in `packages/adapter-opencode`) against the
documented OpenCode HTTP + SSE surface using the global `fetch` (no extra dependency), keeping
all behaviour behind the `HarnessAdapter` interface in `@arke/contracts` so the coordinator
and client never learn an OpenCode-specific fact. Normalisation and validation happen in the
adapter; capability differences are absorbed here, not upstream.

### Target architecture
`@arke/adapter-opencode` (`OpenCodeAdapter implements HarnessAdapter`) ⇄ `opencode serve`
(HTTP for commands, SSE `GET /global/event` for state). The coordinator constructs the adapter
in place of `MockAdapter` and pumps its `streamEvents()` through validate → read model → trace
→ push. Endpoint/event mapping is the one in
[`docs/analysis/opencode-integration-guide.md`](../analysis/opencode-integration-guide.md);
the existing `src/sse.ts` parser is reused. A reconnect loop wraps `streamEvents()` and calls
a `resyncViaRest()` on (re)connect.

### Identifiers (canonical)
These are distinct and MUST NOT be conflated:
- **`spec_id`** — the specification's frontmatter id (e.g. `SPEC-2026-06-28-x`); the unit of work.
  Carried on every `DomainEvent` as `specId`. It is **not** a harness session id.
- **harness `sessionId`** — the OpenCode session id returned by `POST /session`; the execution
  unit, and the board **card id** for that session.
- **parent `sessionId`** — a task session's parent (its spec session), from OpenCode `parentID`.
- **`kind`** — `spec` (a parent session) or `task` (a child session).
- **correlation id (`messageID`)** — caller-supplied per send/dispatch; attributes events to a turn.

A `DomainEvent` therefore carries: `specId` (the frontmatter `spec_id`), `sessionId` (harness),
`kind`, and — where applicable — the correlation id. The board keys a card on `sessionId` and
groups cards by `specId`. (Adding the correlation id to the event envelope is part of the domain
model work in SPEC-003.)

### Session ownership graph & recovery
The adapter maintains a **session graph** mapping `sessionId → { kind, parentSessionId, spec_id }`.
It is built and rebuilt from the harness on connect and reconnect — list sessions, follow
`parentID` to recover parent/child relationships, and resolve each session's owning `spec_id` from
the title/metadata set at creation — and is **persisted** so a coordinator restart recovers it
(the durable store lives with the coordinator; see SPEC-003 for the read model and SPEC-015 for the
trace). In-memory state is a cache, never the source of truth. An event for an unknown `sessionId`
triggers a targeted REST resolve before emission.

### Data model
The durable session graph above. Tier resolution via an injected
`resolveModel(tier) -> { provider, name }`, defaulting to the `gateway` provider. The adapter holds
no other persistent store of its own.

### Interfaces and contracts
- Consumes `HarnessAdapter`, `DomainEvent`, `Capability`, `ModelTier` from `@arke/contracts`.
- OpenCode endpoints used: `POST /session` (+`parentID`), `GET /session` and
  `GET /session/:id/children` (graph rebuild), `POST /session/:id/message` and `/prompt_async`
  (with `messageID`), `GET /session/:id/todo` and `/diff`, `GET /permission/` and
  `POST /permission/:id/reply`, `POST /session/:id/command`, `GET /global/health` (probe),
  `GET /global/event` (SSE). Every request passes a canonicalised, validated `directory`.
- `OpenCodeConfig` carries `resolveModel` (present); the adapter adds: `probeCapabilities()`
  (startup), `rebuildSessionGraph()` (connect/reconnect), `resolveSession(sessionId)` (REST resolve
  for unknown sessions), and the permission-confirmation timeout (configurable).
- The durable session graph and the dead-letter records are written through the coordinator's
  persistence/trace (SPEC-003 / SPEC-015), keeping the adapter's own state a cache.

### Cross-cutting
- **Configuration:** all project configuration — the harness/model registry, coordinator settings, and integrations — lives in the single root file `.arke/config.json` (`ARKE_*` env vars override individual keys; runtime artefacts under `.arke/` are state, not config). See SPEC-005 for the model.
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.
- **Security (NFR-1/5):** credentials stay on the host; the adapter runs inside the coordinator,
  never in the browser; the provider points at the internal gateway. The `directory` is the
  canonicalised configured project root, validated against path traversal (`..`, symlinks,
  absolute escapes) and **never** taken from client or user input.
- **Reliability (NFR-8):** reconnecting subscription; on connect/reconnect the session ownership
  graph is **rebuilt from REST and persisted**, and session/todo/diff/permission state is
  re-fetched; unmappable events are **dead-lettered to the trace** (not silently dropped);
  capabilities are **probed at startup**, not assumed. **Caveats** from the integration guide are
  honoured: no `Last-Event-ID`; stale-permission HTTP 200; the v1.14.42+ message-event regression —
  validated against the live `GET /doc` for the pinned version (see Testing).

### Testing
- Unit: event normalisation table (each OpenCode event → expected `DomainEvent`, with identity
  attached, or dead-letter); dead-letter record shape; tier→model resolution; auth/header
  construction; directory canonicalisation rejects traversal; capability probe (missing endpoint →
  omitted or readiness fail); correlation-id attribution.
- Integration: a stub HTTP/SSE server exercising create→message→todo→diff→permission; a forced
  disconnect proving the graph rebuilds from `parentID`→`spec_id` and state re-fetches; an event for
  an unknown session triggering a REST resolve; permission timeout / stale / duplicate / reconnect.
- Contract: snapshot the pinned server's `GET /doc` (OpenAPI) shape for the endpoints/events used and
  fail on drift, so a server upgrade can't silently break normalisation.
- Manual: run a real `opencode serve`, drive one session from the coordinator, watch the trace.

### Definition of done
All scenarios pass; the conformance and contract suites are green; the coordinator runs end-to-end
on the live adapter with a real (pinned) OpenCode server, recovering correctly across a reconnect
and a coordinator restart; a reviewer has signed off.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | `fetch` over `@opencode-ai/sdk` | Zero extra dependency; the surface used is small and stable |
| 2 | REST resync on reconnect | OpenCode has no `Last-Event-ID`; re-fetch is the only correct recovery |
| 3 | Confirm permissions via event, not HTTP status | The reply endpoint returns 200 for stale ids (issue #15386) |
| 4 | Session ownership rebuilt from REST and persisted, not in-memory only | Events carry only a session id; reconnects and coordinator restarts must recover spec/task identity, not guess it |
| 5 | Canonical identifiers fixed and separated (`spec_id` ≠ harness `sessionId`) | Removes the conflation that made enrichment ambiguous; every event carries `specId`, `sessionId`, `kind` |
| 6 | Capabilities probed at startup, not hard-coded | Server version varies (archive/forks, SSE regressions); degrade or fail readiness honestly |
| 7 | Unmappable events dead-lettered to the trace | Silent drops hide staleness; containment must stay actionable |
| 8 | Correlation via OpenCode `messageID`; completion via `session.idle`/quiescence | The coordinator must attribute events to a request and know when a turn is done |
| 9 | `directory` is the canonicalised configured project root, validated | Prevents path-traversal/escape from client or user input (NFR-1) |

## Open questions
- Which OpenCode version/build to pin, given the upstream archive and the SSE regression? (Drives

- Should message/part (transcript) events be normalised here or in SPEC-003 when the domain model
  gains message events and the correlation id is added to the envelope? (Currently deferred to 003.)
- Where the durable session graph physically lives — an adapter-local store vs the coordinator's
  persistence — to be finalised with SPEC-003 / SPEC-015.
- Default permission-confirmation timeout, and whether it is per-tenant configurable.
