# Managed harness lifecycle, richer approvals & portable agent images

## Why
Reviewing the Omnigent meta-harness ([ADR-0002](../decisions/0002-omnigent-as-candidate-harness-substrate.md)) surfaced three patterns worth adopting now, all on the harness/agent integration layer: Arke should **own the harness process** (start and stop it) rather than only probing whether one happens to be running; the human-in-the-loop decision is too coarse at `approve`/`deny` and should gain **once / always / reject + a message**; and the agents Arke dispatches should be **portable images** (a self-contained, parseable artifact) rather than per-harness markdown the adapter hand-writes. These are independently shippable hardening steps that make the harness layer production-grade and harness-agnostic.

## What changes
- ADDED harness-lifecycle — the adapter can **start, supervise and stop** an OpenCode server it owns (spawn `opencode serve`, track the child, shut it down cleanly), so a fresh machine needs no manually-started harness. (breaking: no)
- MODIFIED opencode-integration — readiness reflects a **managed process** (started-by-us, healthy, owned) rather than only "something answered a probe"; the SPEC-002 startup probe becomes one input to a lifecycle, not the whole story. (breaking: no)
- ADDED human-in-the-loop — the permission decision vocabulary becomes **`once` | `always` | `reject`** with an optional **message**; an `always` decision persists a remembered grant that auto-resolves matching future requests, every auto-grant recorded in the trace. (breaking: yes — `PermissionDecision.granted: boolean` is replaced; see Decision log #2)
- ADDED agent-images — Arke defines a **portable agent image** (a directory: `config.yaml` + `AGENTS.md` + `skills/` + `tools/` + recursive `agents/`), parsed and validated into a typed `AgentImage`, and **materialised per harness** (for OpenCode, into `.opencode/agents/<name>.md`) so the same agent definition runs on any backend. (breaking: no)

## Requirements

### Requirement: Arke owns the harness process lifecycle
`capability: harness-lifecycle` · `delta: ADDED (feat/harness-lifecycle-approvals-and-agent-images)`

The adapter SHALL be able to **start** an OpenCode server it manages (spawning `opencode serve` scoped to the configured project root, on a chosen or ephemeral port, with host-only credentials in the child's environment), **supervise** it (know its process id and health), and **stop** it cleanly on coordinator shutdown. When configured to manage the harness, readiness SHALL reflect a process Arke started and verified — not merely that some server answered a probe. When configured to attach to an externally-running server, the adapter SHALL connect without spawning, and SHALL NOT stop a server it did not start.

#### Scenario: Start a managed server on a fresh machine
- **WHEN** the coordinator starts with `manageHarness: true` and no OpenCode server is running
- **THEN** the adapter spawns `opencode serve` scoped to the configured project root
- **AND** readiness becomes ready only after the spawned server passes the health/capability probe

#### Scenario: Clean shutdown stops only what we started
- **WHEN** the coordinator shuts down (or is aborted) after starting a managed server
- **THEN** the adapter terminates the child process it spawned
- **AND** if the server was external (attach mode), the adapter disconnects but does **not** terminate it

#### Scenario: A managed server that dies is surfaced, not silently degraded
- **WHEN** a managed server process exits unexpectedly
- **THEN** readiness flips to not-ready with a reason naming the harness exit
- **AND** the event is recorded in the trace rather than presenting a healthy board over a dead harness

### Requirement: Richer human decision vocabulary
`capability: human-in-the-loop` · `delta: ADDED (feat/harness-lifecycle-approvals-and-agent-images)`

A human's decision on a gated action SHALL be one of **`once`** (allow this single occurrence), **`always`** (allow and remember for matching future requests), or **`reject`** (deny), and MAY carry a free-text **message** relayed to the harness. The decision SHALL still be confirmed by the matching `permission.replied` event, never by HTTP status (SPEC-002), and the adapter SHALL map the vocabulary onto whatever the target server accepts.

#### Scenario: Once allows a single occurrence
- **WHEN** a human decides `once` on a pending permission
- **THEN** the adapter relays the decision and confirms it by the matching reply event
- **AND** a subsequent identical request still prompts a human

#### Scenario: A message accompanies the decision
- **WHEN** a human decides `reject` with a message "use the migration tool instead"
- **THEN** the message is relayed to the harness with the decision

### Requirement: `always` persists an auditable remembered grant
`capability: human-in-the-loop` · `delta: ADDED (feat/harness-lifecycle-approvals-and-agent-images)`

An `always` decision SHALL persist a **remembered grant** scoped to a stable key (the agent/session and the tool/action class), durable across reconnects and coordinator restarts. A later permission request matching a remembered grant SHALL be auto-resolved without prompting a human, and **every auto-grant SHALL be recorded in the trace** with the id of the rule that authorised it, so propose·decide·execute remains fully auditable. A remembered grant SHALL be revocable.

#### Scenario: Matching request auto-resolves from a remembered grant
- **WHEN** a human has decided `always` for an action, and a matching request later arrives
- **THEN** the coordinator auto-resolves it without a human prompt
- **AND** the trace records the auto-grant and the rule id that authorised it

#### Scenario: Remembered grants survive a restart and are revocable
- **WHEN** the coordinator restarts after an `always` decision
- **THEN** the remembered grant is restored from durable storage
- **AND** revoking the grant causes the next matching request to prompt a human again

### Requirement: Agents are portable images
`capability: agent-images` · `delta: ADDED (feat/harness-lifecycle-approvals-and-agent-images)`

Arke SHALL define an agent as a **portable image** — a directory containing a required `config.yaml` (logical model tier, interaction contract, tool declarations) and optional `AGENTS.md` (instructions), `skills/`, `tools/`, and recursive `agents/` (sub-agents). The image SHALL be parsed and validated into a typed `AgentImage`; an invalid image SHALL be rejected with a reason, never partially loaded. Images SHALL reference a **logical tier**, never a vendor model id (FR-4) — resolution stays in the registry (SPEC-005).

#### Scenario: A valid image parses into a typed AgentImage
- **WHEN** an agent image directory with a valid `config.yaml` is loaded
- **THEN** it parses into a typed `AgentImage` (tier, instructions, tools, sub-agents)
- **AND** no vendor model id appears in the image

#### Scenario: An invalid image is rejected whole
- **WHEN** an image is missing its required `config.yaml` or declares an unknown field
- **THEN** loading fails with a reason naming the problem
- **AND** nothing from the image is partially registered

### Requirement: An agent image materialises into the harness convention
`capability: agent-images` · `delta: ADDED (feat/harness-lifecycle-approvals-and-agent-images)`

The adapter SHALL materialise an `AgentImage` into the connected harness's native agent convention so the same image runs on any backend. For OpenCode this is committed markdown under `.opencode/agents/<name>.md`, with the logical `tier` resolved to a concrete model at session-create time (per [the roster doc](../agent-roster-and-model-resolution.md)), and scoped permissions carried through.

#### Scenario: Image materialises to the OpenCode convention
- **WHEN** an `AgentImage` named `implementer` is materialised for the OpenCode adapter
- **THEN** `.opencode/agents/implementer.md` is produced with the role's instructions and `tier:` reference
- **AND** the tier is resolved to a concrete model by the registry at session-create time, not hard-coded in the file

## Design

### Architectural decision
Adopt the three meta-harness patterns from [ADR-0002](../decisions/0002-omnigent-as-candidate-harness-substrate.md), all **behind the existing `HarnessAdapter` seam** so the coordinator and client learn no new harness fact. Lifecycle management and agent-image materialisation are adapter responsibilities; the richer decision vocabulary and remembered grants are shared contract + coordinator concerns (they outlive any one harness).

### Target architecture
- **Lifecycle:** `OpenCodeAdapter` gains an optional managed mode — a small process supervisor that spawns/owns `opencode serve` (reusing the start/stop/health pattern Omnigent's `OpenCodeNativeServer` uses) and feeds the existing SPEC-002 probe. `init()` either spawns-then-probes (manage) or probes-only (attach); coordinator `stop()` (SPEC-003 abort) also stops a managed server.
- **Approvals:** `PermissionDecision` carries `decision: "once" | "always" | "reject"` + optional `message`. A `GrantStore` (durable, NDJSON, co-located with the session store from SPEC-003) holds remembered grants; the coordinator consults it on `permission.asked` before prompting a human, and records every auto-grant in the trace.
- **Agent images:** a `@arke/contracts` `AgentImage` schema + a loader/validator in a small `@arke/agent-image` module; the adapter materialises an image into `.opencode/agents/`. The canonical roster (spec-author, architect, reviewer-a/b, implementer, researcher) is expressed as images.

### Data model
- `PermissionDecision { permissionId: string; decision: "once" | "always" | "reject"; message?: string }` (replaces `{ permissionId, granted }`).
- `RememberedGrant { id; key: string; scope: { sessionId?; agent?; actionClass }; createdAt; createdBy }` — durable; `key` is the stable match for auto-resolution.
- `AgentImage { name; description?; tier: ModelTier; instructions?; interaction; tools: ToolDecl[]; skills: SkillRef[]; subAgents: AgentImage[] }` — parsed from the image directory; tier is logical only.

### Interfaces and contracts
- `@arke/contracts`: `PermissionDecision` updated; `PermissionAck` unchanged (SPEC-002); new `AgentImage`, `RememberedGrant`. `HarnessAdapter` gains optional `startServer?()/stopServer?()` (lifecycle) and `materializeAgent?(image)`.
- The OpenCode reply mapping translates `once|always|reject` to what the server accepts (the integration guide notes the UI surfaces once/always/reject while the server has historically accepted approve/deny — the adapter owns the mapping and confirms by event).
- The `GrantStore` is read by the coordinator on `permission.asked`; writes are synchronous with the `always` decision and recorded in the trace (SPEC-015).

### Cross-cutting
- **Configuration:** all project configuration — registry, coordinator settings, integrations — lives in the single root file `.arke/config.json` (`ARKE_*` env vars override; runtime artefacts under `.arke/` are state). New keys: `manageHarness` (bool), the managed harness command/port, and `grantStorePath`. See SPEC-005.
- **Design template:** the approval UI (the `once/always/reject` control + message field, and a remembered-grants management view) follows the `arke-design` skill (`.claude/skills/arke-design/`) — shadcn neutral tokens, Geist, Lucide, the prototype screens; see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.
- **Security (NFR-1/5):** managed-server credentials are placed only in the child process environment on the host, never returned to the client; the spawned server is scoped to the canonical project root (SPEC-002 directory validation). `always` grants are a privilege-escalation surface — they are narrowly scoped, fully auditable, and revocable; an auto-grant is never silent.
- **Reliability (NFR-8):** a managed server that dies flips readiness and is traced; remembered grants and the harness lifecycle both survive a coordinator restart (durable store + re-probe on restart).

### Testing
- **Unit:** decision-vocabulary mapping (`once/always/reject` → server reply); `GrantStore` match/persist/restore/revoke; `AgentImage` parse (valid → typed; invalid → rejected whole); materialisation output shape; managed-lifecycle start→ready, stop-only-what-we-started.
- **Integration:** against the stub harness — start a managed server then stop it; an `always` decision auto-resolves a later matching request with a trace record; an image materialises and a session is created naming it; a managed-server crash flips readiness.
- **Manual:** on a fresh machine with no running harness, start the coordinator in manage mode and confirm it brings OpenCode up and tears it down.

### Definition of done
All scenarios pass; the breaking `PermissionDecision` change is migrated across coordinator + client with no `granted` references left; managed lifecycle brings a harness up and down cleanly and survives a restart; remembered grants are durable, auditable, and revocable; agent images parse, validate, and materialise; typecheck + build + tests green; a reviewer has signed off.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Manage the harness process, not just probe it | A fresh machine should need no manually-started server; owning the process gives clean start/stop and honest readiness (ADR-0002) |
| 2 | Replace `granted: boolean` with `once/always/reject` (+message) — breaking | The coarse boolean cannot express "remember this" or carry rationale; the richer vocabulary is proven by Omnigent and small to adopt now, before more code depends on the boolean |
| 3 | `always` grants are durable, auto-resolving, **and always traced** | Remembering a decision is a real convenience but a privilege-escalation surface; auditability (every auto-grant traced + revocable) keeps propose·decide·execute intact |
| 4 | Agent definitions are portable images, materialised per harness | Decouples agent definition from any one harness's convention; the same image runs on OpenCode today and another backend tomorrow (the substrate thesis, ADR-0002) |
| 5 | Keep everything behind `HarnessAdapter` | No coordinator/client learns a harness-specific fact; lifecycle + materialisation are adapter-local |

## Open questions
- Exact match key for a remembered grant — per (session, tool) vs (agent, action-class) vs command-pattern; too broad re-introduces the risk auto-approval is meant to bound.
- Does the pinned OpenCode server accept `once/always/reject` on `/permission/:id/reply`, or only `approve/deny`? (Confirm against `/doc`; the adapter mapping depends on it — ties to SPEC-002's pin.)
- Whether agent-image `tools/` execution is in scope here or deferred to the generation spec (SPEC-013); this spec covers *definition + materialisation*, not a new tool runtime.
- Relationship to SPEC-012 (human-in-the-loop) — this spec introduces the vocabulary + grants; SPEC-012 owns the broader elicitation UX. Reconcile ownership before approval.
