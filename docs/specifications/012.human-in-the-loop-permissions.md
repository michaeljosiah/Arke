# Human-in-the-loop permissions & elicitation

## Why

Agents must not act on gated operations without a human decision between the request and the execution. Permission requests and agent elicitation questions must surface as structured, auditable overlays — not as free text in the transcript — so every decision is visible, loggable, and irreversible only once the engineer has acted.

## What changes

- ADDED permissions — `permission.asked` and `permission.replied` domain events drive structured overlays on the board and in the session-detail view; the engineer's approve/deny decision is sent to the harness and confirmed via the `permission.replied` event.   (breaking: no)
- ADDED elicitation — agent-prompted questions (`question.asked`, `question.replied`, `question.rejected`) surface as the same structured overlay pattern; decisions are traced identically.   (breaking: no)
- ADDED permissions — every permission and elicitation decision is written to the append-only trace with identity and timestamp before the response is relayed to the adapter.   (breaking: no)
- MODIFIED coordinator — the coordinator routes approve/deny decisions from the client through to the adapter's `respondToPermission()` method and writes the outcome to the trace; it does not execute any agent action itself.   (breaking: no)

---

## Requirements

### Requirement: Surface permission requests as structured overlays
`capability: permissions` · `delta: ADDED (feat/human-in-the-loop-permissions)`

The system SHALL render a structured overlay whenever a `permission.asked` event arrives for an active session — both on the delivery board (above the affected card) and inside the session-detail view (above the composer). The overlay SHALL display the permission title and detail from the event, and SHALL present exactly two actions: Approve and Deny. No other interaction SHALL be possible on the affected session until the overlay is resolved. A `permission.asked` event for an unknown session (no matching card in the read model) SHALL be recorded as a `warn` trace entry and discarded — the overlay SHALL NOT be rendered for an unrecognised session.

#### Scenario: Overlay appears on board
- **WHEN** the coordinator pushes a `permission.asked` event for a known session
- **THEN** the board card for that session shows a permission overlay or a `needs-human` badge that opens the overlay on click
- **AND** the card moves to the `needs-human` column (SPEC-010)

#### Scenario: Overlay appears in session detail
- **WHEN** the engineer has the session-detail view open and a `permission.asked` event arrives
- **THEN** the permission overlay renders above the composer
- **AND** the composer is disabled until the overlay is dismissed

#### Scenario: Overlay cannot be dismissed without a decision
- **WHEN** the permission overlay is open
- **THEN** clicking outside the overlay, pressing Escape, or navigating away does NOT dismiss it without a decision
- **AND** the overlay remains until Approve or Deny is activated

#### Scenario: Permission for unknown session is discarded safely
- **WHEN** a `permission.asked` event arrives for a `sessionId` that has no matching card in the read model
- **THEN** no overlay is rendered
- **AND** a `warn` entry is written to the trace recording the unknown `sessionId` and `permissionId`

### Requirement: Route the decision to the harness and confirm via event
`capability: permissions` · `delta: ADDED (feat/human-in-the-loop-permissions)`

The system SHALL send the engineer's approve or deny decision to the coordinator via `ArkeTransport.send()`, which routes it to the adapter's `respondToPermission()` method. The decision SHALL be treated as confirmed only when the coordinator receives the matching `permission.replied` event from the adapter, not on HTTP 200 from the `respondToPermission()` call alone. This follows the confirm-by-event rule in SPEC-002 (`docs/specifications/002.opencode-harness-adapter.md` §Answer a gated action), which documents that OpenCode returns HTTP 200 for stale permission ids (issue #15386). The timeout, stale-id, duplicate, and reconnect-during-pending failure modes are handled at the adapter layer per SPEC-002; this spec does not duplicate those behaviours but DOES require that the coordinator surface the resulting unconfirmed state to the UI so the engineer can retry.

The coordinator SHALL validate the incoming decision message: it SHALL verify that the `permissionId` matches an open `pendingPermission` in the read model for the stated `sessionId`, that the `sessionId` is known, and that the `granted` field is a boolean. A message that fails any of these checks SHALL be rejected with a `warn` trace entry; `respondToPermission()` SHALL NOT be called.

A successful relay that does not produce a `permission.replied` event within a configurable timeout (`ARKE_PERMISSION_TIMEOUT_MS`, default 30 000 ms) SHALL be surfaced as a warning in the session detail; the overlay SHALL remain open so the engineer can retry.

#### Scenario: Approval is relayed and confirmed
- **WHEN** the engineer approves a pending permission
- **THEN** the coordinator calls `adapter.respondToPermission({ permissionId, granted: true })`
- **AND** the overlay closes only when the matching `permission.replied` event with `granted: true` arrives

#### Scenario: Stale or no reply surfaces a warning
- **WHEN** `respondToPermission()` returns but no `permission.replied` event arrives within `ARKE_PERMISSION_TIMEOUT_MS`
- **THEN** the overlay shows a warning ("Awaiting confirmation from harness")
- **AND** the Approve / Deny controls remain available for retry

#### Scenario: Denial closes the session-blocked state
- **WHEN** the engineer denies a permission
- **THEN** `adapter.respondToPermission({ permissionId, granted: false })` is called
- **AND** on the matching `permission.replied` event, `needsHuman` is cleared on the card and the `needs-human` column is vacated (SPEC-010)

#### Scenario: Decision for unknown permission id is rejected
- **WHEN** the coordinator receives a decision for a `permissionId` that has no matching open `permission.asked` in the read model
- **THEN** a `warn` entry is written to the trace recording the unknown id
- **AND** `adapter.respondToPermission()` is NOT called

#### Scenario: Decision with invalid payload is rejected
- **WHEN** the coordinator receives a `permission.decide` message with a missing or non-boolean `granted` field, an unknown `sessionId`, or a `permissionId` that does not belong to the stated session
- **THEN** the message is rejected
- **AND** a `warn` trace entry is written
- **AND** no adapter call is made

### Requirement: Log every decision to the trace with identity and timestamp before relay
`capability: permissions` · `delta: ADDED (feat/human-in-the-loop-permissions)`

The coordinator SHALL write every permission decision to the append-only trace (`packages/coordinator/src/trace.ts`) **before** relaying it to the adapter. The trace record SHALL include the decision (approve/deny), the `permissionId`, the `sessionId`, the approver identity (the authenticated principal making the request, or `"anonymous"` if no auth layer is present — see Open questions), and a wall-clock timestamp. A failure to write the trace SHALL prevent the relay (fail-safe): the coordinator SHALL NOT call `adapter.respondToPermission()` if `trace.write()` throws. This record is the audit source of truth for who authorised what, and when.

The trace record format is: `{ kind: "permission.decision", at, permissionId, sessionId, granted, identity, harness }`.

Elicitation decisions use the parallel format: `{ kind: "elicitation.decision", at, questionId, sessionId, replied: boolean, answer?: string, identity }`.

#### Scenario: Decision appears in trace before relay
- **WHEN** the coordinator receives a valid permission decision from the client
- **THEN** `trace.write({ kind: "permission.decision", ... })` is called before `adapter.respondToPermission()` is invoked
- **AND** no decision is relayed without a corresponding trace entry

#### Scenario: Trace write failure blocks relay
- **WHEN** `trace.write()` throws (e.g. disk full)
- **THEN** `adapter.respondToPermission()` is NOT called
- **AND** the overlay remains open with an error state ("Could not record decision — retry")
- **AND** no decision is silently lost

#### Scenario: Identity is recorded
- **WHEN** an engineer approves a permission
- **THEN** the trace entry contains the engineer's identity string (not null or undefined; may be `"anonymous"` in Phase 1)

#### Scenario: Trace is queryable for audit
- **WHEN** an operator reads the NDJSON trace and filters on `kind === "permission.decision"`
- **THEN** every approve and deny decision for the session is present in timestamp order
- **AND** no decision is missing

### Requirement: Elicitation questions use the same structured overlay
`capability: elicitation` · `delta: ADDED (feat/human-in-the-loop-permissions)`

The system SHALL handle agent-prompted questions (elicitation, mapped to OpenCode's `question.asked` / `question.replied` / `question.rejected` per `docs/analysis/opencode-integration-guide.md` §Questions) through the same structured overlay pattern as permission requests. The `ElicitationEvent` type in the `DomainEvent` union SHALL be introduced by this spec and imported by SPEC-011 (session detail) — see Open questions. An `ElicitationEvent` arrival drives the overlay; the engineer's typed response or rejection is sent to the coordinator, which calls the adapter's corresponding reply endpoint. The overlay SHALL be labelled distinctly ("Agent is asking…") so the engineer knows this is a question, not a tool-permission gate.

The confirm-by-event pattern applies to elicitation in the same way as permissions: the overlay closes only on the matching `question.replied` or `question.rejected` event, not on HTTP status from the relay call. A relay that produces no confirming event within `ARKE_PERMISSION_TIMEOUT_MS` SHALL display a warning and keep the overlay open. Because OpenCode does not persist questions across restarts (`docs/analysis/opencode-integration-guide.md` §Caveats), the coordinator SHALL re-fetch pending questions via `GET /question/` on every connect and reconnect and reconcile the `pendingElicitation` state in the read model, mirroring the permission re-fetch behaviour.

Elicitation decisions SHALL also be written to the trace with identity and timestamp under `kind: "elicitation.decision"` before the relay call, subject to the same fail-safe rule as permissions.

#### Scenario: Question overlay appears
- **WHEN** the coordinator pushes a `question.asked` DomainEvent for a known session
- **THEN** the elicitation overlay renders with the question text
- **AND** the engineer can type a reply or activate "Decline to answer"

#### Scenario: Reply is sent and confirmed by event
- **WHEN** the engineer submits a reply to a question
- **THEN** the coordinator calls the adapter's reply endpoint with the answer
- **AND** the overlay closes only on the matching `question.replied` event, not on HTTP status

#### Scenario: Question timeout surfaces a warning
- **WHEN** the relay returns but no `question.replied` or `question.rejected` event arrives within `ARKE_PERMISSION_TIMEOUT_MS`
- **THEN** the overlay shows a warning ("Awaiting confirmation from harness")
- **AND** the reply controls remain available for retry

#### Scenario: Elicitation decision is traced before relay
- **WHEN** the engineer replies to or rejects an agent question
- **THEN** `trace.write({ kind: "elicitation.decision", ... })` is called before the relay
- **AND** a trace write failure blocks the relay

#### Scenario: Pending questions re-fetched after harness reconnect
- **WHEN** the harness restarts or the event stream reconnects
- **THEN** the coordinator re-fetches `GET /question/` and updates `pendingElicitation` in the read model
- **AND** any question that was pending before the reconnect is surfaced again if it is still open on the harness

### Requirement: Session blocked on a decision shows needs-human until answered
`capability: permissions` · `delta: MODIFIED (feat/human-in-the-loop-permissions)`

The system SHALL keep a session's card in the `needs-human` column for the entire duration that a permission request or elicitation question is open — from the `permission.asked` / `question.asked` event until the matching `permission.replied` / `question.replied` / `question.rejected` event is received. The coordinator's `ReadModel` already sets `needsHuman = true` on `permission.asked` and `false` on `permission.replied`; this requirement extends the same logic to `question.asked` / `question.replied` / `question.rejected` events.

A session SHALL NOT leave the `needs-human` column solely because a different event (e.g. `todo.updated`, `message.part`) arrives while a permission or question is still open. The `needsHuman` flag is sticky until its clearing event arrives.

#### Scenario: Card stays in needs-human during open permission
- **WHEN** a `permission.asked` event is open and no reply has arrived
- **THEN** the card remains in `needs-human` regardless of any other events for that session
  (including `todo.updated` and `message.part`)

#### Scenario: Card leaves needs-human on reply
- **WHEN** a `permission.replied` event arrives for the open permission
- **THEN** the card's column is recomputed (SPEC-010) without `needsHuman` — it may return to `implementing` or another column based on session and spec status

#### Scenario: Elicitation follows the same needs-human lifecycle
- **WHEN** a `question.asked` event arrives for a session
- **THEN** `needsHuman` is set to `true` on the card
- **AND** it is cleared only on `question.replied` or `question.rejected`

#### Scenario: Multiple overlapping permissions are handled sequentially
- **WHEN** a second `permission.asked` event arrives for a session that already has an open permission
- **THEN** the first permission is surfaced until resolved
- **AND** the second permission is queued and shown after the first is resolved (or both are surfaced with the earliest-arrived as primary)
- **AND** `needsHuman` remains set until all pending permissions and questions are resolved

---

### Requirement: Coordinator routes decisions without executing agent actions
`capability: permissions` · `delta: MODIFIED (feat/human-in-the-loop-permissions)`

The coordinator SHALL relay the engineer's decision to the adapter and SHALL NOT itself execute any agent action, write any file, or call any external service as a consequence of the decision. The adapter is the only actor that may act on the decision; the coordinator's role is to be the faithful, traceable relay. This is the propose-decide-execute boundary for gated operations: the agent proposes, the engineer decides, and the adapter executes.

#### Scenario: Coordinator relays but does not act
- **WHEN** the coordinator receives an approve decision for a valid permission
- **THEN** it writes to the trace and calls `adapter.respondToPermission()`
- **AND** it does NOT write any files, emit any side-effectful commands, or trigger any other adapter method as a consequence

#### Scenario: Coordinator ignores unknown permission ids
- **WHEN** the coordinator receives a decision for a `permissionId` that has no matching open `permission.asked` in the read model
- **THEN** it writes a `warn` entry to the trace
- **AND** it does NOT call `adapter.respondToPermission()` for the unknown id

## Design

### Architectural decision

Permission and elicitation handling shares a single overlay component (`<PermissionOverlay>`) parameterised by `type: "permission" | "elicitation"` and the payload from the domain event. All state lives in the coordinator's read model (`pendingPermission` and `pendingElicitation` per session); the client's `useSession()` hook (SPEC-011) exposes them as nullable fields. The coordinator enforces trace-before-relay so the audit record is never dependent on the adapter acknowledging the call. Input validation (permissionId ownership, known sessionId, boolean granted) is performed in the coordinator's `onClientMessage` handler before any trace or relay step.

### Target architecture

```
DomainEvent stream
  └─▶ permission.asked / question.asked
        └─▶ ReadModel: needsHuman = true, pendingPermission/pendingElicitation set
              ├─▶ board: card → needs-human (SPEC-010)
              └─▶ session-detail: <PermissionOverlay> renders (SPEC-011)

Engineer activates Approve / Deny / Reply
  └─▶ ArkeTransport.send({ type: "permission.decide", permissionId, granted })
        └─▶ coordinator.onClientMessage()
              ├─▶ validate: permissionId ∈ pendingPermission for sessionId; granted is boolean
              ├─▶ trace.write({ kind: "permission.decision", ... })  ← before relay; fail-safe
              └─▶ adapter.respondToPermission({ permissionId, granted })
                    └─▶ (adapter) → OpenCode POST /permission/:id/reply
                          └─▶ permission.replied event arrives
                                └─▶ ReadModel: needsHuman = false
                                      └─▶ overlay closes · card leaves needs-human

On harness connect / reconnect:
  └─▶ re-fetch GET /permission/ and GET /question/
        └─▶ reconcile pendingPermission / pendingElicitation in ReadModel
              └─▶ overlays re-rendered for any still-open requests
```

### Data model

No new persistent store. The `ReadModel` is extended with:

```typescript
interface PendingPermission {
  permissionId: string;
  title: string;
  detail?: string;
  askedAt: number;  // epoch ms from event ts
}

interface PendingElicitation {
  questionId: string;
  text: string;
  askedAt: number;
}

// Added to CardState:
pendingPermission: PendingPermission | null;
pendingElicitation: PendingElicitation | null;
```

Multiple concurrent permissions for a single session are held in an ordered array internally
(`PendingPermission[]`); the primary permission (earliest `askedAt`) is exposed as the scalar
`pendingPermission` field for UI simplicity. If needed in a future phase, the UI can receive
the full array.

Trace records:

```
{ kind: "permission.decision", at, permissionId, sessionId, granted, identity, harness }
{ kind: "elicitation.decision", at, questionId, sessionId, replied: boolean, answer?: string, identity }
{ kind: "permission.warn", at, reason, permissionId?, sessionId? }   // for validation failures
```

**New client messages (outbound via `ArkeTransport.send()`):**

```typescript
{ type: "permission.decide"; sessionId: string; permissionId: string; granted: boolean }
{ type: "elicitation.reply"; sessionId: string; questionId: string; answer: string }
{ type: "elicitation.reject"; sessionId: string; questionId: string }
```

`sessionId` is included on every decision message so the coordinator can validate that the
`permissionId` / `questionId` belongs to the stated session, preventing cross-session
decision injection.

These are added to the coordinator's `onClientMessage` handler in `packages/coordinator/src/server.ts`.

### Interfaces and contracts

- Reads `PermissionAskedEvent`, `PermissionRepliedEvent`, `ElicitationEvent` from `@arke/contracts/src/events.ts`.
- `ElicitationEvent` (`question.asked`, `question.replied`, `question.rejected`) is defined in this spec and re-exported for SPEC-011 consumption.
- `adapter.respondToPermission()` is already defined in `@arke/contracts/src/adapter.ts`; no change needed.
- New adapter methods: `respondToElicitation(questionId, answer)` and `rejectElicitation(questionId)` — added to `@arke/adapter-opencode` only; not required in the base `HarnessAdapter` interface (optional capability).
- New coordinator message handler verbs: `permission.decide`, `elicitation.reply`, `elicitation.reject`.
- `<PermissionOverlay>` in `packages/client/src/components/permission-overlay.tsx` (new file); used by both board card and session detail.
- Confirmation timeout configurable via `ARKE_PERMISSION_TIMEOUT_MS` env var (default 30 000 ms).

### Cross-cutting
- **Configuration:** all project configuration — the harness/model registry, coordinator settings, and integrations — lives in the single root file `.arke/config.json` (`ARKE_*` env vars override individual keys; runtime artefacts under `.arke/` are state, not config). See SPEC-005 for the model.
- **Multi-project scoping (SPEC-018):** the coordinator supervises N isolated **project contexts**, each keyed by `projectId` with its own read model, trace, and durable stores (session store, grant store, proposal buffer) under its own `.arke/`. Each client connection resolves against its **active project**; events carry `projectId`. The grant store and trace are therefore **per project context** — a permission decision in project A is written only to A's `.arke/trace.ndjson` and never touches B's trace or read model — and `ARKE_PERMISSION_TIMEOUT_MS`, though a global env var, is enforced **independently per project context**. See [SPEC-018](018.multi-project-workspaces.md).
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.

**Security (NFR-1):** the browser never holds a harness credential; the client sends only a structured decision message over the local WebSocket; the coordinator holds the adapter reference and performs the actual HTTP call to the harness. The `sessionId` field on decision messages is used for cross-session injection validation only — it is never used to derive credentials or routing.

**Authorisation (Phase 1):** Arke has no authentication layer in Phase 1; every decision is attributed as `"anonymous"`. The identity field is recorded explicitly so a Phase 2 auth layer can replace it without a schema change. An authenticated coordinator session (e.g. a local session token) will supply the identity string; until then `"anonymous"` is the required explicit value (not null, not omitted).

**Audit (NFR-7):** trace-before-relay is enforced in the coordinator's `onClientMessage` handler by calling `trace.write` before `adapter.respondToPermission()`; a failure in `trace.write` MUST prevent the relay (fail-safe). Every trace entry is append-only NDJSON with `at` (epoch ms). Validation failures (unknown id, invalid payload) are also traced as `permission.warn` entries so silence-on-rejection is never invisible.

**Permissions not persisted across harness restart:** per the OpenCode integration guide (§Caveats, issue #15386), pending permissions are not persisted on the harness across restarts. The coordinator re-fetches `GET /permission/` (and `GET /question/`) on every connect and reconnect and reconciles the read model accordingly. Any permission that disappears from the harness after a restart is treated as resolved-unknown and its overlay is closed with a `warn` trace entry.

**Accessibility:** `<PermissionOverlay>` uses `role="dialog"`, `aria-modal="true"`, and traps focus within the overlay until resolved. Button labels are explicit ("Approve" / "Deny", not "Yes" / "No"). Focus is returned to the triggering element on close.

**Reliability (NFR-8):** the confirm-by-event rule (decision confirmed only on `permission.replied`, not on HTTP 200) handles OpenCode's known stale-reply behaviour (issue #15386, `docs/analysis/opencode-integration-guide.md` §Caveats). A timeout produces a warning without closing the overlay, so the engineer retains control. Per SPEC-002, the adapter is responsible for timeout enforcement, stale-id detection, duplicate-idempotency, and reconnect-reconciliation; this spec requires the coordinator to surface the resulting unconfirmed / stale / error state to the overlay.

### Testing

- **Unit (read model):** `question.asked` sets `needsHuman` and `pendingElicitation`; `question.replied` clears both; `question.rejected` clears both; `permission.asked` for unknown session is no-op (warn); `todo.updated` does NOT clear `needsHuman` while a permission is open; two concurrent permissions: primary is first-asked, `needsHuman` cleared only when both are resolved.
- **Unit (coordinator routing):** unknown permission id → trace warn → no adapter call; valid permission id from wrong session → trace warn → no adapter call; non-boolean `granted` → trace warn → no adapter call; known permission id → trace write → adapter call → overlay closes on reply event.
- **Unit (reconnect reconcile):** re-fetch returns empty list for a previously-open permission → overlay closed + warn trace; re-fetch returns the permission still open → overlay remains.
- **Unit (component):** `<PermissionOverlay>` renders title + detail; disables Approve/Deny while awaiting confirm; shows timeout warning after `ARKE_PERMISSION_TIMEOUT_MS`; shows trace-failure error state when trace write fails; does not close on click-away or Escape; focus is trapped inside.

- **Manual:** run against real OpenCode, trigger a file-write permission, approve it, verify the trace contains the decision; trigger a stale permission reply and confirm the timeout warning; restart OpenCode mid-pending permission and confirm the overlay reconciles on reconnect.

## Decision log

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Confirm permission via `permission.replied` event, not HTTP 200 | OpenCode returns 200 for stale ids (issue #15386); event-confirmed semantics are the only reliable signal. Adapter-layer failure modes (stale/duplicate/reconnect) are handled per SPEC-002 and surfaced here — not duplicated. |
| 2 | Trace-before-relay with fail-safe: relay blocked if trace write fails | Audit completeness is non-negotiable; a missing trace entry would be invisible and unrecoverable |
| 3 | Single `<PermissionOverlay>` component for both permission and elicitation | Structural parity enforces identical audit treatment; avoids a bespoke elicitation path that bypasses trace logic |
| 4 | Overlay not dismissible without a decision | A dismissed overlay with no decision would leave the session in an unknown state and create a gap in the trace |
| 5 | Coordinator does not act on decisions, only relays | Enforces propose-decide-execute: the coordinator is the traceable relay, the adapter is the executor |
| 6 | Input validation before trace/relay (sessionId + permissionId cross-check) | Prevents cross-session decision injection; ensures every adapter call is for a permission that belongs to the stated session |
| 7 | `ElicitationEvent` defined in SPEC-012 (not SPEC-011) | Elicitation is a governed action requiring trace/audit, owned by the permissions spec; SPEC-011 imports the type for UI rendering |
| 8 | Identity recorded as `"anonymous"` (not null) in Phase 1 | Explicit sentinel is queryable and replaceable without schema change when Phase 2 auth lands |
| 9 | Re-fetch `GET /permission/` and `GET /question/` on every reconnect | OpenCode does not persist permissions across restarts; the read model must reconcile or orphaned overlays remain forever open |
| 10 | `needsHuman` is sticky — only cleared by the matching reply event | Other events (todo, message parts) must not race the permission clear; the human controls the exit condition |

## Open questions

- `ElicitationEvent` is also referenced in SPEC-011. This spec owns the definition; SPEC-011 should import it. Confirm the import direction during implementation to avoid double-definition.
- Identity field in Phase 1: `"anonymous"` is the required explicit value. Phase 2 may add a lightweight coordinator auth (local session token or OS user). The trace schema accommodates this without change.
- Timeout for confirmation (`ARKE_PERMISSION_TIMEOUT_MS`): 30 s is a reasonable default for interactive permissions, but some tool operations (e.g. a slow bash command waiting on a permission) may take longer.
