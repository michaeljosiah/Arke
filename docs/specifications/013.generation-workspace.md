# Generation workspace — propose · decide · execute

## Why
An approved specification is the source of truth for everything downstream — docs, tickets,
test scaffolds, tracking entries — but today nothing generates them. Engineers create
these artefacts manually, duplicating information already in the spec and introducing drift
the moment either side is edited. This spec adds the generation workspace: an agent-assisted
propose–decide–execute flow that produces downstream artefacts from the approved spec, shows
them to the human for review, and writes nothing until approval is given.

## What changes
- ADDED generation — from an approved specification an agent proposes the set of downstream
  artefacts; the proposals are previewed in the generation workspace before any write occurs
- ADDED artifact-fanout — once the human approves, the coordinator fans artefacts to their
  targets through the harness; writes to systems of record are deferred to SPEC-014
- MODIFIED generation — the workspace regenerates artefacts when the specification changes;
  downstream edits outside the spec do not flow back

## Requirements

### Requirement: Agent proposes downstream artefacts from an approved spec
`capability: generation` · `delta: ADDED (feat/generation-workspace)`

The system SHALL, when a specification reaches `approved` status (or the human explicitly
triggers regeneration), dispatch an agent session that reads the specification and proposes
the full set of downstream artefacts — documentation, ticket descriptions, test scaffolds,
and tracking entries — returning them as structured proposals, not live writes. The agent
SHALL be given the canonical spec markdown read from git and SHALL NOT be given any
downstream artefact content as input; the spec is always the sole source for generation
input.

#### Scenario: Generation triggered on approval
- **WHEN** a `spec.status approved` event is received and no live generation session
  already exists for that spec
- **THEN** the coordinator creates a generation session and dispatches the artefact-proposal
  agent via `adapter.dispatchAsync()`
- **AND** the session's proposed artefacts are buffered, not written anywhere

#### Scenario: Sequential re-approval does not accumulate orphaned sessions
- **WHEN** a spec transitions `approved → in-review → approved` and a prior generation
  session for that spec has already completed
- **THEN** the coordinator treats the new approval as a fresh regeneration trigger
- **AND** no more than one live generation session exists for the spec at any time; any
  prior completed session's proposal is superseded

#### Scenario: Human triggers explicit regeneration
- **WHEN** the human clicks "Regenerate" in the generation workspace for an approved spec
- **THEN** a new generation session is created, superseding any previous proposal
- **AND** if a prior generation session's in-flight writes are still executing, those writes
  are completed to their current state (not aborted mid-write) before the old proposal is
  discarded
- **AND** the workspace shows the new proposal once the new session reaches `session.idle`

#### Scenario: Duplicate trigger while session is live is a no-op
- **WHEN** `trigger-generation` is received while a generation session for that spec is
  already running
- **THEN** the coordinator ignores the duplicate and the client shows a "generating…" state

#### Scenario: Generation session is isolated from task sessions
- **WHEN** task execution sessions and a generation session run concurrently for the same
  spec
- **THEN** the generation session operates in its own session scope
- **AND** it does not interfere with the task sessions' git worktrees or file states

### Requirement: Proposals are presented for human review before any write
`capability: generation` · `delta: ADDED (feat/generation-workspace)`

The system SHALL present the agent's proposed artefacts in the generation workspace as a
reviewable preview. The human SHALL be able to approve individual artefacts, approve the
whole proposal, edit individual artefacts, or reject individual artefacts or the proposal
entirely. Nothing SHALL be written to any system of record before the human issues an
approval decision. The coordinator SHALL validate that the structured output from the agent
session is well-formed before presenting a proposal; if the output cannot be parsed or is
empty, the coordinator SHALL surface a user-visible error and return the workspace to a
state where regeneration can be triggered.

#### Scenario: Preview rendered before any write
- **WHEN** the agent session completes and proposals are buffered
- **THEN** the generation workspace renders each proposed artefact as a preview card
- **AND** no write to any external system has occurred at this point

#### Scenario: Unparseable or empty agent output surfaces an error
- **WHEN** the agent session reaches `session.idle` but the structured output cannot be
  parsed or is empty
- **THEN** the coordinator does NOT emit a `generation.proposed` event
- **AND** the generation workspace shows a user-visible error (e.g. "Generation failed —
  could not parse agent output")
- **AND** the workspace returns to a state where regeneration can be triggered

#### Scenario: Human edits a proposal before approval
- **WHEN** the human modifies a proposed artefact in the workspace
- **THEN** the edited text replaces the agent's proposal in the pending set
- **AND** the approval decision applies to the edited version, not the original proposal

#### Scenario: Human partially approves a proposal
- **WHEN** the human approves a subset of the proposed artefacts (selecting specific items)
- **THEN** only the approved artefacts are written; the remaining artefacts are neither
  written nor preserved as pending
- **AND** the coordinator records which artefact ids were approved in `GenerationDecidedEvent`

#### Scenario: Human rejects the proposal
- **WHEN** the human clicks "Reject" on the full proposal
- **THEN** the pending artefacts are discarded
- **AND** the generation workspace returns to a state where regeneration can be triggered

#### Scenario: Generation hangs and times out
- **WHEN** a generation session is dispatched but `session.idle` is not received within a
  configured timeout (default: 5 minutes)
- **THEN** the coordinator marks the generation session as timed out
- **AND** the generation workspace shows a user-visible timeout error with a "Retry"
  option
- **AND** the coordinator emits no `generation.proposed` event for the timed-out session

### Requirement: Approval command is anchored to the proposal it approves
`capability: generation` · `delta: ADDED (feat/generation-workspace)`

The system SHALL require every `approve-generation` and `reject-generation` client command
to carry a `proposalId` (equal to the `sessionId` of the generation session that produced
the proposal) so the coordinator can verify the decision targets the current pending
proposal for that `specId` and reject stale or mismatched decisions.

#### Scenario: Approval targets the current proposal
- **WHEN** the human approves a proposal and the `approve-generation` command carries a
  `proposalId` matching the current pending proposal's `sessionId` for that `specId`
- **THEN** the coordinator proceeds with the approval
- **AND** the `GenerationDecidedEvent` carries the matching `proposalId` / `sessionId`

#### Scenario: Stale approval is rejected
- **WHEN** an `approve-generation` command arrives with a `proposalId` that does not match
  the current pending proposal for that `specId` (e.g. the proposal was superseded by a
  re-run)
- **THEN** the coordinator rejects the command with an error response
- **AND** no write is attempted

### Requirement: Artefacts are written only after human approval
`capability: artifact-fanout` · `delta: ADDED (feat/generation-workspace)`

The system SHALL execute artefact writes only after the human issues an explicit approval
decision, routing approved artefacts through the harness (and, for systems of record, via
the deterministic projection mechanism in SPEC-014). The coordinator SHALL persist the
approval decision — including the set of approved artefact ids and their final content — in
the trace before executing any write. On a write-path restart, the coordinator SHALL
re-derive which artefacts are pending completion from the trace rather than from in-memory
state alone.

#### Scenario: Approval triggers writes through the harness
- **WHEN** the human approves the proposal (in full or in part)
- **THEN** the coordinator records the approval decision in the trace, including the
  `specId`, `proposalId` / `sessionId`, approved artefact ids, and final artefact content
- **AND** dispatches each approved artefact write through the harness
- **AND** the client receives write-status events for each artefact as they complete

#### Scenario: Approval is recorded before any write is attempted
- **WHEN** the approval decision is issued
- **THEN** the coordinator writes the approval to `Trace` before calling any write method
- **AND** if the coordinator restarts mid-write the trace shows approval was given and
  which writes were pending, so recovery can re-drive the remaining writes without
  duplicating already-completed ones

#### Scenario: Mid-write restart does not duplicate completed writes
- **WHEN** the coordinator restarts after some artefact writes succeeded and some did not
- **THEN** on restart the coordinator reads the trace to identify which artefact ids were
  approved and which write completions are recorded
- **AND** only the unconfirmed artefact writes are retried; already-confirmed writes are
  not re-attempted

#### Scenario: Write failure does not silently succeed
- **WHEN** one artefact write fails
- **THEN** the coordinator records the failure in the trace with the artefact id and error
- **AND** the generation workspace shows the failed artefact with an error state and a
  "Retry" option
- **AND** other artefact writes that succeeded are not retried

#### Scenario: Approved content is what was displayed, not the original buffered proposal
- **WHEN** the human edits one or more artefacts and then approves
- **THEN** the coordinator uses the content from the `approve-generation` command's
  `edits` array (not the original buffered `ArtifactProposal.content`) for any edited
  artefact
- **AND** the trace records the final, human-reviewed content for each approved artefact

### Requirement: Artefacts with an SoR target are validated before fan-out
`capability: artifact-fanout` · `delta: ADDED (feat/generation-workspace)`

The system SHALL validate each `ArtifactProposal` that carries a `sorTarget` before
fanning it to SPEC-014. An artefact with `target: "ticket"` or `target: "tracking"` that
has no `sorTarget` SHALL be flagged as invalid and shown to the human as an error in the
workspace; it SHALL NOT be silently skipped or written to an unintended destination.

#### Scenario: Missing sorTarget on a ticket artefact is surfaced
- **WHEN** the agent returns an `ArtifactProposal` with `target: "ticket"` and no
  `sorTarget`
- **THEN** the coordinator marks that artefact as invalid in the preview
- **AND** the workspace shows the artefact with an "Invalid — no integration target
  specified" label
- **AND** the artefact cannot be approved until the human either edits it to add a
  `sorTarget` or rejects it

### Requirement: Regeneration from the spec; downstream edits do not flow back
`capability: generation` · `delta: ADDED (feat/generation-workspace)`

The system SHALL treat the specification file as the sole source of truth for generated
artefacts. The coordinator SHALL detect a spec change by comparing a content hash of the
current spec file against the hash at the time of the last approved proposal (the
`updated` frontmatter date is informational only and SHALL NOT be used as the sole change
signal). When the specification changes, the generation workspace SHALL offer to regenerate.
Changes made to downstream artefacts outside the specification SHALL NOT be reflected back
into the specification.

#### Scenario: Spec change prompts regeneration offer
- **WHEN** a specification that has already produced approved artefacts is updated and its
  content hash changes
- **THEN** the generation workspace shows a "Spec changed — regenerate?" prompt
- **AND** the previous artefacts remain visible until regeneration is triggered and approved

#### Scenario: Editing a ticket does not update the spec
- **WHEN** someone edits a generated ticket in Jira or GitHub
- **THEN** the specification file is unchanged
- **AND** the generation workspace does not reflect the downstream edit

## Design

### Architectural decision
The generation workspace is a client-side surface backed by a coordinator-orchestrated
generation session. The agent proposes artefacts as structured output; the coordinator
buffers them and pushes them to the client as a `generation.proposed` event (a new event
type added to `@arke/contracts`). The client renders proposals; the human edits and
approves; the client sends an `approve-generation` command carrying a `proposalId`; the
coordinator validates the anchor, logs the decision (including final artefact content) and
fans out writes. Systems of record (Jira, GitHub, Azure DevOps) are written exclusively
through the deterministic plugin mechanism in SPEC-014 — this spec covers everything up to
and including that hand-off. Partial approval (approve a subset of artefacts) is supported
from the outset; the default UI gesture approves all, but per-artefact selection is
available.

### Target architecture

```
spec approved (SpecStatusEvent)
        │
        ▼
 Coordinator orchestration step
    └── adapter.dispatchAsync(generationAgent, spec markdown)
              │
              ▼ (session.idle — or timeout → error surface)
    └── parse + validate structured output
    └── buffer GenerationProposal[] (keyed specId → proposalId)
    └── push generation.proposed → client
              │
              ▼ (human reviews in Generation Workspace — edit / partial select)
    └── client: approve(proposalId, approvedIds, edits?) / reject(proposalId)
              → WS command "approve-generation" / "reject-generation"
              │
              ▼
    └── validate proposalId matches current pending proposal
    └── trace.write({ kind: "approval", decision: "approved", specId,
                      proposalId, approvedIds, finalContent })
    └── fan-out approved artefacts:
          ├── docs/tests → harness file writes (dispatchAsync write session)
          └── SoR targets → ProjectionWriteEvent (consumed by SPEC-014 plugin)
    └── trace.write({ kind: "write-complete"/"write-failed", artefactId, ... })
              │
              ▼ (restart recovery — reads trace to find unconfirmed artefacts)
```

The `@arke/client` generation workspace screen (already scaffolded at baseline) renders
the proposals and hosts the approve/edit/reject controls. Per-artefact selection is
available in the review UI. The client holds NO authoritative state; the proposal is
authoritative on the coordinator.

### Data model
A `GenerationProposal` is held in-memory on the coordinator, keyed by `specId`. The
proposal's `sessionId` doubles as its `proposalId` for command anchoring. Per SPEC-018 the
buffer is **per project context** (e.g. keyed `Map<projectId, Map<specId, GenerationProposal>>`),
so generation in project A is independent of B; approval records are written to **that
project's** `.arke/trace.ndjson`.

```typescript
interface ArtifactProposal {
  id: string;                                       // stable per proposal item
  target: "docs" | "tests" | "ticket" | "tracking";
  title: string;
  content: string;                                  // the proposed text/markdown
  sorTarget?: "jira" | "azure-devops" | "github";  // required when target is "ticket"/"tracking"
  invalid?: string;                                 // validation error message, if any
}

interface GenerationProposal {
  specId: string;
  sessionId: string;           // also serves as proposalId for command anchoring
  artifacts: ArtifactProposal[];
  proposedAt: number;          // epoch ms
  specContentHash: string;     // SHA-256 of spec file at time of proposal
  status: "pending-review" | "approved" | "rejected" | "timed-out" | "error";
}
```

Proposals are not persisted to disk in Phase 1; regeneration on restart is acceptable for
the proposal buffer. However, the approval record (including `proposalId`, approved artefact
ids, and final content) is written to the trace before any write is attempted, enabling
recovery of in-progress fan-out without re-hydrating the in-memory proposal.

Phase-2 durability (SQLite via SPEC-003) can persist full proposals.

### Interfaces and contracts
New domain events in `packages/contracts/src/events.ts`:

```typescript
export const GenerationProposedEvent = base.extend({
  type: z.literal("generation.proposed"),
  specId: z.string(),
  sessionId: z.string(),       // proposalId — included on all downstream commands
  artifacts: z.array(z.object({
    id: z.string(),
    target: z.enum(["docs", "tests", "ticket", "tracking"]),
    title: z.string(),
    content: z.string(),
    sorTarget: z.enum(["jira", "azure-devops", "github"]).optional(),
    invalid: z.string().optional(),
  })),
});

export const GenerationDecidedEvent = base.extend({
  type: z.literal("generation.decided"),
  specId: z.string(),
  sessionId: z.string(),       // proposalId anchor — disambiguates concurrent proposals
  decision: z.enum(["approved", "rejected"]),
  approvedArtifactIds: z.array(z.string()).optional(), // absent = all approved
  editedArtifacts: z.array(z.object({ id: z.string(), content: z.string() })).optional(),
});
```

These are added to the `DomainEvent` discriminated union.

Client → coordinator WebSocket commands:
```typescript
{ type: "approve-generation", specId: string, proposalId: string,
  approvedArtifactIds?: string[], edits?: { id: string; content: string }[] }
{ type: "reject-generation",  specId: string, proposalId: string }
{ type: "trigger-generation", specId: string }
```

`proposalId` equals the `sessionId` of the generation session that produced the proposal.

### Cross-cutting
- **Configuration:** all project configuration — the harness/model registry, coordinator settings, and integrations — lives in the single root file `.arke/config.json` (`ARKE_*` env vars override individual keys; runtime artefacts under `.arke/` are state, not config). See SPEC-005 for the model.
- **Multi-project scoping (SPEC-018):** the coordinator supervises N isolated **project contexts**, each keyed by `projectId` with its own read model, trace, and durable stores (session store, grant store, proposal buffer) under its own `.arke/`. Each client connection resolves against its **active project**; events carry `projectId`. See [SPEC-018](018.multi-project-workspaces.md).
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.
- **Propose · decide · execute:** nothing written before approval; the approval is
  persisted to `Trace` (including final artefact content and approved ids) before any
  write begins — this is the durable proof of intent and the recovery anchor.
- **Spec is the truth:** the generation agent is given the canonical spec markdown read
  from git; it never reads downstream artefacts as input. This is enforced by the
  coordinator's session dispatch, not by convention.
- **Spec-change detection:** the coordinator computes a SHA-256 hash of the spec file
  content and stores it on the proposal. Change detection compares this hash, not the
  `updated` frontmatter date.
- **Partial approval:** per-artefact approval is the unit of granularity. The default
  UI gesture approves all, but the command always carries `approvedArtifactIds` (absent
  means all); rejected artefacts are neither written nor kept pending.
- **Idempotency:** if `trigger-generation` is received while a generation session is
  already running, the coordinator ignores the duplicate and the client shows a
  "generating…" state.
- **Recovery:** the trace record written before fan-out carries `approvedArtifactIds` and
  `finalContent`, enabling the coordinator to re-drive unconfirmed writes after a restart
  without duplicating confirmed ones.
- **Observability:** every `GenerationProposedEvent` and `GenerationDecidedEvent` is
  written to `Trace` (feeds SPEC-015).

### Testing
- Unit (coordinator): `spec.status approved` → generation session dispatched; `session.idle`
  with valid structured output → `GenerationProposedEvent` emitted; unparseable output →
  error event emitted, no proposal; duplicate trigger on live session → no-op; sequential
  re-approval → new session, old proposal superseded.
- Unit: `approve-generation` with matching `proposalId` → `Trace.write` called before any
  write method; mismatched `proposalId` → error returned; partial `approvedArtifactIds` →
  only those artefacts written; `reject-generation` → proposal discarded.
- Unit: edited artefact content from command used in write (not original buffer);
  `target: "ticket"` with no `sorTarget` → marked `invalid` in proposal.
- Unit: generation timeout → error surface, no proposal emitted.
- Integration: stub adapter returning a fixed proposal; assert client receives
  `generation.proposed`; send partial approval with edits; assert `generation.decided`
  carries correct `approvedArtifactIds` and edited content; assert write commands use
  edited content; confirm trace records approval before any write call.
- Integration: simulate coordinator restart mid-write; confirm recovery reads trace and
  re-drives only unconfirmed artefacts.
- Manual: approve a real spec in the cockpit; review proposals in the generation workspace;
  partially approve, editing one artefact; confirm only approved artefacts written; SoR
  write event emitted; restart coordinator mid-write and verify recovery.

### Definition of done
All scenarios pass; the generation workspace renders proposals from a real approved spec;
partial approval, edit, and reject all work; no write occurs before approval; the trace
records the full decision chain including final content; restart recovery re-drives
unconfirmed writes without duplicating completed ones; stale `proposalId` commands are
rejected; generation timeout surfaces a user-visible error; a reviewer has signed off.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Proposals buffered on coordinator, not in client | Client holds no authoritative state (PRD §4); the buffer is the pending decision |
| 2 | SoR writes delegated to SPEC-014 projection plugin | Keeps deterministic plugin code separate from the orchestration layer |
| 3 | In-memory proposal store (Phase 1) | Sufficient; durability added when SPEC-003 adds SQLite |
| 4 | `trace.write` before any write method, including `finalContent` and `approvedArtifactIds` | Provides durable proof of approval and a recovery anchor for mid-write restart without in-memory state |
| 5 | `proposalId` (= generation `sessionId`) required on approval/rejection commands | Prevents stale or mismatched approvals from triggering writes; needed because rapid re-runs can produce multiple proposals for the same `specId` |
| 6 | Partial approval supported from the outset | The per-artefact `id` field already exists; deferring per-artefact granularity would leave an irresolvable open question and force an all-or-nothing UX |
| 7 | Spec-change detection by content hash, not `updated` date | The `updated` frontmatter date is human-authored and can be stale; SHA-256 of file content is machine-reliable |
| 8 | Generation timeout (default 5 min) | An agent that never reaches `session.idle` must not leave the workspace stuck indefinitely |
| 9 | `target: "ticket"/"tracking"` without `sorTarget` is invalid, not silently skipped | Silent skips hide missing integration config; surfacing to the human keeps the proposal actionable |

## Open questions
- What is the structured output format the generation agent should produce — JSON embedded
  in a markdown fence, a structured tool call, or free-form markdown parsed by the
  coordinator? (Must be resolved before implementation; drives the parse/validation step.)
- Should the generation workspace show a diff between the last approved proposal and the
  newly regenerated proposal when the spec changes?
