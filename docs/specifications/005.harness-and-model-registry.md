# Harness & model registry and routing

## Why
Roles ask for a logical tier; the system must resolve tier → model → instance and route across harnesses, with vendor model IDs living in exactly one configured place behind the internal gateway — so models change without touching agent definitions, specifications, or the client.

## What changes
- ADDED registry — the coordinator maintains a registry of harness instances, their capabilities, and the tier → model mappings they serve; the client reads it as a live projection (FR-4, FR-19)
- ADDED model-resolution — a registry resolver maps any role to a `ModelSelection {instanceId, tier, model}` following the rule in `docs/agent-roster-and-model-resolution.md`, enforcing that reviewer-a and reviewer-b land on different models
- ADDED routing — sessions are routed to a harness by the model it needs, not by an explicit harness mode; multi-harness routing aggregates across all registered instances on one board
- MODIFIED opencode-integration — `OpenCodeAdapter` receives its `resolveModel` function from the registry resolver rather than the default stub   (breaking: no)

---

## Requirements

### Requirement: Maintain a registry of instances and their capabilities
`capability: registry` · `delta: ADDED (feat/harness-and-model-registry)`

The system SHALL maintain a registry of harness instances. Each instance SHALL record a stable identifier, a driver name, a host, a working directory, a credentials reference, the capabilities it advertises (derived from the SPEC-002 startup capability probe, not from static config), and the set of tier → model pairs it serves. The registry SHALL be the single place vendor model identifiers are stored; they SHALL NOT appear in agent definitions, specification files, or the client. The `credentialsRef` string itself SHALL NOT appear in any WebSocket frame, snapshot, event, or log entry visible to the client.

#### Scenario: Registry loaded from config
- **WHEN** the coordinator starts with a registry configuration file present
- **THEN** the registry contains one entry per configured instance
- **AND** each entry carries the driver, host, cwd, credentialsRef (host-only), capabilities set, and serves list

#### Scenario: Connected instance capabilities derived from probe, not static config
- **WHEN** a harness instance responds to the SPEC-002 startup capability probe
- **THEN** the registry entry for that instance records the `Capability` flags returned by its adapter at probe time
- **AND** those flags (not the static config) are included in the snapshot pushed to the client

#### Scenario: Vendor model IDs absent from non-registry files
- **WHEN** any `.opencode/agents/*.md` file, any `docs/specifications/*.md` file, or any client bundle is inspected
- **THEN** no vendor model identifier appears in any of them; only logical tier names (`capable`, `mid`) are present

#### Scenario: `credentialsRef` absent from all client-facing surfaces
- **WHEN** a client connects and receives the coordinator snapshot or any `registry.updated` event
- **THEN** no `credentialsRef` value, no Basic-auth token, no API key, and no credential path or naming string appears in the payload

### Requirement: Resolve a role to a ModelSelection — deterministically
`capability: model-resolution` · `delta: ADDED (feat/harness-and-model-registry)`

The system SHALL expose a resolver that maps a roster role name to a `ModelSelection {instanceId, tier, model}`. The resolver SHALL follow the rule: if the role pins an instance, use it and take the model that instance serves for the tier; otherwise pick among instances currently serving the role's tier using a stable deterministic order (the order instances appear in the config file), preferring one already running on the correct host and working directory as a tie-breaker. The resolved `model` value SHALL be the concrete provider/model string as configured in the registry. Every resolution call SHALL be written to the trace with `{ role, instanceId, tier, model: '<label>' }` where the model label is a human-readable tier description, not the vendor model string.

#### Scenario: Role without a pinned instance resolves to any serving instance — stable order
- **WHEN** the resolver is called for the `spec-author` role (tier: capable, no instance pin)
- **THEN** it returns a `ModelSelection` pointing to the first instance in config order whose `serves` list includes `{ tier: 'capable' }`
- **AND** the `model` field contains the concrete model string from that instance's configuration
- **AND** repeated calls with the same role return the same `ModelSelection` as long as the registry is unchanged

#### Scenario: Pinned-instance role resolves to the pinned instance
- **WHEN** the resolver is called for the `reviewer-a` role (tier: capable, instance: claude-local)
- **THEN** the returned `ModelSelection.instanceId` is `claude-local`
- **AND** the model is the one `claude-local` serves for the capable tier

#### Scenario: No serving instance returns a resolver error, not a silent default
- **WHEN** the resolver is called for a role whose required tier is not served by any registered instance
- **THEN** the resolver throws a typed `NoInstanceForTierError` naming the tier and the role
- **AND** the coordinator emits a `registry.warning` event with `reason: 'no-instance-for-tier'` and a `detail` string naming the role and the missing tier
- **AND** no session is created; the client receives the warning event and renders an error state for that role

#### Scenario: Trace records resolution without leaking vendor model string
- **WHEN** `resolve(role)` is called
- **THEN** a trace entry is written with `{ role, instanceId, tier, model: '<label>' }` where `<label>` is a human-readable description (e.g. "capable — Internal Gateway") not the vendor model identifier

### Requirement: Reviewer-a and reviewer-b resolve to different model strings on distinct instances
`capability: model-resolution` · `delta: ADDED (feat/harness-and-model-registry)`

The system SHALL enforce that `reviewer-a` and `reviewer-b` are configured to resolve to distinct instances and that those instances serve different model strings for the capable tier. The check SHALL compare model strings (not family names, which are not machine-readable) and SHALL be performed when a project context loads its registry (at project-open, per SPEC-018) and on every registry reload, before any routing is permitted for that context. When both reviewers would resolve to the same model string, the resolver SHALL reject the review session with a `RegistryConfigError` and emit a `registry.warning` event with `reason: 'reviewer-models-identical'`; the registry reload SHALL atomically block routing until the warning is resolved or the config is corrected.

#### Scenario: Reviewers on different instances resolve to different models
- **WHEN** the resolver is called for both `reviewer-a` (instance: claude-local) and `reviewer-b` (instance: opencode-local)
- **THEN** `reviewer-a` resolves to the capable-tier model on `claude-local`
- **AND** `reviewer-b` resolves to the capable-tier model on `opencode-local`
- **AND** the two `model` strings are not equal

#### Scenario: Same-model configuration is rejected atomically at load time
- **WHEN** the registry config is loaded or reloaded and both reviewer roles resolve to the same model string
- **THEN** `assertReviewersDistinct()` throws `RegistryConfigError`
- **AND** the coordinator emits a `registry.warning` event with `reason: 'reviewer-models-identical'`
- **AND** routing remains unavailable until the config is corrected and reloaded cleanly
- **AND** no review session starts before the check passes

#### Scenario: Reload race window — routing blocked until post-load check passes
- **WHEN** the registry config file changes and `RegistryLoader` begins reloading
- **THEN** no new session is routed until the new config has fully parsed and `assertReviewersDistinct()` has passed
- **AND** sessions already in flight on the previous config are not interrupted

### Requirement: Credentials resolved on the host, never sent to the client
`capability: registry` · `delta: ADDED (feat/harness-and-model-registry)`

The system SHALL resolve credentials references (`credentialsRef`) on the coordinator host at session-create time and SHALL NOT include any credential value, nor the `credentialsRef` string itself, in any WebSocket frame, snapshot, or event pushed to the client. The client SHALL receive only instance identifiers, driver names, capability flags, and tier labels. The `file:` backend of `CredentialResolver` SHALL canonicalise the file path and restrict it to a configured safe root, rejecting paths that traverse outside it.

#### Scenario: Snapshot contains no credentials or refs
- **WHEN** a client connects and receives the coordinator snapshot
- **THEN** the snapshot includes the instance list with id, driver, host, caps, and tier labels
- **AND** no `credentialsRef` value, no Basic-auth token, no API key, and no credential naming string appears in the payload

#### Scenario: Adapter receives resolved credentials at session create
- **WHEN** the coordinator creates a session on a harness instance
- **THEN** it reads the credential from the host environment (env var, secrets file, or OS keychain) using the `credentialsRef` key
- **AND** passes it to the adapter constructor; the credential never leaves the coordinator process

#### Scenario: `file:` credential backend path validated
- **WHEN** a `credentialsRef` of the form `file:../../etc/passwd` is encountered
- **THEN** the `CredentialResolver` rejects it with a `CredentialPathError` before reading the file
- **AND** no file content is returned or logged

### Requirement: Route sessions by model, not harness mode
`capability: routing` · `delta: ADDED (feat/harness-and-model-registry)`

The system SHALL route every session to the harness instance that serves the required model, derived from tier resolution. No caller SHALL specify a harness instance directly; routing is always a consequence of the resolved `ModelSelection`. Sessions from multiple instances SHALL be aggregated on one delivery board.

#### Scenario: Session routed to the instance serving the model
- **WHEN** the coordinator creates a session for a role whose resolver returns `{ instanceId: 'opencode-local', tier: 'mid', model: 'gateway/mid-tier' }`
- **THEN** the session is opened on the adapter whose id is `opencode-local`
- **AND** the `SendMessageInput` carries `tier: 'mid'` and the adapter's `resolveModel` returns the configured concrete model

#### Scenario: Multi-harness board aggregates sessions
- **WHEN** two instances are registered — one for `capable` sessions and one for `mid` sessions — and both have active sessions
- **THEN** the delivery board shows cards from both instances under a common project view
- **AND** each card's `harness` field identifies its originating instance

#### Scenario: `resolveModel` closure is scoped to a single tier — mismatch is an error, not a guess
- **WHEN** the injected `resolveModel` closure (which captures a single `ModelSelection`) is called with a tier other than the one it was created for
- **THEN** it throws a typed `TierMismatchError` naming the expected and actual tiers
- **AND** no model is returned for the mismatched tier; the caller must obtain a new `ModelSelection` for a different tier via the registry

### Requirement: Instance unreachable mid-session — explicit failover or failure
`capability: routing` · `delta: ADDED (feat/harness-and-model-registry)`

When a registered harness instance becomes unreachable after sessions have been started on it, the coordinator SHALL mark the instance `reachable: false` in the snapshot and emit a `harness.reachability` event. If another instance serves the same tier, the coordinator SHALL route new sessions to the fallback and emit a `registry.warning` event with `reason: 'instance-failover'` identifying both the failed and fallback instances. In-flight sessions on the failed instance SHALL NOT be silently migrated; they SHALL be marked as `interrupted` and the client SHALL surface an explicit "Instance lost — session interrupted" state for each affected session.

#### Scenario: Instance lost, fallback available — new sessions rerouted
- **WHEN** a registered instance becomes unreachable and another instance serves the same tier
- **THEN** the coordinator marks the failed instance `reachable: false` and emits `harness.reachability` with the reason
- **AND** a `registry.warning` event is emitted with `reason: 'instance-failover'`, naming the failed and fallback instances
- **AND** new sessions for that tier are routed to the fallback instance

#### Scenario: Instance lost, no fallback — routing fails with clear error
- **WHEN** a registered instance becomes unreachable and no other instance serves the same tier
- **THEN** the coordinator emits a `registry.warning` event with `reason: 'no-instance-for-tier'`
- **AND** new sessions for that tier are refused with a `NoInstanceForTierError`
- **AND** the client renders an error state for the affected role

#### Scenario: In-flight sessions on the failed instance marked interrupted
- **WHEN** an instance becomes unreachable while sessions are running on it
- **THEN** each affected session is marked `interrupted` in the read model
- **AND** the client surfaces "Instance lost — session interrupted" for each affected card
- **AND** no automatic migration of in-flight sessions occurs

### Requirement: Harness & model configuration screen shows the live registry
`capability: registry` · `delta: ADDED (feat/harness-and-model-registry)`

The system SHALL present a harness and model configuration screen that shows the registered instances (id, driver, endpoint, reachability status, capability flags), the tier → model mapping label for each instance, and the full roster resolution table (role → instance → tier label). The screen SHALL reflect the live coordinator registry, not a static client-side seed. The screen SHALL NOT display vendor model IDs, `credentialsRef` values, or raw model strings; tier labels (e.g. "capable — Internal Gateway") are the only model-facing text shown.

#### Scenario: Screen reflects live registry data
- **WHEN** the engineer opens the harness and model configuration screen
- **THEN** each registered instance is shown with its driver, endpoint, reachability dot, capability flags, and the tier labels it serves
- **AND** the data matches what the coordinator's current snapshot reports
- **AND** no vendor model ID or credential string appears anywhere in the rendered screen

#### Scenario: Tier-to-model mapping is visible as a label and read-only
- **WHEN** the engineer views the "Model tiering" section
- **THEN** each logical tier (capable, mid) is shown with its human-readable label and the instance that serves it
- **AND** no editing of vendor model IDs is possible from the client; edits require a config file change on the host

## Design

### Architectural decision
The registry is a **per-project-context** component (SPEC-018) — not a coordinator-side singleton — loaded from that project's config file on the host when the project is opened, and released on close/evict. It is the only place vendor model identifiers live, accessed only by that context's `RegistryResolver`; two open projects have two independent registries with no shared model/credential state. The client receives a projection of registry state — instance IDs, capabilities, tier labels, reachability — but never raw credentials, credential ref strings, or raw model IDs. Instance capabilities are derived from the SPEC-002 startup probe, not from static config, so the registry reflects the server's real surface. This mirrors the resolution mechanism described in `docs/agent-roster-and-model-resolution.md §2` and satisfies NFR-1 (credentials on host) and the agent roster's portability principle.

### Target architecture

```
Client (Harnesses screen / Settings)
  ↕ WebSocket snapshot + registry.updated events
Coordinator
  ├── RegistryLoader          — loads .arke/config.json at project-open time (per ProjectContext, not coordinator startup); watches for file changes with debounce; blocks routing during reload until assertReviewersDistinct passes
  ├── RegistryResolver        — resolve(role) → ModelSelection (deterministic config order); assertReviewersDistinct(); listInstances() (no credentials)
  ├── CredentialResolver      — env:/file:/keychain: backends; file: paths canonicalised + safe-root restricted
  ├── SessionRouter           — maps ModelSelection → adapter instance; opens session; handles failover + interrupted marking
  └── HarnessAdapters[]       — one per registered instance (OpenCodeAdapter, …); capabilities from SPEC-002 probe
       ↕ HTTP / SSE (per instance)
Harness hosts (opencode serve, claude-code serve, …)
```

> **Per-project scoping (SPEC-018):** `RegistryLoader` and `RegistryResolver` are **not** coordinator singletons — they live inside each `ProjectContext` alongside its `ReadModel`, `Trace`, and stores. A `ProjectContext` instantiates them when its project is **opened** (loading that project's `.arke/config.json`) and releases them on close/evict. The boxes above describe one project context's slice, not a process-wide registry.

### Configuration model
`.arke/config.json` at the repo root is — **per project** (SPEC-018) — the **single source of that project's context-local configuration**: within a project context that file is the single source for THAT project, and N open projects have N config files (one per root). It carries the
harness/model `registry` (instances + roster bindings, below) and `integrations` config — the
context-local settings.

**Process-wide settings are NOT per project (SPEC-018).** A single coordinator cannot run on
multiple WebSocket ports or apply two different concurrency caps, so genuinely process-wide
settings — the coordinator `port`, the `maxProjects` concurrency cap, the project idle-TTL, the
`OTLP endpoint`, and global query limits — come from a **global/launch-time source** (`ARKE_*`
environment / a global config under `ARKE_HOME`), NOT from any one project's `.arke/config.json`.
A `settings` block in a per-project config is advisory for that context (e.g. its permission
timeout) and MUST NOT set process-wide values; the launch-time source wins. Individual keys MAY
be overridden by `ARKE_*` environment variables (env wins; `ARKE_*` are overrides, not
the source of truth). Runtime artefacts under `.arke/` — `trace.ndjson`, `sessions.ndjson`,
`scaffold-manifest.json`, `projection-fallback.ndjson` — are local **state, not configuration**,
and are git-ignored; `.arke/config.json` is the one tracked file there. Vendor model ids live
only in this file, behind the internal gateway, never in the client.

**Config file shape** — the `registry` and `roster` sections of `.arke/config.json` (shown here
as YAML for readability; the file itself is JSON):

```yaml
instances:
  - id: claude-local
    driver: claude-code
    host: localhost
    cwd: .
    credentialsRef: claude-code/default
    serves:
      - { tier: capable, model: <claude-capable-model> }

  - id: opencode-local
    driver: opencode
    host: localhost
    cwd: .
    credentialsRef: opencode/gateway
    serves:
      - { tier: capable, model: <other-capable-model> }
      - { tier: mid,     model: <mid-model> }

roster:
  spec-author:  { tier: capable }
  architect:    { tier: capable }
  reviewer-a:   { tier: capable, instance: claude-local }
  reviewer-b:   { tier: capable, instance: opencode-local }
  implementer:  { tier: mid }
  researcher:   { tier: mid }
```

This is the identical YAML shape documented in `docs/agent-roster-and-model-resolution.md §2`; the spec makes it the authoritative source and the coordinator's `RegistryLoader` the sole consumer.

**`ModelTier`** is imported from `@arke/contracts/src/spec.ts`: `z.enum(['capable', 'mid', 'fast'])` — three logical tiers (capable = authoring/review, mid = implementation, fast = routine/classification & projection drafts). A registry instance's `serves` maps each tier to a concrete model; an unmapped tier falls back to the gateway default for that tier.

**`SessionRouter`** (`packages/coordinator/src/session-router.ts`):

```typescript
class SessionRouter {
  route(role: string): { adapter: HarnessAdapter; selection: ModelSelection } {
    const selection = this.resolver.resolve(role);   // throws NoInstanceForTierError if no instance
    const adapter = this.adapters.get(selection.instanceId);
    if (!adapter) throw new NoAdapterError(selection.instanceId);
    return { adapter, selection };
  }

  markInstanceUnreachable(instanceId: string): void {
    // marks instance reachable:false; emits harness.reachability + registry.warning (instance-failover or no-instance-for-tier)
    // marks in-flight sessions on this instance as 'interrupted'; does NOT migrate them
  }
}
```

The adapter's `resolveModel` closure is injected from `InstanceConfig.serves` for the single resolved tier; it throws `TierMismatchError` if called for any other tier. A sub-session requiring a different tier must go back through `RegistryResolver` for a new `ModelSelection`.

**RegistryLoader debounce:** file-watch events are debounced by 200 ms before triggering a reload. During reload, `routingEnabled` is set to `false`; it is restored to `true` only after `assertReviewersDistinct()` passes. Sessions in flight are not interrupted by a reload; only new routing requests are held.

**Client changes** (`packages/client/src/`):
- `screens/harnesses-settings.tsx` — `Harnesses`: replace the hardcoded `HARNESSES` and `TIERS` store seeds with live data from the coordinator snapshot's `registry` field; render tier labels (not model IDs); add the roster resolution table (role → instance → tier label). The "model tiering" section shows no raw model string at any point.
- `store.ts` — `harnesses` and `tiers` initial values become empty arrays; populated by the first snapshot frame after WS connection.

### Data model
No database. The registry is loaded from a config file on the coordinator host. The in-memory `RegistryResolver` is the runtime authority. Credential values are read from host environment variables or a secrets file at session-create time and held in memory only for the duration of the request.

**Resolution rule** (from `docs/agent-roster-and-model-resolution.md §2`, made normative here):

```
resolve(role):
  entry = roster[role]                     // { tier, instance? }
  if entry.instance:
    inst = instances[entry.instance]
    model = inst.serves.find(s => s.tier === entry.tier).model
    return { instanceId: inst.id, tier: entry.tier, model }
  else:
    // deterministic: config order; host/cwd match preferred as tie-breaker
    inst = instances.find(i => i.serves.some(s => s.tier === entry.tier))
          ?? throw NoInstanceForTierError(role, entry.tier)
    model = inst.serves.find(s => s.tier === entry.tier).model
    return { instanceId: inst.id, tier: entry.tier, model }
```

### Cross-cutting
- **Multi-project scoping (SPEC-018):** the coordinator supervises N isolated **project contexts**, each keyed by `projectId` with its own root, harness runner, read model, trace, and durable stores under its own `.arke/`. Everything this spec places on "the coordinator" — `RegistryLoader`, `RegistryResolver`, `SessionRouter`, and the per-instance adapters — is per–project context, loading that project's `.arke/config.json` at project-open time; each client connection resolves against its **active project**. See [SPEC-018](018.multi-project-workspaces.md).
- **Configuration:** context-local configuration — the harness/model registry and integrations — lives per project in that project's `.arke/config.json`, loaded when the project is opened (SPEC-018). Process-wide settings (coordinator port, `maxProjects`, idle-TTL, OTLP endpoint, global query limits) come from a global/launch-time source (`ARKE_*` env / a global config under `ARKE_HOME`), never from a single project's file. `ARKE_*` env vars override individual keys; runtime artefacts under `.arke/` are state, not config.
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.
- **Security (NFR-1):** `credentialsRef` values AND the ref strings themselves are resolved and used only inside the coordinator host. The `CredentialResolver` `file:` backend canonicalises paths and restricts them to a configured safe root, rejecting traversal. The client snapshot and all events contain tier labels only; no credential value, no ref string, and no raw model string is ever sent to the client.
- **Observability (NFR-7):** Every `resolve(role)` call is written to the trace with `{ role, instanceId, tier, model: '<label>' }` — the human-readable tier label is logged for audit; the vendor model string is not included in the trace entry.
- **Resilience (NFR-4):** If a registered instance becomes unreachable mid-session, the coordinator marks it `reachable: false`, emits `harness.reachability`, and routes new sessions to a fallback if one exists. In-flight sessions are marked `interrupted`; they are not silently migrated. The reviewer-distinct check blocks routing on a bad reload until the config is corrected; sessions in flight are unaffected. `RegistryLoader` debounces file-watch events by 200 ms to prevent reload storms.
- **SPEC-002 alignment:** Instance capabilities in the registry are derived from the SPEC-002 startup capability probe, not from static config. The `resolveModel` injected into `OpenCodeAdapter` comes from `SessionRouter`, which derives it from the registry (connects SPEC-002 and SPEC-005).

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | YAML config file on the host, not a UI-editable database | Vendor model IDs must stay on the host behind the gateway; config-as-code fits git-backed audit trail |
| 2 | Resolution rule is a coordinator-side function, not client-side | Keeps model IDs off the wire and ensures the same rule governs every session regardless of which client connected |
| 3 | `resolveModel` injected into adapters by `SessionRouter`, not hard-coded in adapter | Adapters stay harness-adapter–interface-compliant; the registry is the single source of model truth |
| 4 | Reviewer-distinct rule enforced at startup and on reload; routing blocked during reload window | A misconfigured registry that silently produces same-model reviews undermines independence; blocking new sessions during reload prevents a race where the bad config briefly wins |
| 5 | Client `Harnesses` screen driven by snapshot projection, not a static seed | The design-baseline seed (`HARNESSES`, `TIERS` in store.ts) was a placeholder; live data is required by FR-19 |
| 6 | Capabilities derived from SPEC-002 startup probe, not static config | The server's real surface governs what is advertised; static config can drift from reality |
| 7 | `credentialsRef` string itself excluded from all client-facing payloads | Even the ref string leaks the naming convention for credentials; only ids, labels, and caps reach the client |
| 8 | Resolution order is deterministic (config order; host/cwd tie-breaker) | Non-deterministic "any instance" resolution is untestable and produces inconsistent session assignment; deterministic order is auditable |
| 9 | In-flight sessions marked `interrupted` on instance loss; no auto-migration | Silent migration would change the session's effective model without the engineer's knowledge; explicit `interrupted` state is honest and recoverable |
| 10 | `resolveModel` closure scoped to one tier; throws `TierMismatchError` for others | Prevents a sub-session from silently resolving to the wrong model by using a parent session's closure; sub-sessions must get their own `ModelSelection` |
| 11 | `CredentialResolver` `file:` backend restricted to safe root | A `credentialsRef: file:../../etc/passwd` in a compromised config could read arbitrary files; safe-root restriction contains the blast radius |
