# Harness & model registry and routing

## Why
Roles ask for a logical tier; the system must resolve tier → model → instance and route across harnesses, with vendor model IDs living in exactly one configured place behind the internal gateway — so models change without touching agent definitions, specifications, or the client.

## What changes
- ADDED registry — the coordinator maintains a registry of harness instances, their capabilities, and the tier → model mappings they serve; the client reads it as a live projection (FR-4, FR-19)
- ADDED model-resolution — a registry resolver maps any role to a `ModelSelection {instanceId, tier, model}` following the rule in `docs/agent-roster-and-model-resolution.md`, enforcing that reviewer-a and reviewer-b land on different models
- ADDED routing — sessions are routed to a harness by the model it needs, not by an explicit harness mode; multi-harness routing aggregates across all registered instances on one board
- MODIFIED opencode-integration — `OpenCodeAdapter` receives its `resolveModel` function from the registry resolver rather than the default stub   (breaking: no)

## Requirements

### Requirement: Maintain a registry of instances and their capabilities
`capability: registry` · `delta: ADDED (feat/harness-and-model-registry)`

The system SHALL maintain a registry of harness instances. Each instance SHALL record a stable identifier, a driver name, a host, a working directory, a credentials reference, the capabilities it advertises, and the set of tier → model pairs it serves. The registry SHALL be the single place vendor model identifiers are stored; they SHALL NOT appear in agent definitions, specification files, or the client.

#### Scenario: Registry loaded from config
- **WHEN** the coordinator starts with a registry configuration file present
- **THEN** the registry contains one entry per configured instance
- **AND** each entry carries the driver, host, cwd, credentialsRef, capabilities set, and serves list

#### Scenario: Connected instance capabilities reflected in registry
- **WHEN** a harness instance responds to a capabilities probe
- **THEN** the registry entry for that instance records the `Capability` flags returned by its adapter
- **AND** those flags are included in the snapshot pushed to the client

#### Scenario: Vendor model IDs absent from non-registry files
- **WHEN** any `.opencode/agents/*.md` file, any `docs/specifications/*.md` file, or any client bundle is inspected
- **THEN** no vendor model identifier appears in any of them; only logical tier names (`capable`, `mid`) are present

### Requirement: Resolve a role to a ModelSelection
`capability: model-resolution` · `delta: ADDED (feat/harness-and-model-registry)`

The system SHALL expose a resolver that maps a roster role name to a `ModelSelection {instanceId, tier, model}`. The resolver SHALL follow the rule: if the role pins an instance, use it and take the model that instance serves for the tier; otherwise pick any instance currently serving the role's tier, preferring one already running on the correct host and working directory. The resolved `model` value SHALL be the concrete provider/model string as configured in the registry.

#### Scenario: Role without a pinned instance resolves to any serving instance
- **WHEN** the resolver is called for the `spec-author` role (tier: capable, no instance pin)
- **THEN** it returns a `ModelSelection` pointing to an instance whose `serves` list includes `{ tier: 'capable' }`
- **AND** the `model` field contains the concrete model string from that instance's configuration

#### Scenario: Pinned-instance role resolves to the pinned instance
- **WHEN** the resolver is called for the `reviewer-a` role (tier: capable, instance: claude-local)
- **THEN** the returned `ModelSelection.instanceId` is `claude-local`
- **AND** the model is the one `claude-local` serves for the capable tier

#### Scenario: No serving instance returns a resolver error
- **WHEN** the resolver is called for a role whose required tier is not served by any registered instance
- **THEN** the resolver throws a typed `NoInstanceForTierError`
- **AND** the coordinator emits a `harness.reachability` event reflecting the gap rather than silently failing

### Requirement: Reviewer-a and reviewer-b resolve to different models
`capability: model-resolution` · `delta: ADDED (feat/harness-and-model-registry)`

The system SHALL enforce that `reviewer-a` and `reviewer-b` resolve to models from different families, as defined by distinct instance pins in the roster configuration. When both reviewers would resolve to the same model, the resolver SHALL reject the review session and surface a configuration warning rather than proceeding with a non-independent critique.

#### Scenario: Reviewers on different instances resolve to different models
- **WHEN** the resolver is called for both `reviewer-a` (instance: claude-local) and `reviewer-b` (instance: opencode-local)
- **THEN** `reviewer-a` resolves to the capable-tier model on `claude-local`
- **AND** `reviewer-b` resolves to the capable-tier model on `opencode-local`
- **AND** the two `model` strings are not equal

#### Scenario: Same-model configuration is rejected
- **WHEN** both reviewer roles resolve to the same model string
- **THEN** the coordinator emits a `registry.warning` event with reason `reviewer-models-identical`
- **AND** the review panel shows a warning that the review cannot be considered independent

### Requirement: Credentials resolved on the host, never sent to the client
`capability: registry` · `delta: ADDED (feat/harness-and-model-registry)`

The system SHALL resolve credentials references (`credentialsRef`) on the coordinator host at session-create time and SHALL NOT include any credential value in any WebSocket frame, snapshot, or event pushed to the client. The client SHALL receive only instance identifiers, driver names, capability flags, and tier labels.

#### Scenario: Snapshot contains no credentials
- **WHEN** a client connects and receives the coordinator snapshot
- **THEN** the snapshot includes the instance list with id, driver, host, caps, and tier labels
- **AND** no `credentialsRef` value, no Basic-auth token, and no API key appears in the payload

#### Scenario: Adapter receives resolved credentials at session create
- **WHEN** the coordinator creates a session on a harness instance
- **THEN** it reads the credential from the host environment (env var, secrets file, or OS keychain) using the `credentialsRef` key
- **AND** passes it to the adapter constructor; the credential never leaves the coordinator process

### Requirement: Route sessions by model, not harness mode
`capability: routing` · `delta: ADDED (feat/harness-and-model-registry)`

The system SHALL route every session to the harness instance that serves the required model, derived from tier resolution. No caller SHALL specify a harness instance directly; routing is always a consequence of the resolved `ModelSelection`. Sessions from multiple instances SHALL be aggregated on one delivery board.

#### Scenario: Session routed to the instance serving the model
- **WHEN** the coordinator creates a session for a role whose resolver returns `{ instanceId: 'opencode-local', tier: 'mid', model: 'gateway/mid-tier' }`
- **THEN** the session is opened on the adapter whose id is `opencode-local`
- **AND** the `SendMessageInput` carries `tier: 'mid'` and the adapter's `resolveModel` returns the configured concrete model

#### Scenario: Multi-harness board aggregates sessions
- **WHEN** two instances are registered — one for `capable` sessions and one for `mid` sessions — and both have active sessions
- **THEN** the delivery board shows cards from both instances under a common project view
- **AND** each card's `harness` field identifies its originating instance

### Requirement: Harness & model configuration screen shows the live registry
`capability: registry` · `delta: ADDED (feat/harness-and-model-registry)`

The system SHALL present a harness and model configuration screen that shows the registered instances (id, driver, endpoint, reachability status, capability flags), the tier → model mapping for each instance, and the full roster resolution table (role → instance → model). The screen SHALL reflect the live coordinator registry, not a static client-side seed.

#### Scenario: Screen reflects live registry data
- **WHEN** the engineer opens the harness and model configuration screen
- **THEN** each registered instance is shown with its driver, endpoint, reachability dot, capability flags, and the models it serves per tier
- **AND** the data matches what the coordinator's current snapshot reports

#### Scenario: Tier-to-model mapping is visible and read-only
- **WHEN** the engineer views the "Model tiering" section
- **THEN** each logical tier (capable, mid) is shown with its resolved model name and the instance that serves it
- **AND** no editing of vendor model IDs is possible from the client; edits require a config file change on the host

## Design

### Architectural decision
The registry is a coordinator-side singleton loaded from a YAML/JSON config file on the host. It is the only place vendor model identifiers live, accessed only by the coordinator's `RegistryResolver`. The client receives a projection of registry state — instance IDs, capabilities, tier labels, reachability — but never raw credentials or model IDs beyond what is needed to label the UI. This mirrors the resolution mechanism described in `docs/agent-roster-and-model-resolution.md §2` and satisfies NFR-1 (credentials on host) and the agent roster's portability principle.

### Target architecture

```
Client (Harnesses screen / Settings)
  ↕ WebSocket snapshot + registry.updated events
Coordinator
  ├── RegistryLoader          — loads registry.yaml at startup; watches for file changes
  ├── RegistryResolver        — resolve(role) → ModelSelection; enforces reviewer-distinct rule
  ├── SessionRouter           — maps ModelSelection → adapter instance; opens session
  └── HarnessAdapters[]       — one per registered instance (OpenCodeAdapter, …)
       ↕ HTTP / SSE (per instance)
Harness hosts (opencode serve, claude-code serve, …)
```

**New coordinator module** (`packages/coordinator/src/registry.ts`):

```typescript
export interface InstanceConfig {
  id: string;
  driver: 'opencode' | 'claude-code' | string;
  host: string;
  cwd: string;
  credentialsRef: string;           // resolved on host; never sent to client
  serves: { tier: ModelTier; model: string }[];
}

export interface RosterEntry {
  tier: ModelTier;
  instance?: string;                // if present, pins to this instanceId
}

export interface RegistryConfig {
  instances: InstanceConfig[];
  roster: Record<string, RosterEntry>;  // role name → tier + optional instance pin
}

export interface ModelSelection {
  instanceId: string;
  tier: ModelTier;
  model: string;                    // concrete provider/model string for the adapter
}

export class RegistryResolver {
  resolve(role: string): ModelSelection { /* … */ }
  assertReviewersDistinct(): void { /* throws RegistryConfigError if same model */ }
}
```

**Config file shape** (`registry.yaml` at repo root or `~/.arke/registry.yaml`):

```yaml
instances:
  - id: claude-local
    driver: claude-code
    host: localhost
    cwd: .
    credentialsRef: claude-code/default
    serves:
      - { tier: capable, model: <claude-capable-model> }

  - id: opencode-local
    driver: opencode
    host: localhost
    cwd: .
    credentialsRef: opencode/gateway
    serves:
      - { tier: capable, model: <other-capable-model> }
      - { tier: mid,     model: <mid-model> }

roster:
  spec-author:  { tier: capable }
  architect:    { tier: capable }
  reviewer-a:   { tier: capable, instance: claude-local }
  reviewer-b:   { tier: capable, instance: opencode-local }
  implementer:  { tier: mid }
  researcher:   { tier: mid }
```

This is the identical YAML shape documented in `docs/agent-roster-and-model-resolution.md §2`; the spec makes it the authoritative source and the coordinator's `RegistryLoader` the sole consumer.

**`ModelTier`** is imported from `@arke/contracts/src/spec.ts` (existing: `z.enum(['capable', 'mid'])`). No schema change needed.

**`SessionRouter`** (`packages/coordinator/src/session-router.ts`):

```typescript
class SessionRouter {
  route(role: string): { adapter: HarnessAdapter; selection: ModelSelection } {
    const selection = this.resolver.resolve(role);
    const adapter = this.adapters.get(selection.instanceId);
    if (!adapter) throw new NoAdapterError(selection.instanceId);
    return { adapter, selection };
  }
}
```

The adapter's `resolveModel` function is injected from `InstanceConfig.serves` so no model ID appears anywhere but the registry config.

**Client changes** (`packages/client/src/`):
- `screens/harnesses-settings.tsx` — `Harnesses`: replace the hardcoded `HARNESSES` and `TIERS` store seeds with live data from the coordinator snapshot's `registry` field; render the same card layout but driven by `registry.instances`; add the roster resolution table (role → instance → model label).
- `store.ts` — `harnesses` and `tiers` initial values become empty arrays; populated by the first snapshot frame after WS connection.

### Data model
No database. The registry is loaded from a config file on the coordinator host. The in-memory `RegistryResolver` is the runtime authority. Credential values are read from host environment variables or a secrets file at session-create time and held in memory only for the duration of the request.

**Resolution rule** (from `docs/agent-roster-and-model-resolution.md §2`, made normative here):

```
resolve(role):
  entry = roster[role]                     // { tier, instance? }
  if entry.instance:
    inst = instances[entry.instance]
    model = inst.serves.find(s => s.tier === entry.tier).model
    return { instanceId: inst.id, tier: entry.tier, model }
  else:
    inst = instances.find(i => i.serves.some(s => s.tier === entry.tier))
          ?? throw NoInstanceForTierError(entry.tier)
    model = inst.serves.find(s => s.tier === entry.tier).model
    return { instanceId: inst.id, tier: entry.tier, model }
```

### Cross-cutting
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.
- **Security (NFR-1):** `credentialsRef` values are read via a `CredentialResolver` abstraction (`env:<VAR>`, `file:<path>`, `keychain:<service>`) that runs on the coordinator host only. The client never sees the ref string or the resolved value.
- **Observability (NFR-7):** Every `resolve(role)` call is written to the trace with `{ role, instanceId, tier, model: '<redacted>' }` — the model label is logged for audit but the vendor model string is redacted in the trace to avoid leaking configuration.
- **Resilience (NFR-4):** If a registered instance becomes unreachable mid-session, the coordinator marks it `reachable: false` in the snapshot and emits a `harness.reachability` event. The `SessionRouter` falls back to another instance serving the same tier if one exists, logging the failover.
- **Extensibility:** New harness drivers are added by implementing `HarnessAdapter` and adding an instance entry to `registry.yaml`. No coordinator code changes are needed.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | YAML config file on the host, not a UI-editable database | Vendor model IDs must stay on the host behind the gateway; config-as-code fits git-backed audit trail |
| 2 | Resolution rule is a coordinator-side function, not client-side | Keeps model IDs off the wire and ensures the same rule governs every session regardless of which client connected |
| 3 | `resolveModel` injected into adapters by `SessionRouter`, not hard-coded in adapter | Adapters stay harness-adapter–interface-compliant; the registry is the single source of model truth |
| 4 | Reviewer-distinct rule enforced at startup and on reload | A misconfigured registry that silently produces same-model reviews would undermine the independence guarantee in the roster doc |
| 5 | Client `Harnesses` screen driven by snapshot projection, not a static seed | The design-baseline seed (`HARNESSES`, `TIERS` in store.ts) was a placeholder; live data is required by FR-19 |

## Open questions
- Should `registry.yaml` be per-project (repo root) or per-engineer (`~/.arke/registry.yaml`), or both with merge semantics?
- What is the credential resolution order when both a project-level and a user-level credential exist for the same `credentialsRef` key?
- Should the roster resolution table in the `Harnesses` screen be read-only or should it offer an "Edit in config" action that opens `registry.yaml` in the harness editor?
- When a running instance becomes unreachable and a fallback instance is used, should in-flight sessions be migrated or left on the original instance until they complete?
