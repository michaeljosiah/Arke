# Delivery board as a live projection

## Why

A kanban where a card moves because the work moved — from real signals — replaces the simulated board baseline. Every column transition must be computed from spec frontmatter, session state, and CI/PR signals arriving over the live event stream, never from a manual drag.

## What changes

- ADDED delivery-board — a live, signal-driven Kanban in `packages/client` where each card's column is computed from `CardState` delivered by the coordinator event stream; the simulated board baseline is replaced.   (breaking: no)
- ADDED delivery-board — cards aggregate across harnesses with a harness + model label per card, so multi-harness runs are visible on a single board.   (breaking: no)
- MODIFIED delivery-board — selecting a card navigates to the session-detail view for that card (SPEC-011).   (breaking: no)

## Requirements

### Requirement: Card column computed deterministically from real signals
`capability: delivery-board` · `delta: ADDED (feat/delivery-board)`

The system SHALL compute each card's `BoardColumn` exclusively from spec frontmatter status, session execution status, PR/CI state, and the `needsHuman` flag as folded by the `ReadModel` in `packages/coordinator/src/read-model.ts`. No drag-and-drop or manual column assignment SHALL alter a card's column; the column is a read-only projection of the underlying state. The derivation function MUST be a pure function of `CardState` — identical inputs produce identical output with no side effects — so that the board is deterministic and testable in isolation.

Cards are keyed on the **harness `sessionId`** (the `CardState.id` field, as defined in SPEC-002 and SPEC-003): this is the unit of execution on the board. Cards are grouped visually by `specId` when multiple sessions belong to the same specification. The board MUST NOT key cards on `spec_id` alone, because multiple sessions (spec + tasks) exist per specification.

The column derivation follows `BoardColumn` in `@arke/contracts/src/events.ts`:

| Condition (evaluated in order) | Column |
|---|---|
| `needsHuman === true` | `needs-human` |
| `kind === "task"` and `status === "done"` | `diff` |
| `kind === "task"` and status ≠ `"done"` | `implementing` |
| `specStatus === "draft"` | `authoring` |
| `specStatus === "in-review"` | `review` |
| `specStatus === "approved"` | `approved` |
| `specStatus === "merged"` | `merged` |
| _(no matching condition)_ | `authoring` (default) |

The board's `useBoard()` hook applies this logic by reading `CardState.column` from the coordinator's read model — it does NOT re-derive the column independently. The read model is the single point of column computation.

#### Scenario: Spec promoted to in-review moves its card
- **WHEN** the coordinator emits a `spec.status` event with `status: "in-review"` for a spec
- **THEN** the board moves the spec's card from `authoring` to `review` within one render cycle
- **AND** no user action is required

#### Scenario: Blocked session surfaces in needs-human
- **WHEN** the coordinator emits a `permission.asked` event for a session
- **THEN** the corresponding card moves to the `needs-human` column
- **AND** it remains there until a `permission.replied` event arrives for that same `permissionId`

#### Scenario: Permission replied returns card to its prior column
- **WHEN** a `permission.replied` event arrives for a session currently in `needs-human`
- **THEN** the card moves back to the column it would occupy based on the remaining derivation rules (e.g. `implementing` for a running task)
- **AND** the `needsHuman` flag is `false`

#### Scenario: Drag has no effect on column
- **WHEN** a user attempts to drag a card from `authoring` to `review`
- **THEN** the card snaps back to its computed column at the next render cycle
- **AND** no state change is written to the coordinator or the trace

#### Scenario: Card with unknown specId appears in authoring
- **WHEN** a `session.status` event arrives carrying a `specId` for which no `spec.status` event has been seen
- **THEN** the card defaults to the `authoring` column (matching the `deriveColumn` default)
- **AND** it moves to the correct column when the corresponding `spec.status` event arrives

### Requirement: Board updates from the live event stream with reconnect safety
`capability: delivery-board` · `delta: ADDED (feat/delivery-board)`

The board SHALL subscribe to the coordinator via the `ArkeTransport` defined in SPEC-003 and update cards in real time as `DomainEvent` frames arrive. On initial connection it SHALL render from the `snapshot` frame before any events arrive, so the board is never blank after the first paint. On reconnect it SHALL re-render from the fresh `snapshot` without visual flicker and without orphaned cards from the previous connection's state — the snapshot is the authoritative replacement of prior client-side state, not a merge on top of it.

Cards that were visible before a disconnect and are absent from the reconnect snapshot SHALL be removed from the board. The board MUST NOT accumulate ghost cards across reconnects.

#### Scenario: Board renders from snapshot on first connect
- **WHEN** the `ArkeTransport` transitions to `open` and receives a `snapshot` frame
- **THEN** all cards present in `CardState[]` are rendered in their correct columns
- **AND** no event loop is needed before the board is visible

#### Scenario: Reconnect snapshot replaces, not merges
- **WHEN** the transport reconnects after a gap during which a session was deleted on the coordinator
- **THEN** the board renders only the cards present in the reconnect snapshot
- **AND** the card for the deleted session is no longer shown

#### Scenario: New card appears without reload
- **WHEN** a `session.status` event arrives for a session id not yet on the board
- **THEN** a new card appears in the correct column within one render cycle
- **AND** no page reload is required

#### Scenario: Connection-state indicator reflects transport state
- **WHEN** the `ArkeTransport` is in `reconnecting`
- **THEN** the board shows a visible reconnecting indicator
- **AND** existing cards remain visible at their last-known columns (stale but not blank)

### Requirement: Cards aggregated across harnesses with identity labels
`capability: delivery-board` · `delta: ADDED (feat/delivery-board)`

The board SHALL display cards from all connected harnesses on a single surface. Each card SHALL show a `harness` label (e.g. "OpenCode") and a `model` label (e.g. "capable") drawn from the `harness` and `model` fields on `CardState`, so a product engineer can distinguish work running on different harnesses at a glance. Cards from different harnesses that belong to the same `specId` SHALL be visually grouped or annotated so the engineer can identify all work for a specification in one place.

No cross-harness aggregation logic lives in the board component — the `CardState[]` from the coordinator snapshot already contains all sessions from all harnesses. The board renders them; the coordinator normalises them. Per SPEC-018, the board shows the **active project context's** sessions only: the snapshot is scoped to the connection's active project, so all aggregated cards belong to that one project.

#### Scenario: Two harnesses on one board
- **WHEN** the coordinator has sessions from both an OpenCode adapter and a mock adapter
- **THEN** both sets of cards appear on the same board
- **AND** each card's harness label matches the `harness` field on its `CardState`

#### Scenario: Sessions for the same spec from different harnesses are visually grouped
- **WHEN** two cards share a `specId` but have different `harness` values
- **THEN** the board renders them with a visual link or grouping indicating they belong to the same specification
- **AND** each card still shows its own harness and model label

#### Scenario: Model label updates when session status changes
- **WHEN** a `session.status` event arrives with a `model` field
- **THEN** the card's model label updates to reflect the new value
- **AND** no other card is affected

### Requirement: Selecting a card opens session detail
`capability: delivery-board` · `delta: ADDED (feat/delivery-board)`

The system SHALL navigate to the session-detail view (SPEC-011) when a user selects a card on the board. The navigation SHALL pass the card's `id` (the harness `sessionId`) and `specId` so the detail view can load the correct transcript, todos, and diff without an additional coordinator round-trip.

#### Scenario: Card selection navigates to detail
- **WHEN** a user clicks or activates a card on the board
- **THEN** the application navigates to the session-detail route for that card's `id` (the harness `sessionId`)
- **AND** the detail view opens with the correct session's transcript visible

#### Scenario: Blocked card selection opens detail with permission overlay
- **WHEN** a user selects a card in the `needs-human` column
- **THEN** the session-detail view opens
- **AND** the pending permission overlay is immediately visible (per SPEC-012)

### Requirement: Empty-state and error-state handling
`capability: delivery-board` · `delta: ADDED (feat/delivery-board)`

The board SHALL render a meaningful empty state when the coordinator snapshot contains no cards (e.g. on first launch before any sessions are started). If the transport is in `closed` or `disposed` state (permanent failure, not a transient reconnect), the board SHALL render a distinct error state explaining that the coordinator connection is lost and showing a manual reconnect control. The error state SHALL be distinguishable from the transient `reconnecting` state.

#### Scenario: Empty board on first launch
- **WHEN** the coordinator snapshot contains zero cards
- **THEN** the board renders an empty-state illustration with guidance text (e.g. "No sessions running yet")
- **AND** no column containers are shown as empty rows

#### Scenario: Permanent connection failure
- **WHEN** the `ArkeTransport` enters `closed` or `disposed` state
- **THEN** the board replaces the connection indicator with a prominent error banner
- **AND** a "Reconnect" button is shown that transitions the transport back to `connecting`
- **AND** the error banner is visually distinct from the `reconnecting` spinner

### Requirement: Human correction without drag
`capability: delivery-board` · `delta: ADDED (feat/delivery-board)`

Because column assignment is a read-only projection and drag-and-drop is disabled, the board SHALL provide a mechanism for human corrections to the underlying state (e.g. marking a stuck session as done, or promoting a spec to `in-review`) without requiring the engineer to edit files manually. Corrections SHALL be issued as coordinator commands via `ArkeTransport.send()`, which routes them to the coordinator and thence to the adapter or spec-file writer. The board itself holds no authority over column placement; it sends commands that change the underlying state, which in turn changes the column.

#### Scenario: Promote spec to in-review from the board
- **WHEN** the engineer activates a "Promote to review" context action on a card in `authoring`
- **THEN** `ArkeTransport.send()` emits a `spec.promote` command carrying the `specId`
- **AND** the coordinator routes the command, updates the spec frontmatter, and emits `spec.status: in-review`
- **AND** the card moves to the `review` column as a result of that event — not as a direct client mutation

#### Scenario: No direct column write from the client
- **WHEN** any board action is triggered (promote, abort, etc.)
- **THEN** no `column` field is ever written directly from the client to any coordinator state
- **AND** the action is recorded in the coordinator trace before execution

### Requirement: Command palette shortcut (optional)
`capability: delivery-board` · `delta: ADDED (feat/delivery-board)`

The board SHOULD provide a command palette (FR-13, *could*) reachable by keyboard shortcut that lets a user jump to a card by spec id or title without scrolling the board. If implemented, it SHALL be read-only (no mutations from the palette); if omitted in v1, the board still satisfies all MUST requirements.

#### Scenario: Jump to card via palette
- **WHEN** a user opens the command palette and types a spec id prefix
- **THEN** matching cards are listed and selecting one navigates to the session detail
- **AND** no card state is modified by the interaction

## Design

### Architectural decision

The board is a pure projection of `CardState[]` received over the `ArkeTransport` (SPEC-003); it holds no authoritative state. The React component tree folds the live `CardState[]` into a column map (`Map<BoardColumn, CardState[]>`) by a pure transform, re-rendering only the columns that changed. On reconnect, the column map is fully replaced from the incoming snapshot — not merged — so ghost cards cannot accumulate. Drag-and-drop is explicitly disabled. Human corrections are sent as coordinator commands, never as direct column writes.

### Target architecture

```
ArkeTransport (SPEC-003)
  └─▶ snapshot → CardState[]  (replaces client state on every connect/reconnect)
  └─▶ event → DomainEvent (incremental fold)
        └─▶ useBoard() hook (packages/client/src/hooks/use-board.ts)
              └─▶ column map: Map<BoardColumn, CardState[]>
                    └─▶ <Board>
                          ├─▶ <BoardColumn> × 7
                          │     └─▶ <BoardCard> (harness label, model label, status dot, specId group marker)
                          ├─▶ <ConnectionIndicator>  ← transport.state$
                          └─▶ <EmptyState> / <ErrorState>
```

`BoardColumn` values come from `@arke/contracts/src/events.ts`; no string literals in the component code. Cards are keyed on `CardState.id` (the harness `sessionId`).

### Data model

No persistent client state. The `useBoard()` hook maintains:

```typescript
interface BoardState {
  columns: Map<BoardColumn, CardState[]>;
  transportState: TransportState;         // from ArkeTransport
  isEmpty: boolean;                       // true when snapshot has zero cards
}
```

State is derived entirely from the server: snapshot on connect (full replacement), incremental folds on events. The hook does NOT write back; corrections go through `ArkeTransport.send()`.

### Interfaces and contracts

- Reads `CardState`, `BoardColumn`, `DomainEvent` from `@arke/contracts`.
- Uses `ArkeTransport` (SPEC-003, `packages/client/src/transport.ts`) for the WebSocket connection.
- Navigation to session detail via React Router (or equivalent) with `{ sessionId, specId }` as route params (using `sessionId`, not a generic `cardId`, to align with SPEC-002/003 identifier naming).
- Coordinator commands (e.g. `spec.promote`) are sent via `ArkeTransport.send({ type: "spec.promote", specId })` — no new endpoint is required for Phase 1; the coordinator routes by `type`.
- Command palette (optional): a `cmdk`-based overlay reading from `BoardState.columns`; no new coordinator contract required.

### Cross-cutting
- **Configuration:** all project configuration — the harness/model registry, coordinator settings, and integrations — lives in the single root file `.arke/config.json` (`ARKE_*` env vars override individual keys; runtime artefacts under `.arke/` are state, not config). See SPEC-005 for the model.

- **Multi-project scoping (SPEC-018):** the coordinator supervises N isolated **project contexts**, each keyed by `projectId` with its own read model, trace, and durable stores (session store, grant store, proposal buffer) under its own `.arke/`. Each client connection resolves against its **active project**; events carry `projectId`. See [SPEC-018](018.multi-project-workspaces.md).

- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`). Do not re-derive styling.

**Security:** the board never sends state mutations based on card position; all client-to-coordinator messages are routed through `ArkeTransport.send()` and traced. The `spec.promote` command is validated by the coordinator before execution — the client cannot directly mutate spec frontmatter.

**Performance (NFR-3):** the `useBoard()` hook derives the column map with a stable reference per column so React can skip re-rendering unchanged columns. `CardState` identity is keyed on `id` (the harness `sessionId`); only the changed card triggers a re-render within its column. On reconnect, the full column map is replaced atomically, not merged field-by-field, to avoid render thrash.

**Observability:** the connection indicator surfaces `ArkeTransport.state$` so product engineers see transport health without opening the browser console. Distinct empty vs error states reduce support burden.

**Accessibility:** column headings are labelled; cards are keyboard-navigable (arrow keys within a column, Tab between columns); the connection indicator uses `aria-live="polite"` for reconnecting/connected status changes; the error state uses `role="alert"`.

## Decision log

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Column computed from `ReadModel` output only, drag disabled | Signal-driven board is the core Arke principle; a manually-moved card would diverge from actual delivery state |
| 2 | `useBoard()` hook owns all transport subscription logic | Centralises reconnect/snapshot/event folding in one place; components are pure render functions of `BoardState` |
| 3 | Reconnect snapshot replaces state, not merges | Merging would accumulate ghost cards for deleted sessions; a full replacement is idempotent and correct |
| 4 | Card key is `sessionId` (harness session id), not `specId` | Multiple sessions exist per spec; keying on `specId` would collapse task cards; `sessionId` is the unique execution unit per SPEC-002/003 |
| 5 | Command palette is *optional* (FR-13 could) | The PRD marks it as a `could`; the board satisfies all MUSTs without it; the palette rides in this spec rather than getting its own to avoid a tiny orphan spec |
| 6 | Seven fixed columns from `BoardColumn` enum | The `BoardColumn` enum in `@arke/contracts` is the single source of column identity; no string literals in component code |
| 7 | Human corrections as coordinator commands, not direct column writes | Preserves the invariant that column is a read-only projection; every correction is traceable |
| 8 | Empty-state and error-state as distinct UI states | Prevents the board from showing a blank page on first launch or a permanent disconnect, which are operationally different from a transient reconnect |
