# Delivery board as a live projection

## Why

A kanban where a card moves because the work moved — from real signals — replaces the simulated board baseline. Every column transition must be computed from spec frontmatter, session state, and CI/PR signals arriving over the live event stream, never from a manual drag.

## What changes

- ADDED delivery-board — a live, signal-driven Kanban in `packages/client` where each card's column is computed from `CardState` delivered by the coordinator event stream; the simulated board baseline is replaced.   (breaking: no)
- ADDED delivery-board — cards aggregate across harnesses with a harness + model label per card, so multi-harness runs are visible on a single board.   (breaking: no)
- MODIFIED delivery-board — selecting a card navigates to the session-detail view for that card (SPEC-011).   (breaking: no)

---

## Requirements

### Requirement: Card column computed from real signals
`capability: delivery-board` · `delta: ADDED (feat/delivery-board)`

The system SHALL compute each card's `BoardColumn` exclusively from spec frontmatter status, session execution status, PR/CI state, and the `needsHuman` flag as folded by the `ReadModel` in `packages/coordinator/src/read-model.ts`. No drag-and-drop or manual column assignment SHALL alter a card's column; the column is a read-only projection of the underlying state.

The column derivation follows `BoardColumn` in `@arke/contracts/src/events.ts`:

| Condition (evaluated in order) | Column |
|---|---|
| `needsHuman === true` | `needs-human` |
| `kind === "task"` and `status === "done"` | `diff` |
| `kind === "task"` and status ≠ "done" | `implementing` |
| `specStatus === "draft"` | `authoring` |
| `specStatus === "in-review"` | `review` |
| `specStatus === "approved"` | `approved` |
| `specStatus === "merged"` | `merged` |

#### Scenario: Spec promoted to in-review moves its card
- **WHEN** the coordinator emits a `spec.status` event with `status: "in-review"` for a spec
- **THEN** the board moves the spec's card from `authoring` to `review` without any user action

#### Scenario: Blocked session surfaces in needs-human
- **WHEN** the coordinator emits a `permission.asked` event for a session
- **THEN** the corresponding card moves to the `needs-human` column
- **AND** it remains there until a `permission.replied` event arrives

#### Scenario: Drag has no effect on column
- **WHEN** a user attempts to drag a card from `authoring` to `review`
- **THEN** the card snaps back to its computed column
- **AND** no state change is written to the coordinator or the trace

### Requirement: Board updates from the live event stream
`capability: delivery-board` · `delta: ADDED (feat/delivery-board)`

The board SHALL subscribe to the coordinator via the `ArkeTransport` defined in SPEC-003 and update cards in real time as `DomainEvent` frames arrive. On initial connection it SHALL render from the `snapshot` frame before any events arrive, so the board is never blank after the first paint. On reconnect it SHALL re-render from the fresh `snapshot` without visual flicker.

#### Scenario: Board renders from snapshot on first connect
- **WHEN** the `ArkeTransport` transitions to `open` and receives a `snapshot` frame
- **THEN** all cards present in `CardState[]` are rendered in their correct columns
- **AND** no event loop is needed before the board is visible

#### Scenario: New card appears without reload
- **WHEN** a `session.status` event arrives for a session id not yet on the board
- **THEN** a new card appears in the correct column within one render cycle
- **AND** no page reload is required

#### Scenario: Connection-state indicator reflects transport state
- **WHEN** the `ArkeTransport` is in `reconnecting`
- **THEN** the board shows a visible reconnecting indicator
- **AND** existing cards remain visible at their last-known columns

### Requirement: Cards aggregated across harnesses with labels
`capability: delivery-board` · `delta: ADDED (feat/delivery-board)`

The board SHALL display cards from all connected harnesses on a single surface. Each card SHALL show a `harness` label (e.g. "OpenCode") and a `model` label (e.g. "capable") drawn from the `harness` and `model` fields on `CardState`, so a product engineer can distinguish work running on different harnesses at a glance.

#### Scenario: Two harnesses on one board
- **WHEN** the coordinator has sessions from both an OpenCode adapter and a mock adapter
- **THEN** both sets of cards appear on the same board
- **AND** each card's harness label matches the `harness` field on its `CardState`

#### Scenario: Model label updates when session status changes
- **WHEN** a `session.status` event arrives with a `model` field
- **THEN** the card's model label updates to reflect the new value
- **AND** no other card is affected

---

### Requirement: Selecting a card opens session detail
`capability: delivery-board` · `delta: ADDED (feat/delivery-board)`

The system SHALL navigate to the session-detail view (SPEC-011) when a user selects a card on the board. The navigation SHALL pass the card's `id` and `specId` so the detail view can load the correct transcript, todos, and diff without an additional lookup.

#### Scenario: Card selection navigates to detail
- **WHEN** a user clicks or activates a card on the board
- **THEN** the application navigates to the session-detail view for that card's `id`
- **AND** the detail view opens with the correct session's transcript visible

#### Scenario: Blocked card selection opens detail with permission overlay
- **WHEN** a user selects a card in the `needs-human` column
- **THEN** the session-detail view opens
- **AND** the pending permission overlay is immediately visible (per SPEC-012)

---

### Requirement: Command palette shortcut (optional)
`capability: delivery-board` · `delta: ADDED (feat/delivery-board)`

The board SHOULD provide a command palette (FR-13, *could*) reachable by keyboard shortcut that lets a user jump to a card by spec id or title without scrolling the board. If implemented, it SHALL be read-only (no mutations from the palette); if omitted in v1, the board still satisfies all MUST requirements.

#### Scenario: Jump to card via palette
- **WHEN** a user opens the command palette and types a spec id prefix
- **THEN** matching cards are listed and selecting one navigates to the session detail
- **AND** no card state is modified by the interaction

## Design

### Architectural decision

The board is a pure projection of `CardState[]` received over the `ArkeTransport` (SPEC-003); it holds no authoritative state. The React component tree folds the live `CardState[]` into a column map (`Map<BoardColumn, CardState[]>`) by a pure transform, re-rendering only the columns that changed. Drag-and-drop is explicitly disabled to enforce signal-driven column placement.

### Target architecture

```
ArkeTransport (SPEC-003)
  └─▶ snapshot → CardState[]
  └─▶ event → DomainEvent (incremental)
        └─▶ useBoard() hook (packages/client/src/hooks/use-board.ts)
              └─▶ column map: Map<BoardColumn, CardState[]>
                    └─▶ <Board>
                          ├─▶ <BoardColumn> × 7
                          │     └─▶ <BoardCard> (harness label, model label, status dot)
                          └─▶ <ConnectionIndicator>  ← transport.state$
```

`BoardColumn` values come from `@arke/contracts/src/events.ts`; no string literals in the component code.

### Data model

No persistent client state. The `useBoard()` hook maintains:

```typescript
interface BoardState {
  columns: Map<BoardColumn, CardState[]>;
  transportState: TransportState;         // from ArkeTransport
}
```

State is derived entirely from the server: snapshot on connect, incremental folds on events. The hook does NOT write back; mutations go through `ArkeTransport.send()` where applicable.

### Interfaces and contracts

- Reads `CardState`, `BoardColumn`, `DomainEvent` from `@arke/contracts`.
- Uses `ArkeTransport` (SPEC-003, `packages/client/src/transport.ts`) for the WebSocket connection.
- Navigation to session detail via React Router (or equivalent) with `{ cardId, specId }` as route params.
- Command palette (optional): a `cmdk`-based overlay reading from the same `BoardState`; no new coordinator contract required.

### Cross-cutting
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.

**Security:** the board never sends state mutations based on card position; all client-to-coordinator messages are routed through `ArkeTransport.send()` and traced.

**Performance (NFR-3):** the `useBoard()` hook derives the column map with a stable reference per column so React can skip re-rendering unchanged columns. `CardState` identity is keyed on `id`; only the changed card triggers a re-render within its column.

**Observability:** the connection indicator surfaces `ArkeTransport.state$` so product engineers see transport health without opening the browser console.

**Accessibility:** column headings are labelled; cards are keyboard-navigable (arrow keys within a column, Tab between columns); the connection indicator uses `aria-live="polite"` for reconnecting/connected status changes.

### Testing

- **Unit (hook):** snapshot populates column map; `session.status` event updates the correct card; `permission.asked` moves card to `needs-human`; `permission.replied` moves it back; unknown event type is ignored without throwing.
- **Unit (components):** `<BoardCard>` renders harness + model labels; column heading renders correct label for each `BoardColumn` value.
- **Integration:** coordinator running with `MockAdapter` (SPEC-003 scripted sequence) → board renders four cards in correct columns → force-reconnect → board re-renders from snapshot without blank flash.
- **Manual:** open board, dispatch a task from another terminal, watch the card appear in `implementing`; approve a pending permission, watch the card leave `needs-human`; click a card, confirm navigation to session detail.

### Definition of done

All scenarios pass; the simulated board is fully replaced; TypeScript strict-mode compiles; columns are signal-driven with no drag override; a reviewer has signed off; SPEC-003 is merged (transport dependency).

## Decision log

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Column computed from `ReadModel` output only, drag disabled | Signal-driven board is the core Arke principle; a manually-moved card would diverge from actual delivery state |
| 2 | `useBoard()` hook owns all transport subscription logic | Centralises reconnect/snapshot/event folding in one place; components are pure render functions of `BoardState` |
| 3 | Command palette is *optional* (FR-13 could) | The PRD marks it as a `could`; the board satisfies all MUSTs without it; the palette rides in this spec rather than getting its own to avoid a tiny orphan spec |
| 4 | Seven fixed columns from `BoardColumn` enum | The `BoardColumn` enum in `@arke/contracts` is the single source of column identity; no string literals in component code |

## Open questions

- Should the board support multi-project views (cards from multiple `docs/specifications/` directories) in Phase 1, or is single-project scope sufficient?
- PR/CI state is referenced in column derivation but no `pr.status` or `ci.status` DomainEvent exists yet (SPEC-014 covers projections). Should the board treat missing CI state as "not blocking" or as "unknown"?
