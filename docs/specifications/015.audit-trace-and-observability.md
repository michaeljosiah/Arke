# Audit, activity trace and observability

## Why
Teams governed by change-control processes need to answer, precisely and durably: what did
every agent do, who approved what, and when was each system of record written? The current
coordinator writes events to an NDJSON trace file and that is the whole story. This spec
promotes the trace to a proper observability layer — OpenTelemetry spans at every boundary,
a persisted, queryable NDJSON audit source of truth, an OTLP export path for standard
backends, and a client surface that turns the trace into a readable record of agent
activity.

## What changes
- ADDED observability — the coordinator emits OpenTelemetry spans at every boundary
  (RPC, orchestration command, provider call, external process, persistence, queue
  hand-off) with span attributes carrying ids and paths; metric labels are kept
  low-cardinality
- ADDED audit — completed spans are appended to the local NDJSON trace (`Trace` in
  `packages/coordinator/src/trace.ts`) as the persisted audit source of truth, and are
  exportable over OTLP to a standard backend
- ADDED audit — every permission decision and deterministic projection write is logged
  with its trigger, extending the existing `Trace.write` call sites
- ADDED audit — an audit and activity-trace surface on the client renders what every
  agent did, including approvals and projections, with a traceable line from intent to
  released change

## Requirements

### Requirement: OpenTelemetry spans at every coordinator boundary
`capability: observability` · `delta: ADDED (feat/audit-trace-and-observability)`

The system SHALL emit an OpenTelemetry span for every significant boundary crossing in the
coordinator: inbound client RPC, orchestration command dispatch (fan-out, generation),
harness adapter call (createSession, dispatchAsync, sendMessage, respondToPermission),
external process invocation (git worktree), trace persistence write, and WebSocket push.
Span attributes SHALL carry the `specId`, `sessionId`, `harness`, and operation name.
Metric labels SHALL be low-cardinality (no per-message or per-line values as label
dimensions).

#### Scenario: Adapter call is spanned
- **WHEN** the coordinator calls `adapter.dispatchAsync()` for a task session
- **THEN** an OpenTelemetry span is started before the call and ended after it returns
- **AND** the span carries `arke.specId`, `arke.sessionId`, `arke.harness`, and
  `arke.operation = "dispatchAsync"` as attributes

#### Scenario: Low-cardinality metric labels
- **WHEN** the coordinator emits a metric counter or histogram for events processed
- **THEN** the label set contains only `harness`, `event.type`, and `operation`
- **AND** no per-spec, per-session, or per-message id appears as a metric label dimension

#### Scenario: Span propagation to child sessions
- **WHEN** a task session is dispatched as a child of a spec session
- **THEN** the child dispatch span is linked to the parent spec span via
  `traceId` / `parentSpanId`
- **AND** the full causal chain from spec approval to task dispatch is visible in a
  waterfall view

### Requirement: Completed spans persisted to the local NDJSON trace
`capability: audit` · `delta: ADDED (feat/audit-trace-and-observability)`

The system SHALL append every completed span to the local NDJSON trace file
(`.arke/trace.ndjson`, written by `Trace` in `packages/coordinator/src/trace.ts`) as
a structured record, and SHALL export the same spans over OTLP when an OTLP endpoint is
configured, so teams can route to Jaeger, Honeycomb, Grafana Tempo, or any compatible
backend.

#### Scenario: Completed span written to trace
- **WHEN** an OpenTelemetry span ends
- **THEN** a structured NDJSON record is appended to `.arke/trace.ndjson` containing
  `traceId`, `spanId`, `parentSpanId`, `name`, `startTime`, `endTime`, and the span
  attributes
- **AND** the record is appended atomically (no partial writes visible on tail)

#### Scenario: OTLP export when endpoint configured
- **WHEN** `ARKE_OTLP_ENDPOINT` is set in the environment
- **THEN** completed spans are exported over OTLP/HTTP to that endpoint
- **AND** the local NDJSON trace is still written regardless of export success or failure

#### Scenario: OTLP failure does not stop the coordinator
- **WHEN** the OTLP endpoint is unreachable
- **THEN** the coordinator continues operating
- **AND** spans continue to be written to the local trace
- **AND** a warning is logged once per failure run, not per span

### Requirement: Permission decisions and projection writes are logged with their trigger
`capability: audit` · `delta: ADDED (feat/audit-trace-and-observability)`

The system SHALL log every permission approval or denial (from `PermissionRepliedEvent`)
and every deterministic projection write (from `ProjectionWriteEvent`) as a structured
audit record in the trace, carrying the triggering change (the permission request id or
the spec status change that caused the write) so the chain of cause and effect is
unambiguous.

#### Scenario: Permission approval logged with request context
- **WHEN** the coordinator receives a `permission.replied` event with `granted: true`
- **THEN** a trace record is appended with `kind: "permission.decision"`, `permissionId`,
  `sessionId`, `specId`, `granted: true`, and the `trigger` (the original
  `permission.asked` permissionId that prompted it)
- **AND** the record's timestamp is that of the coordinator, not the client

#### Scenario: Projection write logged with its triggering spec change
- **WHEN** a `projection.write` event is received
- **THEN** a trace record is appended with `kind: "projection.write"`, `target`,
  `specId`, `ok`, and `trigger` (the `generation.decided` event id or spec status that
  caused the write)
- **AND** the record is appended whether the write succeeded or failed

#### Scenario: Trace is append-only
- **WHEN** any trace record is written
- **THEN** no existing record is modified or deleted
- **AND** the trace file grows monotonically; there is no compaction in Phase 1

### Requirement: Audit and activity-trace surface on the client
`capability: audit` · `delta: ADDED (feat/audit-trace-and-observability)`

The system SHALL provide an audit and activity-trace surface on the client that shows,
for each specification, what every agent did — session starts, tool calls, permission
requests and decisions, artefact proposals, approvals, and projection writes — in
chronological order, with a traceable link from an intent (spec approval) to a released
change (projection write confirmed). The surface SHALL be read-only.

#### Scenario: Full activity visible for a spec
- **WHEN** a human opens the audit surface for a specification
- **THEN** they see a chronological list of trace records for that specId: session
  created, dispatches, permission asked/replied, generation proposed/decided, projection
  writes
- **AND** each record shows its timestamp, kind, actor (harness/agent id), and outcome

#### Scenario: Traceable line from approval to write
- **WHEN** a human inspects a `projection.write` record in the audit surface
- **THEN** they can navigate to the `generation.decided approved` record that triggered it
- **AND** from there to the `spec.status approved` record that triggered generation

#### Scenario: Audit surface is read-only
- **WHEN** a human views the audit surface
- **THEN** no action on the surface modifies any record
- **AND** no approval, rejection, or command can be issued from the audit surface

## Design

### Architectural decision
Use the `@opentelemetry/sdk-node` package as the OTel SDK in the coordinator. A single
`NodeTracerProvider` is initialised at coordinator startup with two span processors: a
`BatchSpanProcessor` targeting an `OTLPTraceExporter` (no-op when endpoint not configured)
and a custom `NdjsonSpanExporter` that appends to `Trace`. All coordinator code calls
`tracer.startActiveSpan(name, attrs, fn)` at each boundary. The T3 learnings pattern of
observability-as-audit (typed runtime receipts as signals, OTel spans at RPC/queue/provider
boundaries) is adopted here directly.

### Target architecture

```
coordinator/src/server.ts
    tracer.startActiveSpan("coordinator.pump") {
        for await (event of adapter.streamEvents()) {
            tracer.startActiveSpan("event.ingest", { ... }) { ... }
        }
    }

coordinator/src/otel.ts   (new)
    initTracing()
    ├── NodeTracerProvider
    ├── BatchSpanProcessor → OTLPTraceExporter   (when ARKE_OTLP_ENDPOINT set)
    └── SimpleSpanProcessor → NdjsonSpanExporter  (always; appends to Trace)

coordinator/src/trace.ts  (extended)
    Trace.write(record)          — unchanged; called for domain-level records
    NdjsonSpanExporter.export()  — appends completed OTel spans to the same file

client audit screen (existing baseline)
    ← websocket push "audit.records" for the requested specId
    renders chronological list of trace records
```

The `Trace` class in `packages/coordinator/src/trace.ts` is extended: its `write()` method
remains for domain-level records (client requests, approval decisions, projection writes);
OTel span export is handled by a separate `NdjsonSpanExporter` that calls the same
underlying `appendFile`. Both write to `.arke/trace.ndjson`.

### Data model
Every trace record is one NDJSON line with `at` (epoch ms) and a `kind` discriminant. The
existing `kind: "event"` and `kind: "client.request"` records remain unchanged. New kinds:

```typescript
// Existing (unchanged)
{ at, kind: "event", event: DomainEvent }
{ at, kind: "client.request", request: object }

// New — OTel span record
{ at, kind: "span", traceId, spanId, parentSpanId?, name, startTime, endTime,
  attributes: { "arke.specId"?, "arke.sessionId"?, "arke.harness"?, "arke.operation" },
  status: "ok" | "error", errorMessage? }

// New — permission decision (enriched beyond the raw event)
{ at, kind: "permission.decision", permissionId, sessionId, specId, granted,
  trigger: string /* permissionId of the .asked event */ }

// New — projection write (enriched)
{ at, kind: "projection.write", target, specId, ok, trigger,
  artifactId?, errorMessage? }
```

The client queries the coordinator for trace records filtered by `specId` via a new
WebSocket command:
```typescript
{ type: "get-audit-records", specId: string, since?: number /* epoch ms */ }
```
The coordinator responds with a `push` of `{ type: "audit.records", specId, records: TraceRecord[] }`.

### Interfaces and contracts
New file `packages/coordinator/src/otel.ts`:
```typescript
export function initTracing(tracePath: string, otlpEndpoint?: string): void;
export function getTracer(): Tracer;  // @opentelemetry/api Tracer
```

`Coordinator.start()` calls `initTracing()` before starting the WebSocket server.

Every existing call site that calls `adapter.*` or `trace.write()` is wrapped with
`tracer.startActiveSpan(...)`. A helper `withSpan(name, attrs, fn)` utility function
reduces boilerplate.

New coordinator WebSocket command handler:
```typescript
case "get-audit-records": {
  const records = await this.trace.query(msg.specId, msg.since);
  ws.send(JSON.stringify({ type: "audit.records", specId: msg.specId, records }));
}
```

`Trace.query(specId, since?)` scans the NDJSON file and returns records where
`event.specId === specId` or `record.specId === specId`, after the `since` timestamp.

### Cross-cutting
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.
- **Incremental build:** spans accrue across every feature spec; each feature adds
  `startActiveSpan` at its new boundaries; the exporter and trace query work from day one.
- **Performance (NFR-2):** `BatchSpanProcessor` flushes asynchronously; span overhead on
  the hot `pump()` loop is sub-millisecond. `Trace.query` scans NDJSON on demand; no
  in-memory index in Phase 1; a file-size cap or rotation policy is an open question.
- **Security:** the trace file may contain prompt text or diff snippets; its permissions
  should be restricted to the coordinator process user.
- **Privacy:** no PII should appear in span attributes; spec content is not included in
  attributes (only ids and operation names).

### Testing
- Unit (`NdjsonSpanExporter`): a completed span → exactly one NDJSON line appended with
  correct fields; concurrent exports do not interleave lines.
- Unit (`Trace.query`): records filtered correctly by `specId`; `since` cursor excludes
  older records.
- Unit (OTLP failure): stub OTLP endpoint returning 503; coordinator continues; local
  trace still written; warning logged once.
- Integration: end-to-end trace from `spec.status approved` through fan-out dispatch to
  `session.idle`; assert span chain has `parentSpanId` linking.
- Manual: set `ARKE_OTLP_ENDPOINT=http://localhost:4318`; run Jaeger locally; approve a
  spec; confirm trace visible in Jaeger waterfall view.

### Definition of done
All scenarios pass; the audit surface shows a readable timeline for a real approved spec
from approval through projection write; spans appear in Jaeger when OTLP is configured;
the NDJSON trace contains the full causal chain; no credentials or PII appear in span
attributes; a reviewer has signed off.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | OTel as the span/metric API | Standard, exportable to any backend; avoids a proprietary tracing format |
| 2 | `NdjsonSpanExporter` writes to the same `Trace` file | One durable file is the source of truth; the OTel exporter and domain-level writes coexist there |
| 3 | OTLP export optional, local file always written | Teams without an OTLP backend still have a full local audit trail |
| 4 | `Trace.query` by scan (Phase 1) | Sufficient for Phase 1 file sizes; index or SQLite FTS deferred to Phase 2 |
| 5 | Audit surface is read-only | Auditability requires immutability; actions live in the cockpit, not the audit view |

## Open questions
- Should `.arke/trace.ndjson` rotate by file size or date, and if so, how should
  `Trace.query` span rotated files?
- Should span attributes include the spec title (useful for human reading) or only the
  specId (avoids large attribute values for long titles)?
- Is a `since` cursor sufficient for the client, or should the audit surface support
  pagination over large trace files?
