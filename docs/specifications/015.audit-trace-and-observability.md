# Audit, activity trace and observability

## Why
Teams governed by change-control processes need to answer, precisely and durably: what did
every agent do, who approved what, and when was each system of record written? The current
coordinator writes events to an NDJSON trace file and that is the whole story. This spec
promotes the trace to a proper observability layer ŌĆö OpenTelemetry spans at every boundary,
a persisted, queryable NDJSON audit source of truth, an OTLP export path for standard
backends, and a client surface that turns the trace into a readable record of agent
activity.

## What changes
- ADDED observability ŌĆö the coordinator emits OpenTelemetry spans at every boundary
  (RPC, orchestration command, provider call, external process, persistence, queue
  hand-off) with span attributes carrying ids and paths; metric labels are kept
  low-cardinality
- ADDED audit ŌĆö completed spans are appended to the local NDJSON trace (`Trace` in
  `packages/coordinator/src/trace.ts`) as the persisted audit source of truth, and are
  exportable over OTLP to a standard backend
- ADDED audit ŌĆö every permission decision and deterministic projection write is logged
  with its trigger, extending the existing `Trace.write` call sites
- ADDED audit ŌĆö an audit and activity-trace surface on the client renders what every
  agent did, including approvals and projections, with a traceable line from intent to
  released change

## Requirements

### Requirement: OpenTelemetry spans at every coordinator boundary
`capability: observability` ┬Ę `delta: ADDED (feat/audit-trace-and-observability)`

The system SHALL emit an OpenTelemetry span for every significant boundary crossing in the
coordinator: inbound client RPC, orchestration command dispatch (fan-out, generation),
harness adapter call (createSession, dispatchAsync, sendMessage, respondToPermission),
external process invocation (git worktree), trace persistence write, and WebSocket push.
Span attributes SHALL carry the `specId`, `sessionId`, `harness`, and operation name.
Metric labels SHALL be low-cardinality (no per-message or per-line values as label
dimensions). A root trace context SHALL be created per spec (keyed by `specId`) at the
point a spec session is first dispatched, and SHALL be propagated as the parent context
for all downstream spans (generation, fan-out, projection) so the full causal chain is
visible as a single waterfall trace.

#### Scenario: Adapter call is spanned
- **WHEN** the coordinator calls `adapter.dispatchAsync()` for a task session
- **THEN** an OpenTelemetry span is started before the call and ended after it returns
- **AND** the span carries `arke.specId`, `arke.sessionId`, `arke.harness`, and
  `arke.operation = "dispatchAsync"` as attributes

#### Scenario: Low-cardinality metric labels
- **WHEN** the coordinator emits a metric counter or histogram for events processed
- **THEN** the label set contains only `harness`, `event.type`, and `operation`
- **AND** no per-spec, per-session, or per-message id appears as a metric label dimension

#### Scenario: Span propagation to child sessions
- **WHEN** a task session is dispatched as a child of a spec session
- **THEN** the child dispatch span is linked to the parent spec span via
  `traceId` / `parentSpanId`
- **AND** the full causal chain from spec approval to task dispatch is visible in a
  waterfall view

#### Scenario: Root trace context propagated through generation and projection
- **WHEN** a spec is approved and the coordinator dispatches a generation session, then
  fans out writes, then the SPEC-014 plugin emits `projection.write`
- **THEN** all spans in that causal chain share the same root `traceId` established when
  the spec session was first created
- **AND** the waterfall view from spec approval to projection write is fully connected
  without orphaned spans

### Requirement: Concurrent NDJSON appends are serialised; writes are atomic
`capability: audit` ┬Ę `delta: ADDED (feat/audit-trace-and-observability)`

The system SHALL serialise all appends to the NDJSON trace file through a write queue or
mutex so that concurrent span exports and domain-level `Trace.write()` calls do not
produce interleaved or partial JSON lines. Each line SHALL be a complete, parseable JSON
object before being committed; no partial write SHALL be visible on a tail of the file.

#### Scenario: Concurrent exports do not interleave lines
- **WHEN** the `NdjsonSpanExporter` and `Trace.write()` are called concurrently from
  different async contexts
- **THEN** each line in `.arke/trace.ndjson` is a complete, parseable JSON object
- **AND** no line is truncated or interleaved with another

#### Scenario: Write queue drains on coordinator shutdown
- **WHEN** the coordinator receives a shutdown signal
- **THEN** the write queue is drained (all pending appends completed) before the process
  exits
- **AND** no enqueued trace records are silently dropped on clean shutdown

### Requirement: Completed spans persisted to the local NDJSON trace
`capability: audit` ┬Ę `delta: ADDED (feat/audit-trace-and-observability)`

The system SHALL append every completed span to the local NDJSON trace file
(`.arke/trace.ndjson`, written by `Trace` in `packages/coordinator/src/trace.ts`) as
a structured record, using a `BatchSpanProcessor` for both the NDJSON exporter and the
OTLP exporter so that span export does not block the coordinator's hot `pump()` loop.
The system SHALL export the same spans over OTLP when an OTLP endpoint is configured,
so teams can route to Jaeger, Honeycomb, Grafana Tempo, or any compatible backend.
OTLP export is best-effort: the local trace is the durability guarantee; OTLP is a
secondary export path. The `Trace` and its `.arke/trace.ndjson` are per **project
context** (SPEC-018): each project owns its trace file, written under its own `.arke/`.
The coordinator SHALL additionally emit a **supervisor-level** NDJSON log of context
lifecycle (create / open / close / evict / idle-timeout), each record stamped with
`projectId`, distinct from any one project's trace.

#### Scenario: Completed span written to trace
- **WHEN** an OpenTelemetry span ends
- **THEN** a structured NDJSON record is appended to `.arke/trace.ndjson` containing
  `traceId`, `spanId`, `parentSpanId`, `name`, `startTime`, `endTime`, `arke.specId`
  (from attributes), and `arke.operation`
- **AND** the record is appended atomically (no partial writes visible on tail)

#### Scenario: OTLP export when endpoint configured
- **WHEN** `ARKE_OTLP_ENDPOINT` is set in the environment
- **THEN** completed spans are exported over OTLP/HTTP to that endpoint
- **AND** the local NDJSON trace is still written regardless of export success or failure

#### Scenario: OTLP failure does not stop the coordinator
- **WHEN** the OTLP endpoint is unreachable
- **THEN** the coordinator continues operating
- **AND** spans continue to be written to the local trace
- **AND** a warning is logged once per failure run, not per span

#### Scenario: BatchSpanProcessor queue-full drops are logged, not silent
- **WHEN** the `BatchSpanProcessor` export queue is full (OTLP endpoint permanently
  unreachable and queue has reached capacity)
- **THEN** the processor logs a warning with the number of dropped spans
- **AND** the local NDJSON trace is not affected (it has its own `BatchSpanProcessor`)

#### Scenario: Span export does not block the event pump
- **WHEN** the coordinator's `pump()` loop processes an event that triggers a span
- **THEN** the span export to NDJSON and OTLP is asynchronous (via `BatchSpanProcessor`)
- **AND** the `pump()` loop does not await the file write before processing the next event

### Requirement: Span attributes contain no PII, secrets, or spec content
`capability: audit` ┬Ę `delta: ADDED (feat/audit-trace-and-observability)`

The system SHALL restrict span attributes to a declared allowlist: `arke.specId`,
`arke.sessionId`, `arke.harness`, `arke.operation`, `arke.target`, `http.status_code`,
`error.type`, `error.message` (truncated to 256 characters). Span attributes SHALL NOT
include spec body text, artefact content, prompt text, agent output, credential fragments,
or any value derived from user input beyond a stable id. The `NdjsonSpanExporter` SHALL
apply this allowlist before writing; any attribute not on the allowlist is dropped
silently. Error messages in `error.message` SHALL be truncated and SHALL NOT include raw
API responses (which may contain credential fragments).

#### Scenario: Spec content excluded from span attributes
- **WHEN** the coordinator spans a generation dispatch carrying the spec markdown
- **THEN** the span record in the trace contains only `arke.specId` and `arke.operation`
- **AND** no spec body, artefact content, or prompt text appears in any attribute

#### Scenario: Error messages are truncated and sanitised
- **WHEN** a SoR API call fails and the error response body contains a credential fragment
  (e.g. a partial API key in a 403 body)
- **THEN** the `error.message` attribute in the span is truncated to 256 characters
- **AND** no raw API response body is included in any span attribute or trace record

#### Scenario: Unknown attributes are stripped
- **WHEN** code adds a span attribute not on the declared allowlist
- **THEN** the `NdjsonSpanExporter` drops that attribute before writing
- **AND** no warning is required; the allowlist is the source of truth

### Requirement: Dead-letter records are a defined trace record kind
`capability: audit` ┬Ę `delta: ADDED (feat/audit-trace-and-observability)`

The system SHALL define `kind: "dead-letter"` as a named trace record type (per SPEC-002,
which requires unmappable harness events to be written to the audit trace as dead-letter
records). Dead-letter records SHALL be queryable via the `get-audit-records` command and
SHALL appear in the audit surface for the relevant `specId` (or under a system-level view
when no `specId` can be resolved).

#### Scenario: Dead-letter record appears in audit surface
- **WHEN** the adapter writes a dead-letter record (SPEC-002: unmappable event)
- **THEN** the record is appended to the trace as
  `{ at, kind: "dead-letter", rawPayload: string, reason: string,
     seq: number, specId?: string }`
- **AND** the audit surface for the affected spec (or system view) shows the dead-letter
  entry with the reason and raw payload (truncated to 1024 characters)

#### Scenario: Dead-letter count is queryable
- **WHEN** a `get-audit-records` query is issued for a `specId` (or without a `specId`
  for the system view)
- **THEN** dead-letter records are included in the results alongside domain events
- **AND** the audit surface can filter to show only dead-letter records

### Requirement: Trace is monotonically sequenced and restart-resilient
`capability: audit` ┬Ę `delta: ADDED (feat/audit-trace-and-observability)`

The system SHALL assign a monotonically increasing sequence number to each trace record,
independent of the `seq` on the `EventEnvelope` (which is per-connection). This sequence
SHALL be a file-level monotonic counter, incremented by the write queue before each
append, so that records can be ordered unambiguously after a restart. On coordinator
restart, the system SHALL read the last sequence number from the tail of the existing
trace file and resume from there. If the trace file is unreadable or missing on startup,
the coordinator SHALL log a warning and start a new trace file; it SHALL NOT crash.

#### Scenario: Trace records are monotonically ordered
- **WHEN** two records are written concurrently
- **THEN** each has a unique, monotonically increasing `seq` in the trace file
- **AND** the ordering of records by `seq` is unambiguous

#### Scenario: Restart resumes sequence numbering from last record
- **WHEN** the coordinator restarts and an existing trace file is present
- **THEN** the coordinator reads the `seq` of the last record in the file
- **AND** new records start from `lastSeq + 1`
- **AND** no duplicate sequence numbers appear across the restart boundary

#### Scenario: Unreadable trace file does not crash the coordinator
- **WHEN** the trace file exists but is unreadable (permissions error or corrupt)
- **THEN** the coordinator logs a warning identifying the file and the error
- **AND** starts a new trace file (with the old file left in place)
- **AND** continues operating normally

### Requirement: Permission decisions and projection writes are logged with their trigger
`capability: audit` ┬Ę `delta: ADDED (feat/audit-trace-and-observability)`

The system SHALL log every permission approval or denial (from `PermissionRepliedEvent`)
and every deterministic projection write (from `ProjectionWriteEvent`) as a structured
audit record in the trace, carrying the triggering change (the permission request id or
the `generation.decided` event id that caused the write) so the chain of cause and effect
is unambiguous. The `trigger` field on projection write records SHALL be the stable event
id of the `generation.decided` event, not a free-form string, so navigational links in
the audit surface are resolvable.

#### Scenario: Permission approval logged with request context
- **WHEN** the coordinator receives a `permission.replied` event with `granted: true`
- **THEN** a trace record is appended with `kind: "permission.decision"`, `permissionId`,
  `sessionId`, `specId`, `granted: true`, and `trigger` (the `permissionId` of the
  original `permission.asked` event)
- **AND** the record's timestamp is that of the coordinator, not the client

#### Scenario: Projection write logged with its triggering event id
- **WHEN** a `projection.write` event is received
- **THEN** a trace record is appended with `kind: "projection.write"`, `target`,
  `specId`, `ok`, `artifactId`, and `trigger` set to the stable event id of the
  `generation.decided` event that caused the write
- **AND** the record is appended whether the write succeeded or failed

#### Scenario: Trigger field is a resolvable event id
- **WHEN** a human navigates from a `projection.write` trace record to its trigger
- **THEN** the `trigger` value resolves to a `generation.decided` record in the trace
  with a matching event id
- **AND** from there navigates to the `spec.status approved` record that triggered
  generation (via `specId` and timestamp ordering)

#### Scenario: Trace is append-only
- **WHEN** any trace record is written
- **THEN** no existing record is modified or deleted
- **AND** the trace file grows monotonically; there is no compaction in Phase 1

### Requirement: Audit and activity-trace surface on the client
`capability: audit` ┬Ę `delta: ADDED (feat/audit-trace-and-observability)`

The system SHALL provide an audit and activity-trace surface on the client that shows,
for each specification, what every agent did ŌĆö session starts, tool calls, permission
requests and decisions, artefact proposals, approvals, and projection writes ŌĆö in
chronological order, with a traceable link from an intent (spec approval) to a released
change (projection write confirmed). The surface SHALL be read-only. The coordinator SHALL
cap the records returned per `get-audit-records` query at 500 rows (configurable via
`ARKE_AUDIT_QUERY_LIMIT`), returning the most recent records first when the cap is
reached, and SHALL indicate how many total records exist for that `specId` so the client
can prompt the user to refine the `since` cursor. Per SPEC-018, every span attribute and
audit record carries a `projectId` dimension; a `get-audit-records` query SHALL resolve
against the connection's **active project** and return only that project's trace, and the
response SHALL carry its `projectId`.

#### Scenario: Full activity visible for a spec
- **WHEN** a human opens the audit surface for a specification
- **THEN** they see a chronological list of trace records for that specId: session
  created, dispatches, permission asked/replied, generation proposed/decided, projection
  writes, dead-letter records (if any)
- **AND** each record shows its `seq`, timestamp, kind, actor (harness/agent id), and
  outcome

#### Scenario: Traceable line from approval to write
- **WHEN** a human inspects a `projection.write` record in the audit surface
- **THEN** they can navigate to the `generation.decided approved` record whose event id
  matches the `trigger` field
- **AND** from there to the `spec.status approved` record that triggered generation

#### Scenario: Audit surface is read-only
- **WHEN** a human views the audit surface
- **THEN** no action on the surface modifies any record
- **AND** no approval, rejection, or command can be issued from the audit surface

#### Scenario: Large result set is capped with a hint
- **WHEN** a `get-audit-records` query matches more than 500 records
- **THEN** the coordinator returns the 500 most recent records and a `total` count
- **AND** the audit surface shows "Showing 500 of N ŌĆö use the time filter to narrow"

## Design

### Architectural decision
Use the `@opentelemetry/sdk-node` package as the OTel SDK in the coordinator. A single
`NodeTracerProvider` is initialised at coordinator startup with **two `BatchSpanProcessor`
instances**: one targeting an `OTLPTraceExporter` (no-op when endpoint not configured)
and one targeting a custom `NdjsonSpanExporter` that appends to `Trace`. Using
`BatchSpanProcessor` for both ensures span export is asynchronous and does not block the
hot `pump()` loop. The T3 learnings pattern of observability-as-audit (typed runtime
receipts as signals, OTel spans at RPC/queue/provider boundaries) is adopted here
directly. All appends to the NDJSON file are serialised through a write queue to prevent
interleaving.

### Target architecture

```
coordinator/src/server.ts
    tracer.startActiveSpan("coordinator.pump") {
        for await (event of adapter.streamEvents()) {
            tracer.startActiveSpan("event.ingest", { specId, sessionId }) { ... }
        }
    }

coordinator/src/otel.ts   (new)
    initTracing()
    Ōö£ŌöĆŌöĆ NodeTracerProvider
    Ōö£ŌöĆŌöĆ BatchSpanProcessor ŌåÆ OTLPTraceExporter    (when ARKE_OTLP_ENDPOINT set)
    ŌööŌöĆŌöĆ BatchSpanProcessor ŌåÆ NdjsonSpanExporter   (always; appends to Trace via write queue)

coordinator/src/trace.ts  (extended)
    WriteQueue                  ŌĆö serialises appendFile calls; drains on shutdown
    Trace.write(record)         ŌĆö appends domain-level records through WriteQueue
    NdjsonSpanExporter.export() ŌĆö appends OTel spans through the same WriteQueue;
                                  applies attribute allowlist before writing

client audit screen (existing baseline)
    ŌåÉ websocket push "audit.records" { records, total } for the requested specId
    renders chronological list of trace records; navigable trigger links
```

A per-spec root trace context is established when the spec session is first created
(storing `traceId` keyed by `specId`). All downstream spans (generation dispatch,
fan-out, SPEC-014 projection writes) use this stored context as their parent, giving a
single unified waterfall per spec.

### Data model
Every trace record is one NDJSON line with `at` (epoch ms), `seq` (file-level monotonic
counter), and a `kind` discriminant. The existing `kind: "event"` and `kind:
"client.request"` records gain a `seq` field. New kinds:

```typescript
// Existing (seq added)
{ at, seq, kind: "event", event: DomainEvent }
{ at, seq, kind: "client.request", request: object }

// New ŌĆö OTel span record
{ at, seq, kind: "span", traceId, spanId, parentSpanId?, name, startTime, endTime,
  attributes: {
    "arke.specId"?: string,      // from allowlist only
    "arke.sessionId"?: string,
    "arke.harness"?: string,
    "arke.operation": string,
    "arke.target"?: string,
    "http.status_code"?: number,
    "error.type"?: string,
    "error.message"?: string,    // truncated to 256 chars
  },
  status: "ok" | "error" }

// New ŌĆö permission decision (enriched beyond the raw event)
{ at, seq, kind: "permission.decision", permissionId, sessionId, specId, granted,
  trigger: string /* permissionId of the .asked event */ }

// New ŌĆö projection write (enriched)
{ at, seq, kind: "projection.write", target, specId, ok, trigger,
  artifactId?, idempotencyKey?, errorMessage? }

// New ŌĆö dead-letter (from SPEC-002)
{ at, seq, kind: "dead-letter", rawPayload: string, reason: string, specId?: string }
```

The `NdjsonSpanExporter` applies the attribute allowlist before writing the `kind: "span"`
record. Any attribute not on the allowlist is silently dropped; no additional fields from
the raw span object are included.

The client queries the coordinator for trace records filtered by `specId` via a new
WebSocket command:
```typescript
{ type: "get-audit-records", specId: string, since?: number /* epoch ms */ }
```
The coordinator responds with:
```typescript
{ type: "audit.records", specId, records: TraceRecord[], total: number }
```
`records` is capped at `ARKE_AUDIT_QUERY_LIMIT` (default 500), most-recent-first when
capped. `total` is the full count matching the query, allowing the client to show a
truncation hint.

`Trace.query(specId, since?)` scans the NDJSON file and returns records where the
`specId` matches either:
- top-level `record.specId`, or
- `record.event.specId` (for `kind: "event"` records), or
- `record.attributes["arke.specId"]` (for `kind: "span"` records)

This unified lookup ensures spans are not missed in audit queries.

### Interfaces and contracts
New file `packages/coordinator/src/otel.ts`:
```typescript
export function initTracing(tracePath: string, otlpEndpoint?: string): void;
export function getTracer(): Tracer;                  // @opentelemetry/api Tracer
export function getRootContext(specId: string): Context | undefined;
export function setRootContext(specId: string, ctx: Context): void;
```

`Coordinator.start()` calls `initTracing()` before starting the WebSocket server.

`packages/coordinator/src/trace.ts` gains:
```typescript
class WriteQueue {
  enqueue(line: string): Promise<void>;  // serialises appendFile; resolves when written
  drain(): Promise<void>;                // call on shutdown; waits for all enqueued writes
}

class Trace {
  // Reads last seq from tail of file on construction; resumes from lastSeq + 1
  constructor(path: string);
  async write(record: Record<string, unknown>): Promise<void>;  // assigns seq
  async query(specId: string, since?: number, limit?: number):
    Promise<{ records: TraceRecord[], total: number }>;
}
```

Every existing call site that calls `adapter.*` or `trace.write()` is wrapped with
`tracer.startActiveSpan(...)`. A helper `withSpan(name, attrs, fn)` utility function
reduces boilerplate.

New coordinator WebSocket command handler:
```typescript
case "get-audit-records": {
  const { records, total } = await this.trace.query(
    msg.specId, msg.since,
    Number(process.env.ARKE_AUDIT_QUERY_LIMIT ?? 500)
  );
  ws.send(JSON.stringify({ type: "audit.records", specId: msg.specId, records, total }));
}
```

### Cross-cutting
- **Multi-project scoping (SPEC-018):** the coordinator supervises N isolated **project contexts**, each keyed by `projectId` with its own root, harness runner, read model, trace, and durable stores under its own `.arke/`. Everything this spec places on "the coordinator" is perŌĆōproject context; each client connection resolves against its **active project**. See [SPEC-018](018.multi-project-workspaces.md).
- **Configuration:** all project configuration ŌĆö the harness/model registry, coordinator settings, and integrations ŌĆö lives in the single root file `.arke/config.json` (`ARKE_*` env vars override individual keys; runtime artefacts under `.arke/` are state, not config). See SPEC-005 for the model.
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) ŌĆö the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.
- **Incremental build:** spans accrue across every feature spec; each feature adds
  `startActiveSpan` at its new boundaries; the exporter and trace query work from day one.
- **Performance (NFR-2):** both `BatchSpanProcessor` instances flush asynchronously; span
  overhead on the hot `pump()` loop is sub-millisecond. `Trace.query` scans NDJSON on
  demand and is capped; no in-memory index in Phase 1.
- **Security:** the trace file may contain prompt text or diff snippets; its file
  permissions SHALL be restricted to the coordinator process user (mode 0600 or equivalent).
  The attribute allowlist is the enforcement point for PII/secret redaction in spans.
- **Trace file path not writable:** if `mkdir` or `appendFile` fails on the trace path,
  the coordinator SHALL log the error and continue operating (degraded audit, not crash).
  A metric counter SHALL be incremented for each dropped write so operators can detect the
  degraded state.
- **Trigger field format:** the `trigger` field on `projection.write` and
  `permission.decision` trace records SHALL be the stable event id of the triggering event
  (not a free-form description), enabling programmatic navigation between causally related
  records.

### Testing
- Unit (`WriteQueue`): concurrent enqueues produce non-interleaved NDJSON lines in the
  file; `drain()` waits for all writes before resolving.
- Unit (`NdjsonSpanExporter`): a completed span ŌåÆ exactly one NDJSON line with allowlisted
  attributes only; non-allowlisted attribute dropped; `error.message` > 256 chars is
  truncated; concurrent exports do not interleave lines (via `WriteQueue`).
- Unit (`Trace.seq`): first write after construction gets `seq = lastSeq + 1`; restart
  reads tail and resumes correctly; unreadable file ŌåÆ warning logged, new file started.
- Unit (`Trace.query`): records filtered by `specId` at all three locations; `since`
  cursor excludes older records; cap respected with correct `total` returned; dead-letter
  records included.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | OTel as the span/metric API | Standard, exportable to any backend; avoids a proprietary tracing format |
| 2 | `NdjsonSpanExporter` writes to the same `Trace` file | One durable file is the source of truth; the OTel exporter and domain-level writes coexist there |
| 3 | OTLP export optional, local file always written | Teams without an OTLP backend still have a full local audit trail; OTLP is best-effort |
| 4 | `Trace.query` by scan with cap (Phase 1) | Sufficient for Phase 1 file sizes; index or SQLite FTS deferred to Phase 2 |
| 5 | Audit surface is read-only | Auditability requires immutability; actions live in the cockpit, not the audit view |
| 6 | `WriteQueue` serialises all `appendFile` calls | `appendFile` is not atomic for concurrent callers; a write queue prevents interleaved lines and enables `drain()` on shutdown |
| 7 | `BatchSpanProcessor` for both NDJSON and OTLP exporters | `SimpleSpanProcessor` is synchronous I/O on the hot `pump()` loop; `BatchSpanProcessor` is async and non-blocking |
| 8 | Attribute allowlist enforced at export time | PII/secret redaction cannot rely on call-site discipline; the exporter is the last line of defence |
| 9 | Per-spec root trace context propagated through all downstream spans | A unified `traceId` per spec is required for a connected waterfall from approval to projection write; ad hoc `parentSpanId` linkage alone is insufficient |
| 10 | File-level monotonic `seq` on every trace record | Per-connection `seq` on `EventEnvelope` is not a file-level order; a file-level counter enables unambiguous ordering after restart |
| 11 | `kind: "dead-letter"` as a named trace record kind | SPEC-002 requires dead-letter records in the audit trace; without a defined kind they are unqueryable and invisible in the audit surface |
| 12 | `trigger` field is a stable event id, not a free-form string | Navigation between causally related records in the audit surface requires a resolvable foreign key, not a label |
| 13 | Graceful degradation on unwritable trace file (no crash) | A permissions error should not take down the coordinator; the degraded state is surfaced via a metric counter |
