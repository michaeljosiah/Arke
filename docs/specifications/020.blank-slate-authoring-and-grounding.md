# Blank-slate specification authoring & file grounding

## Why
The MVP is the specification spine — **author it, review it, deliver it** — and the heart of that is
the authoring loop. Today the cockpit (SPEC-006) opens on an *existing* working file; there is no
first-class way to start a **brand-new** specification from nothing and watch it take shape through
conversation, and there is no way to feed the discussion **your own files** (a brief, notes, an
existing doc). This spec adds the blank-slate start and file grounding so a Product Engineer can sit
down, talk to the agents, drop in reference material, and see the specification build itself section
by section — with everything else (Generation, system-of-record projection) held back so the loop
stays focused.

## What changes
- ADDED authoring-cockpit — starting a **new specification** creates a working file from
  `specification.template.md` on a fresh feature branch, with the frontmatter seeded and every body
  section present but **empty**; the cockpit opens on this blank slate and the document grows in the
  live preview as the conversation develops (building on SPEC-006's section-by-section preview).
- ADDED grounding — the engineer can **upload files** into the discussion; they are stored on the
  harness host under the project's `.arke/grounding/` (never routed or held by the browser) and read
  by the authoring agent as context. Grounding files are inputs to the conversation, not part of the
  specification, and are never committed into `docs/specifications/`.

> **Scope stance (MVP).** The downstream surfaces (Generation, Record sync, Integrations) are
> temporarily hidden from the nav so the product is exactly: **Specifications · Authoring cockpit ·
> Review panel · Delivery board** (+ Audit). This spec owns the authoring half; the loop it completes
> is author → review (SPEC-007) → approve (SPEC-006) → deliver (SPEC-009/010).
>
> **Persistence stance (unchanged from SPEC-006).** The agent writes the emerging spec into the
> working file on the feature branch as you talk — visible live — and it is **committed only on
> approval** (SPEC-006 `approveDraft`). Propose · decide · execute holds: the growing document is an
> uncommitted working draft until the human approves it.

## Requirements

### Requirement: A new specification starts from the template as a blank slate
`capability: authoring-cockpit` · `delta: ADDED (feat/mvp-focus)`

When the engineer starts a new specification, the system SHALL create a working spec file under the
active project's `docs/specifications/` on a fresh feature branch, generated from
`specification.template.md`, with the frontmatter seeded (`spec_id`, a title, `status: draft`,
`branch`, `created`/`updated`) and every body section from the template present but **empty** (Why,
What changes, Requirements, Design, Tasks). The cockpit SHALL open on this file immediately and, per
SPEC-006, render the empty sections as placeholders. No requirement, scenario, or prose SHALL be
invented at creation time — the blank slate is a skeleton the human and agents fill through
conversation. The file SHALL remain an uncommitted draft on its branch until approval (SPEC-006).

#### Scenario: Starting a new spec scaffolds the template on a fresh branch
- **WHEN** the engineer starts a new specification with a title
- **THEN** the coordinator writes a new file under `docs/specifications/` derived from the template, on a new feature branch, with `status: draft` and the title in the frontmatter
- **AND** every template body section is present and empty

#### Scenario: The blank slate renders as the section skeleton
- **WHEN** the cockpit opens on the newly-created spec
- **THEN** the preview shows the section headings (Why / What changes / Requirements / Design / Tasks) with empty-placeholder bodies (SPEC-006 section-by-section rendering)
- **AND** the status bar shows the new branch and `draft` status

#### Scenario: Nothing is authored until the conversation begins
- **WHEN** the spec is first created
- **THEN** no requirements or scenarios exist in the file
- **AND** the file is an uncommitted draft on its feature branch, not yet the authoritative version

### Requirement: The specification is authored progressively as the conversation develops
`capability: authoring-cockpit` · `delta: ADDED (feat/mvp-focus)`

As the engineer and the authoring agents converse, the agent SHALL write the emerging content into
the working file **section by section and incrementally** — e.g. one requirement (with at least one
WHEN/THEN scenario) at a time — rather than emitting the whole document in a single turn, so the
human watches the specification take shape and can steer it. Each write SHALL be reflected in the
live preview per SPEC-006. Sections not yet discussed SHALL remain empty placeholders. When the human
redirects, the agent SHALL revise the affected section in place rather than appending a contradictory
duplicate.

#### Scenario: A discussed requirement materialises in the preview
- **WHEN** the engineer and agent agree on a requirement during the conversation
- **THEN** that requirement — as a normative SHALL statement with a WHEN/THEN scenario — appears in the Requirements section of the live preview
- **AND** sections not yet discussed remain empty placeholders

#### Scenario: Steering revises a section in place
- **WHEN** the engineer redirects an already-drafted part of the specification
- **THEN** the agent updates that section of the working file in place
- **AND** the preview reflects the revision without leaving a contradictory duplicate

### Requirement: Files can be uploaded as grounding for the discussion
`capability: grounding` · `delta: ADDED (feat/mvp-focus)`

The cockpit SHALL let the engineer upload one or more files as grounding for the authoring
discussion. The coordinator SHALL store each uploaded file on the **harness host** under the active
project's `.arke/grounding/` directory; the browser SHALL NOT be in the file's storage path beyond
transmitting its content to the coordinator (the host is the trust boundary, NFR-1). The authoring
agent SHALL be able to read grounding files as context when shaping the specification. Grounding
files are **inputs to the conversation, not part of the specification**: they SHALL NOT be written
into the spec file and SHALL NOT be committed to `docs/specifications/` on approval. Each stored
filename SHALL be validated so it cannot escape `.arke/grounding/`, and uploads SHALL be bounded by a
configurable maximum size.

#### Scenario: An uploaded file lands on the host and is acknowledged
- **WHEN** the engineer uploads a file in the cockpit
- **THEN** the coordinator writes it under the active project's `.arke/grounding/`
- **AND** the cockpit lists it as available grounding for the discussion

#### Scenario: The agent uses grounding when authoring
- **WHEN** grounding files are present and the engineer asks the agent to draft from them
- **THEN** the agent reads the grounding file content and reflects it in the specification it writes

#### Scenario: Grounding is never part of the committed specification
- **WHEN** the specification is approved and committed (SPEC-006)
- **THEN** the commit contains only the spec file under `docs/specifications/`
- **AND** no grounding file is written into the spec or committed

#### Scenario: An unsafe upload name is rejected
- **WHEN** an upload names a path that would resolve outside `.arke/grounding/` (e.g. `../../etc/x`)
- **THEN** the coordinator rejects it with a `validation-error` and writes nothing

## Design

### Architectural decision
Reuse SPEC-006 wholesale — the split surface, the live preview reading the on-disk working file, and
commit-on-approval — and add only the two missing edges: a **blank-slate creation** step (scaffold
the template onto a branch, then hand the cockpit the new file) and a **host-side grounding store**
the agent reads from. Nothing about the persistence or trust model changes: the agent writes the
working file via the harness, the preview projects it, and approval commits it; grounding is host-only
context, never client-held and never part of the spec.

### Target architecture

```
Engineer                                   Coordinator (active project context, SPEC-018)
  │  "new spec: <title>"                      │
  ├──────────── spec.create ────────────────▶ │  next number + slug → new branch → write
  │                                           │  docs/specifications/<nnn>.<slug>.md from
  │  ◀──── { specId, branch, path } ───────── │  specification.template.md (seeded frontmatter,
  │        (cockpit opens on the blank slate) │  empty sections)  [status: draft, uncommitted]
  │                                           │
  │  upload file(s)                           │
  ├──────────── grounding.upload ───────────▶ │  validate name within .arke/grounding →
  │  ◀──── { name, path } ──────────────────── │  write file on host
  │                                           │
  │  converse (SPEC-006 sendMessage) ────────▶ │  agent reads .arke/grounding/* as context,
  │  ◀──── live preview grows (spec.file) ──── │  writes the working file section by section
  │                                           │
  │  approve (SPEC-006 approveDraft) ────────▶ │  commit ONLY docs/specifications/<file>
```

- **`spec.create`** (new coordinator op): allocates the next spec number + a slug from the title,
  creates the feature branch, renders `specification.template.md` with seeded frontmatter and empty
  sections (reusing `@arke/contracts` `parseSpecDoc` / `setFrontmatterStatus`), writes it under the
  active project's `docs/specifications/`, and returns `{ specId, branch, path }`. The client then
  routes into the cockpit on that `specId` (existing SPEC-006 flow).
- **`grounding.upload` / `grounding.list`** (new coordinator ops): `upload` takes `{ name, content }`,
  canonicalises `name` within the `.arke/grounding/` safe root (`InputValidator.canonicalisePath`),
  writes the content on the host, and returns `{ name, path }`; `list` returns the current grounding
  files (name + size). The authoring agent reads them through the harness's normal filesystem access
  to the project root (`.arke/grounding/` is inside it).
- **Client**: the "New specification" action opens a small title prompt → `spec.create` → cockpit.
  The cockpit gains a grounding affordance (file input / drag-drop) that sends `grounding.upload` and
  shows the grounding list; a chip/section indicates which files are in play for the discussion.
- **Agent behaviour**: the `spec-author` image's instructions are extended to (a) author
  incrementally, one requirement + scenario at a time into the correct section, and (b) treat
  `.arke/grounding/` as read-only source material, never inventing requirements not grounded in the
  conversation or the uploaded files.

### Data model
No new persistent domain entities. New runtime state: files under `.arke/grounding/` (git-ignored
project state, not configuration and not specification content). The spec file itself is the existing
SPEC-006 working file. Grounding is deliberately outside `docs/specifications/`.

### Interfaces and contracts
**New coordinator WS ops (client → coordinator):**
```typescript
{ type: "request", op: "spec.create",     args: { title: string } }
  // → { specId, branch, path }
{ type: "request", op: "grounding.upload", args: { name: string, content: string } }
  // → { name, path }   (content is UTF-8 text or base64 for binary; bounded by a max size)
{ type: "request", op: "grounding.list" }
  // → Array<{ name: string, size: number }>
```
`spec.create` reuses the SPEC-006 `spec.file` op to serve the new file to the cockpit; the growing
document flows through the existing `sendMessage` / `spec.file` / preview path. Approval is the
unchanged SPEC-006 `approveDraft` (branch-guarded, transactional).

### Cross-cutting
- **Builds on SPEC-006 / SPEC-004 / SPEC-008:** the cockpit surface, live preview, delta rendering,
  and `approveDraft` are SPEC-006; the template and scaffold conventions are SPEC-004; the spec
  lifecycle/library is SPEC-008. This spec adds the creation entry point and grounding only.
- **MVP scope:** the downstream nav surfaces (Generation, Record sync, Integrations) are hidden
  (client `HIDDEN_FOR_MVP`); Delivery board and Audit are kept. This is a scoping choice, reversible
  in one line, not a removal.
- **Security (NFR-1/5):** grounding files are written and read only on the coordinator host; the
  browser transmits content but is never in the storage path and never holds credentials. Upload
  names are canonicalised within `.arke/grounding/` (traversal rejected); uploads are size-bounded.
  The spec commit contains only the spec file — grounding never crosses into `docs/specifications/`.
- **Multi-project scoping (SPEC-018):** `spec.create` and the grounding ops resolve against the
  connection's **active project** context — its root, `docs/specifications/`, and `.arke/grounding/`.
- **Design template:** the new-spec prompt and the grounding affordance follow the `arke-design` skill
  (shadcn neutral tokens, Geist, Lucide, the prototype screens); do not re-derive styling.
- **Observability (NFR-7):** `spec.create` (specId, branch) and each `grounding.upload` (name, size —
  never the content) are written to the append-only trace.

## Tasks
- [ ] Coordinator `spec.create`: next-number + slug, create feature branch, render the template with
      seeded frontmatter + empty sections, write under the active project's `docs/specifications/`,
      return `{ specId, branch, path }`; trace the creation
- [ ] Coordinator `grounding.upload` / `grounding.list`: canonicalise the name within
      `.arke/grounding/` (reject traversal), write on host, size-bound; return name/path/list; trace
      name + size (never content)
- [ ] Client: "New specification" → title prompt → `spec.create` → route into the cockpit on the new
      spec (reuse SPEC-006 cockpit)
- [ ] Client: grounding affordance in the cockpit (file input / drag-drop) → `grounding.upload`;
      render the grounding list + an in-play indicator
- [ ] Agent image: extend the `spec-author` instructions for incremental, section-by-section authoring
      and for treating `.arke/grounding/` as read-only source material
- [ ] Ensure `.arke/grounding/` is git-ignored (project state, not tracked)
- [ ] Tests (below)

### Testing
- **Unit:** `spec.create` renders the template with empty sections + seeded frontmatter (draft, branch,
  title); slug/number allocation; grounding name canonicalisation rejects traversal and accepts a
  safe name; size bound enforced.
- **Integration:** create a new spec → the cockpit opens on an empty-section skeleton on a fresh
  branch; upload a grounding file → it lands under `.arke/grounding/` and appears in `grounding.list`;
  author a requirement via `sendMessage` → the preview grows that section; approve → the commit
  contains only the spec file, no grounding.
- **Manual:** start a new spec from a title, drop in a reference doc, converse, watch the requirements
  section fill in, steer a revision, approve — confirm the branch commit contains only the spec file.

### Definition of done
All scenarios pass; a new specification starts as the empty template on a fresh branch and grows in
the preview as the conversation develops; uploaded grounding lands under `.arke/grounding/` on the
host, is readable by the agent, and never enters the committed spec; traversal-unsafe upload names are
rejected; the MVP nav shows only the author→review→deliver spine; `npm run typecheck` + `npm run
build` green; a reviewer has signed off.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | The blank slate IS the real spec template with empty sections, not a bespoke form | The document is the artifact; growing the actual template keeps one authoring model and one renderer (SPEC-006), and what you approve is exactly what you watched build |
| 2 | Draft-in-cockpit, commit on approval — reuse SPEC-006 unchanged | Keeps propose·decide·execute intact: the growing file is an uncommitted working draft until the human approves; no new persistence model |
| 3 | Grounding files live on the host under `.arke/grounding/`, never committed | They are inputs to the discussion, not the spec; keeping them host-side (browser never in the file path) preserves the trust boundary and keeps `docs/specifications/` clean |
| 4 | Incremental, section-by-section authoring over one-shot generation | So the human sees the specification take shape and can steer mid-flight, rather than reviewing a wall of text after the fact |
| 5 | Downstream surfaces hidden (not removed) for the MVP | Focuses the product on the authoring/review/delivery spine while keeping every screen one line away from restoration |

## Open questions
- Grounding scope: per-spec (`.arke/grounding/<specId>/`) or project-wide `.arke/grounding/`? Per-spec
  is tidier but complicates cross-spec reuse; project-wide is simpler for the MVP.
- Grounding lifecycle: when are uploaded files cleaned up (never / on spec merge / manual)? And the
  max upload size + allowed types (text-first, or images for multimodal agents?).
- Spec numbering on `spec.create`: strictly monotonic next number, or a date-based id? (The library
  uses `NNN.slug.md`; confirm the allocation rule under concurrent creation.)
- Should `spec.create` also open a PR/branch remotely, or stay local until approval + the normal PR
  flow (SPEC-008)? (Lean: local branch until approval.)
