# Arke CLI ‚Äî spin up, open, and drive the system headlessly

## Why
Today you bring Arke up by hand ‚Äî a coordinator process here, a Vite client there, a browser tab, a config file ‚Äî and there is no scriptable way to drive a session end to end. That is friction for a human and a wall for an **agent**: the whole point of Arke is that agents run the loop, but they cannot start the services or exercise an operation without a human at a keyboard. A single `arke` command that spins up the services, opens the UI, and exposes the system's operations as scriptable, JSON-speaking commands makes Arke runnable and testable by both.

## What changes
- ADDED cli ‚Äî a new `@arke/cli` package exposing an `arke` binary: `up` / `down` / `open` / `status` / `doctor` bring the services up (coordinator, client, and ‚Äî via SPEC-016 ‚Äî a managed harness), wait for readiness, print the URLs, and optionally launch the browser. (breaking: no)
- ADDED operations-api ‚Äî `arke` subcommands that map to the system's operations (`session`, `prompt`, `watch`, `todos`, `diff`, `permission`, `grants`, `agents`, `trace`), so a human or an agent can run a spec‚Üítask‚Üídiff loop and test operations without the browser. (breaking: no)
- ADDED service-lifecycle ‚Äî a small request/response command surface on the coordinator (over its existing WebSocket) that backs those operations and is recorded in the trace, so the CLI drives the system *through* the coordinator, never around it. (breaking: no)

## Requirements

### Requirement: Bring the services up, ready, with one command
`capability: service-lifecycle` ¬∑ `delta: ADDED (feat/arke-cli)`

`arke up` SHALL start the coordinator and (unless told otherwise) the client, optionally start a managed harness (SPEC-016), wait until each reports ready, and print the client URL and coordinator address. It SHALL fail with a non-zero exit and a clear reason if a service does not become ready, and `arke down` SHALL stop exactly the services `arke up` started ‚Äî and no others.

#### Scenario: up starts and reports ready
- **WHEN** `arke up` is run in a project with a valid `.arke/config.json`
- **THEN** the coordinator and client start, the command waits until both are ready
- **AND** it prints the client URL (e.g. `http://localhost:5188`) and the coordinator address

#### Scenario: open launches the browser
- **WHEN** `arke up --open` (or `arke open` against a running stack) is run
- **THEN** the default browser is launched to the client URL

#### Scenario: down stops only what up started
- **WHEN** `arke down` is run after `arke up`
- **THEN** the coordinator, client, and any managed harness that `up` started are stopped
- **AND** a harness that was already running externally (attach mode) is left running

#### Scenario: a service that fails to start is surfaced
- **WHEN** a service cannot become ready (port in use, harness unreachable, invalid config)
- **THEN** `arke up` exits non-zero with a reason naming the failing service

### Requirement: Report health and diagnose the environment
`capability: service-lifecycle` ¬∑ `delta: ADDED (feat/arke-cli)`

`arke status` SHALL report the readiness of the coordinator, the harness, and the client, plus a config summary; `arke doctor` SHALL diagnose the environment (config validity, harness reachability, port availability, versions). Both SHALL support `--json` for machine consumption.

#### Scenario: status as JSON
- **WHEN** `arke status --json` is run
- **THEN** it prints a JSON object with the readiness of each service and a config summary
- **AND** exits zero only if the coordinator is reachable

### Requirement: Drive operations through the coordinator
`capability: operations-api` ¬∑ `delta: ADDED (feat/arke-cli)`

The CLI SHALL expose the system's operations as subcommands that drive the running coordinator ‚Äî never the harness directly ‚Äî so every action flows through the single read-model + audit trace (NFR-7) and the trust boundary holds. The mapped operations SHALL include: create a session for a spec and a child task; list sessions; send a prompt (blocking) and dispatch one (non-blocking) naming an agent role and a logical tier; read a session's todos and diff; list pending permissions and decide them; list and revoke remembered grants; list and materialise agent images; and tail the trace. The CLI SHALL NOT hold or print credentials ‚Äî they stay on the host with the coordinator/harness (NFR-1).

#### Scenario: an agent runs a spec‚Üítask‚Üídiff loop headlessly
- **WHEN** an agent runs `arke session create --spec SPEC-X --json`, then `arke prompt <id> --agent implementer --tier mid --async`, then reads `arke diff <id> --json`
- **THEN** each command drives the coordinator and returns structured results
- **AND** the actions appear in the trace, attributed like any other governed action

#### Scenario: operations never bypass the coordinator
- **WHEN** any operation subcommand runs
- **THEN** it issues a request to the coordinator (not a direct call to the harness)
- **AND** no command places a credential in an argument, environment passed to the client, or printed output

### Requirement: Machine-friendly, non-interactive contract
`capability: cli` ¬∑ `delta: ADDED (feat/arke-cli)`

Every command SHALL offer a `--json` machine-readable output and SHALL use meaningful exit codes (zero on success, non-zero on failure) so it composes in scripts and agent harnesses. The CLI SHALL be **non-interactive** ‚Äî no chat shell or REPL; reading the system is `arke watch` (a stream), driving it is discrete commands. `arke watch` SHALL stream normalised domain events as NDJSON to stdout, and turn completion SHALL be observable via the `turn.quiescent` event (SPEC-003), never inferred by polling.

#### Scenario: watch streams events as NDJSON
- **WHEN** `arke watch --json` is run while a turn streams
- **THEN** each normalised domain event is written as one JSON line to stdout
- **AND** a `turn.quiescent` line marks the end of a turn

#### Scenario: failure is scriptable
- **WHEN** a command fails (coordinator unreachable, unknown session, validation error)
- **THEN** it exits non-zero and prints a structured error under `--json`

### Requirement: Human-in-the-loop is preserved, not bypassed
`capability: operations-api` ¬∑ `delta: ADDED (feat/arke-cli)`

Permission decisions SHALL be explicit CLI commands (`once` / `always` / `reject`, SPEC-016); the CLI SHALL NOT silently auto-approve a gated mutation. An agent driving the system decides explicitly (or relies on a previously human-granted remembered grant), so propose¬∑decide¬∑execute holds whether the operator is a person or an agent.

#### Scenario: a gated action waits for an explicit decision
- **WHEN** an operation raises a permission and no matching remembered grant exists
- **THEN** the action does not proceed until `arke permission decide <id> --once|--always|--reject` is run
- **AND** the CLI never approves it on the agent's behalf without that command

## Design

### Architectural decision
A thin `@arke/cli` Node package exposing an `arke` binary that is a **client of the coordinator**, plus a small request/response command surface added to the coordinator's existing WebSocket. The CLI never embeds its own coordinator/adapter for operations (that would fork the read model and the trace) ‚Äî it ensures one coordinator is running and talks to it. `arke up` supervises the coordinator and client as child processes the same way SPEC-016 supervises the harness, reusing that lifecycle pattern. The CLI is **headless and scriptable**: chat/authoring lives in the cockpit and the harness, not in a terminal REPL.

### Target architecture
```
arke (CLI, @arke/cli)
  ‚îú‚îÄ up/down: spawn + supervise coordinator (+ client, + managed harness via SPEC-016)
  ‚îú‚îÄ open:    OS browser ‚Üí client URL
  ‚îî‚îÄ ops:     WS request/response ‚Üí Coordinator ‚Üí HarnessAdapter
                                       ‚îÇ
                                       ‚îî‚îÄ‚ñ∂ read model + trace (single source of truth)
  watch: subscribe to the coordinator event stream ‚Üí NDJSON on stdout
```
The CLI resolves the coordinator address and client URL from `.arke/config.json` (+ `ARKE_*` + flags). `arke up` checks whether a coordinator is already listening; if so it attaches, if not it spawns one.

### Data model
No new persistent entities. The CLI is stateless between invocations except for a small run handle for the processes `up` started (e.g. pids recorded under `.arke/` so `down` can find them), which is runtime state, not config.

### Interfaces and contracts
- **Coordinator command surface (added):** request/response over the existing WS ‚Äî `{ type: "request", id, op, args }` ‚Üí `{ type: "response", id, ok, result | error }`. Ops back the operation subcommands: `session.create`, `session.list`, `prompt.send`, `prompt.dispatch`, `todos.get`, `diff.get`, `permission.list`, `permission.decide`, `grant.list`, `grant.revoke`, `agents.list`, `agents.materialize`. Each op maps to an existing `HarnessAdapter`/coordinator capability (SPEC-002/003/016) and is traced as a `client.request` (NFR-7).
- **Command map (CLI ‚Üí op):**

  | Command | Backs |
  |---|---|
  | `arke up [--open] [--no-client] [--manage-harness]` | spawn coordinator/client/harness |
  | `arke down` ¬∑ `arke open` ¬∑ `arke status [--json]` ¬∑ `arke doctor [--json]` | lifecycle/health |
  | `arke session create --spec <id> [--parent <sid>]` ¬∑ `arke session list` | `session.create` / `session.list` |
  | `arke prompt <sid> --agent <role> --tier <capable\|mid> [--message ..] [--async]` | `prompt.send` / `prompt.dispatch` |
  | `arke watch [--session <id>]` | event-stream subscription |
  | `arke todos <sid>` ¬∑ `arke diff <sid>` | `todos.get` / `diff.get` |
  | `arke permission list` ¬∑ `arke permission decide <id> --once\|--always\|--reject [--message ..]` | `permission.list` / `permission.decide` |
  | `arke grants list` ¬∑ `arke grants revoke <id>` | `grant.list` / `grant.revoke` |
  | `arke agents list` ¬∑ `arke agents materialize [--dir agents/]` | `agents.list` / `agents.materialize` |
  | `arke trace tail [--json]` | read the NDJSON trace |

- **Global flags:** `--json`, `--coordinator <url>` (default `ws://127.0.0.1:4319`), `--project <dir>`.

### Cross-cutting
- **Configuration:** all project configuration ‚Äî registry, coordinator settings, integrations ‚Äî lives in the single root file `.arke/config.json` (`ARKE_*` env vars override; runtime artefacts under `.arke/`, including the CLI's run handle, are state). See SPEC-005.
- **Design template:** the CLI has no GUI; `arke open` launches the existing client, which follows the `arke-design` skill. No styling is re-derived here.
- **Security (NFR-1/5):** the CLI runs on the host inside the trust boundary; it talks only to `ws://127.0.0.1:<port>`; credentials stay on the host and are never passed as arguments, forwarded to the client, or printed. Operations route through the coordinator so the audit trace is complete.
- **Agent-first ergonomics:** `--json` everywhere, stable exit codes, NDJSON `watch`, and `turn.quiescent` for deterministic turn-completion make the CLI a first-class tool surface an agent harness can call directly ‚Äî Arke driving Arke.

- **Unit:** arg parsing + exit codes; JSON vs human formatting; command‚Üíop mapping; credential-never-printed guard.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | The CLI is a client of the coordinator, not its own embedded runtime | One read model and one audit trace; the CLI must not fork system state by spinning a second coordinator for operations |
| 2 | Headless and scriptable ‚Äî no chat/REPL | Driving is discrete commands, reading is `arke watch`; chat/authoring belongs in the cockpit and the harness, not the terminal. Keeps the CLI a clean tool surface for agents |
| 3 | `--json` + stable exit codes + NDJSON `watch` everywhere | The primary consumer is an agent harness; machine-readable, composable output is the point, not an afterthought |
| 4 | Operations route through a coordinator command surface, not direct harness calls | Preserves the trust boundary (credentials on host) and ensures every action is traced |
| 5 | Permission decisions are explicit commands; the CLI never auto-approves | propose¬∑decide¬∑execute must hold whether the operator is a human or an agent; remembered grants (SPEC-016) are the only non-interactive allow, and they were authored by a human |
| 6 | `up` supervises coordinator/client like SPEC-016 supervises the harness | Reuse one process-lifecycle pattern; `down` stops only what was started |

## Open questions
- Should the coordinator command surface stay on the WS, or should a sibling local HTTP control endpoint be added for simpler one-shot request/response from scripts? (WS keeps one transport; HTTP is easier to `curl`.)
- Packaging: ship `arke` via `npm exec`/`npx` from the workspace, a global install, or bundle it into the Electron desktop app (ADR-0001) as a CLI entry?
- Does `arke up` own the managed-harness decision, or defer entirely to `manageHarness` in `.arke/config.json` (SPEC-016)? (Proposed: config is the default; `--manage-harness`/`--no-manage-harness` override.)
- How much of SPEC-005's registry (agents/instances) should `arke agents`/`arke status` surface before SPEC-005 lands?
