# Arke CLI â€” spin up, open, and drive the system headlessly

## Why
Today you bring Arke up by hand â€” a coordinator process here, a Vite client there, a browser tab, a config file â€” and there is no scriptable way to drive a session end to end. That is friction for a human and a wall for an **agent**: the whole point of Arke is that agents run the loop, but they cannot start the services or exercise an operation without a human at a keyboard. A single `arke` command that spins up the services, opens the UI, and exposes the system's operations as scriptable, JSON-speaking commands makes Arke runnable and testable by both.

## What changes
- ADDED cli â€” a new **.NET / C# console app** (`Arke.Cli`, built on **Spectre.Console.Cli**) exposing the `arke` command: `up` / `down` / `open` / `status` / `doctor` bring the services up (coordinator, client, and â€” via SPEC-016 â€” a managed harness, **on by default**), wait for readiness, print the URLs, and optionally launch the browser. (breaking: no)
- ADDED operations-api â€” `arke` subcommands that map to the system's operations (`session`, `prompt`, `watch`, `todos`, `diff`, `permission`, `grants`, `agents`, `trace`), so a human or an agent can run a specâ†’taskâ†’diff loop and test operations without the browser. (breaking: no)
- ADDED service-lifecycle â€” a small request/response command surface on the coordinator (over its existing WebSocket) that backs those operations and is recorded in the trace, so the CLI drives the system *through* the coordinator, never around it. The C#â†”Node seam is WebSocket + JSON. (breaking: no)

## Requirements

### Requirement: Bring the services up, ready, with one command
`capability: service-lifecycle` Â· `delta: ADDED (feat/arke-cli)`

`arke up` SHALL start the coordinator and (unless told otherwise) the client, **start a managed harness by default** (SPEC-016), wait until each reports ready, and print the client URL and coordinator address. It SHALL fail with a non-zero exit and a clear reason if a service does not become ready, and `arke down` SHALL stop exactly the services `arke up` started â€” and no others.

#### Scenario: up starts and reports ready, managing the harness
- **WHEN** `arke up` is run in a project with a valid `.arke/config.json`
- **THEN** the coordinator and client start, the coordinator is brought up in managed-harness mode so it owns an `opencode serve`, and the command waits until all are ready
- **AND** it prints the client URL (e.g. `http://localhost:5188`) and the coordinator address

#### Scenario: managed harness can be opted out
- **WHEN** `arke up --no-manage-harness` is run (or the running coordinator should attach to an existing server)
- **THEN** the coordinator attaches to an already-running harness instead of spawning one, and `arke down` leaves that external server running

#### Scenario: open launches the browser
- **WHEN** `arke up --open` (or `arke open` against a running stack) is run
- **THEN** the default browser is launched to the client URL

#### Scenario: down stops only what up started
- **WHEN** `arke down` is run after `arke up`
- **THEN** the coordinator, client, and any managed harness that `up` started are stopped
- **AND** a harness that was already running externally (attach mode) is left running

#### Scenario: a service that fails to start is surfaced
- **WHEN** a service cannot become ready (port in use, harness unreachable, invalid config)
- **THEN** `arke up` exits non-zero with a reason naming the failing service

### Requirement: Report health and diagnose the environment
`capability: service-lifecycle` Â· `delta: ADDED (feat/arke-cli)`

`arke status` SHALL report the readiness of the coordinator, the harness, and the client, plus a config summary; `arke doctor` SHALL diagnose the environment (config validity, harness reachability, port availability, versions). Both SHALL support `--json` for machine consumption.

#### Scenario: status as JSON
- **WHEN** `arke status --json` is run
- **THEN** it prints a JSON object with the readiness of each service and a config summary
- **AND** exits zero only if the coordinator is reachable

### Requirement: Drive operations through the coordinator
`capability: operations-api` Â· `delta: ADDED (feat/arke-cli)`

The CLI SHALL expose the system's operations as subcommands that drive the running coordinator â€” never the harness directly â€” so every action flows through the single read-model + audit trace (NFR-7) and the trust boundary holds. The mapped operations SHALL include: create a session for a spec and a child task; list sessions; send a prompt (blocking) and dispatch one (non-blocking) naming an agent role and a logical tier; read a session's todos and diff; list pending permissions and decide them; list and revoke remembered grants; list and materialise agent images; and tail the trace. The CLI SHALL NOT hold or print credentials â€” they stay on the host with the coordinator/harness (NFR-1).

#### Scenario: an agent runs a specâ†’taskâ†’diff loop headlessly
- **WHEN** an agent runs `arke session create --spec SPEC-X --json`, then `arke prompt <id> --agent implementer --tier mid --async`, then reads `arke diff <id> --json`
- **THEN** each command drives the coordinator and returns structured results
- **AND** the actions appear in the trace, attributed like any other governed action

#### Scenario: operations never bypass the coordinator
- **WHEN** any operation subcommand runs
- **THEN** it issues a request to the coordinator (not a direct call to the harness)
- **AND** no command places a credential in an argument, environment passed to the client, or printed output

### Requirement: Machine-friendly, non-interactive contract
`capability: cli` Â· `delta: ADDED (feat/arke-cli)`

Every command SHALL offer a `--json` machine-readable output and SHALL use meaningful exit codes (zero on success, non-zero on failure) so it composes in scripts and agent harnesses. The CLI SHALL be **non-interactive** â€” no chat shell or REPL; reading the system is `arke watch` (a stream), driving it is discrete commands. `arke watch` SHALL stream normalised domain events as NDJSON to stdout, and turn completion SHALL be observable via the `turn.quiescent` event (SPEC-003), never inferred by polling.

#### Scenario: watch streams events as NDJSON
- **WHEN** `arke watch --json` is run while a turn streams
- **THEN** each normalised domain event is written as one JSON line to stdout
- **AND** a `turn.quiescent` line marks the end of a turn

#### Scenario: failure is scriptable
- **WHEN** a command fails (coordinator unreachable, unknown session, validation error)
- **THEN** it exits non-zero and prints a structured error under `--json`

### Requirement: Human-in-the-loop is preserved, not bypassed
`capability: operations-api` Â· `delta: ADDED (feat/arke-cli)`

Permission decisions SHALL be explicit CLI commands (`once` / `always` / `reject`, SPEC-016); the CLI SHALL NOT silently auto-approve a gated mutation. An agent driving the system decides explicitly (or relies on a previously human-granted remembered grant), so proposeÂ·decideÂ·execute holds whether the operator is a person or an agent.

#### Scenario: a gated action waits for an explicit decision
- **WHEN** an operation raises a permission and no matching remembered grant exists
- **THEN** the action does not proceed until `arke permission decide <id> --once|--always|--reject` is run
- **AND** the CLI never approves it on the agent's behalf without that command

## Design

### Architectural decision
A **.NET / C# console app** (`Arke.Cli`, targeting `net10.0`) built on **Spectre.Console.Cli** for command/branch/settings parsing and rich human rendering, exposing the `arke` command. It is a **client of the coordinator** â€” it never embeds its own coordinator/adapter for operations (that would fork the read model and the trace); it ensures one coordinator is running and talks to it. The cross-language seam is **WebSocket + JSON** (`System.Net.WebSockets.ClientWebSocket` + `System.Text.Json`), so a C# CLI over a Node coordinator is clean and stable. `arke up` spawns and supervises the coordinator and client as child processes (`System.Diagnostics.Process`) and brings the coordinator up in **managed-harness mode** so the coordinator (SPEC-016) owns the `opencode serve` lifecycle â€” the CLI does not manage the harness directly, it asks the coordinator to. The CLI is **headless and scriptable**: chat/authoring lives in the cockpit and the harness, not in a terminal REPL.

> Tradeoff (Decision #7): this introduces a **second toolchain** â€” a `dotnet` project alongside the npm workspaces. Accepted deliberately: the team's primary CLI/.NET expertise and Spectre.Console.Cli's command model + rendering outweigh the toolchain split, and the WS+JSON seam keeps the coupling thin.

### Data model
No new persistent entities. The CLI is stateless between invocations except for a small run handle for the processes `up` started (e.g. pids recorded under `.arke/` so `down` can find them), which is runtime state, not config.

### Interfaces and contracts

Each op maps to an existing `HarnessAdapter`/coordinator capability (SPEC-002/003/016) and is traced as a `client.request` (NFR-7).

### Cross-cutting
- **Configuration:** all project configuration â€” registry, coordinator settings, integrations â€” lives in the single root file `.arke/config.json` (`ARKE_*` env vars override; runtime artefacts under `.arke/`, including the CLI's run handle, are state). The C# CLI reads the same `.arke/config.json` (no separate .NET config file). See SPEC-005.
- **Toolchain:** `Arke.Cli` is a standalone `dotnet` project under `cli/Arke.Cli/`, not an npm workspace; it builds/tests with the .NET SDK. It shells out to the workspace npm scripts to start the Node coordinator/client. This is the one place Arke spans two toolchains (Decision #7).
- **Design template:** the CLI has no GUI; `arke open` launches the existing client, which follows the `arke-design` skill. No styling is re-derived here.
- **Security (NFR-1/5):** the CLI runs on the host inside the trust boundary; it talks only to `ws://127.0.0.1:<port>`; credentials stay on the host and are never passed as arguments, forwarded to the client, or printed. Operations route through the coordinator so the audit trace is complete.
- **Agent-first ergonomics:** `--json` everywhere, stable exit codes, NDJSON `watch`, and `turn.quiescent` for deterministic turn-completion make the CLI a first-class tool surface an agent harness can call directly â€” Arke driving Arke.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | The CLI is a client of the coordinator, not its own embedded runtime | One read model and one audit trace; the CLI must not fork system state by spinning a second coordinator for operations |
| 2 | Headless and scriptable â€” no chat/REPL | Driving is discrete commands, reading is `arke watch`; chat/authoring belongs in the cockpit and the harness, not the terminal. Keeps the CLI a clean tool surface for agents |
| 3 | `--json` + stable exit codes + NDJSON `watch` everywhere | The primary consumer is an agent harness; machine-readable, composable output is the point, not an afterthought |
| 4 | Operations route through a coordinator command surface, not direct harness calls | Preserves the trust boundary (credentials on host) and ensures every action is traced |
| 5 | Permission decisions are explicit commands; the CLI never auto-approves | proposeÂ·decideÂ·execute must hold whether the operator is a human or an agent; remembered grants (SPEC-016) are the only non-interactive allow, and they were authored by a human |
| 6 | `up` supervises coordinator/client; the coordinator (not the CLI) owns the harness | Reuse SPEC-016's one harness-lifecycle owner; the CLI asks the coordinator to manage, so there is a single process-owner and `down` stops only what was started |
| 7 | The CLI is a **C#/.NET app on Spectre.Console.Cli**, not a Node package | The team's primary CLI/.NET expertise; Spectre's command/branch/settings model + rich rendering; cross-language is clean over WS+JSON. Accepted cost: a second (`dotnet`) toolchain in the repo |
| 8 | `arke up` manages the harness **by default** | A single `arke up` should give a fully running stack with no manual `opencode serve`; `--no-manage-harness` (or attach config) opts out |
