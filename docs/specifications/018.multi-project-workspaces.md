# Multi-project workspaces — one coordinator, per-project harness runners

## Why
Today the coordinator is single-project: one process is bound to one `ARKE_PROJECT_ROOT`, with one harness, one read model, one trace, and one set of stores (SPEC-002, SPEC-003). The client's picker therefore can only ever reflect that single project — the "Recent projects" list and "Open folder" actions in the [`arke-design`](../../.claude/skills/arke-design/app/screens_spec.jsx) launch template imply a multi-project model that the backend does not have, which is the gap SPEC-004 surfaced.

An engineer works across several repositories. Opening, switching between, and remembering projects must be first-class — without spawning a separate coordinator (and a separate WebSocket port) per project, and without collapsing each project's audit trail, harness, or state into a shared one.

## What changes
- ADDED workspaces — the coordinator becomes a **single control plane** that supervises **N isolated project contexts**, each with its own canonical root, harness runner, read model, trace, and stores (FR-NEW)
- ADDED onboarding — a **durable project registry** records known/recent projects (path, name, last-opened, last-known state) so the picker lists *real* recents and "Open folder / Clone / New" register a project (FR-NEW)
- ADDED lifecycle — opening a project **lazily spawns or attaches its harness runner** (reusing the SPEC-016 managed lifecycle); closing it (or the last client disconnecting) stops only a harness the coordinator started; idle contexts are evicted under a bound (FR-NEW)
- MODIFIED transport — every client connection has **one active project**; the `snapshot` frame and pushed events are scoped to it and carry a `projectId`; switching the active project re-snapshots (breaking: no — single-project clients default to the sole context) (cross-ref SPEC-003)
- MODIFIED onboarding — SPEC-004's `folder.inspect` / `repo.clone` / `scaffold.run` and the command/op surface resolve against the connection's **active project**, each validated against **that project's** safe root (breaking: no)

### Affected specifications
This pivot scopes the existing single-project specs to per-project contexts. It is non-breaking for a single-project deployment (the connection defaults to the sole context), but on implementation SPEC-018 **MODIFIES** the following — the coordinator singletons each spec describes (read model, trace, stores, registry/resolver, integrations, proposal buffer, harness adapter) move *inside* a per-`projectId` `ProjectContext`:

## Requirements

### Requirement: One coordinator, per-project harness runners
`capability: workspaces` · `delta: ADDED (feat/multi-project-workspaces)`

The system SHALL run a single coordinator process that supervises multiple **project contexts**. The coordinator SHALL NOT spawn a separate coordinator process (or a separate client-facing WebSocket port) per project. Each project context SHALL own its harness runner, read model, trace, and durable stores, keyed by a stable `projectId`, so that the projects are isolated at runtime. The harness runner for a project SHALL be created through the SPEC-016 managed lifecycle (spawned if managed, attached otherwise).

#### Scenario: Two open projects share one coordinator but isolate their state
- **WHEN** an engineer opens project A and then project B
- **THEN** there is still exactly one coordinator process and one client-facing WebSocket port
- **AND** project A and project B each have their own harness runner process, read model, and trace file
- **AND** an event, session, or scaffold in project A never appears in project B's read model or trace

#### Scenario: A harness crash is contained to its project
- **WHEN** the harness runner for project A exits unexpectedly
- **THEN** the coordinator records the exit for project A and (per SPEC-016) may restart it
- **AND** project B's runner, snapshot, and in-flight work are unaffected

### Requirement: Durable project registry and real recents
`capability: onboarding` · `delta: ADDED (feat/multi-project-workspaces)`

The coordinator SHALL maintain a durable registry of known projects, stored in a **global** Arke state location (not inside any one project's `.arke/`), recording for each: `projectId`, display name, canonical root path, last-opened timestamp, and last-known folder state (`method-ready` / `partial-scaffold` / `has-code` / `empty`). A `project.list` request SHALL return this registry so the picker renders real recents. Opening, cloning, or scaffolding a folder SHALL register it (or refresh its entry). The registry SHALL contain only display-safe metadata (name, path, state, timestamps) — never credentials or file contents.

#### Scenario: Recents reflect real opened projects, not fixtures
- **WHEN** the client requests `project.list`
- **THEN** the response is the set of projects the engineer has actually opened, most-recent first
- **AND** before any project has been opened the list is empty (the picker shows its empty state)

#### Scenario: Opening a folder registers it
- **WHEN** an engineer opens a folder that is not yet in the registry
- **THEN** a registry entry is created with the canonical path, detected name, and folder state
- **AND** it appears at the top of `project.list` on the next request

### Requirement: Open / switch the active project per connection
`capability: workspaces` · `delta: ADDED (feat/multi-project-workspaces)`

Each client connection SHALL have at most one **active project**. A `project.open` request (by `projectId` or validated path) SHALL ensure the project context exists — validating and canonicalising the root against its own safe root, binding or spawning its harness runner, and loading its read model — then set it as the connection's active project and send that project's `snapshot`. Subsequent pushed events for that connection SHALL be the active project's events and SHALL carry its `projectId`. Opening an already-open project SHALL reuse the existing context rather than creating a second one. A connection with no explicit selection on a single-project deployment SHALL default to the sole context (back-compat with SPEC-003/004).

#### Scenario: Opening a project streams its snapshot
- **WHEN** the client sends `project.open` for a registered project
- **THEN** the coordinator ensures the context, binds its harness, and sends that project's `snapshot` frame (cards + onboarding state)
- **AND** every event subsequently pushed on that connection carries the project's `projectId`

#### Scenario: Switching projects re-snapshots
- **WHEN** the client, already active on project A, sends `project.open` for project B
- **THEN** the connection's active project becomes B and the client receives B's snapshot
- **AND** A's harness runner keeps running (it is not stopped merely because a client looked away), subject to idle eviction

#### Scenario: Re-opening reuses the context
- **WHEN** two `project.open` requests for the same project arrive (same or different connections)
- **THEN** only one project context and one harness runner exist for that project

### Requirement: Per-project isolation and path safety
`capability: workspaces` · `delta: ADDED (feat/multi-project-workspaces)`

Every project context SHALL treat its own canonical root as the safe root for all path validation (SPEC-004 `InputValidator`). A path or operation issued against the active project SHALL never resolve into, read from, or write to another project's root. Each project's trace, session store, and grant store SHALL live under that project's own `.arke/` directory. Credentials remain host-only and per-project (SPEC-002, NFR-1) and SHALL never cross project boundaries or reach the client.

#### Scenario: A path cannot escape into another project
- **WHEN** a `folder.inspect` / `scaffold.run` request on active project A supplies a path that resolves inside project B's root
- **THEN** the coordinator rejects it with a `validation-error` (it is outside A's safe root)
- **AND** no filesystem access occurs in B

#### Scenario: Traces are per-project
- **WHEN** governed actions occur in projects A and B
- **THEN** A's actions are written only to A's `.arke/trace.ndjson` and B's only to B's
- **AND** neither trace contains the other project's events

### Requirement: Lifecycle, idle eviction, and resource bounds
`capability: lifecycle` · `delta: ADDED (feat/multi-project-workspaces)`

The coordinator SHALL bound the number of concurrently active project contexts (and therefore running harness runners) by a configurable maximum. A project context whose harness the coordinator started SHALL be stopped when the project is explicitly closed, or after a configurable idle TTL with no active client and no in-flight work — and SHALL NEVER stop a harness it merely attached to (SPEC-016). Eviction or stop SHALL be recorded (never silent). `project.close` SHALL stop the runner but keep the registry entry; `project.forget` SHALL remove the registry entry and SHALL NOT delete any files on disk.

#### Scenario: Idle managed harness is evicted with a record
- **WHEN** a project context has had no active client and no in-flight work for longer than the idle TTL, and the coordinator started its harness
- **THEN** the coordinator stops that harness runner and records the eviction with the reason and `projectId`
- **AND** the project remains in the registry and can be re-opened

#### Scenario: Closing does not delete; forgetting does not delete
- **WHEN** the engineer closes a project
- **THEN** its managed harness is stopped but its registry entry, `.arke/`, and source remain
- **AND WHEN** the engineer forgets a project, the registry entry is removed but no files are deleted

#### Scenario: Concurrency bound is enforced loudly
- **WHEN** opening a new project would exceed the configured maximum of active contexts
- **THEN** the coordinator evicts the least-recently-used idle context (recording it) or refuses with a clear reason if none is idle
- **AND** it never silently drops a project or its harness

## Design

### Architectural decision
**One coordinator (control plane), per-project harness runners (data plane) — not a coordinator per project.** This mirrors Omnigent, whose always-on `server` (multi-tenant control plane) holds specs/sessions while each `session` names a `workspace` and binds to a `runner` spawned per workspace/host (optionally inside a `bwrap`/`seatbelt`/managed sandbox). The control plane is never cloned per project; isolation and process spawning happen at the runner layer.

Mapping to Arke: the **coordinator = Omnigent's server**, and Arke's **managed harness (SPEC-016) = Omnigent's runner**. Today's singletons — `ReadModel`, `Trace`, `CoordinatorSessionStore`, `GrantStore`, and the `HarnessAdapter` — move *inside* a `ProjectContext`. The `Coordinator` becomes a **supervisor** over a map of contexts, plus a global `ProjectRegistry`. The single client WebSocket and SPEC-003's snapshot/per-connection-seq model are preserved by scoping each connection to one **active project**.

Coordinator-per-project was considered and rejected: it multiplies WebSocket ports, duplicates the server and its reconnect/backpressure machinery, forces the client to discover and connect to N endpoints, and fragments the audit/observability story — all to buy isolation that the per-project-runner boundary already provides.

### Target architecture

```
Client window A ─┐                         ┌─ ProjectContext "alpha" ── HarnessRunner (managed, SPEC-016)
                 │   one WebSocket each     │   ├─ ReadModel  ├─ Trace(.arke/trace.ndjson)
Client window B ─┤ ↔  Coordinator (single)  │   └─ SessionStore  GrantStore   (root: ~/code/alpha)
                 │     ├─ ProjectRegistry ──┤
                 │     └─ ProjectSupervisor─┼─ ProjectContext "beta"  ── HarnessRunner …
                 │        (active-project       (root: ~/code/beta)
                 │         per connection)
```

- **`ProjectContext`** (`packages/coordinator/src/project-context.ts`): owns `{ projectId, root, config, adapter, read, trace, sessions, grants, harnessReachable, projectState, tierDefaults }` — i.e. everything the coordinator holds singularly today. Built lazily on first open.
- **`ProjectSupervisor`** (the evolved `Coordinator`): a `Map<projectId, ProjectContext>`, LRU/idle eviction, the WebSocket server, and per-connection active-project routing. `pump()` runs per context; `emit()` fans out only to connections whose active project matches.
- **`ProjectRegistry`** (`packages/coordinator/src/project-registry.ts`): durable JSON in the **global** Arke home (`ARKE_HOME` ?? OS default — `%APPDATA%/arke` on Windows, `$XDG_STATE_HOME/arke` or `~/.local/state/arke` on Linux, `~/Library/Application Support/arke` on macOS), file `projects.json`. Display-safe metadata only.

### Data model

```typescript
// global registry (one per machine/user)
interface ProjectEntry {
  projectId: string;        // stable hash of the canonical root
  name: string;             // display name (basename or detected repo name)
  root: string;             // canonical absolute path
  lastOpenedAt: number;     // epoch ms
  lastState: 'method-ready' | 'partial-scaffold' | 'has-code' | 'empty' | null;
}
```

`projectId = sha256(canonicalRoot)` (short prefix) so the same folder always maps to the same context. No credentials, config, or file contents are persisted in the registry.

### Interfaces and contracts

**New client→coordinator ops / messages** (same WS, request/response per SPEC-017):

```typescript
{ type: 'request', op: 'project.list' }                              // → ProjectEntry[]
{ type: 'request', op: 'project.open',  args: { projectId? , path? } } // ensure+activate → snapshot follows
{ type: 'request', op: 'project.close', args: { projectId } }         // stop managed runner, keep entry
{ type: 'request', op: 'project.forget',args: { projectId } }         // remove entry, never delete files
```

`snapshot` and every pushed event gain `projectId`. SPEC-004's `folder.inspect` / `repo.clone` / `scaffold.run` / `harness.probe` and SPEC-017's `session.*`, `prompt.*`, `todos.*`, `diff.*`, `permission.*`, `agents.*` resolve against the connection's **active project** and that project's safe root.

### Cross-cutting
- **Security (NFR-1):** each context's root is its own safe root; `InputValidator` is constructed per context; no path or credential crosses a project boundary; the registry is metadata-only.
- **Observability (NFR-7):** per-project `.arke/trace.ndjson` plus a coordinator-level supervisor log (context create/evict/stop, with `projectId`). The SPEC-015 observability surface gains a project dimension.
- **Resilience (NFR-4):** a harness fault is contained to its context and handled by the SPEC-016 lifecycle; eviction and refusal are explicit and logged.
- **Configuration (SPEC-005):** harness/model config stays per project in its `.arke/config.json`; the global registry is separate state. `ARKE_HOME` overrides the registry location.
- **CLI (SPEC-017):** `arke` gains `project list|open|close|forget`; `arke up` may take a `--project <path>` (defaults to cwd) and opens it on launch.
- **Design:** the picker already follows the `arke-design` launch template; this spec makes its Recent projects and entry cards real and multi-project.

### Testing
- Unit: `ProjectRegistry` upsert/list/forget; `projectId` stable for a root; forget never touches files.
- Unit: `ProjectContext` isolation — two contexts have independent read models/traces.
- Unit: per-project safe root rejects a path that resolves into another project's root.
- Integration: open A then B over one WS — one port, two runners, two traces; an event in A is absent from B's snapshot.
- Integration: switch active project re-snapshots; re-opening reuses the context (no second runner).
- Integration: idle eviction stops a managed runner and records it; an attached harness is never stopped.
- Integration: concurrency bound evicts LRU idle or refuses loudly.
- Manual: open two real repos, scaffold one, confirm the other is untouched; close/forget semantics.

### Definition of done
All scenarios pass; one coordinator supervises ≥2 isolated projects with per-project harness runners and traces; the picker lists and opens real projects; idle eviction and concurrency bounds work and are logged; `typecheck` and `build` green across workspaces; a reviewer has signed off.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | One coordinator + per-project harness runners; **not** a coordinator per project | Mirrors Omnigent's server(control-plane)+runner(per-workspace) split; avoids N WebSocket ports and a duplicated server; reuses SPEC-016 managed lifecycle; preserves SPEC-003's single-connection snapshot via active-project scoping |
| 2 | One **active project per connection** | Matches one-window-one-project UX; multiple windows = multiple connections; defers the cross-project board (SPEC-010) without blocking multi-project work |
| 3 | Global project registry, separate from any project's `.arke/` | Recents must span projects and survive deleting/forgetting one; metadata-only keeps it credential-free |
| 4 | Per-project isolated read model / trace / stores | Audit and failure isolation; a fault or scaffold in one project cannot corrupt another's state |
| 5 | Lazy spawn on open + idle eviction + concurrency bound | Don't run every known project's harness; bound resource use; never silently drop a harness |
| 6 | `forget` removes the registry entry but never deletes files | The tool manages references to projects, not the user's source |
| 7 | `projectId = hash(canonical root)` | Same folder ⇒ same context deterministically; safe to use as a key and in events |

## Open questions
- Should the cockpit ever present a **cross-project** board/queue (resolving SPEC-010's open question), or is per-project scope the long-term model?
- Exact default location and migration for the global registry per OS (`ARKE_HOME` precedence, first-run creation).
- Should per-project harness concurrency have its own cap in addition to the global active-context cap?
- **Remote / sandboxed runners** (Omnigent-style e2b/daytona/modal, or local `bwrap`/`seatbelt`): in scope as a later runner-isolation spec? This is the natural home for the deferred `@arke/adapter-omnigent` spike.
- Multi-window: is the active project tracked purely per WebSocket connection, or should the coordinator also expose "which projects are open" to all clients?
