# Coordinator domain model & resilient transport

## Why

The board and cockpit need a live, ordered, resilient stream from the coordinator — but today the client renders simulated data, the domain model has no streaming message or transcript events, and the client transport has no reconnection logic. This makes the normalised event model complete and the client transport production-grade.

## What changes

- ADDED domain-events — the canonical `DomainEvent` union in `@arke/contracts/events.ts` gains `message.part` (streaming transcript delta), `message.updated` (full turn state), and `turn.quiescent` (typed idle receipt); the coordinator normalises these at the boundary before any client sees them.   (breaking: no)
- ADDED domain-events — the `EventEnvelope` gains an optional `correlationId` field (the OpenCode `messageID`) so every event can be attributed to the request that triggered it; SPEC-002's correlation requirement is satisfied here.   (breaking: no)
- ADDED client-transport — a five-state reconnecting WebSocket transport (`connecting → open → reconnecting → closed → disposed`) in `packages/client` that queues outbound requests while disconnected and drains them in order on reconnect; replaces the current placeholder.   (breaking: no)
- MODIFIED coordinator — `packages/coordinator/src/server.ts` sends a `snapshot` frame on connect and on every reconnect; per-connection `seq` is scoped to the connection object, not a shared counter; snapshot is traced before push; client messages are traced.   (breaking: no)
- MODIFIED domain-events — the `ReadModel` in `packages/coordinator/src/read-model.ts` is extended to fold the new transcript events into per-session transcript state, updating `CardState` without changing the column-derivation logic.   (breaking: no)
- ADDED coordinator — a durable session ownership store (NDJSON or embedded KV, co-located with the trace) persists `sessionId → { kind, parentSessionId, specId }` across coordinator restarts; SPEC-002 depends on this.   (breaking: no)

## Requirements

### Requirement: Extended canonical DomainEvent model with correlation id
`capability: domain-events` · `delta: ADDED (feat/coordinator-domain-model-and-transport)`

The system SHALL extend the `DomainEvent` discriminated union in `@arke/contracts/events.ts` to include streaming message and transcript events and a typed runtime-signal receipt. The `EventEnvelope` SHALL gain an optional `correlationId: string` field carrying the caller-supplied correlation id (the OpenCode `messageID`) so the coordinator can attribute events to the request that triggered them; SPEC-002's turn-correlation requirement is satisfied here and nowhere else. The coordinator SHALL stamp `seq`, `ts`, `harness`, and (where provided by the adapter) `correlationId` on every event before persisting or pushing.

The new event shapes are:

- `message.part` — a streaming delta from an in-progress assistant turn; carries `sessionId`, `messageId`, `partIndex`, `delta` (string), `role` ("assistant" | "tool"), and `done` (boolean, `true` signals the part is complete).
- `message.updated` — a full turn state snapshot once a message is complete; carries `sessionId`, `messageId`, `role`, `text` (full accumulated content), `toolCalls` (array of `{ id, name, result? }`), and `isStreaming` (boolean).
- `turn.quiescent` — a typed runtime-signal receipt emitted by the coordinator when a session's `message.updated` with `isStreaming: false` arrives after one or more parts; carries `sessionId` and `turnId`. Consumers MUST use this signal to detect turn completion — they SHALL NOT poll or infer idleness from timeouts alone.

#### Scenario: Streaming part arrives
- **WHEN** the adapter emits a `message.part` event for an in-progress turn
- **THEN** the coordinator stamps it (`seq`, `ts`, `harness`, `correlationId` if present), folds it into the read model, writes it to the trace, and pushes it to all connected clients
- **AND** the `done` field is `false`

#### Scenario: Turn completes and quiescent receipt is emitted
- **WHEN** the adapter emits `message.updated` with `isStreaming: false` for a session that had at least one `message.part` in this turn
- **THEN** the coordinator emits a `turn.quiescent` event carrying the same `sessionId` and the `turnId`
- **AND** `turn.quiescent` is persisted to the trace and pushed to all connected clients before any subsequent event for that session

#### Scenario: Correlation id flows through to clients
- **WHEN** a `message.part` event arrives from the adapter carrying `correlationId: "m1"`
- **THEN** the stamped event pushed to clients carries `correlationId: "m1"`
- **AND** the trace entry for that event contains the same `correlationId`

#### Scenario: Unknown event shape at the boundary
- **WHEN** the adapter yields an object whose `type` is not in the `DomainEvent` union
- **THEN** the coordinator rejects it: the object is NOT pushed to clients and NOT folded into the read model
- **AND** it is written to the trace as a dead-letter record with `kind: "dead-letter"`, `rawType`, `reason`, and `seq` (a monotonic dead-letter sequence for the connection), so the gap is actionable rather than invisible — matching the dead-letter contract in SPEC-002

### Requirement: Durable session ownership store
`capability: coordinator` · `delta: ADDED (feat/coordinator-domain-model-and-transport)`

The coordinator SHALL maintain a durable, append-only session ownership store persisted to `ARKE_SESSION_STORE_PATH` (defaulting to `.arke/sessions.ndjson`). This store maps `sessionId → { kind, parentSessionId?, specId, harness, createdAt }` and is the single source of truth for ownership across coordinator restarts. SPEC-002's requirement that the adapter persist the session graph is satisfied by writing through this store.

The store SHALL be read on coordinator startup to restore the ownership map in memory before any events are processed. Writing to the store SHALL be synchronous with session creation — the record is written before `createSession` returns to the caller.

#### Scenario: Coordinator restart restores ownership
- **WHEN** the coordinator process restarts after one or more sessions have been created
- **THEN** on startup it reads `ARKE_SESSION_STORE_PATH` and restores the full `sessionId → ownership` map in memory
- **AND** the first events arriving from a reconnecting adapter are enriched with correct `specId`, `kind`, and `parentSessionId` — no event is emitted with unknown or guessed ownership

#### Scenario: New session record is durable on creation
- **WHEN** the adapter calls `createSession` and the harness returns a `sessionId`
- **THEN** the ownership record is written to the store before the `sessionId` is returned to the caller
- **AND** a subsequent crash before any events arrive does not lose the ownership mapping

#### Scenario: Unknown session triggers a targeted REST resolve, then a store write
- **WHEN** the adapter emits an event for a `sessionId` not present in the in-memory ownership map
- **THEN** the coordinator requests the adapter to resolve that session via REST (`resolveSession`)
- **AND** the resolved ownership is written to the durable store before the event is folded into the read model

### Requirement: Ordered, sequenced, snapshot-on-connect stream
`capability: coordinator` · `delta: MODIFIED (feat/coordinator-domain-model-and-transport)`

The coordinator SHALL push events to each connected client in arrival order with a monotonically increasing sequence number (`seq`) that is **scoped to the connection object** — not a shared counter on the `Coordinator` class instance. On every new WebSocket connection — including reconnects — it SHALL immediately send a `snapshot` frame containing the current `CardState[]` from the read model as the very first message, before any buffered or live events. The snapshot frame SHALL be written to the trace before it is sent. No event from before the snapshot can arrive at the client before the snapshot itself.

#### Scenario: Fresh connection receives snapshot as first message
- **WHEN** a client opens a new WebSocket connection to the coordinator
- **THEN** the coordinator writes a `{ kind: "snapshot", cardCount }` trace entry
- **AND** sends `{ type: "snapshot", cards: CardState[] }` as the very first WebSocket frame
- **AND** all subsequent events carry `seq` values starting from 1 for that connection

#### Scenario: Per-connection seq is independent
- **WHEN** two clients are connected concurrently
- **THEN** each receives its own independent `seq` starting at 1
- **AND** a new connection from the same client after a disconnect restarts `seq` at 1 without resetting the sequence of any other connection

#### Scenario: Reconnecting client converges without replaying the event log
- **WHEN** a client drops and reconnects after missing some events
- **THEN** the coordinator sends a fresh `snapshot` frame as the first message on reconnect
- **AND** the client's read model converges to the current state from the snapshot without needing to replay missed events

#### Scenario: Shared seq counter bug is absent
- **WHEN** two events arrive from the adapter in rapid succession while two clients are connected
- **THEN** both clients receive the same two events with seq values `n` and `n+1` (relative to each client's own counter)
- **AND** no client observes a gap caused by the other client consuming seq increments — the current skeleton's shared `this.seq` is replaced with per-connection counters

### Requirement: Client reconnecting transport state machine
`capability: client-transport` · `delta: ADDED (feat/coordinator-domain-model-and-transport)`

The client transport SHALL implement a five-state machine (`connecting → open → reconnecting → closed → disposed`) over the coordinator WebSocket, queue outbound requests (steering messages, permission decisions) while not in the `open` state, and drain the queue in order on every transition to `open`. The transport SHALL use capped exponential back-off between reconnect attempts and SHALL expose an observable state so the board and session-detail views can render a connection indicator. Messages queued during `reconnecting` SHALL be delivered at most once — a message acknowledged by the server SHALL be removed from the queue.

#### Scenario: Outbound request while disconnected is queued
- **WHEN** the client sends a steering message while the transport is in `reconnecting`
- **THEN** the message is added to the outbound queue
- **AND** it is NOT sent until the state transitions to `open`

#### Scenario: Queue drains on reconnect, in order, once
- **WHEN** the transport transitions from `reconnecting` to `open`
- **THEN** every queued message is sent in the order it was enqueued
- **AND** the queue is empty after the drain completes
- **AND** no message is sent a second time due to the reconnect

#### Scenario: Back-off caps correctly
- **WHEN** reconnect attempts are made
- **THEN** each attempt waits for `min(attempt * base, cap)` milliseconds before retrying
- **AND** the delay never exceeds the configured cap

#### Scenario: Disposed transport does not reconnect
- **WHEN** the transport enters the `disposed` state
- **THEN** no further reconnect attempts are made
- **AND** queued messages are discarded

#### Scenario: Snapshot triggers a re-render, not a queue drain
- **WHEN** the transport transitions to `open` and receives a `snapshot` frame
- **THEN** the board's `useBoard()` hook re-renders from the snapshot
- **AND** the outbound queue drain begins after the snapshot is processed, not before

### Requirement: Read model folds transcript events
`capability: coordinator` · `delta: MODIFIED (feat/coordinator-domain-model-and-transport)`

The `ReadModel` in `packages/coordinator/src/read-model.ts` SHALL fold `message.part` and `message.updated` events into per-session transcript state: an ordered list of `TranscriptEntry` records, each with its accumulated text and tool calls and an `isStreaming` flag. Deltas SHALL be appended in `partIndex` order, not arrival order — out-of-order parts SHALL be buffered and inserted at the correct index. Column derivation SHALL remain purely a function of spec status, session status, PR/CI state, and `needsHuman` — transcript events SHALL NOT move a card between columns.

#### Scenario: Part deltas accumulate in partIndex order
- **WHEN** the read model receives `message.part` events for `partIndex` 0, 2, 1 (out of order)
- **THEN** the transcript entry for that message concatenates deltas in index order: 0, 1, 2
- **AND** `isStreaming` is `true` until a part with `done: true` arrives

#### Scenario: message.updated closes the streaming window
- **WHEN** the read model receives a `message.updated` event for a message
- **THEN** the transcript entry's `text` is replaced with the authoritative full text from the event
- **AND** `isStreaming` is set to `false`
- **AND** any out-of-order-buffered parts for that `messageId` are discarded

#### Scenario: Column is unaffected by transcript events
- **WHEN** the read model processes a `message.part` or `message.updated` event
- **THEN** the card's `column` value is unchanged
- **AND** `deriveColumn()` is not called as a result of these events

### Requirement: Every event and snapshot persisted to the append-only trace
`capability: coordinator` · `delta: MODIFIED (feat/coordinator-domain-model-and-transport)`

The coordinator SHALL write every normalised, validated `DomainEvent` — including `message.part`, `message.updated`, and `turn.quiescent` — to the append-only NDJSON trace at `ARKE_TRACE_PATH` **before** pushing to any client. Dead-letter records for unmappable events SHALL also be written to the same trace. Snapshot frames SHALL be traced as `{ kind: "snapshot", cardCount }`. Client requests received over WebSocket SHALL be traced as `{ kind: "client.request", request }`. No event that reaches the read model SHALL be absent from the trace.

Whether `message.part` events are persisted verbatim (full fidelity) or whether only `message.updated` (the snapshot) is persisted is left as a configurable option (see Open questions), but the behaviour chosen MUST be consistent — partial trace coverage of parts is not permitted.

#### Scenario: Event appears in trace before client push
- **WHEN** the coordinator processes any domain event
- **THEN** `trace.write` is called and resolves before the event is pushed to any WebSocket client
- **AND** the trace entry contains `at` (epoch ms), `seq` (per-connection, from the event), `ts`, `harness`, `type`, and `correlationId` (if present)

#### Scenario: Dead-letter record is in the trace
- **WHEN** the adapter yields an object the coordinator cannot map to a `DomainEvent`
- **THEN** the trace records `{ kind: "dead-letter", rawType, reason, at }` before any further events are processed

#### Scenario: Snapshot frame is traced before send
- **WHEN** a client connects and the coordinator prepares a snapshot frame
- **THEN** `trace.write({ kind: "snapshot", cardCount })` is called before `ws.send()`

---

### Requirement: Backpressure and slow-client handling
`capability: coordinator` · `delta: ADDED (feat/coordinator-domain-model-and-transport)`

The coordinator SHALL not allow a slow or unresponsive WebSocket client to block the pump loop or starve other clients. Each client connection SHALL have a bounded outbound buffer; when the buffer is full, the coordinator SHALL drop the oldest events to that client (with a trace record: `{ kind: "client.drop", clientId, dropped }`) rather than blocking on `ws.send()`. A client that has missed too many events due to a full buffer SHALL receive a fresh snapshot on its next message, allowing it to re-converge.

#### Scenario: Slow client does not block the pump
- **WHEN** a client's WebSocket buffer is full and a new event arrives
- **THEN** the coordinator enqueues or pushes the event to all other connected clients without waiting for the slow client
- **AND** the slow client's oldest buffered event is dropped and recorded in the trace

#### Scenario: Schema validation failure at the boundary does not crash the pump
- **WHEN** `DomainEvent.parse()` throws a validation error on an event from the adapter
- **THEN** the coordinator catches the error, writes a dead-letter trace record, and continues pumping the next event
- **AND** the coordinator process does not crash or stop processing

## Design

### Architectural decision

Extend the existing contracts and coordinator in place — no new packages — so the domain model gains the transcript events the board and cockpit need without a second state channel. The session ownership store is a co-located NDJSON file (same directory as the trace, same format) — not a new database. The client transport is the one architectural novelty: a thin state-machine class in `packages/client` isolating all reconnect logic from the board and session-detail components. The shared `this.seq` in the coordinator skeleton is corrected to a per-connection counter.

### Target architecture

```
adapter (OpenCode / Mock)
  └─▶ streamEvents() → DomainEvent (zod-validated)
        └─▶ Coordinator.pump()
              ├─▶ ReadModel.apply(event)      — folds all event types including transcript
              ├─▶ Trace.write(event)          — NDJSON, before push
              └─▶ push to clients (seq stamped per connection, not shared)
                        │
                        ▼
              ArkeTransport (packages/client)   ← five-state machine
                        │
              board · session-detail · cockpit

Session ownership store (.arke/sessions.ndjson)
  └─▶ read on startup → in-memory Map<sessionId, OwnershipRecord>
  └─▶ written on createSession + REST resolve
  └─▶ queried by coordinator on event enrichment
```

The `@arke/adapter-opencode` (SPEC-002) normalises OpenCode's `message.part.updated` and `message.updated` provider events into the new `DomainEvent` variants; the coordinator is unaware of OpenCode specifics. The `correlationId` is set by the adapter when it emits events for a turn it initiated with a known `messageID`.

### Data model

**New Zod schemas added to `@arke/contracts/src/events.ts`:**

```typescript
// EventEnvelope gains an optional correlationId
export const EventEnvelope = z.object({
  seq: z.number().int().nonnegative(),
  ts: z.number().int(),
  harness: z.string(),
  correlationId: z.string().optional(), // caller-supplied per SPEC-002; the OpenCode messageID
});

export const MessagePartEvent = base.extend({
  type: z.literal("message.part"),
  sessionId: z.string(),
  messageId: z.string(),
  partIndex: z.number().int().nonneg(),
  delta: z.string(),
  role: z.enum(["assistant", "tool"]),
  done: z.boolean(),
});

export const MessageUpdatedEvent = base.extend({
  type: z.literal("message.updated"),
  sessionId: z.string(),
  messageId: z.string(),
  role: z.enum(["user", "assistant", "tool"]),
  text: z.string(),
  toolCalls: z.array(z.object({
    id: z.string(),
    name: z.string(),
    result: z.string().optional(),
  })).default([]),
  isStreaming: z.boolean(),
});

export const TurnQuiescentEvent = base.extend({
  type: z.literal("turn.quiescent"),
  sessionId: z.string(),
  turnId: z.string(),
});
```

These are added to the `DomainEvent` discriminated union.

**`CardState` transcript extension (read-model.ts):**

```typescript
export interface TranscriptEntry {
  messageId: string;
  role: "user" | "assistant" | "tool";
  text: string;
  toolCalls: { id: string; name: string; result?: string }[];
  isStreaming: boolean;
}
// Added to CardState:
transcript: TranscriptEntry[];
```

**Session ownership store (`OwnershipRecord`, `.arke/sessions.ndjson`):**

```typescript
export interface OwnershipRecord {
  sessionId: string;
  kind: "spec" | "task";
  parentSessionId?: string;
  specId: string;
  harness: string;
  createdAt: number; // epoch ms
}
```

**Client transport state (`packages/client/src/transport.ts`):**

```typescript
type TransportState = "connecting" | "open" | "reconnecting" | "closed" | "disposed";

class ArkeTransport {
  readonly state$: Observable<TransportState>;
  send(msg: unknown): void;   // queues if not open; drains in order on open; at-most-once
  dispose(): void;
}
```

### Interfaces and contracts

- `DomainEvent` in `@arke/contracts/src/events.ts` — union extended with `MessagePartEvent | MessageUpdatedEvent | TurnQuiescentEvent`; `EventEnvelope` gains optional `correlationId`.
- `CardState` in `packages/coordinator/src/read-model.ts` — gains `transcript: TranscriptEntry[]`.
- `ArkeTransport` in `packages/client/src/transport.ts` (new file) — used by board and session-detail hooks.
- Wire frames remain `{ type: "snapshot", cards: CardState[] }` and `{ type: "event", event: DomainEvent }` — no wire-format breaking change.
- Session ownership store: `packages/coordinator/src/session-store.ts` (new file) exposes `SessionStore.write(record)` and `SessionStore.loadAll(): OwnershipRecord[]`, used by the coordinator and SPEC-002's adapter.

### Cross-cutting

- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`). Do not re-derive styling.

**Security (NFR-1):** the transport connects only to `ws://127.0.0.1:<ARKE_COORDINATOR_PORT>`; no cloud relay. Credentials remain on the coordinator host and are never forwarded to the client. Inbound client messages are parsed and validated before routing to the adapter; malformed JSON is discarded and traced.

**Performance (NFR-3):** transcript deltas are appended in place on the `TranscriptEntry`; no full snapshot per delta. Snapshot frames are the read-model snapshot, not a full event replay, keeping initial load O(cards). `message.part` persistence to the trace is configurable (see Open questions) to control volume. Per-connection seq uses a `let seq = 0` closure, not a class field, eliminating the shared-counter race.

**Observability (NFR-7):** every event (including `message.part`) and every dead-letter record is written to the NDJSON trace before pushing; `turn.quiescent` gives tests and operators a deterministic idle signal without polling; the session ownership store is also NDJSON and auditable.

**Resilience (NFR-8):** the five-state transport ensures no outbound message is lost during transient disconnects; snapshot-on-connect means the client always converges without event-log replay; the durable ownership store means a coordinator restart does not lose session-to-spec mappings; backpressure dropping prevents a slow client from stalling the pump; `parse()` errors are caught and dead-lettered so the pump never crashes.

### Testing

- **Unit (contracts):** every new Zod schema parses valid input; rejects missing/extra fields; `EventEnvelope` with and without `correlationId`; the discriminated union dispatches correctly.
- **Unit (session store):** `write()` appends a line; `loadAll()` restores records on fresh read; startup after a crash-before-first-event returns an empty map (not an error).
- **Unit (read model):** `apply` accumulates deltas in `partIndex` order including out-of-order arrival; `message.updated` closes streaming and discards buffered parts; column is unaffected by transcript events; `turn.quiescent` is emitted after `isStreaming: false`.
- **Unit (coordinator pump):** `DomainEvent.parse()` error → dead-letter trace entry → pump continues; slow client buffer full → event dropped → drop trace entry → other clients receive event; per-connection seq resets to 1 on reconnect without affecting the counter for other connections.
- **Unit (transport):** state transitions under normal connect/reconnect/dispose; outbound queue accumulates while not `open`; drains in order on `open`; each message sent at most once; back-off cap respected; snapshot frame does not trigger a double drain.
- **Integration:** coordinator end-to-end with `MockAdapter` emitting transcript events (including out-of-order parts) → trace contains all events including dead-letter → connected client receives snapshot then ordered stream → kill/restart coordinator → restart reads ownership store → subsequent events carry correct `specId`.
- **Manual:** bring up the coordinator, connect the client, verify the connection-state indicator, force-close the WebSocket, confirm automatic reconnect and queue drain, verify no duplicate messages.

### Definition of done

All scenarios pass; TypeScript strict-mode compiles with no errors; the trace NDJSON contains every new event type after an end-to-end run; the shared `this.seq` bug is absent (verified by concurrent-client test); the session ownership store survives a coordinator restart; a reviewer has signed off; SPEC-002's open questions on message normalisation and the durable session graph are closed by this spec.

## Decision log

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Extend the `DomainEvent` union in place rather than adding a parallel transcript model | Keeps the coordinator's pump, trace, and push loop uniform; no consumer needs a second channel |
| 2 | `turn.quiescent` as an explicit typed receipt, not a timeout | Mirrors T3 runtime-signal pattern; deterministic idle signal without polling (NFR-8) |
| 3 | Five-state transport with queued-outbound-drain in `packages/client`, not in the coordinator | The coordinator is stateless per connection; client-side queuing is the correct layer for at-most-once delivery toward the browser |
| 4 | Snapshot on connect rather than full event-log replay | OpenCode has no `Last-Event-ID`; snapshot is the only correct recovery, and keeps the coordinator free of a persistent event store in Phase 1 |
| 5 | `seq` is per-connection (closure-scoped), not a shared class field | A shared counter races when two connections receive events concurrently; per-connection seq is sufficient for ordered delivery and imposes no persistence requirement |
| 6 | `correlationId` added to `EventEnvelope` in this spec, not in SPEC-002 | The envelope is owned by this spec; SPEC-002 depends on it being here; placing it in the adapter would couple the adapter to an envelope field it does not own |
| 7 | Session ownership store is NDJSON co-located with the trace | Keeps the coordinator free of a database dependency; the NDJSON format is auditable and consistent with the trace; satisfies SPEC-002's durability requirement |
| 8 | Dead-letter records go to the same trace file | A separate dead-letter file adds operational complexity with no benefit at this scale; the `kind` field distinguishes them |
| 9 | `message.part` persistence is configurable (verbatim vs updated-only) | Full fidelity is valuable for replay; compactness matters at high throughput; neither choice is universally correct — leave as a config option |
| 10 | Per-client outbound buffer with drop-oldest on full | Blocking `ws.send()` would stall the pump and starve all other clients; a bounded buffer with trace-recorded drops is the safe default |

## Open questions

- Should `message.part` events be persisted to the trace verbatim (potentially high-volume) or should the trace record only `message.updated`? The configurable-option decision defers the policy; the default (verbatim) should be confirmed with the team before the first release.
- The v1.14.42+ OpenCode regression drops `message.*` events over SSE for some versions (issue #27966). If the regression is unresolved on the pinned version, transcript events will be absent from the live adapter; the capability gate should degrade gracefully. Confirm the pinned version before closing SPEC-002.
- What is the bounded outbound buffer size per client? (Candidate: 256 events; drop oldest; log.) Needs confirmation before implementation.
- Should the session ownership store be keyed by `(harness, sessionId)` to handle multiple harnesses with overlapping session id spaces, or is `sessionId` globally unique across harnesses by convention?
