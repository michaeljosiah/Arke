# Coordinator domain model & resilient transport

## Why

The board and cockpit need a live, ordered, resilient stream from the coordinator — but today the client renders simulated data, the domain model has no streaming message or transcript events, and the client transport has no reconnection logic. This makes the normalized event model complete and the client transport production-grade.

## What changes

- ADDED domain-events — the canonical `DomainEvent` union in `@arke/contracts/events.ts` gains `message.part` (streaming transcript delta) and `message.updated` (full turn state) events, plus typed runtime-signal receipts (`turn.quiescent`, `diff.finalized` is already present but gains a richer shape); the coordinator normalises these at the boundary before any client sees them.   (breaking: no)
- ADDED client-transport — a five-state reconnecting WebSocket transport (`connecting → open → reconnecting → closed → disposed`) in `packages/client` that queues outbound requests while disconnected and drains them on reconnect; replaces the current placeholder.   (breaking: no)
- MODIFIED coordinator — `packages/coordinator/src/server.ts` sends a `snapshot` frame on connect and replays the current read-model state on every resubscribe; all incoming events are stamped with a monotonic per-connection sequence before pushing.   (breaking: no)
- MODIFIED domain-events — the `ReadModel` in `packages/coordinator/src/read-model.ts` is extended to fold the new transcript events into per-session transcript state, updating `CardState` without changing the column-derivation logic.   (breaking: no)

## Requirements

### Requirement: Extended canonical DomainEvent model
`capability: domain-events` · `delta: ADDED (feat/coordinator-domain-model-and-transport)`

The system SHALL extend the `DomainEvent` discriminated union in `@arke/contracts/events.ts` to include streaming message and transcript events and typed runtime-signal receipts, so that every part of a live agent turn is represented in the normalized model and no consumer needs to inspect raw provider output.

The new event shapes are:

- `message.part` — a streaming delta from an in-progress assistant turn; carries `sessionId`, `messageId`, `partIndex`, `delta` (string), `role` ("assistant" | "tool"), and `done` (boolean, true signals the part is complete).
- `message.updated` — a full turn state snapshot once a message is complete; carries `sessionId`, `messageId`, `role`, `text` (full accumulated content), `toolCalls` (array of `{ id, name, result? }`), and `isStreaming` (boolean).
- `turn.quiescent` — a typed runtime-signal receipt emitted by the coordinator when a session's turn queue drains and no more deltas are expected; carries `sessionId` and `turnId`. Consumers MUST use this signal rather than polling to detect a completed turn (NFR-8).

The coordinator SHALL stamp `seq`, `ts`, and `harness` on every event before persisting or pushing, regardless of event type.

#### Scenario: Streaming part arrives
- **WHEN** the adapter emits a `message.part` event for an in-progress turn
- **THEN** the coordinator stamps it, folds it into the read model, persists it to the trace, and pushes it to all connected clients
- **AND** the `done` field is `false`

#### Scenario: Turn completes
- **WHEN** the adapter emits the final `message.part` with `done: true` followed by `message.updated`
- **THEN** the coordinator emits a `turn.quiescent` event for the session
- **AND** consumers observing the stream see the quiescent signal and stop waiting

#### Scenario: Unknown event shape is rejected at the boundary
- **WHEN** the adapter yields an object whose `type` is not in the `DomainEvent` union
- **THEN** the coordinator drops the event and does not push it to clients
- **AND** it writes a `warn` entry to the trace recording the unknown type

### Requirement: Ordered, sequenced, snapshot-on-connect stream
`capability: coordinator` · `delta: MODIFIED (feat/coordinator-domain-model-and-transport)`

The coordinator SHALL push events to each connected client in arrival order with a monotonically increasing sequence number (`seq`) that resets per connection (not per session). On every new WebSocket connection — including reconnects — it SHALL immediately send a `snapshot` frame containing the current `CardState[]` from the read model, so the client can render without waiting for a full replay of the event log.

#### Scenario: Fresh connection receives snapshot
- **WHEN** a client opens a new WebSocket connection to the coordinator
- **THEN** the coordinator sends a `{ type: "snapshot", cards: CardState[] }` frame as the first message
- **AND** subsequent events carry `seq` values starting from 1 for that connection

#### Scenario: Sequence is strictly increasing per connection
- **WHEN** the coordinator pushes three events to a given client
- **THEN** their `seq` values are 1, 2, 3 in that order
- **AND** a second concurrent client receives its own independent sequence starting at 1

#### Scenario: Reconnecting client receives current state
- **WHEN** a client drops and reconnects after missing events
- **THEN** the coordinator sends a fresh `snapshot` frame before any further events
- **AND** the client's read model converges to the current state without relying on replaying missed events

### Requirement: Client reconnecting transport state machine
`capability: client-transport` · `delta: ADDED (feat/coordinator-domain-model-and-transport)`

The client transport SHALL implement a five-state machine (`connecting → open → reconnecting → closed → disposed`) over the coordinator WebSocket, queue outbound requests (steering messages, permission decisions) while not in the `open` state, and drain the queue in order on every transition to `open`. The transport SHALL use capped exponential back-off between reconnect attempts, and SHALL expose an observable state so the board and session-detail views can render a connection indicator.

This pattern mirrors the T3 Code queued-outbound-drain transport (`docs/analysis/t3code-learnings.md`, §Transport).

#### Scenario: Outbound request while disconnected is queued
- **WHEN** the client sends a steering message while the transport is in `reconnecting`
- **THEN** the message is added to the outbound queue
- **AND** it is NOT sent until the state transitions to `open`

#### Scenario: Queue drains on reconnect
- **WHEN** the transport transitions from `reconnecting` to `open`
- **THEN** every queued message is sent in the order it was enqueued
- **AND** the queue is empty after the drain completes

#### Scenario: Back-off caps correctly
- **WHEN** reconnect attempts are made
- **THEN** each attempt waits for a delay that is at least `min(attempt * base, cap)` milliseconds before retrying
- **AND** the delay never exceeds the configured cap

#### Scenario: Disposed transport does not reconnect
- **WHEN** the transport enters the `disposed` state
- **THEN** no further reconnect attempts are made
- **AND** queued messages are discarded

### Requirement: Read model folds transcript events
`capability: coordinator` · `delta: MODIFIED (feat/coordinator-domain-model-and-transport)`

The `ReadModel` in `packages/coordinator/src/read-model.ts` SHALL fold `message.part` and `message.updated` events into per-session transcript state: an ordered list of messages, each with its accumulated text and tool calls, and an `isStreaming` flag. Column derivation SHALL remain purely a function of spec status, session status, PR/CI state, and `needsHuman` — transcript events SHALL NOT move a card between columns.

#### Scenario: Part deltas accumulate in order
- **WHEN** the read model receives three consecutive `message.part` events for the same `messageId`
- **THEN** the transcript entry for that message holds the concatenation of all three deltas
- **AND** `isStreaming` is `true` until a part with `done: true` arrives

#### Scenario: message.updated closes the streaming window
- **WHEN** the read model receives a `message.updated` event for a message
- **THEN** the transcript entry's `text` is replaced with the authoritative full text from the event
- **AND** `isStreaming` is set to `false`

---

### Requirement: Every event persisted to the append-only trace
`capability: coordinator` · `delta: MODIFIED (feat/coordinator-domain-model-and-transport)`

The coordinator SHALL write every normalized, validated `DomainEvent` — including the new `message.part`, `message.updated`, and `turn.quiescent` events — to the append-only NDJSON trace at `ARKE_TRACE_PATH` before pushing to any client. The trace MUST also record `snapshot` frames and client requests for a complete audit record. No event that has reached the read model SHALL be absent from the trace.

This underpins NFR-7 (audit) and NFR-2 (observability). The `Trace` class in `packages/coordinator/src/trace.ts` is the write path; it already exists and appends NDJSON lines.

#### Scenario: Event appears in trace before client push
- **WHEN** the coordinator processes a domain event
- **THEN** `trace.write` is called before the event is pushed to any WebSocket client
- **AND** the trace entry contains `at` (epoch ms), `seq`, `ts`, `harness`, and `type`

#### Scenario: snapshot frame is traced
- **WHEN** a client connects and receives a snapshot frame
- **THEN** the trace records a `{ kind: "snapshot", cardCount }` entry at that moment

## Design

### Architectural decision

Extend the existing contracts and coordinator in place — no new packages, no new persistence layer — so the domain model gains the transcript events that the board and cockpit surfaces need without introducing a second source of state. The client transport is the one architectural novelty: it is a thin state-machine class in `packages/client` that wraps the native `WebSocket`, isolating all reconnect logic from the board and session-detail components.

### Target architecture

```
adapter (OpenCode / Mock)
  └─▶ streamEvents() → DomainEvent (zod-validated)
        └─▶ Coordinator.pump()
              ├─▶ ReadModel.apply(event)      — folds all event types including transcript
              ├─▶ Trace.write(event)          — NDJSON, before push
              └─▶ push to clients (seq stamped per connection)
                        │
                        ▼
              ArkeTransport (packages/client)   ← five-state machine
                        │
              board · session-detail · cockpit
```

The `@arke/adapter-opencode` (SPEC-002) normalises OpenCode's `message.part.updated` and `message.updated` provider events into the new `DomainEvent` variants; the coordinator is unaware of OpenCode specifics.

### Data model

**New Zod schemas added to `@arke/contracts/src/events.ts`:**

```typescript
export const MessagePartEvent = base.extend({
  type: z.literal("message.part"),
  sessionId: z.string(),
  messageId: z.string(),
  partIndex: z.number().int().nonnegative(),
  delta: z.string(),
  role: z.enum(["assistant", "tool"]),
  done: z.boolean(),
});

export const MessageUpdatedEvent = base.extend({
  type: z.literal("message.updated"),
  sessionId: z.string(),
  messageId: z.string(),
  role: z.enum(["user", "assistant", "tool"]),
  text: z.string(),
  toolCalls: z.array(z.object({
    id: z.string(),
    name: z.string(),
    result: z.string().optional(),
  })).default([]),
  isStreaming: z.boolean(),
});

export const TurnQuiescentEvent = base.extend({
  type: z.literal("turn.quiescent"),
  sessionId: z.string(),
  turnId: z.string(),
});
```

These are added to the `DomainEvent` discriminated union alongside the existing types.

**`CardState` transcript extension (read-model.ts):**

```typescript
export interface TranscriptEntry {
  messageId: string;
  role: "user" | "assistant" | "tool";
  text: string;
  toolCalls: { id: string; name: string; result?: string }[];
  isStreaming: boolean;
}
// Added to CardState:
transcript: TranscriptEntry[];
```

**Client transport state (packages/client):**

```typescript
type TransportState = "connecting" | "open" | "reconnecting" | "closed" | "disposed";

class ArkeTransport {
  readonly state$: Observable<TransportState>;
  send(msg: unknown): void;   // queues if not open; drains on open
  dispose(): void;
}
```

### Interfaces and contracts

- `DomainEvent` in `@arke/contracts/src/events.ts` — union extended with `MessagePartEvent | MessageUpdatedEvent | TurnQuiescentEvent`.
- `CardState` in `packages/coordinator/src/read-model.ts` — gains `transcript: TranscriptEntry[]`.
- `ArkeTransport` in `packages/client/src/transport.ts` (new file) — used by board and session-detail hooks.
- Wire frames remain `{ type: "snapshot", cards: CardState[] }` and `{ type: "event", event: DomainEvent }` — no wire-format breaking change.

### Cross-cutting
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.

**Security (NFR-1):** the transport connects only to `ws://127.0.0.1:<ARKE_COORDINATOR_PORT>`; no cloud relay. Credentials remain on the coordinator host and are never forwarded to the client.

**Performance (NFR-3):** transcript deltas are appended in place on the `TranscriptEntry`; no full snapshot per delta. Snapshot frames are the read-model snapshot, not a full event replay, keeping initial load O(cards).

**Observability (NFR-7):** every event (including `message.part`) is written to the NDJSON trace before pushing; `turn.quiescent` gives tests and operators a deterministic idle signal without polling, matching the T3 pattern of typed runtime-signal receipts.

**Resilience (NFR-8):** the five-state transport ensures no outbound message is lost during transient disconnects; snapshot-on-connect means the client always converges without event-log replay.

## Decision log

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Extend the `DomainEvent` union in place rather than adding a parallel transcript model | Keeps the coordinator's pump, trace, and push loop uniform; no consumer needs to check a second channel |
| 2 | `turn.quiescent` as an explicit typed receipt, not a timeout | Mirrors T3 runtime-signal pattern; gives tests and operators a deterministic idle signal without polling (NFR-8) |
| 3 | Five-state transport with queued-outbound-drain in `packages/client`, not in the coordinator | The coordinator is stateless per connection; client-side queuing is the correct layer for delivery guarantees toward the browser |
| 4 | Snapshot on connect rather than full event-log replay | OpenCode has no `Last-Event-ID`; re-fetch (snapshot) is the only correct recovery, and it keeps the coordinator free of a persistent event store in Phase 1 |
| 5 | `seq` is per-connection, not global | A global seq would require the coordinator to persist it across restarts; per-connection seq is sufficient for ordered delivery and imposes no persistence requirement |

## Open questions

- Should `message.part` events be persisted to the trace verbatim (potentially high-volume) or should the trace record only `message.updated` (the accumulated snapshot)? Verbatim gives full replay fidelity; the snapshot-only option keeps the trace compact.
- The v1.14.42+ OpenCode regression drops `message.*` events over SSE for some versions (issue #27966). If the regression is unresolved on the pinned OpenCode version, transcript events will be absent from the live adapter; the gate should degrade gracefully rather than failing. Confirm the pinned version before closing SPEC-002.
