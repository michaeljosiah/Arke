# Authoring cockpit — chat and live preview

## Why
A Product Engineer co-authoring a specification with agents needs two things in front of them at the same time: a conversation with the agents that is shaping the document, and a live view of what those agents are actually writing into the repository. Without that split surface, the engineer must context-switch between a chat thread and a file, losing the connection between what was asked and what landed. This surface makes that connection continuous, covers FR-1 (authoring workspace), FR-14 (chat and live preview), and FR-15 (approve and persist), and is the host surface from which the review panel (007) is convened.

## What changes
- ADDED authoring-cockpit — a split surface pairing a conversation with the authoring agents beside a live preview of the working specification file, with per-turn agent role and model visible, delta-tagged requirements highlighted, and approval that commits the specification and advances its status

## Requirements

### Requirement: Split authoring surface with named agent turns
`capability: authoring-cockpit` · `delta: ADDED (feat/authoring-cockpit)`

The system SHALL present a split surface whose left panel is a conversation with the authoring agents and whose right panel is a live preview of the working specification file; each turn in the conversation SHALL display the agent role (e.g. `spec-author`, `architect`) and the concrete model that produced it.

#### Scenario: Agent role and model are visible on every turn
- **WHEN** an agent produces a turn in the conversation
- **THEN** the turn header names the agent role and the resolved model (e.g. `spec-author · claude-opus-4-5`)
- **AND** the human's own turns are labelled with their identity

#### Scenario: Chat input dispatches to the named agent
- **WHEN** the engineer submits a message selecting a named agent role
- **THEN** the coordinator routes the message to that role at its configured tier via `HarnessAdapter.sendMessage`
- **AND** the response streams into the conversation panel in real time via `message.part` events (depends on 003)

### Requirement: Live preview is a view of the working file, never a copy
`capability: authoring-cockpit` · `delta: ADDED (feat/authoring-cockpit)`

The specification preview SHALL be a read-only rendering of the working file in the repository as it exists on the feature branch; the client SHALL NOT hold a separate authoritative copy of the specification body.

#### Scenario: Preview reflects the file on disk
- **WHEN** an agent writes a change to the working specification file via the harness
- **THEN** the preview panel updates to reflect the changed file
- **AND** the client does not maintain its own writable copy of the specification body

#### Scenario: Preview renders section by section against the anatomy
- **WHEN** the preview panel displays the working file
- **THEN** it renders the specification section by section in the order defined by `SPEC_ANATOMY` (frontmatter → requirements → design → tasks)
- **AND** sections absent from the working file are shown as empty placeholders, not hidden

### Requirement: Delta-tagged requirements are highlighted in the preview
`capability: authoring-cockpit` · `delta: ADDED (feat/authoring-cockpit)`

The preview SHALL highlight every requirement that carries a `delta:` tag, visually distinguishing it from settled, untagged requirements, so the preview reads as a contract-level diff whilst the specification is in flight on its branch.

#### Scenario: In-flight ADDED requirement is highlighted
- **WHEN** a requirement in the working file carries `delta: ADDED (feat/authoring-cockpit)`
- **THEN** the preview renders it with a distinct visual treatment (e.g. a green left border and `ADDED` badge)
- **AND** a requirement with no delta tag is rendered in the neutral, settled style

#### Scenario: MODIFIED requirement shows the full updated text
- **WHEN** a requirement in the working file carries `delta: MODIFIED`
- **THEN** the preview renders the full updated requirement text with a `MODIFIED` badge
- **AND** does not attempt to display a fragment or inline diff against a prior version

#### Scenario: REMOVED requirement shows a tombstone
- **WHEN** a requirement carries `delta: REMOVED`
- **THEN** the preview shows a struck-through tombstone line with a `REMOVED` badge
- **AND** the tombstone is not collapsed or hidden by default

### Requirement: Approval commits the specification and advances its status
`capability: authoring-cockpit` · `delta: ADDED (feat/authoring-cockpit)`

The system SHALL expose an explicit approval action; on human approval the system SHALL commit the specification file to `docs/specifications/` on its feature branch, update the `status` frontmatter field to `in-review`, and record the transition in the specification's Change history; before approval the specification SHALL remain on a feature branch and SHALL NOT be the authoritative version.

#### Scenario: Pre-approval specification is on a branch
- **WHEN** a specification has not yet been approved
- **THEN** the working file lives on the feature branch named in its `branch` frontmatter field
- **AND** the cockpit displays the branch name prominently in the status bar

#### Scenario: Approval transitions status and commits the file
- **WHEN** the engineer invokes the approval action
- **THEN** the system commits the specification file via the harness (writing to `docs/specifications/` on the feature branch)
- **AND** the frontmatter `status` advances from `draft` to `in-review`
- **AND** a Change history line is appended recording the branch, the new status, and the date

#### Scenario: Approved specification becomes reviewable
- **WHEN** the approval commit lands
- **THEN** the specification is visible in the spec library (008) with status `in-review`
- **AND** the engineer is notified the specification is ready for the review panel

### Requirement: Review panel is convened without leaving the cockpit
`capability: authoring-cockpit` · `delta: ADDED (feat/authoring-cockpit)`

The system SHALL expose, within the cockpit, an action to convene the multi-model review panel (007) on the current working draft; the engineer SHALL NOT need to navigate away from the cockpit to initiate a review.

#### Scenario: Convene review action is available at any point
- **WHEN** the working draft contains at least one requirement
- **THEN** a "Convene review" action is visible in the cockpit's toolbar
- **AND** invoking it opens the review panel (007) in-context, passing the current working-file state

#### Scenario: Review can be convened before approval
- **WHEN** the specification status is `draft`
- **THEN** the engineer can still convene a review panel
- **AND** the review operates on the current working file, not on an approved version

## Design

### Architectural decision
The cockpit is a pure projection surface: it reads the working file from the repository (via the adapter) and renders it; it never writes the specification body directly. All writes to `docs/specifications/` flow through the harness adapter, keeping the file in git as the single source of truth. The approval action is a coordinator command, not a client-side file write.

### Target architecture
The cockpit (`packages/client/src/screens/cockpit.tsx`) is the existing baseline screen. It is wired to the coordinator WebSocket (via the reconnecting transport from 003) for live event ingestion and to the coordinator's command channel for `sendMessage` and `approve` actions.

```
Engineer input
    │
    ▼
cockpit.tsx (React 19, @arke/client)
    ├── left panel: conversation thread
    │       ├── turn bubbles (role + model label from message.updated / message.part events)
    │       ├── composer → coordinator command: sendMessage { agent, tier, parts }
    │       └── "Convene review" button → navigate to review panel (007)
    └── right panel: spec preview
            ├── reads working file path from SpecFrontmatter.branch
            ├── renders section-by-section against SPEC_ANATOMY
            ├── delta-tag renderer (ADDED / MODIFIED / REMOVED visual treatments)
            └── status bar (specId, status, branch, model-in-use)

coordinator (packages/coordinator/server.ts)
    ├── receives sendMessage command → HarnessAdapter.sendMessage
    ├── streams message.part / message.updated → cockpit left panel
    ├── receives approve command → HarnessAdapter writes file + git commit on branch
    └── emits spec.status event { specId, status: "in-review" } → cockpit status bar
```

The `spec-author` and `architect` roles (capable tier) from the roster in `docs/agent-roster-and-model-resolution.md` are the two authoring agents surfaced in the cockpit's agent selector. The registry (005) resolves tier to concrete model; the cockpit receives the resolved model name in the `model` field of `SessionStatusEvent` / `message.updated` for display.

### Data model
No new persistent entities. The cockpit consumes:
- `SpecFrontmatter` (from `@arke/contracts`) — parsed from the working file's YAML frontmatter for the status bar.
- `SPEC_ANATOMY` (from `@arke/contracts`) — the section outline driving section-by-section rendering.
- `DomainEvent` union — `message.part`, `message.updated` (003), `session.status`, `spec.status` from the coordinator WebSocket.

The delta-tag renderer is a pure function: it parses the `capability: …` · `delta: ADDED|MODIFIED|REMOVED (branch)` metadata line on each `### Requirement:` block and maps it to a visual class. No server state is needed.

### Interfaces and contracts
**Coordinator commands (client → coordinator via WebSocket):**
```typescript
// send a message to a named agent role
{ type: "sendMessage", sessionId: string, agent: "spec-author" | "architect", tier: ModelTier, parts: MessagePart[] }

// approve the working draft: commit file, advance status
{ type: "approveDraft", specId: string, branch: string }
```

**Events consumed by the cockpit (coordinator → client):**
- `message.part` / `message.updated` — stream agent turns into the left panel (003)
- `spec.status` — update the status bar on approval
- `session.status` — show active-model indicator

**File read:** the coordinator exposes a REST endpoint `GET /spec/:specId/file` that returns the current working-file text from the repository (reading off the feature branch); the cockpit polls this on mount and after each `message.updated` to refresh the preview. A future improvement can use a file-watch push event to avoid polling.

### Cross-cutting
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.
- **Security (NFR-1, NFR-5):** all harness writes happen inside the coordinator on the host; the client sends only the `approveDraft` command, never raw file content.
- **Accessibility (NFR-6):** the split surface is keyboard-navigable; focus moves between the chat composer and the preview with a standard shortcut; ARIA live regions announce streaming agent output.
- **Observability (NFR-7):** the `approveDraft` command and the resulting git commit are written to the append-only trace (`Trace.write`) with the specId and branch.
- **Resilience (NFR-8):** the cockpit uses the five-state reconnecting transport (003); the chat composer queues outbound `sendMessage` commands whilst disconnected and drains on reconnect.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Preview reads the file via `GET /spec/:specId/file` (poll after message.updated) rather than a push watch | Simpler and sufficient for authoring cadence; a watch push can be added later without changing the contract |
| 2 | `approveDraft` is a coordinator command, not a client-side git operation | Keeps credentials and git access on the host; the client never holds the spec body or a git credential |
| 3 | Delta-tag rendering is a client-side parse of the markdown text, not a server computation | The file is the source of truth; no derived state is needed server-side for highlighting |

## Open questions
- Should `GET /spec/:specId/file` return raw markdown (client renders it) or pre-parsed sections? Raw markdown keeps the server thin and lets the renderer evolve without a server change.
- When the engineer edits the specification prose inline in the preview (a possible future capability), how is that write funnelled through the harness without a round-trip through an agent? Deferred until 006 is stable.
