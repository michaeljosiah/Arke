# Authoring cockpit — chat and live preview

## Why
A Product Engineer co-authoring a specification with agents needs two things in front of them at the same time: a conversation with the agents that is shaping the document, and a live view of what those agents are actually writing into the repository. Without that split surface, the engineer must context-switch between a chat thread and a file, losing the connection between what was asked and what landed. This surface makes that connection continuous, covers FR-1 (authoring workspace), FR-14 (chat and live preview), and FR-15 (approve and persist), and is the host surface from which the review panel (007) is convened.

## What changes
- ADDED authoring-cockpit — a split surface pairing a conversation with the authoring agents beside a live preview of the working specification file, with per-turn agent role and model visible, delta-tagged requirements highlighted, and approval that commits the specification and advances its status

## Requirements

### Requirement: Split authoring surface with named agent turns
`capability: authoring-cockpit` · `delta: ADDED (feat/authoring-cockpit)`

The system SHALL present a split surface whose left panel is a conversation with the authoring agents and whose right panel is a live preview of the working specification file; each turn in the conversation SHALL display the agent role (e.g. `spec-author`, `architect`) and the concrete model that produced it.

#### Scenario: Agent role and model are visible on every turn
- **WHEN** an agent produces a turn in the conversation
- **THEN** the turn header names the agent role and the resolved model (e.g. `spec-author · claude-opus-4-5`)
- **AND** the human's own turns are labelled with their identity

#### Scenario: Chat input dispatches to the named agent
- **WHEN** the engineer submits a message selecting a named agent role
- **THEN** the coordinator routes the message to that role at its configured tier via `HarnessAdapter.sendMessage`
- **AND** the response streams into the conversation panel in real time via `message.part` events (depends on SPEC-003)

### Requirement: Live preview is a view of the working file, never a copy
`capability: authoring-cockpit` · `delta: ADDED (feat/authoring-cockpit)`

The specification preview SHALL be a read-only rendering of the working file in the repository as it exists on the feature branch; the client SHALL NOT hold a separate authoritative copy of the specification body; writes to `docs/specifications/` SHALL flow exclusively through the harness adapter on the coordinator host.

#### Scenario: Preview reflects the file on disk
- **WHEN** an agent writes a change to the working specification file via the harness
- **THEN** the preview panel updates to reflect the changed file within one poll cycle after the `message.updated` event arrives
- **AND** the client does not maintain its own writable copy of the specification body

#### Scenario: Preview shows in-flight write indicator
- **WHEN** the coordinator receives a `session.status { status: "running" }` event for the spec-author or architect session
- **THEN** the preview panel displays an in-flight indicator (e.g. a spinner in the panel header) to signal that the file may be mid-write
- **AND** the indicator is cleared when the session transitions to `idle`

#### Scenario: Missed message.updated leaves preview stale — bounded recovery
- **WHEN** a `message.updated` event is not received (e.g. due to a transient network loss)
- **THEN** the preview panel SHALL re-poll `GET /spec/:specId/file` within a bounded interval not exceeding 30 seconds, regardless of whether a `message.updated` event was received
- **AND** if the refreshed file differs from the displayed content, the preview updates and a "refreshed" indicator is shown

#### Scenario: Preview renders section by section against the anatomy
- **WHEN** the preview panel displays the working file
- **THEN** it renders the specification section by section in the order defined by `SPEC_ANATOMY` (frontmatter → requirements → design → tasks)
- **AND** sections absent from the working file are shown as empty placeholders, not hidden

### Requirement: Delta-tagged requirements are highlighted in the preview
`capability: authoring-cockpit` · `delta: ADDED (feat/authoring-cockpit)`

The preview SHALL highlight every requirement that carries a `delta:` tag, visually distinguishing it from settled, untagged requirements, so the preview reads as a contract-level diff whilst the specification is in flight on its branch.

#### Scenario: In-flight ADDED requirement is highlighted
- **WHEN** a requirement in the working file carries `delta: ADDED (feat/authoring-cockpit)`
- **THEN** the preview renders it with a distinct visual treatment (e.g. a green left border and `ADDED` badge)
- **AND** a requirement with no delta tag is rendered in the neutral, settled style

#### Scenario: MODIFIED requirement shows the full updated text
- **WHEN** a requirement in the working file carries `delta: MODIFIED`
- **THEN** the preview renders the full updated requirement text with a `MODIFIED` badge
- **AND** does not attempt to display a fragment or inline diff against a prior version

#### Scenario: REMOVED requirement shows a tombstone in the preview
- **WHEN** a requirement carries `delta: REMOVED`
- **THEN** the preview shows a struck-through tombstone line with a `REMOVED` badge
- **AND** the tombstone is not collapsed or hidden by default
- **AND** the tombstone text matches the flatten-on-merge tombstone format defined in SPEC-008 (requirement: Delta tags flatten on merge)

### Requirement: Approval commits the specification atomically and advances its status
`capability: authoring-cockpit` · `delta: ADDED (feat/authoring-cockpit)`

The system SHALL expose an explicit approval action; on human approval the coordinator SHALL atomically: (1) commit the specification file to `docs/specifications/` on the feature branch named in the frontmatter, (2) update the `status` frontmatter field to `in-review`, and (3) append a Change history line; the coordinator SHALL verify that the working HEAD branch matches the frontmatter `branch` field before committing; if the commit fails for any reason the coordinator SHALL emit a `spec.approval-failed` event with the failure reason, leave the status unchanged, and surface a user-visible error; before approval the specification SHALL remain on a feature branch and SHALL NOT be the authoritative version.

#### Scenario: Branch guard — HEAD must match frontmatter branch
- **WHEN** the engineer invokes the approval action
- **THEN** the coordinator verifies that the current git HEAD branch equals the `branch` field in the specification's frontmatter
- **AND** if they differ, the approval is rejected with a user-visible error naming the mismatch, and no commit is made

#### Scenario: Approval succeeds — status committed atomically
- **WHEN** the engineer invokes the approval action and the branch guard passes
- **THEN** the coordinator writes the updated frontmatter (`status: in-review`) and the new Change history line into the file, commits it to the feature branch as a single commit, and emits `spec.status { specId, status: "in-review" }`
- **AND** the cockpit status bar advances to `in-review`

#### Scenario: Commit fails — client sees an error, status unchanged
- **WHEN** the coordinator attempts the approval commit and the git operation fails (e.g. dirty branch, authentication failure, merge conflict)
- **THEN** the coordinator emits `spec.approval-failed { specId, reason }` and does NOT emit `spec.status { status: "in-review" }`
- **AND** the cockpit displays the failure reason and leaves the approval action available for retry

#### Scenario: Pre-approval specification is on a branch
- **WHEN** a specification has not yet been approved
- **THEN** the working file lives on the feature branch named in its `branch` frontmatter field
- **AND** the cockpit displays the branch name prominently in the status bar

#### Scenario: Approved specification becomes reviewable
- **WHEN** the approval commit lands
- **THEN** the specification is visible in the spec library (008) with status `in-review`
- **AND** the engineer is notified the specification is ready for the review panel

### Requirement: Outbound command queue is bounded and surfaced on drain
`capability: authoring-cockpit` · `delta: ADDED (feat/authoring-cockpit)`

The cockpit SHALL queue outbound `sendMessage` commands whilst the coordinator WebSocket is disconnected; the queue SHALL be bounded (maximum 50 commands); when the connection is re-established the coordinator SHALL drain the queue and the cockpit SHALL surface which commands were drained; if a queued `sendMessage` targets a session that is no longer `idle` or `running` on reconnect, the command SHALL be rejected with a user-visible message, not silently dropped.

#### Scenario: Queued commands drain on reconnect
- **WHEN** the WebSocket connection is re-established after a drop
- **THEN** the cockpit drains the outbound queue in order, submitting each queued `sendMessage` to the coordinator
- **AND** the cockpit displays a "reconnected — replaying N queued messages" indicator

#### Scenario: Stale queued command is rejected, not silently executed
- **WHEN** a queued `sendMessage` targets a session whose status is no longer `idle` or `running`
- **THEN** the coordinator rejects the command and returns an error naming the session and its current status
- **AND** the cockpit surfaces the rejection to the engineer and does not submit remaining queued commands for that session

#### Scenario: Queue bound enforced
- **WHEN** the outbound queue reaches 50 commands
- **THEN** further `sendMessage` attempts are blocked with a user-visible "offline — queue full" indicator
- **AND** no command is silently discarded

### Requirement: Review panel is convened without leaving the cockpit
`capability: authoring-cockpit` · `delta: ADDED (feat/authoring-cockpit)`

The system SHALL expose, within the cockpit, an action to convene the multi-model review panel (007) on the current working draft; the action SHALL pass the `specId` and `branch` reference to the coordinator (not a client-side copy of the file content) so the coordinator reads the authoritative working file; the engineer SHALL NOT need to navigate away from the cockpit to initiate a review.

#### Scenario: Convene review action is available at any point
- **WHEN** the working draft contains at least one requirement
- **THEN** a "Convene review" action is visible in the cockpit's toolbar
- **AND** invoking it sends `{ type: "convenePanel", specId, branch }` to the coordinator (not raw file content)

#### Scenario: Review can be convened before approval
- **WHEN** the specification status is `draft`
- **THEN** the engineer can still convene a review panel
- **AND** the review operates on the current working file, not on an approved version

## Design

### Architectural decision
The cockpit is a pure projection surface: it reads the working file from the repository (via the adapter) and renders it; it never writes the specification body directly. All writes to `docs/specifications/` flow through the harness adapter, keeping the file in git as the single source of truth. The approval action is a coordinator command, not a client-side file write. The `approveDraft` command is transactional: the coordinator validates the HEAD branch, writes the file, commits, then emits the status event — any failure at any step is surfaced via `spec.approval-failed`.

### Target architecture
The cockpit (`packages/client/src/screens/cockpit.tsx`) is the existing baseline screen. It is wired to the coordinator WebSocket (via the reconnecting transport from SPEC-003) for live event ingestion and to the coordinator's command channel for `sendMessage` and `approveDraft` actions.

```
Engineer input
    │
    ▼
cockpit.tsx (React 19, @arke/client)
    ├── left panel: conversation thread
    │       ├── turn bubbles (role + model label from message.updated / message.part events)
    │       ├── composer → coordinator command: sendMessage { agent, tier, parts }
    │       ├── outbound command queue (max 50; drains on reconnect; stale-session guard)
    │       └── "Convene review" button → convenePanel { specId, branch } (not file content)
    └── right panel: spec preview
            ├── reads working file via GET /spec/:specId/file (poll after message.updated; max-interval fallback 30s)
            ├── in-flight write indicator (session.status running → idle)
            ├── renders section-by-section against SPEC_ANATOMY
            ├── delta-tag renderer (ADDED / MODIFIED / REMOVED visual treatments; REMOVED tombstone matches SPEC-008 format)
            └── status bar (specId, status, branch from SpecFrontmatter — branch must match HEAD before approval)

coordinator (packages/coordinator/server.ts)
    ├── receives sendMessage command → HarnessAdapter.sendMessage
    ├── streams message.part / message.updated → cockpit left panel (SPEC-003)
    ├── receives approveDraft command:
    │       1. verify HEAD branch == frontmatter branch (else emit spec.approval-failed)
    │       2. write file (updated status + Change history)
    │       3. git commit on branch (else emit spec.approval-failed)
    │       4. emit spec.status { status: "in-review" }
    └── emits spec.status / spec.approval-failed → cockpit status bar
```

**SPEC-003 dependency (cross-spec):** the left panel's streaming capability depends on the `message.part` and `message.updated` event types defined in SPEC-003. The cockpit MUST degrade gracefully when those event types are absent (show a "streaming unavailable" label on turns; still show the completed turn when `message.updated` arrives). This dependency must be tracked in the project build sequence.

The `spec-author` and `architect` roles (capable tier) from the roster in `docs/agent-roster-and-model-resolution.md` are the two authoring agents surfaced in the cockpit's agent selector. The registry (SPEC-005) resolves tier to concrete model; the cockpit receives the resolved model name in the `model` field of `SessionStatusEvent` / `message.updated` for display.

### Data model
No new persistent entities. The cockpit consumes:
- `SpecFrontmatter` (from `@arke/contracts`) — parsed from the working file's YAML frontmatter for the status bar and branch guard.
- `SPEC_ANATOMY` (from `@arke/contracts`) — the section outline driving section-by-section rendering.
- `DomainEvent` union — `message.part`, `message.updated` (SPEC-003), `session.status`, `spec.status`, `spec.approval-failed` from the coordinator WebSocket.

The delta-tag renderer is a pure function: it parses the `capability: …` · `delta: ADDED|MODIFIED|REMOVED (branch)` metadata line on each `### Requirement:` block and maps it to a visual class. No server state is needed.

### Interfaces and contracts
**Coordinator commands (client → coordinator via WebSocket):**
```typescript
// send a message to a named agent role
{ type: "sendMessage", sessionId: string, agent: "spec-author" | "architect", tier: ModelTier, parts: MessagePart[] }

// approve the working draft: verify branch, commit file, advance status
{ type: "approveDraft", specId: string, branch: string }

// convene a review panel — passes reference only, not file content
{ type: "convenePanel", specId: string, branch: string, reviewers?: Array<{ role: string; instanceId?: string }> }
```

**Events consumed by the cockpit (coordinator → client):**
- `message.part` / `message.updated` — stream agent turns into the left panel (SPEC-003)
- `session.status` — in-flight write indicator (running → idle) and active-model indicator
- `spec.status` — update the status bar on approval
- `spec.approval-failed { specId, reason }` — surface commit failure to the engineer

**New event added to `@arke/contracts/src/events.ts`:**
```typescript
{ type: "spec.approval-failed", specId: string, reason: string }
```

**File read:** the coordinator exposes `GET /spec/:specId/file` returning the current working-file text from the repository (reading off the feature branch); the cockpit polls this on mount and after each `message.updated`, with a maximum fallback interval of 30 seconds to guard against missed events.

### Cross-cutting
- **Configuration:** all project configuration — the harness/model registry, coordinator settings, and integrations — lives in the single root file `.arke/config.json` (`ARKE_*` env vars override individual keys; runtime artefacts under `.arke/` are state, not config). See SPEC-005 for the model.
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.
- **Security (NFR-1, NFR-5):** all harness writes happen inside the coordinator on the host; the client sends only the `approveDraft` command, never raw file content; `convenePanel` passes a reference (`specId`/`branch`), not a file copy.
- **Accessibility (NFR-6):** the split surface is keyboard-navigable; focus moves between the chat composer and the preview with a standard shortcut; ARIA live regions announce streaming agent output.
- **Observability (NFR-7):** the `approveDraft` command, the branch-guard check, the resulting git commit, and any `spec.approval-failed` events are written to the append-only trace (`Trace.write`) with the specId, branch, and actor.
- **Resilience (NFR-8):** the cockpit uses the five-state reconnecting transport (SPEC-003); the chat composer queues outbound `sendMessage` commands whilst disconnected (bounded at 50), drains on reconnect with stale-session guard, and surfaces both drain completion and per-command rejections to the engineer; the preview falls back to a 30-second poll interval to guard against missed `message.updated` events.
- **Multi-project scoping (SPEC-018):** the coordinator supervises N isolated **project contexts**, each keyed by `projectId`. Everything this spec places on "the coordinator" is per–project context; each client connection resolves against its **active project** and that project's safe root + `docs/specifications/`. See [SPEC-018](018.multi-project-workspaces.md). Concretely, the cockpit coordinator ops (`sendMessage` / `approveDraft`, and `GET /spec/:specId/file`) resolve against the **active project**'s root, harness, and repository.

### Testing
- Unit: delta-tag parser (ADDED / MODIFIED / REMOVED / no-tag cases; REMOVED tombstone format matches SPEC-008); `SPEC_ANATOMY` section renderer with missing sections; coordinator `approveDraft` handler — branch mismatch → `spec.approval-failed`, commit failure → `spec.approval-failed`, success → `spec.status`; Change history line appended; outbound queue bound (50 commands → reject further); stale-session guard on drain.
- Integration: cockpit end-to-end — send a message, agent writes to the file, preview updates; approve with matching HEAD → `spec.status` + commit on branch; approve with mismatched HEAD → `spec.approval-failed` and no commit; 30-second fallback poll fires when `message.updated` is suppressed; `convenePanel` sends specId/branch reference not file content.
- Manual: open the cockpit on a real working draft, watch streaming output, verify delta highlights including tombstone for REMOVED, approve, confirm branch commit and status advance.

### Definition of done
All scenarios pass; the preview is demonstrably a view of the on-disk file (editing the file externally updates the preview within 30 seconds); delta highlights render correctly for all three delta types including the REMOVED tombstone; approval is atomic (commit failure emits `spec.approval-failed`, no partial state); branch guard rejects mismatched HEAD; queue drain on reconnect is visible and stale-session rejections are surfaced; the "Convene review" action passes a reference, not a file copy; a reviewer has signed off.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Preview reads the file via `GET /spec/:specId/file` (poll after `message.updated`, 30s fallback) rather than a push watch | Simpler and sufficient for authoring cadence; the fallback interval guards against missed events; a watch push can be added later without changing the contract |
| 2 | `approveDraft` is a coordinator command, not a client-side git operation | Keeps credentials and git access on the host; the client never holds the spec body or a git credential |
| 3 | Delta-tag rendering is a client-side parse of the markdown text, not a server computation | The file is the source of truth; no derived state is needed server-side for highlighting |
| 4 | `approveDraft` is a single transaction: branch guard → write → commit → event; failure at any step emits `spec.approval-failed` | Prevents partial approval state where the client thinks approval succeeded but the branch is dirty or the status event was lost |
| 5 | `convenePanel` passes `specId`/`branch` reference, not file content | The coordinator reads the authoritative file from the repo, maintaining the "no client copy" invariant consistently |
| 6 | Outbound queue bounded at 50 commands, with stale-session guard on drain | Prevents silent data loss on reconnect and prevents executing commands against sessions that changed state while offline |

## Open questions
- Should `GET /spec/:specId/file` return raw markdown (client renders it) or pre-parsed sections? Raw markdown keeps the server thin and lets the renderer evolve without a server change.
- When the engineer edits the specification prose inline in the preview (a possible future capability), how is that write funnelled through the harness without a round-trip through an agent? Deferred until 006 is stable.
- Should the 30-second fallback poll be configurable per project, or is a fixed interval sufficient?
