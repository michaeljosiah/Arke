# Session detail, diff review, rescue & steering

## Why

A product engineer must be able to watch a single task live, review the diff it produces before a PR is opened, steer it mid-run, and rescue it when it goes wrong. Without this surface the board is a status display only — agents can fail silently and no human can intervene within the loop.

## What changes

- ADDED session-detail — a detail panel in `packages/client` showing the live transcript, todo list, and diff summary for a single session, updated from the coordinator event stream.   (breaking: no)
- ADDED diff-review — a per-session diff view gating PR creation: the engineer must review and approve the diff before the adapter opens a pull request.   (breaking: no)
- ADDED rescue — revert and restore (unrevert) controls that call the adapter's `revert`/`unrevert` capability via the coordinator; git checkpoints underpin each turn.   (breaking: no)
- ADDED session-detail — two-way steering: a message composer lets the engineer send a message to the running agent; permission and elicitation requests appear as structured overlays, not free text.   (breaking: no)
- ADDED session-detail — a "promote to specification" action that drops a mid-delivery change of intent back into the spec as an `in-review` draft, pausing the session.   (breaking: no)

## Requirements

### Requirement: Live transcript with todos and diff summary
`capability: session-detail` · `delta: ADDED (feat/session-detail-and-rescue)`

The system SHALL display, for a selected session, a live scrolling transcript of all messages (user, assistant, tool calls), the current todo list, and the diff summary. These SHALL update in real time as `message.part`, `message.updated`, `todo.updated`, and `diff.finalized` events arrive from the coordinator without requiring a page reload. The transcript SHALL use the `TranscriptEntry[]` from the `CardState` read model (SPEC-003) as its data source.

The diff panel SHALL display the diff as known at the time of the most recent `diff.finalized` event. If no `diff.finalized` event has arrived, the diff panel SHALL indicate that no diff is available yet — it SHALL NOT call `getDiff()` on the adapter directly from the client. If a stale `diff.finalized` event was received before a reconnect, the diff panel SHALL indicate that the diff may be stale (showing the `ts` of the last event) and offer a manual refresh action that routes a `diff.refresh` command through the coordinator.

The transcript MAY use virtualised rendering to handle long sessions without degrading scroll performance (see T3 timeline approach, `docs/analysis/t3code-learnings.md` §Timeline).

#### Scenario: Streaming part arrives during an active turn
- **WHEN** the coordinator pushes a `message.part` event for the open session
- **THEN** the transcript appends the delta to the current assistant message without a full re-render of prior messages
- **AND** a visual streaming indicator is shown while `isStreaming` is `true`

#### Scenario: Turn completes on quiescent receipt
- **WHEN** a `turn.quiescent` event arrives for the session
- **THEN** the streaming indicator is removed and the message is rendered as complete
- **AND** the `isStreaming` flag on the transcript entry is `false`
- **AND** the composer is re-enabled if it was disabled solely because a turn was in progress

#### Scenario: Todo list updates
- **WHEN** a `todo.updated` event arrives
- **THEN** the todo list reflects the new state within one render cycle
- **AND** completed todos are visually distinguished from pending ones

#### Scenario: Diff is stale after reconnect
- **WHEN** the transport reconnects and the most recent `diff.finalized` event pre-dates the reconnect gap
- **THEN** the diff panel shows the last known diff with a "may be stale — refresh" indicator
- **AND** activating the refresh sends a `diff.refresh` coordinator command rather than calling `getDiff()` directly from the client

### Requirement: Two-way steering with structured overlays
`capability: session-detail` · `delta: ADDED (feat/session-detail-and-rescue)`

The system SHALL provide a message composer in the session detail that lets the engineer send a plain-text steering message to the running agent via `ArkeTransport.send()` (which routes it to the adapter's `sendMessage` or `runCommand` capability). Permission requests (SPEC-012) and agent elicitation (questions) SHALL surface as structured overlays rendered above the composer — not as messages in the transcript — so the engineer's decision is clearly bounded and separate from the conversation flow. The composer SHALL be disabled while a permission or elicitation overlay is open, or while an agent turn is actively streaming.

Steering messages sent while a turn is streaming present a race: the agent may not be in a state to receive steering mid-turn. The composer SHALL warn the engineer if the session's `isStreaming` is `true` when they attempt to submit, and SHALL require a second confirmation before sending. The send is not blocked outright, because some harnesses accept mid-turn steering, but the risk is surfaced.

#### Scenario: Steering message is sent to the agent
- **WHEN** the engineer types a message in the composer and submits while the session is idle
- **THEN** `ArkeTransport.send()` emits a steering request carrying the `sessionId` and message text
- **AND** the message appears in the transcript as a user turn

#### Scenario: Composer is disabled during a pending overlay
- **WHEN** a `permission.asked` event is open (overlay visible) for this session
- **THEN** the composer input is disabled
- **AND** the submit button is not activatable until the overlay is resolved

#### Scenario: Steering race warning while streaming
- **WHEN** the engineer attempts to submit a steering message while `isStreaming` is `true`
- **THEN** the composer shows an inline warning: "Agent is mid-turn — send anyway?"
- **AND** the message is sent only if the engineer confirms the secondary prompt
- **AND** the send (if confirmed) is traced with `{ kind: "client.request", warning: "mid-turn-steering" }`

#### Scenario: Agent elicitation appears as a structured overlay
- **WHEN** the adapter emits an `elicitation.asked` event for the session
- **THEN** a structured overlay renders the question text with Accept / Dismiss controls
- **AND** the overlay is dismissed only after the engineer responds — not on click-away or Escape
- **AND** the composer is disabled until the elicitation is resolved

### Requirement: Per-session diff review before PR creation
`capability: diff-review` · `delta: ADDED (feat/session-detail-and-rescue)`

Before a PR is opened for a session, the system SHALL require the engineer to review the diff in the detail view and explicitly approve it. The diff data SHALL come from `diff.finalized` events arriving over the coordinator stream — NOT from a direct `getDiff()` call from the client. Approval SHALL send a `pr.approve` command to the coordinator via `ArkeTransport.send()`; the adapter opens the PR only after receiving and tracing this command. A PR SHALL never be opened automatically without an explicit human approval on the diff.

This is the propose-decide-execute boundary for code output: the diff is the proposal, the engineer's approval is the decision, and the PR is the execution (PRD §4).

The approve action SHALL be idempotent: submitting it twice for the same session SHALL have no additional effect (the coordinator deduplicates by session + `seq` of the `diff.finalized` event that was approved).

#### Scenario: Diff visible before PR
- **WHEN** a session's `diff.finalized` event arrives (agent turn complete)
- **THEN** the diff panel shows added/removed counts and the changed-file list
- **AND** an "Approve diff & open PR" button is visible but requires interaction

#### Scenario: PR not opened without approval
- **WHEN** a session reaches `done` state
- **THEN** no PR is opened automatically
- **AND** the PR only opens after the engineer activates the approve-diff action

#### Scenario: Diff panel shows patch when available
- **WHEN** the `diff.finalized` event includes a `patch` field (or the coordinator synthesises one from `getDiff()`)
- **THEN** the diff panel renders a syntax-highlighted unified diff alongside the summary counts

#### Scenario: Approve is idempotent
- **WHEN** the engineer submits "Approve diff & open PR" twice (e.g. due to a double-click or queue drain on reconnect)
- **THEN** the coordinator processes the command once and the PR is opened once
- **AND** the second submission returns without error

### Requirement: Revert and restore via harness rescue
`capability: rescue` · `delta: ADDED (feat/session-detail-and-rescue)`

The system SHALL provide revert and restore controls in the session detail. Activating revert SHALL call the adapter's `revert` capability (mapped to `POST /session/:id/revert` in OpenCode) via a coordinator command, rolling the session back to the git snapshot before a specified turn. Restore (unrevert) SHALL call the adapter's `unrevert` capability via a coordinator command. Both actions SHALL be confirmed with a destructive-action prompt before execution.

A revert is a rescue path, not a routine action; it SHALL be visually distinct (a collapsed rescue section) to avoid accidental activation. The rescue controls SHALL be absent (not just disabled) when the adapter does not advertise the `"revert"` capability — the section SHALL not render at all if the harness cannot honour it.

After a revert, the transcript in the detail view SHALL reflect the rolled-back state as soon as the coordinator pushes the updated `session.status` or `message.updated` events. The client SHALL NOT apply a local optimistic update for the revert; it waits for the coordinator to confirm via the event stream.

**Checkpoint integrity:** the revert target is a `messageId` (the checkpoint before that message's turn). The engineer SHOULD be shown the content of that message (from the transcript) before confirming, so they understand which turn they are reverting to. A revert to a `messageId` that no longer exists on the harness (e.g. because the session was forked) SHALL surface a clear error from the coordinator.

#### Scenario: Revert rolls back to a prior checkpoint
- **WHEN** the engineer confirms a revert to a specific `messageId`
- **THEN** `ArkeTransport.send({ type: "revert", sessionId, messageId })` is emitted
- **AND** the coordinator routes the command to `adapter.revert({ sessionId, messageId })`
- **AND** the transcript reflects the rolled-back state once the coordinator pushes the resulting events — no optimistic client update

#### Scenario: Revert requires explicit confirmation naming the target
- **WHEN** the engineer activates the revert control for a `messageId`
- **THEN** a destructive-action prompt appears with `role="alertdialog"` and focus trapped, naming the message text of the target checkpoint
- **AND** the revert command is NOT emitted until the engineer confirms
- **AND** the prompt can be cancelled without sending any command

#### Scenario: Revert to invalid checkpoint surfaces an error
- **WHEN** the coordinator cannot find the `messageId` on the harness (session forked, pruned, etc.)
- **THEN** the coordinator returns an error event to the client
- **AND** the detail view shows a user-visible error message naming the `messageId` and suggesting a page refresh
- **AND** no transcript optimistic update was applied, so the view is not corrupted

#### Scenario: Rescue section absent when capability unavailable
- **WHEN** the current adapter does not advertise the `"revert"` capability
- **THEN** the `<RescueSection>` component is not rendered in the session detail
- **AND** no revert/unrevert controls are visible or focusable

#### Scenario: Agent failure surfaces as needs-human
- **WHEN** the adapter emits a `session.status` event with `status: "error"` for this session (NFR-4)
- **THEN** the card moves to the `needs-human` column on the board (SPEC-010)
- **AND** the session detail shows the error message and offers the revert/restore controls as rescue options (if the capability is present)

### Requirement: Promote intent change to specification
`capability: session-detail` · `delta: ADDED (feat/session-detail-and-rescue)`

The system SHALL provide an action in the session detail that promotes a mid-delivery change of intent back into the open specification, drops the spec's status to `in-review`, and pauses the current session pending re-approval. The promote action SHALL write the intent change as a new requirement stub to the spec file on the feature branch via a coordinator command, not via a direct filesystem write from the client.

**Who approves re-entry:** dropping the spec to `in-review` means any human reviewer can approve it (per the spec lifecycle in SPEC-008). The session-detail view has no authority to advance the spec back to `approved`; that requires a separate approval action in the review panel (SPEC-007). The promote action is not reversible from the session-detail view alone — it requires a reviewer to re-approve.

**Session pause semantics:** "paused" means the coordinator ceases dispatching new turns to the agent for this session. It does not mean the current in-flight turn is aborted; the turn runs to completion (or errors), and no new turns are dispatched until the spec returns to `approved` status. If an in-flight turn is actively streaming when the promote action is triggered, the coordinator queues the pause instruction and applies it after `turn.quiescent`.

#### Scenario: Intent promoted and spec drops to in-review
- **WHEN** the engineer triggers "Promote to spec" and enters the intent change text
- **THEN** `ArkeTransport.send({ type: "promote", sessionId, intentText })` is emitted
- **AND** the coordinator writes the change as a requirement stub to the spec file on the branch
- **AND** emits `spec.status: in-review`
- **AND** the session is paused (no further agent dispatch until re-approved)

#### Scenario: Paused session is visible on the board
- **WHEN** a session is paused following a promote action
- **THEN** the session's card appears in the `review` column (derived from `in-review` spec status)
- **AND** the card does not show `needs-human` (the engineer is not blocked on a permission; the spec is under review)

#### Scenario: In-flight turn runs to completion before pause takes effect
- **WHEN** the promote action is triggered while a turn is actively streaming
- **THEN** the coordinator does not abort the in-flight turn
- **AND** the pause is applied after `turn.quiescent` for that turn
- **AND** the session detail shows a "Promoting after current turn completes" status indicator

#### Scenario: Re-approval is required via the review panel, not the session detail
- **WHEN** a reviewer re-approves the spec in the review panel (SPEC-007)
- **THEN** the spec's status advances to `approved`
- **AND** the coordinator may resume dispatching turns to the session
- **AND** the session-detail view does NOT provide a direct "approve and resume" button — re-entry is governed by SPEC-007/SPEC-008

### Requirement: Permission and elicitation overlays aligned with SPEC-012
`capability: session-detail` · `delta: ADDED (feat/session-detail-and-rescue)`

Permission requests and agent elicitation (questions) SHALL use the structured overlay components defined in SPEC-012. This spec specifies where those overlays appear (above the composer in the session-detail view, and as the primary content when a `needs-human` card is opened from the board) — it does NOT define the overlay internals, which are owned by SPEC-012. The two specs MUST NOT contradict on the overlay behaviour; if a conflict arises, SPEC-012 governs.

The session-detail view SHALL subscribe to `permission.asked` and `permission.replied` events from the coordinator stream for its session. When a `permission.asked` event arrives:
- The `<PermissionOverlay>` (SPEC-012 component) is rendered above the composer.
- The composer is disabled.
- The engineer's approve/deny decision is sent via `ArkeTransport.send()` — never via a direct adapter call.

`ElicitationEvent` (`elicitation.asked`, `elicitation.replied`, `elicitation.rejected`) is the canonical name used in this spec and MUST be aligned with the final name chosen in SPEC-012. If SPEC-012 uses a different name (e.g. `question.asked`), the contracts file is the single source of truth; this spec's implementation follows `@arke/contracts/src/events.ts`.

#### Scenario: Permission overlay appears and disables composer
- **WHEN** a `permission.asked` event arrives for the open session
- **THEN** the `<PermissionOverlay>` (SPEC-012) is rendered above the composer
- **AND** the composer input and submit button are disabled
- **AND** no navigation away from the session detail is required to handle the permission

#### Scenario: Elicitation overlay is not dismissible by click-away
- **WHEN** a `elicitation.asked` event arrives for the open session
- **THEN** the elicitation overlay is rendered with Accept and Dismiss controls
- **AND** clicking outside the overlay does NOT dismiss it
- **AND** pressing Escape does NOT dismiss it
- **AND** the overlay is removed only after the engineer submits a response or explicitly rejects it

## Design

### Architectural decision

Session detail is a read-only projection of the coordinator's event stream for a single `sessionId`, augmented by coordinator-routed commands for mutations. All mutations — steering, permissions, revert, promote, diff approval — flow through `ArkeTransport.send()` to the coordinator and onwards to the adapter, never via direct adapter calls from the client. The diff panel consumes only `diff.finalized` events (not live `getDiff()` calls) for consistency with the event-driven model; a manual refresh sends a coordinator command. Revert state is not applied optimistically; the client waits for the event-stream confirmation. The promote action pauses after the in-flight turn completes, not immediately, to avoid orphaning a streaming turn.

### Target architecture

```
ArkeTransport (SPEC-003)
  └─▶ DomainEvent stream filtered to sessionId
        └─▶ useSession(sessionId) hook
              ├─▶ transcript: TranscriptEntry[]  ← from CardState.transcript (SPEC-003)
              ├─▶ todos: TodoItem[]              ← from todo.updated events
              ├─▶ diff: DiffSummary | null       ← from diff.finalized events (event-driven, not live REST)
              ├─▶ pendingPermission              ← from permission.asked / permission.replied
              └─▶ pendingElicitation             ← from elicitation.asked / elicitation.replied/rejected

  <SessionDetail>
    ├─▶ <Transcript>          (virtualised, streaming delta, tool-call groups, isStreaming indicator)
    ├─▶ <TodoList>
    ├─▶ <DiffPanel>           (diff.finalized event data; stale indicator; approve-diff gate)
    ├─▶ <MessageComposer>     (disabled while overlay open or turn streaming; mid-turn warning)
    ├─▶ <PermissionOverlay>   (SPEC-012 component; not dismissible by click-away)
    ├─▶ <ElicitationOverlay>  (same overlay pattern; not dismissible by click-away)
    └─▶ <RescueSection>       (collapsed; absent if "revert" capability not advertised)
         └─▶ Checkpoint picker → destructive-action prompt → ArkeTransport.send(revert)
```

Git checkpoints (one per agent turn) are the revert targets, matching the T3 per-turn-snapshot pattern. The OpenCode adapter exposes `POST /session/:id/revert` and `/unrevert`.

### Data model

`useSession()` derives its state entirely from the coordinator stream; no client-side persistence. The hook maintains:

```typescript
interface SessionDetailState {
  transcript: TranscriptEntry[];     // from CardState.transcript, kept in sync via events
  todos: TodoItem[];                 // folded from todo.updated events
  diff: DiffSummary | null;          // from diff.finalized events; null = not yet received
  diffStale: boolean;                // true if diff.finalized.ts < last reconnect ts
  sessionStatus: SessionStatus;
  isStreaming: boolean;              // true while any message.part with done:false is in flight
  needsHuman: boolean;
  pendingPermission: PermissionAskedEvent | null;
  pendingElicitation: ElicitationAskedEvent | null;
  error: string | null;             // set from session.status: error
  pausePending: boolean;            // set when promote action is queued behind in-flight turn
}
```

`ElicitationAskedEvent` (canonical name pending alignment with SPEC-012) is a domain event added to `@arke/contracts/src/events.ts` in this spec's implementation; it maps to OpenCode's `question.asked` event. The final name MUST match SPEC-012.

**`HarnessAdapter` additions** (optional capabilities):

```typescript
revert?(ref: SessionRef, messageId: string): Promise<void>;
unrevert?(ref: SessionRef): Promise<void>;
```

`"revert"` is added to the `Capability` enum. The coordinator routes `{ type: "revert", sessionId, messageId }` client messages to `adapter.revert()` when the capability is present; it returns an error event if absent.

### Interfaces and contracts

- Reads `TranscriptEntry[]`, `CardState`, `DomainEvent` from `@arke/contracts`.
- Uses `ArkeTransport.send()` for all mutations: steering, permission replies, revert, promote, diff approval, diff refresh.
- `revert`/`unrevert` are new optional methods on `HarnessAdapter`.
- Route: `/sessions/:sessionId` or a right-panel layout alongside the board; navigation from `<BoardCard>` (SPEC-010) passes `sessionId` and `specId`.
- `ElicitationAskedEvent` / `ElicitationRepliedEvent` / `ElicitationRejectedEvent` added to the `DomainEvent` union — names must align with SPEC-012.
- Coordinator command verbs added: `revert`, `unrevert`, `promote`, `pr.approve`, `diff.refresh`.

### Cross-cutting
- **Configuration:** all project configuration — the harness/model registry, coordinator settings, and integrations — lives in the single root file `.arke/config.json` (`ARKE_*` env vars override individual keys; runtime artefacts under `.arke/` are state, not config). See SPEC-005 for the model.

- **Multi-project scoping (SPEC-018):** the coordinator supervises N isolated **project contexts**, each keyed by `projectId` with its own read model, trace, and durable stores (session store, grant store, proposal buffer) under its own `.arke/`. Each client connection resolves against its **active project**; events carry `projectId`. Because a `sessionId` is unique only **within** a project (two projects' harnesses can mint the same id), the session-detail deep link SHALL carry the owning project — `/projects/:projectId/sessions/:sessionId` — and opening such a link (a fresh tab, or after switching projects) SHALL first activate that project (`project.open`) before resolving the session. A bare `/sessions/:sessionId` resolves only against the already-active project and is not a stable shareable link. See [SPEC-018](018.multi-project-workspaces.md).

- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`). Do not re-derive styling.

**Security (NFR-1):** client never calls the adapter directly; all commands flow through `ArkeTransport → coordinator → adapter`. The coordinator validates `sessionId` ownership before routing revert/promote commands — a client cannot revert a session belonging to a different spec.

**Performance (NFR-3):** transcript uses virtualised rendering (anchored scroll) so long sessions with thousands of parts do not degrade. Only the current session's events are processed in `useSession()`; events for other sessions are filtered at the hook boundary. Diff is received from events, not polled, eliminating periodic REST overhead.

**Resilience (NFR-4):** agent failure (`session.status: error`) is surfaced within the session detail as a rescue prompt; the board independently moves the card to `needs-human` (SPEC-010). The session detail SHALL not crash on an error event; it SHALL render the error message and the rescue controls. Revert confirmation does not apply optimistic updates — no corrupted state on coordinator errors. The promote action is safe across an in-flight turn because the coordinator defers the pause to post-quiescence.

**Observability:** every coordinator command (steering, revert, promote, pr.approve, diff.refresh) is written to the append-only trace by the coordinator before the adapter call, providing a complete action log. Mid-turn steering warnings are also traced.

**Accessibility:** the diff panel uses `aria-label` on added/removed counts; the composer textarea has a `<label>`; destructive-action prompts have `role="alertdialog"` with focus trapped until resolved; elicitation overlays are not dismissible by keyboard Escape (by design — requires an explicit response); the rescue section uses a `<details>` or equivalent collapsed element.

- **Unit (idempotency):** `pr.approve` sent twice → coordinator deduplicates → PR opened once.
- **Unit (revert invalid checkpoint):** coordinator returns error event → detail view shows error message → transcript not corrupted.
- **Integration:** `MockAdapter` scripted sequence (SPEC-003) ending in an error event → session detail shows rescue controls → revert command → transcript rolls back (confirmed via event-stream, not optimistic); promote action while streaming → pause deferred to post-quiescence → `spec.status: in-review` emitted.

### Definition of done

All scenarios pass; TypeScript strict-mode compiles; the diff-review gate cannot be bypassed (integration test proves `pr.approve` is required); agent failure surfaces correctly on board and in detail; revert is absent when capability is not advertised; no optimistic update is applied for revert (verified by test); promote correctly defers pause to post-quiescence; elicitation overlay cannot be dismissed by click-away or Escape; `ElicitationEvent` name is aligned with SPEC-012; a reviewer has signed off; SPEC-002 and SPEC-003 are merged (adapter and transport dependencies); SPEC-012 overlay names are confirmed before `@arke/contracts` is updated.

## Decision log

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | All mutations through `ArkeTransport.send()`, never direct adapter calls from the client | Preserves the invariant that the browser is not in the credential path; ensures every action is traced |
| 2 | Revert/unrevert are optional adapter capabilities, not required | Not every harness supports git checkpoints; the rescue section degrades gracefully (and is absent, not disabled) |
| 3 | Diff approval is a hard gate before PR creation, and is idempotent | Enforces propose-decide-execute for code output; idempotency prevents double-PR on reconnect queue drain |
| 4 | `ElicitationEvent` added to `DomainEvent` union (not a separate channel) | Elicitation is a governed action like permission — must be traced and visible on the same stream |
| 5 | "Promote to spec" writes via coordinator command, not filesystem from client | Maintains direction of truth: spec file in git is the source of truth; client mutations go through the coordinator |
| 6 | No optimistic update for revert | Optimistic state that does not match the coordinator would leave the view corrupted if the revert fails; event-stream confirmation is the correct pattern |
| 7 | Promote defers pause to post-quiescence of in-flight turn | Aborting a streaming turn mid-flight wastes work and may leave the harness in an inconsistent state; the coordinator is the safe point to apply the pause |
| 8 | Re-approval of a promoted spec is out of scope for session-detail | Governance lives in SPEC-007/008; the session-detail cannot unilaterally advance a spec status; this prevents a backdoor around the review gate |
| 9 | Diff panel consumes `diff.finalized` events, not live `getDiff()` REST calls | Keeps the client in the event-driven model; avoids polling; stale indicator + refresh command handles the reconnect case correctly without direct adapter access |
| 10 | RescueSection absent (not disabled) when revert capability not advertised | A disabled button leaks the existence of a feature the harness cannot honour; absent is cleaner and avoids user confusion |
| 11 | Mid-turn steering shows a warning, not a hard block | Some harnesses accept mid-turn steering; a hard block would reduce utility; the warning makes the race explicit and auditable |

## Open questions

- `ElicitationEvent` maps to OpenCode's `question.asked` / `question.replied` / `question.rejected`. The final canonical name in `@arke/contracts` MUST be agreed with SPEC-012 before the contracts file is updated. (This spec uses `elicitation.*`; SPEC-012 may settle on `question.*`.)
- Should the `<Transcript>` virtualisation use a fixed-height container or anchor scroll at the bottom on new messages? T3 uses anchored scroll; this may need a `scrollIntoView` guard to avoid fighting user-initiated upward scroll.
- How many git checkpoints are retained per session? OpenCode snapshots per message turn; should Arke expose a full checkpoint picker or only "revert to last N turns"?
- The coordinator-side `sessionId` ownership validation for revert/promote commands: is ownership checked against the durable session store (SPEC-003) or re-fetched from the adapter? Durable store is preferred for restart safety.
- Should `diff.refresh` trigger a `getDiff()` REST call on the adapter and emit a fresh `diff.finalized` event, or should the coordinator cache the last diff and re-emit it? The former is correct; confirm with the team.
