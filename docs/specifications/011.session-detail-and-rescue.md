# Session detail, diff review, rescue & steering

## Why

A product engineer must be able to watch a single task live, review the diff it produces before a PR is opened, steer it mid-run, and rescue it when it goes wrong. Without this surface the board is a status display only — agents can fail silently and no human can intervene within the loop.

## What changes

- ADDED session-detail — a detail panel in `packages/client` showing the live transcript, todo list, and diff summary for a single session, updated from the coordinator event stream.   (breaking: no)
- ADDED diff-review — a per-session diff view gating PR creation: the engineer must review and approve the diff before the adapter opens a pull request.   (breaking: no)
- ADDED rescue — revert and restore (unrevert) controls that call the adapter's `revert`/`unrevert` capability via the coordinator; git checkpoints underpin each turn.   (breaking: no)
- ADDED session-detail — two-way steering: a message composer lets the engineer send a message to the running agent; permission and elicitation requests appear as structured overlays, not free text.   (breaking: no)
- ADDED session-detail — a "promote to specification" action that drops a mid-delivery change of intent back into the spec as an `in-review` draft, pausing the session.   (breaking: no)

## Requirements

### Requirement: Live transcript with todos and diff summary
`capability: session-detail` · `delta: ADDED (feat/session-detail-and-rescue)`

The system SHALL display, for a selected session, a live scrolling transcript of all messages (user, assistant, tool calls), the current todo list, and the diff summary. These SHALL update in real time as `message.part`, `message.updated`, `todo.updated`, and `diff.finalized` events arrive from the coordinator without requiring a page reload. The transcript SHALL use the `TranscriptEntry[]` from the `CardState` read model (SPEC-003) as its data source.

The transcript MAY use virtualized rendering to handle long sessions without degrading scroll performance (see T3 timeline approach, `docs/analysis/t3code-learnings.md` §Timeline).

#### Scenario: Streaming part arrives during an active turn
- **WHEN** the coordinator pushes a `message.part` event for the open session
- **THEN** the transcript appends the delta to the current assistant message without a full re-render of prior messages
- **AND** a visual streaming indicator is shown while `isStreaming` is `true`

#### Scenario: Turn completes
- **WHEN** a `turn.quiescent` event arrives for the session
- **THEN** the streaming indicator is removed and the message is rendered as complete
- **AND** the `isStreaming` flag on the transcript entry is `false`

#### Scenario: Todo list updates
- **WHEN** a `todo.updated` event arrives
- **THEN** the todo list reflects the new state within one render cycle
- **AND** completed todos are visually distinguished from pending ones

### Requirement: Two-way steering with structured overlays
`capability: session-detail` · `delta: ADDED (feat/session-detail-and-rescue)`

The system SHALL provide a message composer in the session detail that lets the engineer send a plain-text steering message to the running agent via `ArkeTransport.send()` (which routes it to the adapter's `sendMessage` or `runCommand` capability). Permission requests and agent elicitation (questions) SHALL surface as structured overlays rendered above the composer — not as messages in the transcript — so the engineer's decision is clearly bounded and separate from the conversation flow. The composer SHALL be disabled while a permission or elicitation overlay is open.

This mirrors the T3 `ComposerPendingApprovalPanel` / `ComposerPendingUserInputPanel` pattern (`docs/analysis/t3code-learnings.md` §Composer).

#### Scenario: Steering message is sent to the agent
- **WHEN** the engineer types a message in the composer and submits
- **THEN** `ArkeTransport.send()` emits a steering request carrying the session id and message text
- **AND** the message appears in the transcript as a user turn

#### Scenario: Composer is disabled during a pending overlay
- **WHEN** a `permission.asked` event is open (overlay visible) for this session
- **THEN** the composer input is disabled
- **AND** the submit button is not activatable until the overlay is resolved

#### Scenario: Agent elicitation appears as a structured overlay
- **WHEN** the adapter emits an elicitation event (question) for the session
- **THEN** a structured overlay renders the question text with Accept / Dismiss controls
- **AND** the overlay is dismissed only after the engineer responds, not on click-away

### Requirement: Per-session diff review before PR creation
`capability: diff-review` · `delta: ADDED (feat/session-detail-and-rescue)`

Before a PR is opened for a session, the system SHALL require the engineer to review the diff in the detail view and explicitly approve it. The diff SHALL be fetched from the adapter via `getDiff()` and SHALL show added/removed line counts and the list of changed files. Approval SHALL send a `pr.approve` command to the coordinator; the adapter opens the PR only after receiving this command. A PR SHALL never be opened automatically without an explicit human approval on the diff.

This is the propose-decide-execute boundary for code output: the diff is the proposal, the engineer's approval is the decision, and the PR is the execution (PRD §4).

#### Scenario: Diff visible before PR
- **WHEN** a session's `diff.finalized` event arrives (agent turn complete)
- **THEN** the diff panel shows added/removed counts and the changed-file list
- **AND** an "Approve diff & open PR" button is visible but requires interaction

#### Scenario: PR not opened without approval
- **WHEN** a session reaches `done` state
- **THEN** no PR is opened automatically
- **AND** the PR only opens after the engineer activates the approve-diff action

#### Scenario: Diff panel shows patch when available
- **WHEN** the `DiffSummary` returned by `getDiff()` includes a `patch` field
- **THEN** the diff panel renders a syntax-highlighted unified diff alongside the summary counts

### Requirement: Revert and restore via harness rescue
`capability: rescue` · `delta: ADDED (feat/session-detail-and-rescue)`

The system SHALL provide revert and restore controls in the session detail. Activating revert SHALL call the adapter's `revert` capability (mapped to `POST /session/:id/revert` in OpenCode, `docs/analysis/opencode-integration-guide.md` §Revert), rolling the session back to the git snapshot before a specified turn. Restore (unrevert) SHALL call the adapter's `unrevert` capability. Both actions SHALL be confirmed with a destructive-action prompt before execution.

A revert is a rescue path, not a routine action; it SHALL be visually distinct (e.g. a collapsed rescue section) to avoid accidental activation. The adapter's revert/unrevert methods SHALL be added to the `HarnessAdapter` interface in `@arke/contracts/src/adapter.ts` as optional capabilities.

#### Scenario: Revert rolls back to a prior checkpoint
- **WHEN** the engineer confirms a revert to a specific turn
- **THEN** `adapter.revert({ sessionId, messageId })` is called via the coordinator
- **AND** the transcript reflects the rolled-back state in the next update

#### Scenario: Revert requires explicit confirmation
- **WHEN** the engineer activates the revert control
- **THEN** a destructive-action prompt appears naming the target checkpoint
- **AND** the revert is NOT executed until the engineer confirms

#### Scenario: Agent failure surfaces as needs-human
- **WHEN** the adapter emits a `session.status` event with `status: "error"` for this session (NFR-4)
- **THEN** the card moves to the `needs-human` column on the board (SPEC-010)
- **AND** the session detail shows the error detail and offers the revert/restore controls as rescue options

---

### Requirement: Promote intent change to specification
`capability: session-detail` · `delta: ADDED (feat/session-detail-and-rescue)`

The system SHALL provide an action in the session detail that promotes a mid-delivery change of intent back into the open specification, drops the spec's status to `in-review`, and pauses the current session pending re-approval. The promote action SHALL write the intent change as a new requirement stub to the spec file on the feature branch via a coordinator command, not via a direct filesystem write from the client.

#### Scenario: Intent promoted and spec drops to in-review
- **WHEN** the engineer triggers "Promote to spec" and enters the intent change
- **THEN** the coordinator writes the change to the spec file on the branch
- **AND** emits a `spec.status` event with `status: "in-review"`
- **AND** the session is paused (no further agent dispatch until re-approved)

#### Scenario: Paused session is visible on the board
- **WHEN** a session is paused following a promote action
- **THEN** the session's card appears in the `review` column (derived from `in-review` spec status)
- **AND** the card does not show `needs-human`

## Design

### Architectural decision

Session detail is a read-only projection of the coordinator's event stream for a single session id, augmented by on-demand REST calls to the adapter for `getDiff()` and `getTodos()`. All mutations — steering, permissions, revert, promote — flow through `ArkeTransport.send()` to the coordinator and onwards to the adapter, never via direct adapter calls from the client. This preserves the invariant that the browser is never in the credential path (NFR-1) and that every governed action is traceable.

### Target architecture

```
ArkeTransport (SPEC-003)
  └─▶ DomainEvent stream filtered to sessionId
        └─▶ useSession(sessionId) hook
              ├─▶ transcript: TranscriptEntry[]  ← from CardState.transcript (SPEC-003)
              ├─▶ todos: TodoItem[]              ← from todo.updated events
              └─▶ diff: DiffSummary             ← from diff.finalized events

  <SessionDetail>
    ├─▶ <Transcript>         (virtualized, streaming delta, tool-call groups)
    ├─▶ <TodoList>
    ├─▶ <DiffPanel>          (approve-diff gate; syntax-highlighted patch)
    ├─▶ <MessageComposer>    (disabled while overlay open)
    ├─▶ <PermissionOverlay>  (SPEC-012 component, rendered here too)
    └─▶ <RescueSection>      (collapsed; revert / unrevert controls)
```

Git checkpoints (one per agent turn) are the revert targets, matching the T3 per-turn-snapshot pattern (`docs/analysis/t3code-learnings.md` §Checkpoints). The OpenCode adapter already exposes `POST /session/:id/revert` and `/unrevert` (`docs/analysis/opencode-integration-guide.md` §Revert).

### Data model

`useSession()` derives its state entirely from the coordinator stream; no client-side persistence. The hook maintains:

```typescript
interface SessionDetailState {
  transcript: TranscriptEntry[];     // from CardState.transcript, kept in sync
  todos: TodoItem[];                 // folded from todo.updated events
  diff: DiffSummary | null;          // from diff.finalized events
  sessionStatus: SessionStatus;
  needsHuman: boolean;
  pendingPermission: PermissionAskedEvent | null;
  pendingElicitation: ElicitationEvent | null;  // new event type; see Open questions
}
```

`ElicitationEvent` is a new domain event (question asked by the agent) that should be added to `@arke/contracts/src/events.ts` as part of this spec's implementation; it maps to OpenCode's `question.asked` event (`docs/analysis/opencode-integration-guide.md` §Questions).

**`HarnessAdapter` additions** (optional capabilities):

```typescript
revert?(ref: SessionRef, messageId: string): Promise<void>;
unrevert?(ref: SessionRef): Promise<void>;
```

### Interfaces and contracts

- Reads `TranscriptEntry[]`, `CardState`, `DomainEvent` from `@arke/contracts`.
- Uses `ArkeTransport.send()` for all mutations (steering, permission replies, revert, promote).
- `getDiff()` on `HarnessAdapter` is already defined; `revert`/`unrevert` are new optional methods.
- Route: `/sessions/:sessionId` or a right-panel layout alongside the board; navigation from `<BoardCard>` (SPEC-010).
- `ElicitationEvent` (`question.asked`, `question.replied`, `question.rejected`) added to the `DomainEvent` union.

### Cross-cutting
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.

**Security (NFR-1):** client never calls the adapter directly; all commands flow through `ArkeTransport → coordinator → adapter`.

**Performance (NFR-3):** transcript uses virtualized rendering (anchored scroll) so long sessions with thousands of parts do not degrade. Only the current session's events are processed in the `useSession()` hook; events for other sessions are filtered out at the hook boundary.

**Resilience (NFR-4):** agent failure (`session.status: error`) is surfaced within the session detail as a rescue prompt; the board independently moves the card to `needs-human` (SPEC-010). The session detail SHALL not crash on an error event; it SHALL render the error message and the rescue controls.

**Observability:** every coordinator command (steering message, revert, promote) is written to the append-only trace by the coordinator before the adapter call, providing a complete action log.

**Accessibility:** the diff panel uses `aria-label` on added/removed counts; the composer textarea has a `<label>`; destructive-action prompts have `role="alertdialog"` with focus trapped until resolved.

### Testing

- **Unit (hook):** transcript accumulates deltas from `message.part`; closes on `turn.quiescent`; `todo.updated` replaces the list; `diff.finalized` updates the diff; `permission.asked` sets `pendingPermission`; `session.status: error` sets error state.
- **Unit (components):** `<Transcript>` renders streaming indicator while `isStreaming`; `<DiffPanel>` hides approve button when no diff; `<MessageComposer>` is disabled when overlay is open; `<RescueSection>` shows confirmation before calling revert.
- **Integration:** `MockAdapter` scripted sequence (SPEC-003) ending in an error event → session detail shows rescue controls → revert command → transcript rolls back.
- **Manual (NFR-4):** kill the adapter mid-run; confirm card moves to `needs-human` on the board; open session detail; confirm error is shown with rescue options; confirm revert.

### Definition of done

All scenarios pass; TypeScript strict-mode compiles; the diff-review gate cannot be bypassed (integration test); agent failure surfaces correctly on board and in detail; a reviewer has signed off; SPEC-002 and SPEC-003 are merged (adapter and transport dependencies).

## Decision log

| # | Decision | Rationale |
|---|----------|-----------|
| 1 | All mutations through `ArkeTransport.send()`, never direct adapter calls from the client | Preserves the invariant that the browser is not in the credential path; ensures every action is traced |
| 2 | Revert/unrevert are optional adapter capabilities, not required | Not every harness supports git checkpoints; the rescue section degrades gracefully when the capability is absent |
| 3 | Diff approval is a hard gate before PR creation | Enforces propose-decide-execute for code output; a diff is the agent's proposal and the human must decide before execution |
| 4 | `ElicitationEvent` added to `DomainEvent` union (not a separate channel) | Elicitation is a governed action like permission — it must be traced and visible on the same stream the board reads |
| 5 | "Promote to spec" writes via coordinator command, not filesystem from client | Maintains direction of truth: the spec file in git is the source of truth; client mutations go through the coordinator |

## Open questions

- `ElicitationEvent` maps to OpenCode's `question.asked` / `question.replied` / `question.rejected`. Should this be defined in this spec or delegated to SPEC-012 (permissions & elicitation) to keep the elicitation handling co-located with permission handling?
- Should the `<Transcript>` pagination (virtualization) use a fixed-height container or anchor scroll at the bottom on new messages? T3 uses anchored scroll; this may need a `scrollIntoView` guard to avoid fighting user-initiated upward scroll.
- How many git checkpoints are retained? OpenCode snapshots per message turn; should Arke expose a checkpoint picker or only "revert to last good"?
