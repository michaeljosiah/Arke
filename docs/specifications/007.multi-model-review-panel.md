# Multi-model review panel

## Why
A specification authored on a single model inherits that model's blind spots. Cross-model adversarial review grounded in the actual codebase catches what the authoring model misses — different models have different failure modes, and running them independently against the same source stops mutual reinforcement. This covers FR-16 and provides the human-adjudicated critique gate that stands between a draft specification and approval.

## What changes
- ADDED review-panel — the engineer can convene a configurable multi-model review panel at any point during authoring; the panel runs two or more independent reviewers on different models, presents their critiques side by side with agreement and divergence marked, attaches each issue to the relevant specification section, and lets the human accept, dismiss, or send a section back for revision

---

## Requirements

### Requirement: Panel convened on demand, required before finalisation
`capability: review-panel` · `delta: ADDED (feat/multi-model-review-panel)`

The system SHALL allow the engineer to convene a review panel at any point during authoring, and SHALL require that at least one panel has been run and adjudicated before the specification status advances to `approved`; this gate SHALL be enforced by the coordinator (not only by the client UI) so that an `approveDraft` command issued without a completed review is rejected server-side with a `review.gate-failed` error.

#### Scenario: Review convened mid-draft
- **WHEN** the working specification contains at least one requirement and its status is `draft`
- **THEN** the engineer can convene a review panel
- **AND** the panel operates on the current working file, not on an approved version

#### Scenario: Finalisation blocked without a completed review — coordinator enforced
- **WHEN** the coordinator receives an `approveDraft` command for a specification
- **THEN** the coordinator checks that at least one review panel has been completed and adjudicated for the current draft (tracked in the coordinator's durable state)
- **AND** if no completed review exists, the coordinator rejects the command with `{ type: "review.gate-failed", specId, reason: "no completed review panel" }` and does NOT commit the file or emit `spec.status`

#### Scenario: Client approval action reflects gate state
- **WHEN** no completed review exists for the current draft
- **THEN** the client approval action in the cockpit is disabled with an explanatory prompt
- **AND** the gate is also enforced server-side, so a direct `approveDraft` command is rejected regardless of the client state

### Requirement: Two or more reviewers on pairwise-distinct models
`capability: review-panel` · `delta: ADDED (feat/multi-model-review-panel)`

A panel SHALL run at least two reviewer agents; every pair of configured reviewers SHALL resolve to distinct concrete models — the system SHALL reject any panel configuration where any two reviewers resolve to the same model, not only configurations where all reviewers resolve identically; the system SHALL also reject a panel configuration where the registry cannot supply enough distinct capable models to satisfy the configured reviewer count.

#### Scenario: Default two-reviewer panel uses distinct models
- **WHEN** a panel is convened with default configuration
- **THEN** `reviewer-a` runs on one model (e.g. the capable instance registered for `claude-local`)
- **AND** `reviewer-b` runs on a different model (e.g. the capable instance registered for `opencode-local`)
- **AND** the panel header identifies each reviewer's role and concrete model

#### Scenario: Panel rejected if any two reviewers resolve to the same model
- **WHEN** the panel configuration is submitted and any two configured reviewer roles resolve to the same concrete model (e.g. [A, A, B] in a three-reviewer panel)
- **THEN** the system refuses to start the panel
- **AND** returns `panel.config-error` naming the conflicting pair

#### Scenario: Panel rejected if registry cannot supply enough distinct models
- **WHEN** the panel requires N reviewers but the registry has fewer than N distinct capable models available
- **THEN** the system refuses to start the panel and emits `panel.config-error { reason: "insufficient distinct capable models: need N, have M" }`
- **AND** the cockpit surfaces the error with guidance on configuring additional harness instances

#### Scenario: Each reviewer is grounded in the source and blind to peers
- **WHEN** a reviewer agent runs
- **THEN** it receives the current working specification text and a reference to the project's grounding context (codebase summary, vendored references under `.repos/`)
- **AND** it is not given prior critiques from any other reviewer before producing its own output

### Requirement: Critiques presented side by side with agreement and divergence marked
`capability: review-panel` · `delta: ADDED (feat/multi-model-review-panel)`

The panel SHALL present each reviewer's critique beside the others in a single view; the system SHALL mark issues where reviewers agree and flag issues where they diverge, and SHALL attach each issue to the specification section it concerns.

#### Scenario: Side-by-side critique columns
- **WHEN** a panel completes
- **THEN** the review panel screen shows one column per reviewer
- **AND** each column contains that reviewer's issues in the order they arise in the specification

#### Scenario: Agreed issues are marked
- **WHEN** two or more reviewers raise an issue on the same specification section with substantially the same concern
- **THEN** the system marks those issues as agreed and visually groups them

#### Scenario: Divergent issues are flagged
- **WHEN** reviewers raise contradictory concerns about the same section
- **THEN** the system flags the divergence and presents both positions without resolving it
- **AND** the engineer is expected to adjudicate

#### Scenario: Each issue is anchored to a section
- **WHEN** a reviewer raises an issue
- **THEN** the issue is attached to the specification section it concerns (e.g. `requirements > Requirement: Split authoring surface`)
- **AND** clicking the issue scrolls the specification preview to that section

### Requirement: One reviewer failing does not abort the panel
`capability: review-panel` · `delta: ADDED (feat/multi-model-review-panel)`

If one reviewer session errors or times out, the panel SHALL mark that reviewer as `error`, continue displaying the other reviewers' output, and allow adjudication of the received issues; the coordinator SHALL emit `session.status { status: "error" }` for the failed reviewer and a `panel.reviewer-error` event identifying which reviewer failed and the reason; the panel SHALL not be silently blocked or left in an indeterminate state.

#### Scenario: One reviewer errors — panel continues
- **WHEN** one reviewer session errors during a panel run
- **THEN** the coordinator emits `panel.reviewer-error { panelId, reviewerRole, reason }` and marks that reviewer's column as `error` in the panel view
- **AND** the remaining reviewers' issues are presented and remain adjudicatable
- **AND** the panel may be completed (and the review gate satisfied) with the remaining reviewers' output

#### Scenario: All reviewers error — panel fails
- **WHEN** all reviewer sessions error or time out
- **THEN** the panel transitions to `failed` state and emits `panel.complete { panelId, status: "failed" }`
- **AND** the review gate is NOT satisfied by a failed panel; the engineer must re-convene

### Requirement: Human adjudicates — accept, dismiss, or send back
`capability: review-panel` · `delta: ADDED (feat/multi-model-review-panel)`

The system SHALL provide three adjudication actions per issue: accept (feeds the accepted critique back into the working draft via the authoring agent), dismiss (records the dismissal with optional rationale), and send back (marks the section as requiring revision, routing it to the authoring agent for a targeted rewrite); an accepted critique is applied on the current working file at the moment of acceptance — if the working file has changed since the issue was raised (due to a concurrent human edit or agent write), the coordinator SHALL detect the change and warn the engineer before routing the critique to the authoring agent.

#### Scenario: Accepting a critique feeds it into the draft via authoring agent
- **WHEN** the engineer accepts a reviewer issue
- **THEN** the system forwards the accepted critique as a prompt to the `spec-author` agent with the section in context
- **AND** the agent revises the relevant section in the working file
- **AND** the accepted issue is marked resolved in the panel

#### Scenario: Stale working file detected before routing accepted critique
- **WHEN** the engineer accepts an issue and the working file's Requirements section has changed since the panel was run (detected by comparing the section hash at panel-start against the current file)
- **THEN** the coordinator warns the engineer that the file has changed and requests confirmation before routing the critique to the authoring agent
- **AND** the engineer may confirm (critique is applied to the current file) or cancel

#### Scenario: Dismissing a critique records the decision
- **WHEN** the engineer dismisses a reviewer issue
- **THEN** the dismissal is recorded in the trace with the specId, section, issue text, and optional rationale
- **AND** the dismissed issue is marked closed in the panel and does not reappear if the panel is re-run on the same section without a content change (content identity determined by section hash, not section key alone)

#### Scenario: Sending back marks a section for revision
- **WHEN** the engineer chooses "send back" on a section
- **THEN** the section is flagged as requiring revision in the cockpit
- **AND** the cockpit highlights the flagged section and surfaces the reviewer's concern as context for the next authoring turn

### Requirement: Number of reviewers and their models are configurable
`capability: review-panel` · `delta: ADDED (feat/multi-model-review-panel)`

The system SHALL allow the engineer to configure the number of reviewers (minimum two) and the harness instance each reviewer runs on, overriding the default roster-driven resolution, and SHALL persist the configuration per project.

#### Scenario: Three-reviewer panel
- **WHEN** the engineer configures three reviewers
- **THEN** the panel runs three independent reviewer agents, each on a pairwise-distinct instance/model
- **AND** the side-by-side view shows three columns

#### Scenario: Custom instance override
- **WHEN** the engineer pins a reviewer to a specific harness instance
- **THEN** the panel routes that reviewer to the pinned instance and logs the override in the trace
- **AND** the pairwise distinct-model constraint is still enforced across all configured reviewers

### Requirement: Reviewers are read-only and never commit
`capability: review-panel` · `delta: ADDED (feat/multi-model-review-panel)`

Reviewer agents (`reviewer-a`, `reviewer-b`, and any additional reviewer) SHALL be read-only; they SHALL propose critiques into the panel only; they SHALL never write to the repository, commit files, or perform any side-effecting harness action; the adapter SHALL enforce this at the harness permission layer (`edit: deny`, `bash: deny` in the agent profile), and the coordinator SHALL log a `policy.violation` trace event if a reviewer session attempts a write action, identifying the reviewer role and the attempted action.

#### Scenario: Reviewer write attempt is rejected at the harness layer
- **WHEN** a reviewer agent attempts a harness write (e.g. editing a file, committing)
- **THEN** the adapter rejects the action at the permission layer
- **AND** the coordinator logs a `policy.violation` trace event with the reviewer's role, the attempted action, and the panelId

#### Scenario: Critique flows back to the draft only through the authoring agent
- **WHEN** a critique is accepted
- **THEN** only the `spec-author` or `architect` authoring agent makes the resulting change in the repository
- **AND** the reviewer's identity is recorded in the trace as the source of the accepted critique

## Design

### Architectural decision
Reviewers are stateless, read-only harness sessions dispatched in parallel via `HarnessAdapter.dispatchAsync`. Their output arrives as transcript events on the coordinator event stream and is assembled into the panel view client-side. No reviewer session is granted write permissions; the adapter enforces this via the agent's permission profile in the harness (`.opencode/agents/reviewer-a.md` scopes `edit: deny`, `bash: deny`). The authoring agent applies accepted critiques, keeping the write path singular. Panel state is checkpointed to the trace at each adjudication so the coordinator can reconstruct a summary on restart, even though the live panel view is in-memory only.

The finalisation gate (no `approveDraft` without a completed panel) is enforced at the coordinator level — the coordinator checks its durable state before acting, not only the client UI.

### Target architecture
```
cockpit "Convene review" action → convenePanel { specId, branch }
    │
    ▼
coordinator
    ├── validate reviewer configuration:
    │       ├── resolve all reviewer roles to concrete models
    │       ├── check pairwise distinctness (any two same → panel.config-error)
    │       └── check registry has enough distinct capable models (else panel.config-error)
    ├── record panel start in durable state (panelId, specId, section hash at start)
    ├── for each configured reviewer role:
    │       ├── createSession (child of spec session, kind: "task")
    │       ├── dispatchAsync { agent: "reviewer-a"|"reviewer-b", tier: capable, parts: [spec text + grounding] }
    │       └── streams message.part / message.updated → review panel
    ├── on reviewer session error → emit panel.reviewer-error; continue with remaining
    └── emits panel.complete when all reviewer sessions finish (or fail)

review panel screen (packages/client/src/screens/review-generation.tsx)
    ├── side-by-side columns, one per reviewer (error column marked)
    ├── agreement detector: cross-reference section hashes across columns (not section key alone)
    ├── adjudication actions per issue (accept / dismiss / send back)
    │       ├── accept → stale-file check (section hash now vs. at panel-start) → confirm if changed → sendMessage to spec-author
    │       ├── dismiss → trace.write { kind: "review.dismiss", section, sectionHash, … }
    │       └── send back → emits flagged-section event → cockpit highlights section
    └── panel completion indicator + "re-run panel" action

harness (OpenCode)
    ├── reviewer-a: permission edit: deny, bash: deny
    └── reviewer-b: permission edit: deny, bash: deny
```

### Data model
Panel state is checkpointed to the trace. The coordinator maintains in-memory panel state for the lifetime of the review, with adjudications written durably at each step:

```typescript
interface ReviewPanel {
  panelId: string;
  specId: string;
  startedAt: number;
  requirementsSectionHashAtStart: string;  // for stale-file detection on accept
  reviewers: Array<{
    role: "reviewer-a" | "reviewer-b" | string;
    sessionId: string;
    instanceId: string;       // resolved harness instance
    model: string;            // concrete model name
    status: SessionStatus | "error";
    errorReason?: string;
    issues: ReviewIssue[];
  }>;
  adjudications: AdjudicationRecord[];
  status: "running" | "complete" | "failed";
}

interface ReviewIssue {
  issueId: string;
  reviewerRole: string;
  section: string;            // spec anatomy key + requirement name
  sectionHash: string;        // hash of the section content at issue-raise time
  text: string;
  severity: "blocking" | "suggestion" | "question";
  agreedBy?: string[];        // other reviewer roles that raised the same concern
}

interface AdjudicationRecord {
  issueId: string;
  action: "accepted" | "dismissed" | "sent-back";
  rationale?: string;
  ts: number;
}
```

The panel state is written to the trace (`trace.write`) at each adjudication and at panel completion. On coordinator restart, the in-memory panel view is lost, but the trace holds all adjudication records; the coordinator can reconstruct whether a panel was completed (for the review gate) from the trace.

### Interfaces and contracts
**Coordinator commands (client → coordinator):**
```typescript
// start a review panel
{ type: "convenePanel", specId: string, branch: string, reviewers?: Array<{ role: string; instanceId?: string }> }

// adjudicate an issue
{ type: "adjudicateIssue", panelId: string, issueId: string, action: "accepted" | "dismissed" | "sent-back", rationale?: string }
```

**New domain events (coordinator → client):**
```typescript
// panel started
{ type: "panel.started", panelId: string, specId: string, reviewers: Array<{ role: string; model: string }> }

// issue raised by a reviewer (parsed from message.updated)
{ type: "panel.issue", panelId: string, issueId: string, reviewerRole: string, section: string, sectionHash: string, text: string, severity: string }

// agreement detected across reviewers
{ type: "panel.agreed", panelId: string, issueIds: string[], section: string }

// panel complete (all reviewer sessions done or failed)
{ type: "panel.complete", panelId: string, status: "complete" | "failed", issueCount: number, adjudicatedCount: number }

// one reviewer errored — panel continues
{ type: "panel.reviewer-error", panelId: string, reviewerRole: string, reason: string }

// config error (e.g. any two reviewers same model; insufficient distinct models)
{ type: "panel.config-error", panelId: string, reason: string }

// stale-file warning on accept
{ type: "panel.stale-file-warning", panelId: string, issueId: string, specId: string }

// finalisation gate rejection (issued in response to approveDraft)
{ type: "review.gate-failed", specId: string, reason: string }
```

These event types are additions to the `DomainEvent` union in `@arke/contracts/src/events.ts`.

**Grounding payload:** the coordinator constructs the reviewer message from:
1. The current working specification text (read via `GET /spec/:specId/file` from SPEC-006).
2. A grounding summary drawn from the project's `.repos/` context (same mechanism as the authoring agents).

**Issue-extraction prompt:** the prompt used to parse `message.updated` text into structured `panel.issue` events is versioned alongside the coordinator source (not embedded as a string literal). The version is logged in the trace with each `panel.started` event. Changes to the prompt are treated as coordinator releases, not silent edits.

### Cross-cutting
- **Configuration:** all project configuration — the harness/model registry, coordinator settings, and integrations — lives in the single root file `.arke/config.json` (`ARKE_*` env vars override individual keys; runtime artefacts under `.arke/` are state, not config). See SPEC-005 for the model.
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.
- **Security (NFR-1, NFR-5):** reviewers run inside the coordinator's trusted host boundary; their harness sessions are scoped with `edit: deny`, `bash: deny`; the client sends only the `convenePanel` command, not raw spec text; reviewer write attempts log a `policy.violation` trace event.
- **Auditability (NFR-2):** every adjudication (accept / dismiss / send-back) is written to the append-only trace with the issueId, section, sectionHash, rationale, and acting user; panel completion and reviewer errors are also traced; the issue-extraction prompt version is recorded at panel start.
- **Performance (NFR-3):** reviewer sessions are dispatched in parallel via `dispatchAsync`; the panel view updates incrementally as `panel.issue` events arrive, rather than waiting for all reviewers to finish.
- **Resilience (NFR-8):** if one reviewer session errors, the panel marks that reviewer as `error` and continues with the remaining reviewers (see requirement above); on coordinator restart, the review-gate state (whether a completed panel exists for a given spec) is reconstructed from the durable trace, so the gate survives restarts; the live panel view is in-memory and is lost on restart, but all adjudications are durable.
- **Multi-project scoping (SPEC-018):** the coordinator supervises N isolated **project contexts**, each keyed by `projectId`. Everything this spec places on "the coordinator" is per–project context; each client connection resolves against its **active project** and that project's safe root + `docs/specifications/`. See [SPEC-018](018.multi-project-workspaces.md). Concretely, each reviewer session and the grounding context (working spec text + `.repos/` summary) are scoped to the **active project** context.

### Testing
- Unit: pairwise distinct-model validator (same model in any pair → error; three reviewers [A, A, B] → error; [A, B, C] all distinct → pass); registry-insufficiency error (need 3 distinct, have 2 → error); `panel.agreed` detection by section hash (not key); stale-file detection (section hash before vs. after); adjudication trace write including sectionHash; review-gate check blocks `approveDraft` when trace has no completed panel.
- Integration: convene a panel against a MockAdapter producing structured review output; verify `panel.issue` events arrive; accept one issue with matching section hash → `sendMessage` fires; accept one issue with changed section hash → `panel.stale-file-warning` fires; dismiss one issue and verify trace write with sectionHash; reviewer session errors mid-panel → `panel.reviewer-error` emitted, remaining column adjudicatable; coordinator restart → review-gate state reconstructed from trace.
- Manual: convene a panel on a real working draft with two live reviewer sessions; observe side-by-side columns; accept one issue and watch the cockpit preview update; verify approval is blocked until panel completes.

### Definition of done
All scenarios pass; the two-reviewer panel runs to completion with pairwise distinct models and distinct critiques in parallel; [A, A, B] three-reviewer configuration is rejected; registry-insufficiency error is surfaced; adjudication of at least one issue of each kind (accept / dismiss / send-back) is verified end-to-end; the approval guard (coordinator-enforced) blocks finalisation until a panel is completed; reviewer write attempts are rejected and traced; stale-file warning fires on accept when section has changed; a reviewer has signed off.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Reviewers dispatched via `dispatchAsync` in parallel | Reviews are independent; running them in parallel halves the wall-clock time and guarantees neither reviewer sees the other's output before producing its own |
| 2 | `panel.issue` events parsed from `message.updated` by the coordinator, not by the client | The coordinator is the normalisation boundary; the client receives already-structured issue events rather than raw markdown |
| 3 | Distinct-model check enforced at the coordinator, not the client | The registry resolution happens server-side; the client does not know concrete model ids |
| 4 | Accepted critiques flow back through `spec-author`, not through a reviewer write | Keeps the write path singular and attributable; the reviewer's role remains advisory |
| 5 | Pairwise distinctness enforced (any two same → reject), not only all-same | Three reviewers [A, A, B] is a meaningful gap in independence; pairwise is the correct invariant |
| 6 | Review gate enforced at the coordinator (not only client UI) | A direct `approveDraft` command bypasses the UI; server-side enforcement is the only reliable gate |
| 7 | Adjudications checkpointed to trace; review-gate reconstructed from trace on restart | In-memory panel state cannot survive coordinator restart; the trace is the durable source for gate enforcement |
| 8 | Stale-file warning on accept when section hash changed since panel start | Prevents an accepted critique from being applied to a section that no longer matches the reviewed text, causing a confused authoring-agent rewrite |
| 9 | Issue-extraction prompt versioned alongside coordinator source; version logged at panel start | Prompt drift is a silent failure mode; versioning makes prompt changes explicit and auditable |

## Open questions
- How should the issue-extraction prompt be updated when the specification anatomy changes? A formal prompt registry (analogous to the agent roster) may be warranted in Phase 2.
- Should the `panel.agreed` heuristic be a semantic similarity check (embedding distance) or section-hash match? Starting with section-hash is simpler; upgrade to semantic if false negatives are frequent.
- Should a panel that completed with one reviewer in error state satisfy the review gate? Current design: yes (partial completion is accepted). May need policy knob.
- Should dismissed issues carry a TTL tied to section content, so a substantive rewrite resets dismissals automatically?
