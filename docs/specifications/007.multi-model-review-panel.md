# Multi-model review panel

## Why
A specification authored on a single model inherits that model's blind spots. Cross-model adversarial review grounded in the actual codebase catches what the authoring model misses — different models have different failure modes, and running them independently against the same source stops mutual reinforcement. This covers FR-16 and provides the human-adjudicated critique gate that stands between a draft specification and approval.

## What changes
- ADDED review-panel — the engineer can convene a configurable multi-model review panel at any point during authoring; the panel runs two or more independent reviewers on different models, presents their critiques side by side with agreement and divergence marked, attaches each issue to the relevant specification section, and lets the human accept, dismiss, or send a section back for revision

---

## Requirements

### Requirement: Panel convened on demand, required before finalisation
`capability: review-panel` · `delta: ADDED (feat/multi-model-review-panel)`

The system SHALL allow the engineer to convene a review panel at any point during authoring, and SHALL require that at least one panel has been run and adjudicated before the specification status advances to `approved`.

#### Scenario: Review convened mid-draft
- **WHEN** the working specification contains at least one requirement and its status is `draft`
- **THEN** the engineer can convene a review panel
- **AND** the panel operates on the current working file, not on an approved version

#### Scenario: Finalisation blocked without a completed review
- **WHEN** the engineer attempts to advance the specification to `approved`
- **THEN** the system checks that at least one review panel has been completed and adjudicated on the current draft
- **AND** if no completed review exists, the approval action is disabled with an explanatory prompt

### Requirement: Two or more reviewers on distinct models
`capability: review-panel` · `delta: ADDED (feat/multi-model-review-panel)`

A panel SHALL run at least two reviewer agents, and the two SHALL resolve to different concrete models; the system SHALL reject a panel configuration where all reviewers resolve to the same model.

#### Scenario: Default two-reviewer panel uses distinct models
- **WHEN** a panel is convened with default configuration
- **THEN** `reviewer-a` runs on one model (e.g. the capable instance registered for `claude-local`)
- **AND** `reviewer-b` runs on a different model (e.g. the capable instance registered for `opencode-local`)
- **AND** the panel header identifies each reviewer's role and concrete model

#### Scenario: Panel rejected if all reviewers resolve to the same model
- **WHEN** the panel configuration is submitted and all configured reviewer roles resolve to the same concrete model
- **THEN** the system refuses to start the panel
- **AND** returns an error naming the conflict

#### Scenario: Each reviewer is grounded in the source
- **WHEN** a reviewer agent runs
- **THEN** it receives the current working specification text and a reference to the project's grounding context (codebase summary, vendored references under `.repos/`)
- **AND** it is not given prior critiques from the other reviewer before producing its own

### Requirement: Critiques presented side by side with agreement and divergence marked
`capability: review-panel` · `delta: ADDED (feat/multi-model-review-panel)`

The panel SHALL present each reviewer's critique beside the others in a single view; the system SHALL mark issues where reviewers agree and flag issues where they diverge, and SHALL attach each issue to the specification section it concerns.

#### Scenario: Side-by-side critique columns
- **WHEN** a panel completes
- **THEN** the review panel screen shows one column per reviewer
- **AND** each column contains that reviewer's issues in the order they arise in the specification

#### Scenario: Agreed issues are marked
- **WHEN** two or more reviewers raise an issue on the same specification section with substantially the same concern
- **THEN** the system marks those issues as agreed and visually groups them

#### Scenario: Divergent issues are flagged
- **WHEN** reviewers raise contradictory concerns about the same section
- **THEN** the system flags the divergence and presents both positions without resolving it
- **AND** the engineer is expected to adjudicate

#### Scenario: Each issue is anchored to a section
- **WHEN** a reviewer raises an issue
- **THEN** the issue is attached to the specification section it concerns (e.g. `requirements > Requirement: Split authoring surface`)
- **AND** clicking the issue scrolls the specification preview to that section

### Requirement: Human adjudicates — accept, dismiss, or send back
`capability: review-panel` · `delta: ADDED (feat/multi-model-review-panel)`

The system SHALL provide three adjudication actions per issue: accept (feeds the accepted critique back into the working draft via the authoring agent), dismiss (records the dismissal with optional rationale), and send back (marks the section as requiring revision, routing it to the authoring agent for a targeted rewrite).

#### Scenario: Accepting a critique feeds it into the draft
- **WHEN** the engineer accepts a reviewer issue
- **THEN** the system forwards the accepted critique as a prompt to the `spec-author` agent with the section in context
- **AND** the agent revises the relevant section in the working file
- **AND** the accepted issue is marked resolved in the panel

#### Scenario: Dismissing a critique records the decision
- **WHEN** the engineer dismisses a reviewer issue
- **THEN** the dismissal is recorded in the trace with the specId, section, issue text, and optional rationale
- **AND** the dismissed issue is marked closed in the panel and does not reappear if the panel is re-run on the same section without a content change

#### Scenario: Sending back marks a section for revision
- **WHEN** the engineer chooses "send back" on a section
- **THEN** the section is flagged as requiring revision in the cockpit
- **AND** the cockpit highlights the flagged section and surfaces the reviewer's concern as context for the next authoring turn

### Requirement: Number of reviewers and their models are configurable
`capability: review-panel` · `delta: ADDED (feat/multi-model-review-panel)`

The system SHALL allow the engineer to configure the number of reviewers (minimum two) and the harness instance each reviewer runs on, overriding the default roster-driven resolution, and SHALL persist the configuration per project.

#### Scenario: Three-reviewer panel
- **WHEN** the engineer configures three reviewers
- **THEN** the panel runs three independent reviewer agents, each on a distinct instance
- **AND** the side-by-side view shows three columns

#### Scenario: Custom instance override
- **WHEN** the engineer pins a reviewer to a specific harness instance
- **THEN** the panel routes that reviewer to the pinned instance and logs the override in the trace
- **AND** the distinct-model constraint is still enforced across the configured reviewers

### Requirement: Reviewers are read-only and never commit
`capability: review-panel` · `delta: ADDED (feat/multi-model-review-panel)`

Reviewer agents (`reviewer-a`, `reviewer-b`, and any additional reviewer) SHALL be read-only; they SHALL propose critiques into the panel only; they SHALL never write to the repository, commit files, or perform any side-effecting harness action.

#### Scenario: Reviewer write attempt is rejected
- **WHEN** a reviewer agent attempts a harness write (e.g. editing a file, committing)
- **THEN** the adapter rejects the action
- **AND** the coordinator logs a policy violation in the trace with the reviewer's role and the attempted action

#### Scenario: Critique flows back to the draft only through the authoring agent
- **WHEN** a critique is accepted
- **THEN** only the `spec-author` or `architect` authoring agent makes the resulting change in the repository
- **AND** the reviewer's identity is recorded in the trace as the source of the accepted critique

## Design

### Architectural decision
Reviewers are stateless, read-only harness sessions dispatched in parallel via `HarnessAdapter.dispatchAsync`. Their output arrives as transcript events on the coordinator event stream and is assembled into the panel view client-side. No reviewer session is granted write permissions; the adapter enforces this via the agent's permission profile in the harness (`.opencode/agents/reviewer-a.md` scopes `edit: deny`). The authoring agent applies accepted critiques, keeping the write path singular.

### Target architecture
```
cockpit "Convene review" action
    │
    ▼
coordinator
    ├── for each configured reviewer role:
    │       ├── createSession (child of spec session, kind: "task")
    │       ├── dispatchAsync { agent: "reviewer-a"|"reviewer-b", tier: capable, parts: [spec text + grounding] }
    │       └── streams message.part / message.updated → review panel
    └── emits session.status (done) when all reviewers finish

review panel screen (packages/client/src/screens/review-generation.tsx)
    ├── side-by-side columns, one per reviewer
    ├── agreement detector: cross-reference section anchors across columns
    ├── adjudication actions per issue (accept / dismiss / send back)
    │       ├── accept → sendMessage to spec-author with critique as context
    │       ├── dismiss → trace.write { kind: "review.dismiss", … }
    │       └── send back → emits flagged-section event → cockpit highlights section
    └── panel completion indicator + "re-run panel" action

harness (OpenCode)
    ├── reviewer-a: permission edit: deny, bash: deny
    └── reviewer-b: permission edit: deny, bash: deny
```

The registry (005) is the authority on which harness instance each reviewer role binds to. The two-model constraint is checked by the coordinator at panel-start time: it resolves all reviewer roles and compares concrete model names; if they are all identical it returns a `panel.config-error` event and does not start sessions.

### Data model
No persistent entities beyond the trace. The coordinator maintains in-memory panel state for the lifetime of the review:

```typescript
interface ReviewPanel {
  panelId: string;
  specId: string;
  startedAt: number;
  reviewers: Array<{
    role: "reviewer-a" | "reviewer-b" | string;
    sessionId: string;
    instanceId: string;       // resolved harness instance
    model: string;            // concrete model name
    status: SessionStatus;
    issues: ReviewIssue[];
  }>;
  adjudications: AdjudicationRecord[];
}

interface ReviewIssue {
  issueId: string;
  reviewerRole: string;
  section: string;            // spec anatomy key + requirement name
  text: string;
  severity: "blocking" | "suggestion" | "question";
  agreedBy?: string[];        // other reviewer roles that raised the same concern
}

interface AdjudicationRecord {
  issueId: string;
  action: "accepted" | "dismissed" | "sent-back";
  rationale?: string;
  ts: number;
}
```

The panel state is written to the trace (`trace.write`) at each adjudication and at panel completion.

### Interfaces and contracts
**Coordinator commands (client → coordinator):**
```typescript
// start a review panel
{ type: "convenePanel", specId: string, branch: string, reviewers?: Array<{ role: string; instanceId?: string }> }

// adjudicate an issue
{ type: "adjudicateIssue", panelId: string, issueId: string, action: "accepted" | "dismissed" | "sent-back", rationale?: string }
```

**New domain events (coordinator → client):**
```typescript
// panel started
{ type: "panel.started", panelId: string, specId: string, reviewers: Array<{ role: string; model: string }> }

// issue raised by a reviewer (parsed from message.updated)
{ type: "panel.issue", panelId: string, issueId: string, reviewerRole: string, section: string, text: string, severity: string }

// agreement detected across reviewers
{ type: "panel.agreed", panelId: string, issueIds: string[], section: string }

// panel complete (all reviewer sessions done)
{ type: "panel.complete", panelId: string, issueCount: number, adjudicatedCount: number }

// config error (e.g. all reviewers same model)
{ type: "panel.config-error", panelId: string, reason: string }
```

These event types are additions to the `DomainEvent` union in `@arke/contracts/src/events.ts`.

**Grounding payload:** the coordinator constructs the reviewer message from:
1. The current working specification text (read via `GET /spec/:specId/file` from 006).
2. A grounding summary drawn from the project's `.repos/` context (same mechanism as the authoring agents).

### Cross-cutting
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.
- **Security (NFR-1, NFR-5):** reviewers run inside the coordinator's trusted host boundary; their harness sessions are scoped with `edit: deny`; the client sends only the `convenePanel` command, not raw spec text.
- **Auditability (NFR-2):** every adjudication (accept / dismiss / send-back) is written to the append-only trace with the issueId, section, rationale, and acting user.
- **Performance (NFR-3):** reviewer sessions are dispatched in parallel via `dispatchAsync`; the panel view updates incrementally as `panel.issue` events arrive, rather than waiting for all reviewers to finish.
- **Resilience (NFR-8):** if one reviewer session errors, the panel marks that reviewer as `error` and continues with the remaining reviewers; the coordinator emits `session.status { status: "error" }` and the panel surface indicates which reviewer failed without blocking adjudication of the others.

### Testing
- Unit: distinct-model validator (same model → error; different models → pass; three reviewers, two same → error); `panel.agreed` detection logic; adjudication trace write.
- Integration: convene a panel against a MockAdapter producing structured review output; verify `panel.issue` events arrive; accept one issue and verify `sendMessage` to spec-author fires with critique in context; dismiss one and verify trace write.

### Definition of done
All scenarios pass; the two-reviewer panel runs to completion with distinct models and distinct critiques in parallel; adjudication of at least one issue of each kind (accept / dismiss / send-back) is verified end-to-end; the approval guard blocks finalisation until a panel is completed; a reviewer has signed off.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Reviewers dispatched via `dispatchAsync` in parallel | Reviews are independent; running them in parallel halves the wall-clock time and guarantees neither reviewer sees the other's output before producing its own |
| 2 | `panel.issue` events parsed from `message.updated` by the coordinator, not by the client | The coordinator is the normalisation boundary; the client receives already-structured issue events rather than raw markdown |
| 3 | Distinct-model check enforced at the coordinator, not the client | The registry resolution happens server-side; the client does not know concrete model ids |
| 4 | Accepted critiques flow back through `spec-author`, not through a reviewer write | Keeps the write path singular and attributable; the reviewer's role remains advisory |

## Open questions
- How should the issue-extraction prompt be versioned and updated when the specification anatomy changes? The coordinator embeds it; a registry of prompts (similar to the agent roster) might be warranted in Phase 2.
- Should the `panel.agreed` heuristic be a semantic similarity check (embedding distance) or a section-key exact match? Starting with section-key match is simpler; upgrade to semantic if false negatives are frequent.
