# Deterministic projection and integrations registry

## Why
Writing to systems of record (GitHub Issues, Jira, Azure DevOps) is the moment the spec
turns into tracked work. If that write is ad hoc or free-form agent behaviour the result
is unpredictable, unaudited, and unrecoverable. This spec makes every system-of-record
write deterministic — a plugin that reacts to a known event the same way every time — and
pairs it with an integrations registry so teams know which connections are active, what
they enable, and what they are waiting for before projection can proceed.

## What changes
- ADDED projection — a harness plugin under `.opencode/plugins/` drives every system-of-record
  write deterministically, reacting to `generation.decided approved` and `spec.status` events
  and logging each write as a `projection.write` DomainEvent
- ADDED integrations-registry — an integrations registry on the coordinator tracks the
  status of GitHub, Jira, and Azure DevOps connections, confirms they are authorised on the
  host, and exposes their status to the client
- ADDED projection — a `tool.execute.before` policy hook prevents downstream edits from
  being written back into the specification
- ADDED projection — a projections-status surface shows the health of system-of-record
  writes per spec

## Requirements

### Requirement: Deterministic harness plugin drives system-of-record writes
`capability: projection` · `delta: ADDED (feat/deterministic-projection-and-integrations)`

The system SHALL drive every write to a system of record through a harness plugin
(`.opencode/plugins/arke-projection.ts`) that reacts to `generation.decided approved`
events deterministically — generating the same API payload for the same input every time —
and SHALL log each write attempt and outcome as a `projection.write` DomainEvent in the
`@arke/contracts` canonical event schema, including the triggering event id. The plugin
SHALL be verified as loaded and active (via an integration health check) before any
projection write is attempted; an unloaded plugin SHALL surface as a `blocked` state, not
a silent failure.

#### Scenario: Plugin writes to GitHub on approval
- **WHEN** a `generation.decided` event with `decision: "approved"` is received and the
  proposal includes an artefact with `sorTarget: "github"`
- **THEN** the projection plugin creates the corresponding GitHub issue using host-side
  credentials
- **AND** emits a `projection.write` event with `target: "github"`, `specId`, `trigger`
  (the `generation.decided` event id), and `ok: true`

#### Scenario: Failed write is logged, not silently dropped
- **WHEN** the plugin attempts a system-of-record write and the API call fails
- **THEN** a `projection.write` event is emitted with `ok: false`, the failure detail,
  and the triggering event id
- **AND** no automatic retry is attempted; the projections-status surface shows the failure
  with a "Retry" option

#### Scenario: Plugin produces identical payload for the same approved artefact
- **WHEN** the same `generation.decided approved` event is replayed (e.g. after a crash
  and recovery)
- **THEN** the plugin produces the same API payload
- **AND** the stable idempotency key (see Design → Idempotency) prevents a duplicate
  record on the SoR if the first write succeeded; the second attempt updates or is a no-op

#### Scenario: Plugin not loaded surfaces as blocked, not silent
- **WHEN** projection is attempted but the plugin is not loaded in the OpenCode session
  (e.g. the `.opencode/plugins/` directory is absent or the plugin errors at load)
- **THEN** the coordinator marks the projection as `blocked — plugin not loaded`
- **AND** the projections-status surface shows the blocked state with configuration
  guidance

#### Scenario: projection.write event is recorded even when the coordinator log endpoint is unreachable
- **WHEN** the plugin completes a write (success or failure) but `POST /log` to the
  coordinator fails
- **THEN** the plugin retries `POST /log` up to 3 times with exponential backoff before
  writing the record to a local fallback log
- **AND** the coordinator reconciles the fallback log on next startup so the write is not
  permanently unaudited

### Requirement: Idempotency keys prevent duplicate system-of-record records
`capability: projection` · `delta: ADDED (feat/deterministic-projection-and-integrations)`

The system SHALL construct a stable idempotency key for every SoR write as
`SHA-256(specId + artifactId + contentDigest)`, where `contentDigest` is the SHA-256 of
the artefact content at approval time. The plugin SHALL include this key in the SoR API
call (as a creation-dedup key where the SoR supports it, or as an idempotency header) and
SHALL verify on a second write attempt whether a record with this key already exists on the
SoR; if it does, the write is recorded as a no-op success rather than a duplicate creation.
The idempotency key SHALL also cover the case where the same artefact is re-generated
(different `generation.decided` event, same content) to prevent duplicate items.

#### Scenario: Crash-recovery write does not duplicate
- **WHEN** the coordinator restarts mid-write and re-drives an approved artefact
- **THEN** the plugin computes the same idempotency key and checks the SoR for an existing
  record with that key
- **AND** if found, records the outcome as a no-op success and emits `projection.write ok:true`
  without creating a duplicate item

#### Scenario: Same content re-generated does not duplicate
- **WHEN** a spec is re-generated after a minor spec edit, producing the same artefact
  content for one ticket, and the human re-approves
- **THEN** the plugin's idempotency check prevents a second ticket being created for
  identical content
- **AND** the projections-status surface shows the result as "already exists — no-op"

### Requirement: `retry-projection` requires the original approval record, not a new human action
`capability: projection` · `delta: ADDED (feat/deterministic-projection-and-integrations)`

The system SHALL allow the coordinator to re-attempt a failed SoR write using the original
`generation.decided approved` trace record as authorisation, without requiring a new human
approval gesture. The coordinator SHALL verify that the original trace record exists and
has not been superseded before retrying. A `retry-projection` command SHALL NOT be accepted
if the corresponding approval trace record cannot be found or has been superseded.

#### Scenario: Retry uses original approval, no new gesture
- **WHEN** the human clicks "Retry" on a failed projection
- **THEN** the coordinator locates the original `generation.decided approved` trace record
  for that `specId` and artefact
- **AND** uses it as the authorisation to re-drive the projection plugin write
- **AND** records the retry attempt and outcome as a new `projection.write` event in the
  trace

#### Scenario: Retry blocked when original approval not found
- **WHEN** `retry-projection` is received but no matching `generation.decided approved`
  trace record exists for that `specId` / `artifactId`
- **THEN** the coordinator rejects the retry with an error
- **AND** the projections-status surface shows "Cannot retry — original approval not found"

### Requirement: Agents may propose but never author the record
`capability: projection` · `delta: ADDED (feat/deterministic-projection-and-integrations)`

The system SHALL ensure that AI agents propose artefact content (via SPEC-013) but SHALL
NOT permit an agent to call system-of-record APIs directly. The plugin, not the agent,
constructs and submits the API call using the host-held credentials.

#### Scenario: Agent content becomes plugin input
- **WHEN** an agent session produces a proposed artefact with a `sorTarget`
- **THEN** the content is passed to the projection plugin as structured input
- **AND** the agent session has no direct access to the integration credentials or API
  surface

#### Scenario: No agent can call an integration API in-band
- **WHEN** any tool call in any agent session targets a known integration API hostname
  (see Design → Policy hook domain list)
- **THEN** the `tool.execute.before` policy hook intercepts the call and throws a blocking
  error before the call is made
- **AND** the block is logged to the trace with the session id, tool name, and matched domain

### Requirement: Policy hook blocks downstream edits from re-entering the spec; kind is reliably set
`capability: projection` · `delta: ADDED (feat/deterministic-projection-and-integrations)`

The system SHALL deploy a `tool.execute.before` hook in the projection plugin that
inspects every harness tool call and blocks any write to a specification file path that
originates from a projection or integration session. The hook SHALL read the session kind
from the session creation record persisted in the coordinator (not from an ambient runtime
field that could be absent), enforcing the direction-of-truth invariant (spec → artefacts,
never artefacts → spec). Sessions created as projection sessions SHALL have `kind:
"projection"` recorded in the session graph (SPEC-002) at creation time; the hook SHALL
treat a session with a missing or unknown kind as untrusted and block spec-path writes from
it.

#### Scenario: Write to spec path from projection session is blocked
- **WHEN** a projection session attempts a `file.write` (or equivalent) tool call
  targeting a path matching `docs/specifications/**`
- **THEN** the `tool.execute.before` hook throws a blocking error
- **AND** the write does not reach the file system
- **AND** the block is logged to the trace with session id and path

#### Scenario: Session with missing kind is treated as untrusted for spec writes
- **WHEN** a tool call targets `docs/specifications/**` and the session's kind cannot be
  determined from the session graph
- **THEN** the hook blocks the write and logs it as "kind unknown — blocked by policy"
- **AND** does NOT silently permit the write on the grounds that the session might be an
  authoring session

#### Scenario: Authoring session may write to the spec
- **WHEN** the authoring cockpit session (kind: `spec`) writes to a specification file
- **THEN** the hook permits the write
- **AND** no blocking error is thrown

#### Scenario: Shell and MCP tool calls to integration domains are also blocked
- **WHEN** any agent session issues a shell command or MCP tool call whose resolved
  destination matches a domain on the integration block-list (see Design → Policy hook
  domain list)
- **THEN** the `tool.execute.before` hook blocks the call and logs the attempt

### Requirement: Integrations registry confirms and manages connections, with proactive re-validation
`capability: integrations-registry` · `delta: ADDED (feat/deterministic-projection-and-integrations)`

The system SHALL maintain an integrations registry on the coordinator that records the
configured state of GitHub, Jira, and Azure DevOps connections, reports each connection's
status (`connected`, `not-configured`, `error`), describes what it enables, and confirms
that credentials are held on the host and never passed to the client. The coordinator SHALL
re-validate each integration's status at a configurable interval (default: 5 minutes) and
update the registry immediately on a `get-integration-status` request, so that expired
tokens are detected without requiring a coordinator restart.

#### Scenario: Connected integration shows as active
- **WHEN** the GitHub integration is configured (token present in the host environment)
  and the coordinator can reach the API
- **THEN** the registry reports GitHub as `connected` with `lastCheckedAt` updated to the
  probe time
- **AND** the client registry screen shows "GitHub — connected (enables: issue projection)"

#### Scenario: Missing credential shows as not configured
- **WHEN** no Jira API token is present in the host environment
- **THEN** the registry reports Jira as `not-configured`
- **AND** the client shows configuration guidance, not raw credential fields

#### Scenario: Expired or revoked token surfaces as error, not connected
- **WHEN** a token was previously valid but the API rejects it (401/403)
- **THEN** the registry updates the integration status to `error` with a "credential
  rejected" reason
- **AND** the projections-status surface shows any pending writes for that integration as
  `blocked — credential error`

#### Scenario: Credentials stay on the host
- **WHEN** a client requests the integration registry status
- **THEN** the coordinator returns status, `lastCheckedAt`, and enabled capabilities only
- **AND** no credential, token, API key, or partial token (prefix/hash) is included in
  the WebSocket payload or the trace

### Requirement: Pending writes survive a coordinator restart
`capability: projection` · `delta: ADDED (feat/deterministic-projection-and-integrations)`

The system SHALL persist pending (blocked) writes to the local trace so they survive a
coordinator restart. On restart, the coordinator SHALL scan the trace for approved artefact
writes (from `generation.decided approved` records) that have no corresponding
`projection.write ok:true` record and no `blocked — superseded` marker, and SHALL
re-populate the pending-write queue from these trace records. In-memory `PendingWrite`
state is a cache; the trace is the source of truth.

#### Scenario: Pending write survives restart
- **WHEN** a write is held pending (integration missing) and the coordinator restarts
- **THEN** on restart the coordinator reconstructs the pending-write queue from the trace
- **AND** retries the write automatically when the integration connects, without requiring
  a new human approval

#### Scenario: Pending write content is available on restart
- **WHEN** the coordinator restarts and re-populates a pending write from the trace
- **THEN** the artefact content is read from the `generation.decided approved` trace
  record (which was written before any write was attempted, per SPEC-013)
- **AND** no in-memory `GenerationProposal` is required to re-drive the write

### Requirement: Integrations required before generation, not before authoring
`capability: integrations-registry` · `delta: ADDED (feat/deterministic-projection-and-integrations)`

The system SHALL allow specification authoring and cockpit use regardless of integration
status. Integrations SHALL be confirmed as active before the coordinator dispatches
system-of-record artefact writes; if a required integration is not connected, the
projection for that target SHALL be held pending and the projections-status surface SHALL
show it as blocked.

#### Scenario: Authoring proceeds without integrations
- **WHEN** no integrations are configured
- **THEN** the authoring cockpit, multi-model review panel, and spec lifecycle all operate
  normally
- **AND** no warning or error is shown related to missing integrations

#### Scenario: SoR write is held when integration is missing
- **WHEN** an approved artefact targets GitHub but the GitHub integration is not connected
- **THEN** the write is not attempted
- **AND** the pending write is recorded to the trace as `blocked — GitHub not connected`
- **AND** the projections-status surface shows it as `blocked — GitHub not connected`
- **AND** the write is retried automatically once the integration is connected (using the
  original approval trace record as authorisation)

### Requirement: Projections-status surface shows write health with paginated results
`capability: projection` · `delta: ADDED (feat/deterministic-projection-and-integrations)`

The system SHALL provide a projections-status surface on the client that lists every
`projection.write` event for each specification, showing target, trigger, outcome (`ok /
failed`), and timestamp. The coordinator SHALL cap the number of projection.write records
returned per query at 200 rows (configurable via `ARKE_PROJECTION_QUERY_LIMIT`), returning
the most recent records first, to avoid blocking the coordinator on large trace scans.

#### Scenario: Successful write visible in projections status
- **WHEN** a `projection.write ok:true` event is received
- **THEN** the projections-status surface shows a row with target, specId, trigger,
  timestamp, and a success indicator

#### Scenario: Failed write shown with retry option
- **WHEN** a `projection.write ok:false` event is received
- **THEN** the projections-status surface shows the failure alongside a "Retry" action
- **AND** clicking Retry sends a `retry-projection` command to the coordinator for that
  specId and artefact id

#### Scenario: Large history is capped, not unbounded
- **WHEN** a spec has more than 200 projection write records
- **THEN** the coordinator returns the 200 most recent records
- **AND** the surface indicates that older records are available (e.g. "Showing 200 of N")

## Design

### Architectural decision
System-of-record writes live entirely in a single harness plugin
(`.opencode/plugins/arke-projection.ts` in the Arke project directory, loaded by
OpenCode when the adapter runs). The plugin subscribes to `session.idle` and
`message.updated` events to detect approved artefacts, and uses `tool.execute.before` as
the policy gate. Credentials are read from the process environment inside the plugin (Bun
process on the host); they never reach the coordinator WebSocket or the browser. The
coordinator's role is to emit `ProjectionWriteEvent` records received back via the event
stream, fold them into the read model, and push them to the client.

### Target architecture

```
Coordinator emits generation.decided approved
        │
        ▼ (coordinator pushes to OpenCode session via adapter)
.opencode/plugins/arke-projection.ts (Bun, on host)
    ├── session.idle hook → read approved artefact proposals
    ├── for each artefact with sorTarget:
    │     ├── build deterministic API payload (same input → same payload)
    │     ├── compute idempotency key = SHA-256(specId + artifactId + contentDigest)
    │     ├── check SoR for existing record with idempotency key → no-op if found
    │     ├── call SoR API (GitHub/Jira/Azure DevOps) with host-held credentials
    │     ├── POST /log to coordinator (retry ×3 + fallback local log on failure):
    │     │     projection.write {target, specId, trigger, ok, idempotencyKey}
    │     └── coordinator writes ProjectionWriteEvent to trace + pushes to client
    └── tool.execute.before hook
          ├── if session.kind=projection AND tool writes to docs/specifications/** → block + log
          ├── if tool targets domain in integration block-list → block + log
          └── session with unknown kind writing to docs/specifications/** → block + log
```

### Idempotency
Every SoR write uses a stable key: `SHA-256(specId + ":" + artifactId + ":" + SHA-256(content))`.
This key is stable across crash-recovery re-runs (same content → same key) and across
re-generations that produce identical content. Where the SoR supports deduplication keys
(e.g. GitHub Issues does not natively, but Jira supports idempotency via `addField` on
update), the plugin uses them. Where the SoR does not support them natively, the plugin
queries for existing records by title + label (e.g. GitHub Issues: search by `arke-spec-id`
label and title) before creating.

### Policy hook domain list
The `tool.execute.before` hook blocks tool calls (including shell commands and MCP calls)
whose resolved target domain matches any entry in a configurable list, defaulting to:
- `api.github.com`
- `*.atlassian.net`
- `dev.azure.com`
- `app.vssps.visualstudio.com`

The list is loaded from `.arke/integration-domains.json` at plugin startup (falling back to
the built-in defaults), making it extensible without a plugin code change.

### The `ProjectionWriteEvent` is already defined in `packages/contracts/src/events.ts`:
```typescript
// existing — no change
ProjectionWriteEvent = base.extend({
  type: z.literal("projection.write"),
  target: z.enum(["jira", "azure-devops", "github", "docs", "tests"]),
  specId: z.string(),
  trigger: z.string(),   // the generation.decided event id that caused the write
  ok: z.boolean(),
  // new fields added by this spec:
  artifactId: z.string().optional(),
  idempotencyKey: z.string().optional(),
  errorMessage: z.string().optional(),
});
```

The integrations registry is a map on the coordinator, populated by probing the host
environment at startup, on `get-integration-status` requests, and on the periodic
re-validation interval.

### Data model
Integrations registry (in-memory, coordinator):

```typescript
type IntegrationStatus = "connected" | "not-configured" | "error";

interface IntegrationRecord {
  id: "github" | "jira" | "azure-devops";
  status: IntegrationStatus;
  enables: string[];     // human-readable capability list, e.g. ["issue projection"]
  lastCheckedAt: number; // epoch ms — returned to the client for staleness display
  errorReason?: string;  // present when status is "error", e.g. "credential rejected"
}
```

Projection write history is derived entirely from the `ProjectionWriteEvent` records in
the `Trace` — no separate store. The coordinator queries the trace (capped at
`ARKE_PROJECTION_QUERY_LIMIT`, default 200, most recent first) for the
projections-status surface.

Pending (blocked) writes — **persisted via trace, not in-memory only**:

```typescript
// Trace record: kind "projection.blocked"
{ at, kind: "projection.blocked", specId, target, artifactId, blockedSince, reason }
```

The coordinator reconstructs the pending-write queue on startup from trace records that
have a `projection.blocked` entry with no corresponding `projection.write ok:true`
entry.

### Interfaces and contracts
New coordinator WebSocket events pushed to the client:

```typescript
// new event in @arke/contracts
export const IntegrationStatusEvent = base.extend({
  type: z.literal("integration.status"),
  integrations: z.array(z.object({
    id: z.enum(["github", "jira", "azure-devops"]),
    status: z.enum(["connected", "not-configured", "error"]),
    enables: z.array(z.string()),
    lastCheckedAt: z.number(),
    errorReason: z.string().optional(),
  })),
});
```

New client → coordinator commands:
```typescript
{ type: "get-integration-status" }
{ type: "retry-projection", specId: string, artifactId: string,
  target: "github" | "jira" | "azure-devops" }
```

`retry-projection` now carries `artifactId` (not just `target`) so the coordinator can
look up the exact trace record for that artefact.

The plugin communicates `projection.write` back to the coordinator via
`POST /log` (the OpenCode `POST /log` endpoint, which the coordinator adapter
translates to a `ProjectionWriteEvent`). On `POST /log` failure, the plugin retries
×3 with exponential backoff (100ms, 200ms, 400ms), then appends to a local
`.arke/projection-fallback.ndjson` file. The coordinator checks this file on startup.

### Cross-cutting
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.
- **Security (§14):** credentials are held exclusively in the Bun/Node host process
  environment; they are never serialised to the coordinator WebSocket, the `Trace` file,
  or the client. No partial token, hash, or prefix of a credential is included. The plugin
  is the sole credential consumer.
- **Direction of truth:** the `tool.execute.before` hook is the enforcement point, not a
  convention. Session kind is set at creation and persisted in the session graph (SPEC-002);
  the hook reads from the session graph, not from an ambient runtime field. Sessions with
  an unknown kind are treated as untrusted for spec-path writes.
- **Observability:** every `projection.write` event (success, failure, or no-op/blocked) is
  written to `Trace` and pushed to the client, giving a full audit trail (feeds SPEC-015).
- **Phase 2 note:** this spec is Phase 2 — integrations are not required before the
  Phase 1 authoring and generation loop works. Teams can author, review, approve, and
  generate local artefacts without any integration connected.
- **Azure DevOps scope:** Phase 2 targets Azure DevOps work items only; wiki page
  projection is deferred to Phase 3.

### Testing
- Unit (plugin): same artefact input → identical idempotency key and identical API
  payload (determinism); failed API call → `projection.write ok:false` posted to log;
  `POST /log` failure → retried ×3 → fallback file written.
- Unit (`tool.execute.before`): projection-session file write to spec path → error thrown
  + logged; unknown-kind session write to spec path → error thrown; authoring-session
  write to spec path → allowed; shell command to `api.github.com` → blocked; MCP call to
  `*.atlassian.net` → blocked.
- Unit (idempotency): second write with same content → idempotency key computed identically;
  SoR stub returns existing record → no duplicate creation, `ok:true` no-op.
- Unit (integrations registry): env token present → `connected`; env token absent →
  `not-configured`; API returns 401 → `error` with `errorReason`; no credentials in WS
  payload; periodic re-check updates `lastCheckedAt`.
- Unit (retry-projection): original trace record found → retry authorised; no trace record
  → rejected with error.
- Unit (pending-write persistence): coordinator restart with `projection.blocked` trace
  records → pending-write queue reconstructed; coordinator restart with
  `projection-fallback.ndjson` → records ingested.
- Integration: stub SoR APIs; approve a proposal; assert `projection.write ok:true` event
  received by client; assert idempotency key in event; assert no credentials in WS payload.
- Integration: simulate `POST /log` failure; assert fallback file written; restart
  coordinator; assert fallback records reconciled.
- Manual: connect GitHub integration; approve a real spec; confirm GitHub issue created;
  check projections-status surface shows the row with trigger and idempotency key.

### Definition of done
All scenarios pass; the plugin creates a GitHub issue from an approved artefact with
idempotency (no duplicate on retry); projections-status shows the write with trigger and
idempotency key; `tool.execute.before` blocks a test write to the spec path from a
projection session and from an unknown-kind session; shell and MCP calls to blocked domains
are intercepted; no credentials appear in any WebSocket message or trace file; pending
writes survive a coordinator restart; `retry-projection` verifies against the original
trace record; expired tokens surface as `error` status; a reviewer has signed off.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Deterministic plugin, not free-form agent | Reproducible, auditable, testable; the same input always produces the same SoR record |
| 2 | Plugin runs on host (Bun), credentials in environment | Credentials never leave the trust boundary; the browser and the coordinator never see them |
| 3 | `tool.execute.before` as the direction-of-truth gate | The only point in the stack where a write can be intercepted before it reaches the file system |
| 4 | Projection writes communicated back via `POST /log` with retry + fallback | Reuses the existing OpenCode log endpoint; retry + fallback prevents unaudited writes |
| 5 | Phase 2 positioning | Teams must be able to author and generate without having integrations configured |
| 6 | Idempotency key = SHA-256(specId + artifactId + contentDigest) | Stable across crash-recovery and re-generation with identical content; prevents duplicate SoR records |
| 7 | `retry-projection` uses original trace record as authorisation, no new gesture | A retry is a re-drive of a previously approved write; requiring a new approval for a network failure is friction without security benefit; but the trace record must be verified to exist |
| 8 | Pending writes persisted via trace, not in-memory | A coordinator restart must not silently drop pending writes; the trace is the source of truth |
| 9 | Session kind read from session graph (SPEC-002), not ambient runtime field | Ambient fields can be absent or spoofed; the persisted session graph is authoritative |
| 10 | Unknown-kind sessions are treated as untrusted for spec-path writes | False negatives (write permitted because kind is missing) are a security boundary failure; fail closed |
| 11 | Integration domain block-list loaded from `.arke/integration-domains.json` | Extensible without plugin code change; new integrations add their domains without a deployment |
| 12 | Integrations re-validated on interval (default 5 min) | Expired tokens must surface as `error` without requiring a coordinator restart |
| 13 | Azure DevOps Phase 2 scope = work items only; wiki deferred to Phase 3 | Reduces scope to the well-understood write target; resolves the open question |
