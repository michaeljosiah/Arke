# Spec lifecycle, governance and library

## Why
A specification that cannot be governed is not a contract — it is a note. The lifecycle turns the working draft into an authoritative artefact: status tracks where the specification stands, transitions are tied to pull-request state rather than to manual edits, a second human reviews and gates the merge, and delta tags flatten away on merge so the body always reads as the current contract. The spec library is the single place to discover every specification in a project. This covers FR-5 (lifecycle and governance) and FR-12 (reviewer approval).

## What changes
- ADDED spec-lifecycle — frontmatter status (`draft → in-review → approved`) is governed by pull-request state; a material change to an approved specification drops it back to `in-review`; a second human gates the merge; delta tags flatten on merge per the template lifecycle rules
- ADDED spec-library — the client presents every specification in the project with its current status and supports search by title, capability, and status

---

## Requirements

### Requirement: Status follows pull-request state, including divergence events
`capability: spec-lifecycle` · `delta: ADDED (feat/spec-lifecycle-and-library)`

The system SHALL carry specification status through `draft → in-review → approved` in frontmatter, and SHALL tie each transition to a corresponding pull-request event so that status reflects the state of the change in version control, not a manual edit; the coordinator SHALL handle PR closure without merge (regression to `draft`), PR reopening (restore prior status), and force-push (re-validate the frontmatter branch field against the new HEAD SHA).

#### Scenario: Draft to in-review on approval action
- **WHEN** the engineer invokes the approval action in the authoring cockpit (SPEC-006) after a completed review
- **THEN** the coordinator commits the specification file to `docs/specifications/` on the feature branch, updates the frontmatter `status` to `in-review`, and opens (or updates) a pull request for the feature branch

#### Scenario: In-review to approved on PR approval
- **WHEN** a second human approves the pull request for the feature branch
- **THEN** the coordinator receives the `pull_request_review` webhook event, verifies the approver is not the specification's `owner` (see requirement: second-human gate), and updates the frontmatter `status` to `approved` in a follow-up commit on the branch

#### Scenario: PR closed without merge — status regresses to draft
- **WHEN** the pull request for a specification feature branch is closed without being merged
- **THEN** the coordinator receives the `pull_request.closed` (not merged) webhook event and regresses the specification status to `draft`
- **AND** emits `spec.status { specId, status: "draft", reason: "pr-closed" }` and records the regression in the Change history

#### Scenario: PR reopened — status restored
- **WHEN** a previously closed pull request is reopened
- **THEN** the coordinator receives the `pull_request.reopened` webhook event and restores the specification status to `in-review`
- **AND** emits `spec.status { specId, status: "in-review", reason: "pr-reopened" }`

#### Scenario: Force-push — coordinator re-validates frontmatter branch
- **WHEN** a force-push to the feature branch is received (detected via `push` webhook with `forced: true`)
- **THEN** the coordinator re-reads the specification frontmatter from the new HEAD commit and validates that the `branch` field still matches the pushed branch
- **AND** if the frontmatter branch field no longer matches, emits a `spec.branch-mismatch` warning event and marks the read model status as `unknown` until reconciled

#### Scenario: Approved to in-review on material change
- **WHEN** a material change (any modification to a requirement, design section, or task) is made to a specification whose current status is `approved`
- **THEN** the system drops the status back to `in-review` and records the regression in the Change history
- **AND** the UI reflects the regression immediately via a `spec.status` event

### Requirement: Material-change detector hashes normative sections only, excluding non-normative fields
`capability: spec-lifecycle` · `delta: ADDED (feat/spec-lifecycle-and-library)`

The material-change detector SHALL hash the Requirements section and the Design section of the specification to detect regressions; it SHALL exclude the frontmatter block (including the `updated` date field), the Change history section, and the Open questions section from the hash, so that routine metadata updates do not trigger a regression; the hash algorithm and the sections included SHALL be documented in the coordinator source and logged with each regression event.

#### Scenario: Frontmatter date change does not trigger regression
- **WHEN** only the frontmatter `updated` field changes on an approved specification
- **THEN** the material-change detector does not emit a regression event
- **AND** the status remains `approved`

#### Scenario: Requirements change triggers regression
- **WHEN** a requirement is added, modified, or removed in the Requirements section of an approved specification
- **THEN** the material-change detector emits `spec.status { status: "in-review", reason: "material-change" }` and records the regression in Change history
- **AND** the hash algorithm used and the sections hashed are included in the trace event

#### Scenario: Design section change triggers regression
- **WHEN** the Design section of an approved specification changes
- **THEN** the material-change detector emits a regression event, treating a design change as material

### Requirement: Material change ripples to projections
`capability: spec-lifecycle` · `delta: ADDED (feat/spec-lifecycle-and-library)`

The system SHALL, when an approved specification regresses to `in-review`, emit a `projection.stale` event for each system-of-record record derived from it; these events SHALL be written to the durable event log before being pushed to any downstream subscriber, so that a subscriber that was offline can reconstruct stale state on reconnect.

#### Scenario: Jira tickets marked stale on regression
- **WHEN** an approved specification regresses to `in-review`
- **THEN** the coordinator emits a `projection.stale` event for each system-of-record record derived from that specification
- **AND** the projections status screen reflects the stale state for those records

#### Scenario: Regression does not automatically rewrite projections
- **WHEN** a specification regresses
- **THEN** the system flags the derived projections as stale but does NOT automatically update or delete them
- **AND** the engineer is notified and must re-run the generation workflow (SPEC-013) to produce updated projections after re-approval

#### Scenario: Offline subscriber receives stale events on reconnect
- **WHEN** a downstream projection subscriber was offline when the `projection.stale` event was emitted
- **THEN** on reconnect the subscriber can read the event from the durable event log and reconcile its state

### Requirement: Delta tags flatten on merge — idempotently
`capability: spec-lifecycle` · `delta: ADDED (feat/spec-lifecycle-and-library)`

The system SHALL, on merge of the feature branch, flatten all delta tags in the specification according to the template lifecycle rules: `ADDED` and `MODIFIED` tags are dropped; `REMOVED` requirements are cut from the body and replaced by a tombstone under a `## Removed` note; `RENAMED` changes are applied in place; a Change-history line is appended; the flatten operation SHALL be idempotent — re-running it on an already-flattened file SHALL produce no change and no duplicate tombstone; the coordinator SHALL write a flatten-started marker to the trace before beginning and a flatten-complete marker on success, so that a restart mid-flatten resumes from a known state rather than re-running partially.

#### Scenario: ADDED tag drops on merge
- **WHEN** the feature branch is merged to main
- **THEN** every `delta: ADDED (…)` metadata line is removed from the specification
- **AND** the requirement is indistinguishable from pre-existing settled requirements in the merged file

#### Scenario: MODIFIED tag drops on merge
- **WHEN** the feature branch is merged and a requirement carries `delta: MODIFIED (…)`
- **THEN** the `delta: MODIFIED (…)` metadata line is removed
- **AND** the full requirement body is retained unchanged

#### Scenario: REMOVED requirement becomes a tombstone
- **WHEN** the feature branch is merged and a requirement carries `delta: REMOVED`
- **THEN** the requirement body is cut from the Requirements section
- **AND** a tombstone line is appended under a `## Removed` heading:
  `> REMOVED <capability>/<name> — Reason: <why> · Migration: <what to do instead>`

#### Scenario: RENAMED tag is applied in place
- **WHEN** the feature branch is merged and a requirement carries `delta: RENAMED (from: <old-name>)`
- **THEN** the requirement heading is updated to the new name and the delta tag is removed
- **AND** a note is added to Change history recording `RENAMED <old-name> → <new-name>`

#### Scenario: Change history is appended at merge
- **WHEN** the feature branch is merged
- **THEN** a line is appended to the specification's Change history recording: the date, the branch name, the status reached (`approved`), and a one-line summary of the net delta
- **AND** if the "What changes" section is absent or empty, the summary is auto-derived from the delta operations found (e.g. "ADDED: N requirements; MODIFIED: M requirements; REMOVED: K requirements")
- **AND** the merged file body reads purely as the current contract with no in-flight delta tags remaining

#### Scenario: Flatten is idempotent — re-run on an already-flattened file is a no-op
- **WHEN** `flattenDeltaTags` is run on a specification file that contains no delta tags
- **THEN** the output is identical to the input (no Change history line added, no tombstone duplicated)
- **AND** no commit is made

#### Scenario: Flatten resumes after coordinator restart mid-flatten
- **WHEN** the coordinator restarts after writing the flatten-started trace marker but before completing the flatten commit
- **THEN** on startup the coordinator detects the incomplete flatten (flatten-started without flatten-complete in the trace) and re-runs `flattenDeltaTags` from scratch on the current file
- **AND** because the operation is idempotent, re-running on a partially-flattened file produces a correct result

### Requirement: A second human reviews and gates the merge — enforced at both git host and coordinator
`capability: spec-lifecycle` · `delta: ADDED (feat/spec-lifecycle-and-library)`

The system SHALL require a second human to review the specification and the generated change before the feature branch is merged; self-approval SHALL be blocked both at the git host (branch protection) and at the coordinator (the coordinator SHALL reject any `pull_request_review.approved` webhook where the approver's identity matches the specification's `owner` field, and SHALL NOT emit `spec.status { status: "approved" }` for a self-approval); the coordinator-level check is the durable enforcement, independent of whether the git host is misconfigured.

#### Scenario: Merge blocked without second-human approval
- **WHEN** a pull request for a specification feature branch is open and the first engineer is the only approver
- **THEN** the pull request merge action is blocked (via branch protection rules on the repository)
- **AND** the spec library shows the specification as awaiting second-human approval

#### Scenario: Second-human approval unblocks the merge
- **WHEN** a second human (not the specification's `owner`) reviews the pull request and approves it
- **THEN** the merge is unblocked
- **AND** the coordinator receives the approval webhook, verifies the approver ≠ `owner`, emits `spec.status { status: "approved" }`, and the spec library updates

#### Scenario: Self-approval rejected at the coordinator
- **WHEN** the author of the specification approves their own pull request (bypassing git host protection or in a misconfigured repository)
- **THEN** the coordinator receives the `pull_request_review.approved` webhook, detects the self-approval (approver login == specification `owner`), and does NOT emit `spec.status { status: "approved" }`
- **AND** the coordinator logs a `governance.self-approval-rejected` trace event with the specId, approver, and PR number

#### Scenario: The second human is not the specification author — enforced at the git host
- **WHEN** the author of the specification attempts to approve their own pull request as the sole reviewer
- **THEN** the repository's branch protection rules reject the self-review
- **AND** the system does not count self-approval as second-human approval

### Requirement: Spec library lists every specification with status and search
`capability: spec-library` · `delta: ADDED (feat/spec-lifecycle-and-library)`

The system SHALL present a spec library screen listing every specification in the **active project** with its current status, and SHALL support search by title, capability, and status; the library SHALL be the definitive discovery surface for specifications and SHALL be rebuilt when the client switches projects (SPEC-018 `project.open`). Reconciliation against both the filesystem (frontmatter status) and any open PR state from the git host SHALL run **when each `ProjectContext` is opened** (so a project opened after the process is already running — e.g. after an offline webhook gap — corrects its stale statuses), as well as a startup pass over any already-open contexts; a warning SHALL be surfaced when library status diverges from frontmatter status.

#### Scenario: All specifications are listed
- **WHEN** the engineer opens the spec library
- **THEN** every specification file in `docs/specifications/` (excluding the template and examples) is listed with its `spec_id`, title, status, and last-updated date

#### Scenario: Search filters the list
- **WHEN** the engineer types in the search field
- **THEN** the library filters in real time to show only specifications whose title, `spec_id`, capabilities, or status match the query

#### Scenario: Status badge reflects the current frontmatter value
- **WHEN** a specification's status changes (e.g. a `spec.status` event arrives from the coordinator)
- **THEN** the status badge in the library updates without a full page reload
- **AND** `draft` shows in amber, `in-review` in blue, `approved` in green, `merged` in muted

#### Scenario: Library warns on status divergence
- **WHEN** the coordinator's read-model status for a specification differs from the status field in the specification's frontmatter file (e.g. read model says `approved` but frontmatter says `draft`)
- **THEN** the spec library shows a divergence warning icon on that specification entry
- **AND** the coordinator logs the divergence and surfaces it as a reconciliation task

#### Scenario: Concurrent branches touching the same capability are flagged
- **WHEN** two open pull requests both modify the same capability (same capability string in their frontmatter)
- **THEN** the spec library flags each specification with a "concurrent capability conflict" warning, identifying the other specification and branch
- **AND** the engineer is informed that merging both may produce overlapping or conflicting requirements

## Design

### Architectural decision
Status is carried in the frontmatter of the specification file in git — the file is the source of truth — and reflected as a derived projection in the client's read model. Transitions are driven by git/PR events (webhooks from GitHub or equivalent), not by client-side mutations. Delta-tag flattening is a harness-side file rewrite performed by the coordinator at merge time, committed as a single flatten commit on main. Self-approval is blocked both at the git host (branch protection) and at the coordinator (webhook handler rejects self-approval independently of host enforcement). Flatten is designed for idempotency: re-running it on a file with no delta tags is a no-op; restart recovery uses a flatten-started/flatten-complete trace marker pair.

### Target architecture
```
GitHub (or git host)
    ├── PR opened / commit pushed          → webhook → coordinator
    ├── PR review approved                 → webhook → coordinator (self-approval check here)
    ├── PR closed (not merged)             → webhook → coordinator (status → draft)
    ├── PR reopened                        → webhook → coordinator (status → in-review)
    ├── push (forced: true)                → webhook → coordinator (re-validate frontmatter branch)
    └── PR merged                         → webhook → coordinator

coordinator (packages/coordinator/server.ts)
    ├── on PR opened: confirm branch matches spec frontmatter; emit spec.status { status: "in-review" }
    ├── on PR review approved:
    │       ├── check approver ≠ spec.owner → if same: log governance.self-approval-rejected; return
    │       └── emit spec.status { status: "approved" }
    ├── on PR closed (not merged): emit spec.status { status: "draft", reason: "pr-closed" }; append Change history
    ├── on PR reopened: emit spec.status { status: "in-review", reason: "pr-reopened" }
    ├── on push (forced: true): re-read frontmatter; validate branch field; emit spec.branch-mismatch if mismatched
    ├── on PR merged:
    │       ├── trace.write { kind: "flatten.started", specId }
    │       ├── read working file off branch
    │       ├── flatten delta tags (ADDED/MODIFIED → drop tag; MODIFIED body retained; REMOVED → tombstone; RENAMED → apply; idempotent)
    │       ├── derive Change history summary from delta ops if "What changes" is absent
    │       ├── append Change history line; update frontmatter updated date
    │       ├── commit flattened file to main
    │       ├── trace.write { kind: "flatten.complete", specId }
    │       └── emit spec.status { status: "merged" }
    ├── on spec.status: update read model; trace.write
    ├── on material change to approved spec:
    │       ├── compute normative hash (Requirements + Design sections only, excluding frontmatter/Change history/Open questions)
    │       ├── compare against stored hash; if changed: emit spec.status { status: "in-review", reason: "material-change" }; emit projection.stale for each derived record; write projection.stale to durable event log before pushing
    │       └── store new normative hash
    └── startup reconciliation (per project context — SPEC-018):
            ├── for each ProjectContext: scan that project's docs/specifications/ → build SpecRecord[] from frontmatter
            ├── query that project's repo for open PRs → reconcile its read model
            └── emit spec.divergence-warning (scoped to the project) for any specId where read-model status ≠ frontmatter status

spec library screen (packages/client/src/screens/picker-init-library.tsx)
    ├── reads CardState[] from coordinator snapshot on connect
    ├── subscribes to spec.status events for live badge updates
    ├── search: client-side filter on title / specId / capabilities / status
    ├── divergence warning icon on mismatched entries
    ├── concurrent-capability conflict warning (two open PRs, same capability)
    └── click → navigate to cockpit (006) or board (010) for that spec
```

**Git integration:** the coordinator subscribes to a configured webhook endpoint (`POST /webhooks/github`) and maps GitHub events to internal coordinator actions. The webhook endpoint stays **global** (one endpoint on the single coordinator), but each delivered event must be **routed to the owning `ProjectContext` by the remote repository identity** in the payload (SPEC-018) — `repository.full_name`/`html_url`/`node_id` — **not** by a local filesystem path, which the webhook payload does not carry. The `ProjectRegistry` therefore records each project's remote(s) (the configured git remote URL, normalised) so a delivered event resolves to its project. A remote that maps to **multiple** registered local clones SHALL fan the event to every matching open context (and be a no-op for projects not currently open, reconciled later on their next `project.open`); an event whose remote matches no registered project is logged and dropped. Branch protection rules (required ≥ 1 non-author PR approval; self-merge blocked) are configured on the repository at project-initialisation time (SPEC-004).

**Delta-tag flattening algorithm (run at merge, idempotent):**

1. Check whether the file contains any `delta:` tags. If none, this is a no-op — return the file unchanged and make no commit.
2. Parse the merged specification markdown.
3. For each `### Requirement:` block:
   - If the metadata line contains `delta: ADDED (…)` or `delta: MODIFIED (…)` → remove the metadata line; retain the requirement body unchanged.
   - If the metadata line contains `delta: REMOVED (…)` → remove the entire requirement block; append under `## Removed` (create the heading if absent):
     `> REMOVED <capability>/<name> — Reason: <from-open-questions-or-decision-log> · Migration: see Change history`
     Check for an existing tombstone with the same `<capability>/<name>` before appending; if one already exists, skip (idempotency guard).
   - If the metadata line contains `delta: RENAMED (from: <old-name>)` → apply the new name to the heading; remove the delta tag; note the rename in Change history.
4. Derive the Change history summary: if "What changes" section is non-empty, use its content; otherwise auto-derive from the delta operations found (e.g. "ADDED: N requirements; MODIFIED: M; REMOVED: K").
5. Append to `## Change history`:
   `- <date> · <branch> · approved — <summary>`
6. Update frontmatter `updated` date.
7. Commit to main with message `chore(specs): flatten delta tags for <spec_id>`.

**Normative hash computation (for material-change detection):**

Sections included: `## Requirements` (full block) + `## Design` (full block).
Sections excluded: frontmatter YAML block, `## Change history`, `## Open questions`, `## Decision log`.
Algorithm: SHA-256 of the UTF-8 encoded concatenation of the included sections, after normalising line endings to LF. The algorithm name and sections list are constants in the coordinator source and are logged with each regression event.

### Data model
No new persistent entities. The coordinator maintains in-memory spec state derived from frontmatter reads and PR events:

```typescript
interface SpecRecord {
  specId: string;
  title: string;
  status: SpecStatus;       // from @arke/contracts: "draft" | "in-review" | "approved" | "merged"
  branch: string;
  capabilities: string[];
  updatedAt: string;
  prNumber?: number;
  prApprovers: string[];    // GitHub logins of non-self PR approvers
  normativeHash?: string;   // SHA-256 of Requirements + Design sections (set when status becomes approved)
  hasDivergence?: boolean;  // read-model status ≠ frontmatter status
}
```

The library screen reads `SpecRecord[]` from the coordinator's read model snapshot and subscribes to `spec.status` events for live updates. `SpecFrontmatter` (from `@arke/contracts`) is the schema against which each specification file is parsed on read.

**Durable event log for `projection.stale`:** `projection.stale` events are written to an append-only log (the existing `trace.ts` append-only store, or a dedicated projection-events queue if SPEC-015 introduces one) before being pushed to subscribers, ensuring offline subscribers can reconstruct state on reconnect.

### Interfaces and contracts
**New domain events:**
```typescript
// emitted when a projection derived from a spec becomes stale due to regression
{ type: "projection.stale", specId: string, target: string, recordRef: string }

// emitted when the coordinator detects a PR closed without merge
{ type: "spec.status", specId: string, status: "draft", reason: "pr-closed" }

// emitted when coordinator detects frontmatter branch ≠ pushed branch after force-push
{ type: "spec.branch-mismatch", specId: string, frontmatterBranch: string, pushedBranch: string }

// emitted on startup when read-model status ≠ frontmatter status
{ type: "spec.divergence-warning", specId: string, readModelStatus: SpecStatus, frontmatterStatus: SpecStatus }
```

**New trace events (internal, not pushed to client):**
```typescript
{ kind: "flatten.started", specId: string, branch: string, ts: number }
{ kind: "flatten.complete", specId: string, branch: string, commitSha: string, ts: number }
{ kind: "governance.self-approval-rejected", specId: string, approver: string, prNumber: number, ts: number }
```

**Webhook endpoint (coordinator REST):**
```
POST /webhooks/github
  Body: GitHub webhook payload (pull_request, pull_request_review, push events)
  Action: mapped to coordinator internal spec-lifecycle transitions
```

**Flattening command (internal coordinator, not client-facing):**
```typescript
// triggered on PR merge webhook; idempotent
flattenDeltaTags(specId: string, branch: string): Promise<"flattened" | "no-op">
```

**Coordinator read model additions:**
- `specs: Map<specId, SpecRecord>` — indexed for O(1) lookup on `spec.status` events and library snapshot.

The existing `spec.status` event type in `@arke/contracts/src/events.ts` is extended with an optional `reason` field; `projection.stale`, `spec.branch-mismatch`, and `spec.divergence-warning` are new additions.

### Cross-cutting
- **Configuration:** all project configuration — the harness/model registry, coordinator settings, and integrations — lives in the single root file `.arke/config.json` (`ARKE_*` env vars override individual keys; runtime artefacts under `.arke/` are state, not config). See SPEC-005 for the model.
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.
- **Security (NFR-1):** the webhook endpoint validates the GitHub `X-Hub-Signature-256` header; no secrets are exposed to the client; self-approval is rejected at the coordinator independently of the git host.
- **Auditability (NFR-2):** every lifecycle transition (draft → in-review, regression, flatten.started/complete, self-approval rejection) is written to the append-only trace with the specId, branch, actor (webhook source / human login), and timestamp; the normative hash algorithm and sections are logged with regression events.
- **Performance (NFR-3):** the spec library search is a client-side filter over the snapshot; no server round-trip per keystroke; normative hash computation is O(file size) and runs synchronously in the webhook handler.
- **Resilience (NFR-8):** if a webhook delivery is missed (e.g. coordinator was offline), the coordinator re-fetches open PR state at startup via the GitHub REST API and reconciles the read model; startup reconciliation also compares read-model status against frontmatter status and emits `spec.divergence-warning` for any mismatch; `projection.stale` events are written to the durable event log before delivery so offline subscribers can reconstruct state; flatten restart recovery uses the flatten-started/flatten-complete trace marker pair.
- **Multi-project scoping (SPEC-018):** the coordinator supervises N isolated **project contexts**, each keyed by `projectId`. Everything this spec places on "the coordinator" is per–project context; each client connection resolves against its **active project** and that project's safe root + `docs/specifications/`. See [SPEC-018](018.multi-project-workspaces.md). The library lists the active project's specs and is rebuilt on `project.open`; startup reconciliation runs per context; the global webhook endpoint routes each event by repository path to the owning `ProjectContext`.

### Definition of done
All scenarios pass; the spec library lists all specifications with live status badges; the flatten algorithm produces a clean merged file for all four delta types (ADDED, MODIFIED, REMOVED, RENAMED) and is verified idempotent; the second-human approval gate blocks self-merge at both the git host and coordinator; the coordinator-level self-approval rejection is tested independently of git host enforcement; the regression path (approved → in-review) emits the correct events and logs the normative hash; `projection.stale` events survive coordinator restart; concurrent-capability conflicts are surfaced in the library; a reviewer has signed off.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Status tied to PR state via webhooks, not to manual frontmatter edits | PR state is the authoritative signal in git-hosted projects; tying status to it prevents status from drifting out of sync with the actual change history |
| 2 | Delta-tag flattening is a coordinator-side file rewrite committed at merge, not a client-side transformation | The merge produces a single durable commit; the flatten is part of the merge action, not a separate manual step |
| 3 | Second-human approval enforced at both git host AND coordinator | Branch protection is enforced by the git host independently of the coordinator; the coordinator check is the durable gate that holds even if the git host is misconfigured or the approver bypasses UI |
| 4 | Normative hash covers Requirements + Design sections only, excluding frontmatter/Change history/Open questions | Frontmatter changes (e.g. `updated` date) and non-normative prose changes should not trigger a regression; Design section changes are material and must trigger it |
| 5 | Flatten is idempotent; restart recovery uses flatten-started/flatten-complete trace markers | Coordinator restarts mid-flatten must not corrupt the file; re-running on a partially or fully flattened file must be safe |
| 6 | `projection.stale` written to durable event log before delivery to subscribers | Fire-and-forget delivery loses events for offline subscribers; the event log allows reconstruction on reconnect |
| 7 | PR-closed-not-merged regresses status to `draft` | A closed PR means the change was abandoned; the spec status must reflect this to prevent it from appearing in-review indefinitely |
| 8 | Concurrent-capability conflict surfaced as a warning, not a hard block | Two branches touching the same capability may be legitimate (e.g. one extends, one fixes); the warning gives the engineer visibility without preventing valid parallel work |
| 9 | RENAMED delta type included in flatten scenarios and testing | The template lifecycle defines RENAMED; not covering it in scenarios and tasks leaves a gap that silently fails on real RENAMED deltas |

## Open questions
- Should the material-change detector hash the Tasks section as well as Requirements + Design? Tasks changes may indicate scope change but are less normative.
- How should the coordinator handle a repository hosted on a git provider other than GitHub (e.g. GitLab, Gitea)? The webhook handler is GitHub-specific for Phase 1; a provider-agnostic abstraction is a Phase 2 concern.
- Should `projection.stale` re-delivery to offline subscribers use the existing `trace.ts` store or a dedicated projection-events queue (to be decided with SPEC-015)?
- Should the concurrent-capability conflict detection be a hard block on PR open, or only a warning in the library? Current design: warning only.
