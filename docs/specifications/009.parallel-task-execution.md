# Parallel task execution (fan-out)

## Why
Once a specification is approved its tasks sit idle unless someone fans them out to the
harness one by one. This spec adds the orchestration step that fans an approved spec's task
list into concurrent child sessions — each running in its own isolated git worktree — so
implementation begins without manual intervention and the orchestrator never blocks while
work is in flight.

## What changes
- ADDED task-fanout — the coordinator gains an orchestration/decider step that fans an
  approved specification's task list into asynchronous child sessions via `dispatchAsync`
- ADDED parallel-execution — child sessions run concurrently; the coordinator does not
  wait for any one to complete before starting the next
- ADDED task-fanout — each task session surfaces as its own board card under its parent
  spec, tracked independently by the read model
- ADDED parallel-execution — each child session is dispatched into an isolated git
  worktree off the feature branch so concurrent writers do not collide

---

## Requirements

### Requirement: Fan tasks into asynchronous child sessions
`capability: task-fanout` · `delta: ADDED (feat/parallel-task-execution)`

The system SHALL, when a specification's status transitions to `approved`, read the task
list from the specification file and dispatch each task as an asynchronous child session
by calling `adapter.dispatchAsync()` with the task's text as the prompt, linking the child
to the parent spec session via the `parent` field of `CreateSessionInput`.

#### Scenario: Approved spec triggers fan-out
- **WHEN** a `spec.status` event arrives with `status: "approved"`
- **THEN** the coordinator reads the task list from the specification's `## Tasks` section
- **AND** calls `adapter.dispatchAsync()` once per unchecked task item
- **AND** each dispatch carries the task text and the parent spec session id

#### Scenario: Only unchecked tasks are dispatched
- **WHEN** a specification is fanned out and some task items are already checked (`[x]`)
- **THEN** the coordinator dispatches only the unchecked items
- **AND** already-completed tasks are not re-dispatched

#### Scenario: Fan-out is idempotent
- **WHEN** the coordinator receives a duplicate `spec.status approved` event for a
  specification it has already fanned out
- **THEN** no additional child sessions are created
- **AND** the existing task sessions continue unaffected

### Requirement: Non-blocking concurrent dispatch
`capability: parallel-execution` · `delta: ADDED (feat/parallel-task-execution)`

The system SHALL dispatch all task sessions without waiting for any session to reach
`session.idle` before calling `dispatchAsync()` for the next, so all tasks begin
executing concurrently. The coordinator orchestration loop SHALL NOT block on a running
task session.

#### Scenario: Tasks start concurrently
- **WHEN** a specification with three unchecked tasks is approved
- **THEN** all three `dispatchAsync` calls are issued before any `session.idle` event is
  received
- **AND** all three task sessions transition to `running` state concurrently

#### Scenario: Task completion does not gate remaining tasks
- **WHEN** one task session emits `session.idle` (done)
- **THEN** the other in-flight task sessions continue without interruption
- **AND** the orchestrator's event loop remains available for other events

### Requirement: Each task appears as its own board card
`capability: task-fanout` · `delta: ADDED (feat/parallel-task-execution)`

The system SHALL emit a `session.status` event for each dispatched task session, causing
the read model to create a `CardState` of `kind: "task"` for it, linked to its parent
spec via `specId`, so the board shows each task as a separate card under the spec.

#### Scenario: Task card created on dispatch
- **WHEN** a task session is dispatched
- **THEN** the coordinator emits a `session.status` event with `kind: "task"` and the
  correct `specId`
- **AND** the read model creates a new card in the `implementing` column for that task

#### Scenario: Task card moves on completion
- **WHEN** a task session emits `session.idle`
- **THEN** the corresponding board card transitions from `implementing` to `diff`
- **AND** the spec parent card is not moved by the task's state change alone

### Requirement: Isolated git worktrees per task session
`capability: parallel-execution` · `delta: ADDED (feat/parallel-task-execution)`

The system SHALL dispatch each task session into a separate git worktree branched off the
feature branch, so concurrent agent sessions write to distinct file trees and cannot
overwrite each other's changes mid-run.

#### Scenario: Distinct worktrees for concurrent tasks
- **WHEN** two task sessions are dispatched for the same specification
- **THEN** each session operates in a separate git worktree path
- **AND** a file write in one session's worktree does not appear in the other session's
  working directory until the branches are integrated

#### Scenario: Worktree branch naming is unique
- **WHEN** a worktree is created for a task session
- **THEN** it is branched from the spec's feature branch
- **AND** the branch name is unique per task (e.g. `feat/<slug>/task-<task-index>`)

## Design

### Architectural decision
Add an orchestration *decider* step in the coordinator that reacts to `spec.status approved`
and fans out dispatch calls, following the T3 decider pattern: a pure function maps the
current read-model state and the incoming event to a list of commands (here, `dispatchAsync`
calls), which the effect layer then executes. This keeps orchestration logic deterministic
and testable independently of the adapter.

### Target architecture

```
ReadModel (spec approved)
        │
        ▼
 Decider.onSpecApproved(spec, readModel)
    → TaskCommand[]  (one per unchecked task)
        │
        ▼
 Dispatcher (effect layer)
    ├── git worktree create (feat/<slug>/task-<n>)
    ├── adapter.createSession({ parent: specSessionId, specId })
    └── adapter.dispatchAsync({ sessionId, agent, tier, parts: [taskText] })
        │
        ▼
 adapter.streamEvents()  ──►  ReadModel.apply()  ──►  Trace.write()  ──►  WS push
```

The `Coordinator` in `packages/coordinator/src/server.ts` gains a `Decider` class and a
`Dispatcher` that drives the adapter. The `ReadModel` in `read-model.ts` already tracks
`task` card state correctly once `session.status` events arrive; no read-model changes are
needed beyond folding `todo.updated` into task card progress.

### Data model
Fan-out state is kept in-memory on the coordinator (a `Set<specId>` of already-fanned
specs guards idempotency). No persistent store is required in Phase 1; if the coordinator
restarts, re-approval or an explicit re-fan-out command restarts the tasks. Durability of
fan-out state is deferred to the Phase-2 SQLite addition (see SPEC-003).

The specification's task list is parsed from the markdown `## Tasks` section: lines
matching `/^- \[ \] (.+)/` are unchecked tasks; lines matching `/^- \[x\] (.+)/i` are
done.

### Interfaces and contracts

New types added to the coordinator (not to `@arke/contracts` — these are internal):

```typescript
interface TaskCommand {
  kind: "dispatch-task";
  specId: string;
  specSessionId: string;
  taskIndex: number;
  taskText: string;
  featureBranch: string;   // e.g. feat/parallel-task-execution
}

class Decider {
  onSpecApproved(specId: string, specSessionId: string, tasks: string[], branch: string): TaskCommand[];
}

class Dispatcher {
  async execute(cmd: TaskCommand, adapter: HarnessAdapter): Promise<void>;
}
```

The `HarnessAdapter.dispatchAsync` contract (`packages/contracts/src/adapter.ts`) is
already in place; `CreateSessionInput.parent` is the link field. The worktree shell
command is issued via the Node `child_process` module from the `Dispatcher`.

### Cross-cutting
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.
- **Ordering:** `dispatchAsync` calls are issued in task-list order, though all run
  concurrently; the order is deterministic so logs are readable.
- **Observability:** each `TaskCommand` execution is written to `Trace` before the
  `dispatchAsync` call and on completion, giving a durable record of fan-out intent and
  outcome (feeds SPEC-015).
- **Error isolation:** a failing `dispatchAsync` for one task emits a `session.status
  error` event and moves that card to `needs-human`; the remaining tasks are unaffected.
- **Worktree cleanup:** worktrees are not deleted by the coordinator; that is a dev-machine
  concern deferred to a future tidy-up command.

## Tasks
- [ ] Add `Decider` class: pure function mapping `spec.status approved` + parsed task list
  → `TaskCommand[]`; guard idempotency with a `fannedOut: Set<specId>`
- [ ] Add `Dispatcher` class: `execute(cmd)` creates a git worktree, calls
  `adapter.createSession` with parent, calls `adapter.dispatchAsync`, writes to `Trace`
- [ ] Parse `## Tasks` markdown section in the coordinator (regex for `- [ ]` / `- [x]`)
- [ ] Wire `Decider` + `Dispatcher` into `Coordinator.pump()` on `spec.status approved`
- [ ] Emit `session.status running kind:task` after each `dispatchAsync`; fold
  `todo.updated` into the task card in the read model
- [ ] Update `ReadModel.deriveColumn` to handle task `done → diff` transition on
  `session.idle`

### Testing
- Unit (`Decider`): approved spec with mixed checked/unchecked tasks → correct command
  list; duplicate `approved` event → empty command list (idempotency).
- Unit (`Dispatcher`): verify `adapter.createSession` and `dispatchAsync` called with
  correct parent and task text; verify `Trace.write` called before and after.
- Integration: stub adapter emitting `session.idle` events; assert all three task cards
  move to `diff` column independently.
- Manual: approve a real spec in the cockpit, watch three board cards appear concurrently
  in the `implementing` column, confirm they do not share a worktree.

### Definition of done
All scenarios above pass; the coordinator fans out a three-task spec concurrently in the
integration test; the board shows distinct task cards; a reviewer has signed off.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Pure `Decider` function rather than inline orchestration | Deterministic, testable without the adapter; mirrors T3 decider pattern |
| 2 | In-memory idempotency guard (Phase 1) | Sufficient until coordinator persistence lands in SPEC-003 |
| 3 | Git worktrees per task session | Prevents concurrent write collisions without a merge step mid-flight |
| 4 | `dispatchAsync` called without `await` on each | Non-blocking is the contract; concurrency is the point |

## Open questions
- Should the worktree creation be part of the `Dispatcher` or delegated to the adapter
  (so Claude Code adapters can use their own worktree mechanism)?
- If the coordinator restarts mid-fan-out, should incomplete tasks be resumed
  automatically, or require explicit operator re-approval?
