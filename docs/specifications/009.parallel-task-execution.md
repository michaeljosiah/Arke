# Parallel task execution (fan-out)

## Why
Once a specification is approved its tasks sit idle unless someone fans them out to the
harness one by one. This spec adds the orchestration step that fans an approved spec's task
list into concurrent child sessions — each running in its own isolated git worktree — so
implementation begins without manual intervention and the orchestrator never blocks while
work is in flight.

## What changes
- ADDED task-fanout — the coordinator gains an orchestration/decider step that fans an
  approved specification's task list into asynchronous child sessions via `dispatchAsync`
- ADDED parallel-execution — child sessions run concurrently; the coordinator does not
  wait for any one to complete before starting the next
- ADDED task-fanout — each task session surfaces as its own board card under its parent
  spec, tracked independently by the read model
- ADDED parallel-execution — each child session is dispatched into an isolated git
  worktree off the feature branch so concurrent writers do not collide

---

## Requirements

### Requirement: Fan tasks into asynchronous child sessions
`capability: task-fanout` · `delta: ADDED (feat/parallel-task-execution)`

The system SHALL, when a specification's status transitions to `approved`, read the task
list from the specification file and dispatch each task as an asynchronous child session
by calling `adapter.dispatchAsync()` with the task's text as the prompt, linking the child
to the parent spec session via the `parent` field of `CreateSessionInput`. Each dispatched
task session SHALL carry the canonical identity required by SPEC-002: the harness `sessionId`,
the owning `spec_id` from the specification frontmatter, `kind: "task"`, and the parent
spec `sessionId`. This identity SHALL be written to the durable session graph (SPEC-002
§Create spec and task sessions with resolvable identity) before `dispatchAsync` is called.

#### Scenario: Approved spec triggers fan-out
- **WHEN** a `spec.status` event arrives with `status: "approved"`
- **THEN** the coordinator reads the task list from the specification's `## Tasks` section
- **AND** calls `adapter.dispatchAsync()` once per unchecked task item
- **AND** each dispatch carries the task text and the parent spec session id

#### Scenario: Only unchecked tasks are dispatched
- **WHEN** a specification is fanned out and some task items are already checked (`[x]`)
- **THEN** the coordinator dispatches only the unchecked items
- **AND** already-completed tasks are not re-dispatched

#### Scenario: Fan-out is idempotent on duplicate event
- **WHEN** the coordinator receives a duplicate `spec.status approved` event for a
  specification it has already fanned out
- **THEN** no additional child sessions are created
- **AND** the existing task sessions continue unaffected

#### Scenario: Spec with no parseable Tasks section is rejected gracefully
- **WHEN** a `spec.status approved` event arrives for a specification whose file has no
  `## Tasks` section, or whose `## Tasks` section contains no items matching
  `/^- \[[ xX]\] /`
- **THEN** the coordinator emits a `session.status` event with `status: "error"` on
  the spec session carrying a human-readable reason
- **AND** no `dispatchAsync` calls are issued
- **AND** the trace records a `warn` entry with the spec id and reason (e.g. `"no-tasks"`)
  so the gap is actionable

#### Scenario: Fan-out is idempotent on coordinator restart
- **WHEN** the coordinator restarts and the in-memory idempotency guard is lost
- **THEN** the coordinator checks the durable session graph for existing task sessions
  linked to the spec before re-dispatching
- **AND** tasks that already have an existing session in the graph are NOT re-dispatched

### Requirement: Partial fan-out failure handling
`capability: task-fanout` · `delta: ADDED (feat/parallel-task-execution)`

When a fan-out issues multiple `dispatchAsync` calls and one or more fail, the system SHALL
NOT abort the entire fan-out. Each task dispatch SHALL be attempted independently. A failing
dispatch SHALL emit a `session.status` event with `status: "error"` for that task, move the
corresponding board card to `needs-human`, and write a trace entry. Successful dispatches
SHALL proceed and run normally. The coordinator SHALL record, per fan-out, which tasks were
successfully dispatched and which failed, so an operator can identify and retry failed tasks
without re-dispatching succeeded ones.

#### Scenario: One task fails to dispatch; others proceed
- **WHEN** a specification with three tasks is fanned out and the second `dispatchAsync`
  call returns an error (e.g. harness unreachable, worktree creation failed)
- **THEN** the first and third tasks are dispatched and run normally
- **AND** the second task's card moves to `needs-human` with a visible error
- **AND** a trace entry records `{ kind: "dispatch.failed", specId, taskIndex: 1, reason }`

#### Scenario: Partial failure is visible on the board
- **WHEN** a partial fan-out completes with some tasks failed
- **THEN** the board shows successful task cards in `implementing` and failed task cards
  in `needs-human`
- **AND** the spec parent card is NOT moved to `needs-human` solely because of a partial
  failure

#### Scenario: Failed task can be retried without re-dispatching succeeded tasks
- **WHEN** an operator triggers a retry on a failed task
- **THEN** only the failed task is re-dispatched
- **AND** task sessions that were successfully dispatched earlier are not duplicated

### Requirement: Non-blocking concurrent dispatch
`capability: parallel-execution` · `delta: ADDED (feat/parallel-task-execution)`

The system SHALL dispatch all task sessions without waiting for any session to reach
`session.idle` before calling `dispatchAsync()` for the next, so all tasks begin
executing concurrently. The coordinator orchestration loop SHALL NOT block on a running
task session. The coordinator SHALL enforce a configurable maximum concurrent task limit
(`ARKE_MAX_CONCURRENT_TASKS`, default 10); when a fan-out would exceed this limit the
excess tasks SHALL be queued and dispatched as running tasks complete, not silently dropped
or rejected with an error.

#### Scenario: Tasks start concurrently
- **WHEN** a specification with three unchecked tasks is approved and three is within the
  concurrency limit
- **THEN** all three `dispatchAsync` calls are issued before any `session.idle` event is
  received
- **AND** all three task sessions transition to `running` state concurrently

#### Scenario: Task completion does not gate remaining tasks
- **WHEN** one task session emits `session.idle` (done)
- **THEN** the other in-flight task sessions continue without interruption
- **AND** the orchestrator's event loop remains available for other events

#### Scenario: Fan-out exceeding the concurrency limit queues excess tasks
- **WHEN** a specification with fifteen tasks is approved and `ARKE_MAX_CONCURRENT_TASKS`
  is 10
- **THEN** the first ten tasks are dispatched immediately
- **AND** the remaining five are held in a queue
- **AND** as running tasks emit `session.idle` the queued tasks are dispatched one by one
  until the queue is empty

### Requirement: Each task appears as its own board card
`capability: task-fanout` · `delta: ADDED (feat/parallel-task-execution)`

The system SHALL emit a `session.status` event for each dispatched task session, causing
the read model to create a `CardState` of `kind: "task"` for it, linked to its parent
spec via `specId`, so the board shows each task as a separate card under the spec.

#### Scenario: Task card created on dispatch
- **WHEN** a task session is dispatched
- **THEN** the coordinator emits a `session.status` event with `kind: "task"` and the
  correct `specId`
- **AND** the read model creates a new card in the `implementing` column for that task

#### Scenario: Task card moves on completion
- **WHEN** a task session emits `session.idle`
- **THEN** the corresponding board card transitions from `implementing` to `diff`
- **AND** the spec parent card is not moved by the task's state change alone

### Requirement: Isolated git worktrees per task session
`capability: parallel-execution` · `delta: ADDED (feat/parallel-task-execution)`

The system SHALL dispatch each task session into a separate git worktree branched off the
feature branch, so concurrent agent sessions write to distinct file trees and cannot
overwrite each other's changes mid-run. Worktree creation SHALL be attempted before
`dispatchAsync` is called; a failure to create the worktree SHALL prevent the dispatch
of that task and SHALL be treated as a dispatch failure per the partial fan-out failure
requirement. Worktrees SHALL be given a unique branch name derived from the spec slug and
a stable task index to prevent branch collisions on retry. The coordinator SHALL detect and
refuse to create a worktree whose target branch already exists in the local repository (a
sign of an orphaned worktree from a prior run) and SHALL surface this as a `warn` trace
entry so the operator can clean up manually.

#### Scenario: Distinct worktrees for concurrent tasks
- **WHEN** two task sessions are dispatched for the same specification
- **THEN** each session operates in a separate git worktree path
- **AND** a file write in one session's worktree does not appear in the other session's
  working directory until the branches are integrated

#### Scenario: Worktree branch naming is unique and deterministic
- **WHEN** a worktree is created for a task session
- **THEN** it is branched from the spec's feature branch
- **AND** the branch name is `feat/<spec-slug>/task-<task-index>` — deterministic so a
  retry for the same task index produces the same branch name rather than a collision

#### Scenario: Branch collision on retry is surfaced, not silently overwritten
- **WHEN** the coordinator attempts to create a worktree whose branch name already exists
  (e.g. because a previous run left an orphaned worktree)
- **THEN** the coordinator emits a `warn` trace entry naming the branch
- **AND** does NOT proceed to dispatch the task until the collision is resolved
- **AND** the task's board card shows a `needs-human` error state

#### Scenario: Worktree creation failure prevents dispatch
- **WHEN** `git worktree add` fails (e.g. filesystem error, branch conflict)
- **THEN** `dispatchAsync` is NOT called for that task
- **AND** the failure is treated as a dispatch failure (card → `needs-human`, trace entry)

### Requirement: Approved spec mutation guard during fan-out
`capability: task-fanout` · `delta: ADDED (feat/parallel-task-execution)`

The system SHALL detect when a spec's status changes away from `approved` (e.g. drops back
to `in-review` via a "Promote to spec" action in SPEC-011) while a fan-out is in progress.
For tasks that have already been dispatched, the coordinator SHALL emit a `warn` trace entry
noting that the spec was demoted mid-fan-out; it SHALL NOT automatically abort running task
sessions (those sessions are already in-flight and aborting them requires an explicit operator
action). Tasks that are still queued (not yet dispatched) SHALL NOT be dispatched when the spec
is no longer in `approved` state.

#### Scenario: Spec demoted mid-fan-out — queued tasks are held
- **WHEN** a spec's status transitions from `approved` to `in-review` while queued tasks
  are waiting for dispatch
- **THEN** queued tasks are NOT dispatched
- **AND** a trace entry records `{ kind: "fanout.halted", specId, reason: "spec-demoted" }`

#### Scenario: Already-running tasks are not aborted automatically
- **WHEN** a spec is demoted mid-fan-out and some task sessions are already running
- **THEN** those sessions continue running
- **AND** the board shows them as `implementing` with a visual indicator that their parent
  spec is no longer `approved`

## Design

### Architectural decision
Add an orchestration *decider* step in the coordinator that reacts to `spec.status approved`
and fans out dispatch calls, following the T3 decider pattern: a pure function maps the
current read-model state and the incoming event to a list of commands (here, `dispatchAsync`
calls), which the effect layer then executes. This keeps orchestration logic deterministic
and testable independently of the adapter.

### Target architecture

```
ReadModel (spec approved)
        │
        ▼
 Decider.onSpecApproved(spec, readModel)
    → TaskCommand[]  (one per unchecked task)
        │
        ▼
 Dispatcher (effect layer)
    ├── collision check (branch already exists?)
    ├── git worktree create (feat/<slug>/task-<n>)
    ├── adapter.createSession({ parent: specSessionId, specId })
    │      └── write to durable session graph (SPEC-002)
    └── adapter.dispatchAsync({ sessionId, agent, tier, parts: [taskText] })
           ├── on success: emit session.status kind:task, write trace
           └── on failure: emit session.status error, write trace, card → needs-human
        │
        ▼
 adapter.streamEvents()  ──►  ReadModel.apply()  ──►  Trace.write()  ──►  WS push
```

The `Coordinator` in `packages/coordinator/src/server.ts` gains a `Decider` class and a
`Dispatcher` that drives the adapter. The `ReadModel` in `read-model.ts` already tracks
`task` card state correctly once `session.status` events arrive; no read-model changes are
needed beyond folding `todo.updated` into task card progress.

### Data model
Fan-out state is kept in a `FanOutRecord` persisted alongside the coordinator's durable
store (see SPEC-003). The record tracks, per `specId`, which task indices have been
dispatched and the outcome (pending/succeeded/failed), enabling idempotency after restart.
An in-memory cache of this record guards duplicate events within a single process lifetime;
the persistent store guards cross-restart idempotency.

```typescript
interface FanOutRecord {
  specId: string;
  specSessionId: string;
  featureBranch: string;
  tasks: Array<{
    taskIndex: number;
    taskText: string;
    status: "queued" | "dispatching" | "running" | "done" | "failed";
    sessionId?: string;       // set once the harness session is created
    worktreeBranch?: string;  // set once the worktree is created
    error?: string;
  }>;
  startedAt: number;
}
```

The specification's task list is parsed from the markdown `## Tasks` section: lines
matching `/^- \[ \] (.+)/` are unchecked tasks; lines matching `/^- \[x\] (.+)/i` are
done. A spec with no `## Tasks` section or no matching items triggers a graceful error.

The concurrency limit is controlled by `ARKE_MAX_CONCURRENT_TASKS` (default `10`). The
internal queue drains as `session.idle` events arrive.

### Interfaces and contracts

New types added to the coordinator (not to `@arke/contracts` — these are internal):

```typescript
interface TaskCommand {
  kind: "dispatch-task";
  specId: string;
  specSessionId: string;
  taskIndex: number;
  taskText: string;
  featureBranch: string;   // e.g. feat/parallel-task-execution
  worktreeBranch: string;  // e.g. feat/parallel-task-execution/task-0
}

class Decider {
  onSpecApproved(specId: string, specSessionId: string, tasks: string[], branch: string): TaskCommand[];
}

class Dispatcher {
  async execute(cmd: TaskCommand, adapter: HarnessAdapter): Promise<void>;
  // Writes to FanOutRecord before and after; emits session.status; writes Trace.
}
```

The `HarnessAdapter.dispatchAsync` contract (`packages/contracts/src/adapter.ts`) is
already in place; `CreateSessionInput.parent` is the link field. The worktree shell
command is issued via the Node `child_process` module from the `Dispatcher`.

### Cross-cutting
- **Configuration:** all project configuration — the harness/model registry, coordinator settings, and integrations — lives in the single root file `.arke/config.json` (`ARKE_*` env vars override individual keys; runtime artefacts under `.arke/` are state, not config). See SPEC-005 for the model.
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.
- **Ordering:** `dispatchAsync` calls are issued in task-list order, though all run
  concurrently (up to the concurrency limit); the order is deterministic so logs are
  readable.
- **Observability:** each `TaskCommand` execution is written to `Trace` before the
  `dispatchAsync` call and on completion (success or failure), giving a durable record of
  fan-out intent and outcome. Partial failure trace entries (`dispatch.failed`) carry the
  `taskIndex`, `reason`, and `worktreeBranch` to make recovery actionable.
- **Worktree lifecycle:** worktrees are not deleted automatically by the coordinator on
  task completion. They remain for diff review and PR creation. Orphaned worktrees (from
  failed/abandoned tasks) are surfaced via branch-collision detection on retry, which emits
  a `warn` trace entry. A future tidy-up command (not in this spec) will list and remove
  stale worktrees.
- **Security:** the `git worktree add` command is called with a fixed, coordinator-derived
  branch name (`feat/<slug>/task-<n>`). The slug is derived from the spec's `spec_id`
  frontmatter value, which is validated as an alphanumeric-and-hyphen string at read time.
  No client-supplied or user-supplied value is interpolated into the shell command.

## Tasks
- [ ] Add `Decider` class: pure function mapping `spec.status approved` + parsed task list
  → `TaskCommand[]`; include worktree branch name derivation; validate task-list presence
  and emit graceful error if absent
- [ ] Add `Dispatcher` class: per-task, check branch collision → create git worktree →
  call `adapter.createSession` with parent → write to session graph (SPEC-002) → call
  `adapter.dispatchAsync` → write to `Trace` (before and after); treat each step's
  failure as a per-task dispatch failure, not a fan-out abort
- [ ] Add `FanOutRecord` persistence (alongside SPEC-003 store): write before dispatch,
  update on success/failure; check on startup for existing records to guard restart
  idempotency
- [ ] Parse `## Tasks` markdown section in the coordinator (regex for `- [ ]` / `- [x]`);
  emit graceful error + trace warn if no tasks found
- [ ] Wire `Decider` + `Dispatcher` into `Coordinator.pump()` on `spec.status approved`;
  check `FanOutRecord` for already-dispatched tasks before issuing commands
- [ ] Implement concurrency queue: cap at `ARKE_MAX_CONCURRENT_TASKS`; drain on
  `session.idle`; respect spec-demoted mid-fan-out (do not dispatch queued tasks)
- [ ] Emit `session.status running kind:task` after each successful `dispatchAsync`; emit
  `session.status error` on dispatch failure; fold `todo.updated` into task card in read model
- [ ] Update `ReadModel.deriveColumn` to handle task `done → diff` transition on
  `session.idle`
- [ ] Add spec-demotion guard: on `spec.status in-review` while a fan-out is queued, halt
  remaining queued tasks and write `fanout.halted` trace entry

### Testing
- Unit (`Decider`): approved spec with mixed checked/unchecked tasks → correct command
  list; duplicate `approved` event → empty command list (idempotency); spec with no `##
  Tasks` section → graceful error command, not throw; worktree branch name is
  deterministic (`feat/<slug>/task-<n>`) and does not vary across calls.
- Unit (`Dispatcher`): branch collision detected → no dispatch, warn trace emitted; worktree
  creation failure → no `dispatchAsync` call, failure trace entry, card → `needs-human`;
  partial failure (second of three dispatches fails) → first and third proceed, second
  fails independently; verify session graph write before `dispatchAsync`.
- Unit (concurrency queue): fifteen tasks, limit 10 → ten dispatched immediately, five
  queued; `session.idle` drains queue one by one; spec demoted mid-queue → queued tasks
  not dispatched.
- Unit (restart idempotency): `FanOutRecord` contains two tasks with `status: "running"` →
  third task (status `queued`) is dispatched; running tasks are not re-dispatched.
- Integration: stub adapter emitting `session.idle` events; assert all three task cards
  move to `diff` column independently; assert one card moves to `needs-human` when its
  `dispatchAsync` stub throws.
- Manual: approve a real spec in the cockpit, watch three board cards appear concurrently
  in the `implementing` column, confirm they do not share a worktree; verify restart
  idempotency by killing and restarting the coordinator mid-run.

### Definition of done
All scenarios above pass; the coordinator fans out a three-task spec concurrently in the
integration test; partial failure test demonstrates independent card moves; the board shows
distinct task cards; restart idempotency is verified by the unit test against `FanOutRecord`;
a reviewer has signed off.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Pure `Decider` function rather than inline orchestration | Deterministic, testable without the adapter; mirrors T3 decider pattern |
| 2 | `FanOutRecord` persisted (not in-memory only) for idempotency | In-memory guard is lost on restart; without persistence a coordinator restart mid-fan-out re-dispatches already-running tasks (duplicate session bug); SPEC-003 store is the vehicle |
| 3 | Git worktrees per task session | Prevents concurrent write collisions without a merge step mid-flight |
| 4 | `dispatchAsync` called without `await` on each | Non-blocking is the contract; concurrency is the point |
| 5 | Per-task failure isolation in Dispatcher | A harness blip for one task must not abort a 10-task fan-out; partial progress is preferable and recoverable |
| 6 | Deterministic worktree branch name (`feat/<slug>/task-<n>`) | Makes retry safe: the same task index always maps to the same branch, so a collision on retry means an orphaned worktree exists rather than a naming race |
| 7 | Branch collision check before worktree create | Prevents silent overwrite of in-progress or orphaned worktrees; surfaces the problem as an actionable warn entry |
| 8 | Concurrency cap (`ARKE_MAX_CONCURRENT_TASKS`) with queue | Unbounded fan-out on a large spec would exhaust harness resources and local disk; cap with drain is the safe default |
| 9 | Queued tasks not dispatched when spec demoted | Dispatching tasks for a spec that is no longer approved violates propose-decide-execute; the human must re-approve before work resumes |

## Open questions
- Should the worktree creation be part of the `Dispatcher` or delegated to the adapter
  (so Claude Code adapters can use their own worktree mechanism)? Keeping it in the
  `Dispatcher` is simpler for Phase 1; adapter-delegated creation is the correct
  long-term model when non-OpenCode harnesses land.
- If the coordinator restarts mid-fan-out, `FanOutRecord` enables task-level resume. Should
  `queued` tasks resume automatically on restart, or require explicit operator re-approval?
  Automatic resume risks acting on a spec the engineer may have edited since approval;
  explicit re-approval is safer.
- Default value of `ARKE_MAX_CONCURRENT_TASKS`: 10 is conservative. Should this be
  per-spec (based on task count) or global?
