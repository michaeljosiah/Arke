# 1 Â· Requirements

## 1.1 Summary
Stand up the Arke foundation: an npm-workspaces monorepo with schema-first domain
contracts, a thin local coordinator, the first (OpenCode) harness adapter skeleton, and a
React client that reproduces the approved design baseline. This is Phase 1 ("prove the
loop", PRD Â§16) groundwork â€” the scaffolding everything else builds on.

## 1.2 Scope
**In scope**
- Monorepo structure, shared TypeScript config, contracts package (zod).
- Coordinator skeleton: WebSocket push, normalized read model, append-only trace.
- OpenCode adapter interface + skeleton mapped to OpenCode primitives.
- React client reproducing the design prototype (all v1 screens reachable).
- Application launch/splash screen per the `arke-design` skill (`Arke Launch Screen Light.html`).
- Method scaffolding: agent roster, specification template, grounding file.

**Out of scope (this spec)**
- A live end-to-end loop against a running OpenCode server (follow-up spec).
- Deterministic projection plugins (Jira / Azure DevOps).
- Electron packaging beyond a skeleton; multi-harness routing; team tier.

## 1.3 Acceptance criteria
- **R-1** The repository SHALL be an npm-workspaces monorepo with `packages/contracts`,
  `packages/coordinator`, `packages/adapter-opencode`, `packages/client`, and `apps/desktop`.
  - **WHEN** `npm install` is run at the root **THEN** all workspaces resolve without error.
- **R-2** The contracts package SHALL define, as zod schemas, the spec lifecycle, the
  normalized domain-event union, and the `HarnessAdapter` interface with capability flags.
  - **WHEN** a malformed event is parsed **THEN** validation fails at the boundary.
- **R-3** The coordinator SHALL push ordered, monotonically sequenced, schema-validated
  events over WebSocket and persist each to an append-only trace.
  - **WHEN** the coordinator starts with the mock adapter **THEN** a client receives a
    snapshot followed by sequenced events, and `.arke/trace.ndjson` grows.
- **R-4** The client SHALL reproduce the approved design baseline with every v1 screen
  reachable from the shell navigation.
  - **WHEN** `npm run build -w @arke/client` runs **THEN** it succeeds and emits `dist/`.
- **R-5** The project SHALL ship method-ready: agent roster under `.opencode/agents`, the
  `specification.template.md`, and the `AGENTS.md` grounding file.
  - **WHEN** a new specification is started **THEN** it inherits the template's rigor.
- **R-6** The client SHALL present an application launch (splash) screen on start, rendered
  to the approved design in the `arke-design` skill (`Arke Launch Screen Light.html`, the
  light default; a dark variant is available): the `//Arke` wordmark, the "Specification
  orchestrator" tag, and a live harness-reachability indicator ("probing harness"). It SHALL
  respect reduced-motion, and SHALL hand off to first-run onboarding (SPEC-004) when no
  reachable harness or open project is found, rather than entering the app.
  - **WHEN** the application starts **THEN** the launch screen renders and a harness-reachability
    probe runs, reflected by the live indicator
  - **AND** when a harness is reachable and a project is open it transitions into the app
  - **AND** otherwise it hands off to first-run onboarding (SPEC-004)

## 1.4 Open questions
- Exact OpenCode endpoint/event names (resolved by the integration guide in `docs/analysis`).
- Whether the client wires to `@arke/contracts` types now or after the live-loop spec.
- Electron vs Tauri for the desktop shell (PRD open question; Electron is the default).

# 2 Â· Design

## 2.1 Architectural decision
Thin client over a harness that owns execution, with a thin local coordinator normalising
provider events into one schema-validated domain model (PRD Â§8, Â§8.5; D2, D12). No cloud
backend on the hot path.

## 2.2 Target architecture
React client â‡„ (WebSocket) â‡„ local coordinator â‡„ (HTTP/SSE) â‡„ OpenCode harness host.
Contracts are shared by client and coordinator. See `AGENTS.md` for the package map.

## 2.3 Data model
No database in v1. State homes per PRD Â§8.4: spec in git, code in branches/worktrees,
delivery state derived from events, audit in the local NDJSON trace, client view state
in memory only.

## 2.4 API contracts
The `HarnessAdapter` interface (`packages/contracts/src/adapter.ts`) and the `DomainEvent`
union (`packages/contracts/src/events.ts`). The coordinator WebSocket emits `snapshot` then
`event` frames.

## 2.5 Application services
Coordinator: `ReadModel` (folds events â†’ board cards), `Trace` (audit), adapter pump.

## 2.6 Security
No credentials in the client (NFR-1). All harness/CLI/MCP auth at the host. The coordinator
runs inside the trust boundary.

## 2.7 Performance
Event-driven, never polled (NFR-3). Pushes are ordered and sequenced; the client transport
will be a reconnecting state machine (NFR-8) in the live-loop spec.

## 2.8 Design system (canonical template)
All Arke UI â€” this client, the launch screen, and every screen specified downstream â€” follows
the **`arke-design` skill** at `.claude/skills/arke-design/` (see its
[`SKILL.md`](../../.claude/skills/arke-design/SKILL.md)): the shadcn/ui neutral monochrome
token contract (`_ds/.../tokens/`), Geist typography, Lucide icons, the brand voice, and the
prototype screens (`app/*.jsx`) as the reference implementation. The launch screen renders to
`Arke Launch Screen Light.html`. The skill is the canonical design template; specs reference it
rather than re-deriving styling.
