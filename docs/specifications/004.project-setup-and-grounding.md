# Project setup — onboarding, scaffold & grounding

## Why
A repository must become method-ready in one action, and a first-run engineer must be guided to a reachable harness and an opened or created project — without needing model knowledge or touching source files in an existing codebase.

## What changes
- ADDED onboarding — the client checks harness reachability on launch and guides the engineer to install, start, or configure one before any project is opened (FR-20)
- ADDED scaffolding — opening a new or existing folder triggers folder-state detection and, when scaffolding is chosen, creates the full method-ready structure in one action (FR-18)
- ADDED grounding — after a folder is opened the system uses a harness session to analyse the repository and produce or refresh `AGENTS.md`, surfacing guides and relevant code so authoring is grounded before the first spec is written (FR-3)
- MODIFIED onboarding — the project picker's "Initialise a new project" path is wired to the real coordinator scaffolding action rather than the design-baseline simulation   (breaking: no)

## Requirements

### Requirement: Harness reachability gate
`capability: onboarding` · `delta: ADDED (feat/project-setup-and-grounding)`

The system SHALL, on client launch, probe the configured harness endpoint and, if no harness is reachable, present a dedicated first-run surface that guides the engineer to install, start, or point to a harness before any project action is available. The gate SHALL be non-blocking for engineers who already have a reachable harness. A probe that receives a response other than a successful health reply (e.g. a connection error, an HTTP error status, or a timeout) SHALL be treated as unreachable and SHALL surface a reason string to the guidance surface; a partial response (e.g. the health endpoint exists but capabilities cannot be confirmed) SHALL be reported distinctly from a clean failure. If a "Retry connection" probe itself does not respond within the probe window the guidance surface SHALL remain visible and the engineer SHALL be offered the option to adjust the configured endpoint.

#### Scenario: No harness reachable on launch
- **WHEN** the client starts and no harness responds at any configured endpoint within a 3-second probe
- **THEN** the client renders the harness-setup guidance surface rather than the project picker
- **AND** the guidance surface shows install and start instructions for OpenCode and any other registered harness drivers
- **AND** a "Retry connection" action re-runs the probe without restarting the client

#### Scenario: Harness responds with an error, not silence
- **WHEN** a configured harness endpoint is reachable but returns an HTTP error or a malformed health response
- **THEN** the guidance surface is shown with the reason (e.g. "Harness responded with 503") rather than the generic "unreachable" copy
- **AND** the retry action is still available

#### Scenario: Retry probe itself times out
- **WHEN** the engineer presses "Retry connection" and the probe does not resolve within the probe window
- **THEN** the guidance surface remains visible and offers an "Edit endpoint" action in addition to retry
- **AND** no project action becomes available

#### Scenario: Harness becomes reachable after guidance
- **WHEN** the engineer starts a harness and presses "Retry connection"
- **THEN** the probe succeeds and the client proceeds to the project picker
- **AND** no project action was available before the harness was confirmed reachable

#### Scenario: Harness already reachable on launch
- **WHEN** the client starts and at least one harness responds within the probe window
- **THEN** the harness-setup surface is skipped and the project picker is shown directly

### Requirement: Input validation — folder paths and clone URLs
`capability: onboarding` · `delta: ADDED (feat/project-setup-and-grounding)`

The coordinator SHALL validate and canonicalise every folder path and git clone URL it receives before acting on it. A `folder.inspect` or `scaffold.run` path SHALL be resolved to an absolute path and checked that it does not escape outside a configured safe root (or, where no safe root is configured, outside the filesystem mount). A `repo.clone` URL SHALL be validated as a well-formed `https://` or `ssh://` git URL before the clone is executed; `file://` URLs, bare path strings, and any URL with shell-special characters SHALL be rejected. The canonical path SHALL be used in all subsequent operations; the raw client-supplied value SHALL be discarded after validation.

#### Scenario: Path traversal in `folder.inspect` is rejected
- **WHEN** the coordinator receives a `folder.inspect` message with `path: "../../etc"`
- **THEN** the coordinator rejects the request with a `validation-error` response naming the reason
- **AND** no filesystem access is performed outside the safe root

#### Scenario: `repo.clone` with an invalid URL is rejected
- **WHEN** the coordinator receives a `repo.clone` message with `url: "file:///etc/passwd"` or `url: "; rm -rf /"`)
- **THEN** the coordinator rejects the request with a `validation-error` response before invoking git
- **AND** no clone operation is started

#### Scenario: Valid URL and path proceed normally
- **WHEN** the coordinator receives a `repo.clone` with a valid `https://` URL and a `targetPath` inside the safe root
- **THEN** the clone proceeds and folder-state detection runs on the cloned result

### Requirement: Folder-state detection on open
`capability: onboarding` · `delta: ADDED (feat/project-setup-and-grounding)`

The system SHALL let the engineer open a local folder, supply a git URL to clone, or create a new empty project. Once a folder is resolved the system SHALL inspect it and classify its state — method-ready, has-existing-code, partial-scaffold, or empty — then open or offer to scaffold accordingly, without modifying any source file in an existing codebase. A folder is **partial-scaffold** when some but not all of the three method-ready sentinels (`.opencode/agents/`, `docs/specifications/`, `AGENTS.md`) are present; this state SHALL be reported distinctly from `has-existing-code` so the initialisation screen can explain exactly what is missing rather than offering a full scaffold as if the folder were pristine.

#### Scenario: Opening a method-ready folder
- **WHEN** the engineer opens a folder that already contains `.opencode/agents/`, `docs/specifications/`, and `AGENTS.md`
- **THEN** the client opens directly into the spec library for that project
- **AND** no scaffolding action is offered or run

#### Scenario: Opening a folder with existing code but no method scaffold
- **WHEN** the engineer opens a folder that contains source files but lacks all three method-ready sentinels
- **THEN** the client presents the initialisation screen pre-filled with the detected repository name
- **AND** offers "Scaffold this project" as the primary action, explaining what will be added
- **AND** no source file is read, modified, or deleted before the engineer confirms

#### Scenario: Opening a partially-scaffolded folder
- **WHEN** the engineer opens a folder where one or two (but not all three) method-ready sentinels exist
- **THEN** the client presents the initialisation screen with a `partial-scaffold` classification
- **AND** the screen lists specifically which sentinels are absent and what will be added to complete them
- **AND** existing sentinels are not overwritten by the scaffold step that creates the missing ones

#### Scenario: Opening an empty folder or creating a new project
- **WHEN** the engineer opens an empty folder or chooses "Create new project"
- **THEN** the client presents the initialisation screen with sensible defaults and no warnings about existing code

#### Scenario: Cloning a git URL
- **WHEN** the engineer supplies a validated git clone URL
- **THEN** the coordinator clones the repository into a local folder chosen by the engineer
- **AND** folder-state detection proceeds on the cloned result

### Requirement: One-action method-ready scaffolding — idempotent and non-destructive
`capability: scaffolding` · `delta: ADDED (feat/project-setup-and-grounding)`

The system SHALL scaffold a repository into a method-ready state in a single confirmed action. The scaffold SHALL create the agent roster, the specification directory structure plus the canonical template, the `AGENTS.md` grounding baseline stub, and the policy and projection plugin stubs. Vendored reference copies SHALL be written under `.repos/` as read-only grounding material; if git is unavailable this step SHALL be skipped and a `scaffold.step` event with `status: 'skipped'` and an explanatory `detail` SHALL be emitted (not silently dropped). Tier defaults SHALL be pre-filled so no model knowledge is required from the engineer. The scaffold SHALL NOT modify any existing source file.

Each scaffold step SHALL be idempotent: on a re-run, an artefact is **skipped** if it already exists and its content was produced by a previous scaffold run (detected via a sentinel comment or checksum stored in `.arke/scaffold-manifest.json`); it is **overwritten** only when it is listed as stale in the manifest or is absent. A user-customised artefact (one whose checksum does not match the manifest's recorded scaffold-time checksum) SHALL be left untouched and the step SHALL be reported `skipped` with reason `user-modified`. The manifest is written atomically after each successful step and is the authoritative record of what the scaffold produced.

#### Scenario: Scaffold completes all four steps
- **WHEN** the engineer confirms "Run scaffold" on the initialisation screen
- **THEN** the coordinator executes the four scaffold steps in order: agent roster, specification structure, grounding baseline stub, policy & projection plugins
- **AND** each step is reported in the terminal panel as it completes
- **AND** on completion `.opencode/agents/` contains the six canonical role files, `docs/specifications/` contains `specification.template.md` and `README.md`, `AGENTS.md` exists at the repo root, and at least one projection plugin stub exists
- **AND** `.arke/scaffold-manifest.json` records the scaffold-time checksum and timestamp for every artefact created

#### Scenario: Tier defaults require no model input
- **WHEN** the initialisation screen is displayed before scaffolding
- **THEN** the tier fields (capable, mid) are pre-filled with sensible defaults from the coordinator registry
- **AND** the engineer can proceed through scaffolding without changing them
- **AND** if the registry is unavailable or returns no defaults, the tier fields show placeholder labels (e.g. "capable — not configured") and scaffolding is blocked until the registry is reachable (cross-ref SPEC-005)

#### Scenario: Scaffold is idempotent — existing artefacts untouched unless absent or stale
- **WHEN** the scaffold action is run on a folder that already contains some scaffold artefacts
- **THEN** each artefact is compared against `.arke/scaffold-manifest.json`
- **AND** artefacts that exist and whose checksum matches the manifest are skipped (terminal panel reports `skipped`)
- **AND** artefacts that are absent or stale (manifest marks them stale) are created or overwritten
- **AND** artefacts whose checksum does not match (user-modified) are left untouched and reported `skipped (user-modified)`

#### Scenario: Source files are untouched
- **WHEN** the scaffold runs in a folder that contains application source code
- **THEN** no file outside `.opencode/`, `docs/specifications/`, `AGENTS.md`, `.repos/`, and `.arke/scaffold-manifest.json` is created, modified, or deleted

#### Scenario: Vendored `.repos/` step skipped without git
- **WHEN** the scaffold runs and git is not available on the host
- **THEN** a `scaffold.step` event with `step: 'repos'`, `status: 'skipped'`, and `detail: 'git not found on PATH'` is emitted
- **AND** the remaining steps complete normally
- **AND** the terminal panel shows the skip reason rather than silently omitting the step

### Requirement: Partial scaffold failure and resume
`capability: scaffolding` · `delta: ADDED (feat/project-setup-and-grounding)`

If a scaffold step fails, the coordinator SHALL stop execution at that step, emit a `scaffold.step` event with `status: 'error'` and the failure reason, and offer a "Retry from step" action that resumes from the failed step rather than restarting all steps. The scaffold manifest SHALL record the last successfully completed step so that a resume can skip already-completed steps. Steps that completed successfully before the failure SHALL NOT be re-executed on resume unless explicitly requested.

#### Scenario: Single step fails mid-scaffold
- **WHEN** the scaffold is running and the `specs` step fails (e.g. a write permission error)
- **THEN** the terminal panel reports `agents: done`, `specs: error — <reason>`
- **AND** the `grounding` and `plugins` steps are not attempted
- **AND** a "Retry from specs" action is offered
- **AND** on retry the `agents` step is skipped (already recorded in the manifest as done)

#### Scenario: Resume after process crash
- **WHEN** the coordinator process exits while a scaffold is in progress and is restarted
- **THEN** the coordinator reads `.arke/scaffold-manifest.json` and offers to resume the scaffold from the last incomplete step
- **AND** completed steps are not re-run

### Requirement: Repository analysis and grounding baseline
`capability: grounding` · `delta: ADDED (feat/project-setup-and-grounding)`

The system SHALL, after a project folder is opened or scaffolded, use a harness session to analyse the repository and produce or update `AGENTS.md` with a summary of the codebase structure, key entry points, and relevant conventions. The analysis SHALL NOT modify any source file. The researcher session's trace entry SHALL record: the harness session id, the researcher role, the UTC timestamp of completion, and the git SHA of the `AGENTS.md` commit produced, so that provenance is fully auditable. On completion the client SHALL surface `AGENTS.md`, any existing guides under `docs/`, and relevant code extracts so spec authoring begins grounded.

When `AGENTS.md` already exists and the grounding session produces a different analysis, the coordinator SHALL write the new version in full (not attempt section-level merging), commit it through the harness, and record both the previous and new git SHAs in the trace. The previous version is recoverable from git history.

#### Scenario: Fresh grounding baseline produced after scaffold
- **WHEN** scaffolding completes on a folder that contains existing source code
- **THEN** the coordinator opens a harness session scoped to the researcher role
- **AND** the session analyses the repository and writes an `AGENTS.md` baseline covering module structure, key entry points, and relevant conventions
- **AND** the session closes and the trace records `{ sessionId, role: 'researcher', completedAt, agentsMdSha }`
- **AND** the result is surfaced in the authoring cockpit before the first specification is started

#### Scenario: Grounding baseline refreshed on open
- **WHEN** the engineer opens an existing method-ready project where `AGENTS.md` was last updated more than 24 hours ago (determined from the git commit timestamp recorded in the trace, not the file's mtime)
- **THEN** the client offers a "Refresh grounding" action
- **AND** if accepted the coordinator runs a fresh researcher session that rewrites `AGENTS.md` in full and commits it through the harness

#### Scenario: Existing `AGENTS.md` overwritten on refresh
- **WHEN** a grounding refresh produces an `AGENTS.md` different from the current version
- **THEN** the new version is written in full, committed through the harness, and the trace records both the previous SHA and the new SHA
- **AND** the previous version is recoverable from git history

#### Scenario: Grounding surface shown in the cockpit
- **WHEN** the authoring cockpit opens for a specification
- **THEN** a grounding panel shows the current `AGENTS.md`, any guides under `docs/`, and code extracts relevant to the spec under authorship
- **AND** the agent roster and relevant existing files are reachable from that panel without leaving the cockpit

### Requirement: Credentials and file contents never sent to the client
`capability: onboarding` · `delta: ADDED (feat/project-setup-and-grounding)`

The coordinator SHALL NOT include any credential value, git token, environment variable value, or file content (beyond the project name and folder path returned for display) in any response to a `folder.inspect`, `repo.clone`, `scaffold.run`, or `harness.probe` client message, nor in any `scaffold.step`, `scaffold.done`, `harness.reachability`, or grounding-related event pushed to the client. The client receives structural metadata (step name, status, project path, tier labels) only.

#### Scenario: Scaffold events carry no sensitive content
- **WHEN** a `scaffold.run` completes and `scaffold.step` / `scaffold.done` events are pushed to the client
- **THEN** no event payload contains a credential value, a git token, a filesystem path outside the project root, or the content of any file
- **AND** the `projectPath` in `scaffold.done` is the canonicalised project root, not a raw client-supplied value

### Requirement: Tier defaults pre-filled, no model knowledge required
`capability: onboarding` · `delta: ADDED (feat/project-setup-and-grounding)`

The system SHALL pre-fill logical model tier defaults (capable, mid) on the initialisation screen from the coordinator registry so that the engineer can complete onboarding and scaffolding without knowing any vendor model identifier. The tier labels SHALL be shown alongside a brief description of their intended use. If the registry is unavailable or serves no defaults, the tier fields SHALL show a clearly disabled state and scaffolding SHALL be blocked with a prompt to configure the registry (cross-ref SPEC-005), rather than proceeding with empty or guessed tier values.

#### Scenario: Tier fields shown with descriptions
- **WHEN** the initialisation screen is rendered and the registry is reachable
- **THEN** each logical tier (capable, mid) is shown with a human-readable label, a brief purpose description, and the model name the registry currently resolves for it
- **AND** no vendor model identifier is required to confirm and run the scaffold

#### Scenario: Registry unavailable — scaffolding blocked, not silent
- **WHEN** the initialisation screen is rendered but the coordinator registry returns no tier defaults (registry config absent or unreadable)
- **THEN** the tier fields show "not configured" labels
- **AND** the "Run scaffold" button is disabled with a tooltip directing the engineer to configure the registry
- **AND** no scaffold is run with empty tier values

#### Scenario: Tier defaults propagate into the roster
- **WHEN** the scaffold creates the agent roster files under `.opencode/agents/`
- **THEN** each roster file carries a `tier:` reference, not a hardcoded model identifier
- **AND** the coordinator registry is the single place vendor model IDs are recorded

## Design

### Architectural decision
The scaffolding action is a coordinator-side command, not a client-side file write — so credentials, git access, and file I/O stay on the host (NFR-1). The client sends a `scaffold.run` request over the existing WebSocket; the coordinator executes it step-by-step, emitting progress events that the client projects onto the terminal panel. Grounding analysis is a thin harness session using the `researcher` role; its output is committed to `AGENTS.md` through the harness, keeping the same git-backed audit trail as any other spec write. All client-supplied paths and URLs are validated and canonicalised before any filesystem or git operation.

### Target architecture

```
Client (Initialisation screen)
  ↕ WebSocket (ws://127.0.0.1:4319)
Coordinator
  ├── InputValidator          — canonicalises paths; validates clone URLs; rejects traversal
  ├── ScaffoldRunner          — orchestrates the four scaffold steps; reads/writes scaffold-manifest.json; emits scaffold.step events
  ├── FolderInspector         — detects method-ready / partial-scaffold / has-code / empty state
  ├── HarnessReachabilityProbe— pings configured endpoints; result → snapshot + reachability event
  └── HarnessAdapter          — researcher session for grounding analysis (AGENTS.md)
       ↕ HTTP / SSE
OpenCode server (opencode serve)
```

**New coordinator modules** (all within `packages/coordinator/src/`):
- `input-validator.ts` — `InputValidator.canonicalisePath(raw, safeRoot)` and `InputValidator.validateCloneUrl(raw)`.
- `scaffold.ts` — `ScaffoldRunner` with four steps: `agents`, `specs`, `grounding`, `plugins`, plus `repos` (advisory); reads and writes `.arke/scaffold-manifest.json`; emits `scaffold.step` and `scaffold.done` events into the trace and push channel; supports `resumeFrom(step)`.
- `folder-inspector.ts` — `FolderInspector.classify(path)` returns `'method-ready' | 'partial-scaffold' | 'has-code' | 'empty'` plus `{ missingSentinels: string[] }`.
- `reachability.ts` — `HarnessReachabilityProbe.probe(endpoints)` returns per-endpoint reachability with a `reason` string; wired into startup.

**New domain events** (additions to `@arke/contracts/src/events.ts`):

```typescript
export const ScaffoldStepEvent = base.extend({
  type: z.literal('scaffold.step'),
  step: z.enum(['agents', 'specs', 'grounding', 'plugins', 'repos']),
  status: z.enum(['running', 'done', 'skipped', 'error']),
  detail: z.string().optional(),  // skip reason or error message; no credential values
});

export const ScaffoldDoneEvent = base.extend({
  type: z.literal('scaffold.done'),
  projectPath: z.string(),        // canonicalised project root only
  stepsRun: z.array(z.string()),
});

export const HarnessReachabilityEvent = base.extend({
  type: z.literal('harness.reachability'),
  endpoint: z.string(),
  reachable: z.boolean(),
  reason: z.string().optional(),  // human-readable failure reason; no credential values
});
```

These join the `DomainEvent` discriminated union. Note: `harness.reachability` here is per-endpoint; the SPEC-005 `registry.updated` event is the authoritative registry projection for the client. These are complementary: `harness.reachability` fires from the onboarding probe path; `registry.updated` fires from the full registry. Cross-ref SPEC-005.

**Scaffold manifest** (`.arke/scaffold-manifest.json`):

```json
{
  "version": 1,
  "lastCompletedStep": "agents",
  "artefacts": {
    ".opencode/agents/spec-author.md": { "scaffoldChecksum": "<sha256>", "createdAt": "..." },
    "AGENTS.md": { "scaffoldChecksum": "<sha256>", "createdAt": "..." }
  }
}
```

Written atomically (write to `.arke/scaffold-manifest.json.tmp`, rename) after each successful step.

**Client changes** (`packages/client/src/`):
- `screens/picker-init-library.tsx` — `Initialisation`: replace the simulated `run()` with a `scaffold.run` WebSocket message; consume `scaffold.step` / `scaffold.done` events for the terminal panel; disable "Run scaffold" when tier defaults are absent.
- `screens/picker-init-library.tsx` — `Picker`: on mount, check the snapshot's `harnessReachable` flag; if false, render the harness-setup guidance surface in place of the project list.
- New `HarnessGuidance` component — install/start instructions per driver; "Retry connection" fires a `harness.probe` request; shows reason string on partial/error reachability.

### Data model
No persistent database beyond the scaffold manifest. Scaffold artefacts are files committed by the harness into git (the same trust boundary as spec files). The coordinator's in-memory snapshot gains:
- `harnessReachable: boolean`
- `harnessReachabilityReason?: string`
- `projectState: 'method-ready' | 'partial-scaffold' | 'has-code' | 'empty' | null`
- `missingSentinels?: string[]`

pushed in the `snapshot` frame on each (re)connection.

### Interfaces and contracts

**WebSocket client→coordinator messages** (additions):

```typescript
// Check harness reachability; coordinator responds with harness.reachability events
{ type: 'harness.probe' }

// Detect the state of a local folder (path validated server-side)
{ type: 'folder.inspect', path: string }

// Clone a remote repository (url and targetPath validated server-side)
{ type: 'repo.clone', url: string, targetPath: string }

// Run the scaffold steps on the current project path
// resumeFrom: if present, skip steps before this step (uses manifest)
{ type: 'scaffold.run', path: string, tiers?: { capable?: string; mid?: string }, resumeFrom?: string }
```

**Scaffold artefacts written by `ScaffoldRunner`:**

| Step | Artefacts created |
|------|-------------------|
| `agents` | `.opencode/agents/{spec-author,architect,reviewer-a,reviewer-b,implementer,researcher}.md` |
| `specs` | `docs/specifications/specification.template.md`, `docs/specifications/README.md` |
| `grounding` | `AGENTS.md` at repo root (stub, then enriched by grounding session) |
| `plugins` | `plugins/policy.ts` (permission policy stub), `plugins/projection.ts` (projection stub) |
| `repos` | `.repos/` vendored references (skipped with `detail` if git unavailable) |

**`FolderInspector.classify` heuristic:**

```
method-ready    → all three sentinels present: .opencode/agents/ AND docs/specifications/ AND AGENTS.md
partial-scaffold→ one or two sentinels present (list of missing returned)
has-code        → no sentinels but source files exist (*.ts, *.js, *.py, *.go, *.rs, …)
empty           → no source files found
```

### Cross-cutting
- **Configuration:** all project configuration — the harness/model registry, coordinator settings, and integrations — lives in the single root file `.arke/config.json` (`ARKE_*` env vars override individual keys; runtime artefacts under `.arke/` are state, not config). See SPEC-005 for the model.
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) — the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.
- **Security (NFR-1):** All file I/O, git clone, and harness session creation run inside the coordinator on the host. The client sends intent; the coordinator acts. All client-supplied paths are canonicalised and validated against a safe root before use. Clone URLs are validated as well-formed `https://` or `ssh://` before invoking git. No file paths, credential values, or file contents are included in any event pushed to the client beyond the project name and canonicalised project root.
- **Observability (NFR-7):** Every scaffold step and grounding session is written to `.arke/trace.ndjson` with the same `Trace.write()` path as domain events. The researcher session's trace entry records `{ sessionId, role, completedAt, agentsMdSha, previousAgentsMdSha? }` — full grounding provenance. Skipped steps are traced with their reason.
- **Resilience (NFR-4):** If a scaffold step fails, the terminal panel shows the error and offers a "Retry from step" action. The scaffold manifest preserves completed-step state so resume skips already-completed steps. The harness-reachability probe surfaces a reason string on any failure, not just silent timeout. If tier defaults are absent, the scaffold is blocked (not run with empty values).
- **Accessibility (NFR-6):** The harness-guidance surface and initialisation screen are keyboard-navigable; the terminal panel uses `role="log"` with `aria-live="polite"`.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Scaffold runs in the coordinator, not the client | Keeps file I/O and git access on the host (NFR-1); coordinator already owns the harness trust boundary |
| 2 | Grounding analysis via the researcher harness session, not a static template | The codebase is unknown at scaffold time; only a live analysis can produce an accurate `AGENTS.md` for an existing repo |
| 3 | Tier defaults from the coordinator registry, not hardcoded in the client | Vendor model IDs must live in one place behind the gateway (FR-4, roster doc §2) |
| 4 | Scaffold steps are idempotent via a manifest | Engineers re-run scaffolding after pulling updates; checksum-based detection prevents accidental overwrites and distinguishes system-produced vs user-modified artefacts |
| 5 | `FolderInspector` checks three sentinel paths, not a full tree walk | Fast and deterministic; four-way classification (adding `partial-scaffold`) gives the UI precise information about what is missing |
| 6 | All client-supplied paths validated and canonicalised on the coordinator | Prevents path-traversal and shell-injection from a browser-supplied value; aligns with SPEC-002's directory validation rule |
| 7 | `AGENTS.md` overwritten in full on refresh | Section-level merging requires a schema that does not exist; full rewrite is deterministic and git history preserves the previous version |
| 8 | Grounding provenance recorded in trace with git SHAs | Freshness check anchors on git commit timestamp in trace, not filesystem mtime; provenance is auditable and recoverable |
| 9 | Tier defaults absent → scaffold blocked, not silent | Running a scaffold with empty tier values would produce unusable agent files; fail loudly and direct the engineer to configure the registry |
| 10 | `repos` step emits `skipped` with detail rather than silent omission | Silent skips hide missing grounding material; the engineer must know if vendored references were not written |
