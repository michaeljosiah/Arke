# Project setup ‚Äî onboarding, scaffold & grounding

## Why
A repository must become method-ready in one action, and a first-run engineer must be guided to a reachable harness and an opened or created project ‚Äî without needing model knowledge or touching source files in an existing codebase.

## What changes
- ADDED onboarding ‚Äî the client checks harness reachability on launch and guides the engineer to install, start, or configure one before any project is opened (FR-20)
- ADDED scaffolding ‚Äî opening a new or existing folder triggers folder-state detection and, when scaffolding is chosen, creates the full method-ready structure in one action (FR-18)
- ADDED grounding ‚Äî after a folder is opened the system uses a harness session to analyse the repository and produce or refresh `AGENTS.md`, surfacing guides and relevant code so authoring is grounded before the first spec is written (FR-3)
- MODIFIED onboarding ‚Äî the project picker's "Initialise a new project" path is wired to the real coordinator scaffolding action rather than the design-baseline simulation   (breaking: no)

## Requirements

### Requirement: Harness reachability gate
`capability: onboarding` ¬∑ `delta: ADDED (feat/project-setup-and-grounding)`

The system SHALL, on client launch, probe the configured harness endpoint and, if no harness is reachable, present a dedicated first-run surface that guides the engineer to install, start, or point to a harness before any project action is available. The gate SHALL be non-blocking for engineers who already have a reachable harness.

#### Scenario: No harness reachable on launch
- **WHEN** the client starts and no harness responds at any configured endpoint within a 3-second probe
- **THEN** the client renders the harness-setup guidance surface rather than the project picker
- **AND** the guidance surface shows install and start instructions for OpenCode and any other registered harness drivers
- **AND** a "Retry connection" action re-runs the probe without restarting the client

#### Scenario: Harness becomes reachable after guidance
- **WHEN** the engineer starts a harness and presses "Retry connection"
- **THEN** the probe succeeds and the client proceeds to the project picker
- **AND** no project action was available before the harness was confirmed reachable

#### Scenario: Harness already reachable on launch
- **WHEN** the client starts and at least one harness responds within the probe window
- **THEN** the harness-setup surface is skipped and the project picker is shown directly

### Requirement: Folder-state detection on open
`capability: onboarding` ¬∑ `delta: ADDED (feat/project-setup-and-grounding)`

The system SHALL let the engineer open a local folder, supply a git URL to clone, or create a new empty project. Once a folder is resolved the system SHALL inspect it and classify its state ‚Äî method-ready, has existing code, or empty ‚Äî then open or offer to scaffold accordingly, without modifying any source file in an existing codebase.

#### Scenario: Opening a method-ready folder
- **WHEN** the engineer opens a folder that already contains `.opencode/agents/`, `docs/specifications/`, and `AGENTS.md`
- **THEN** the client opens directly into the spec library for that project
- **AND** no scaffolding action is offered or run

#### Scenario: Opening a folder with existing code but no method scaffold
- **WHEN** the engineer opens a folder that contains source files but lacks the method scaffold
- **THEN** the client presents the initialisation screen pre-filled with the detected repository name
- **AND** offers "Scaffold this project" as the primary action, explaining what will be added
- **AND** no source file is read, modified, or deleted before the engineer confirms

#### Scenario: Opening an empty folder or creating a new project
- **WHEN** the engineer opens an empty folder or chooses "Create new project"
- **THEN** the client presents the initialisation screen with sensible defaults and no warnings about existing code

#### Scenario: Cloning a git URL
- **WHEN** the engineer supplies a git clone URL
- **THEN** the coordinator clones the repository into a local folder chosen by the engineer
- **AND** folder-state detection proceeds on the cloned result

### Requirement: One-action method-ready scaffolding
`capability: scaffolding` ¬∑ `delta: ADDED (feat/project-setup-and-grounding)`

The system SHALL scaffold a repository into a method-ready state in a single confirmed action. The scaffold SHALL create the agent roster, the specification directory structure plus the canonical template, the `AGENTS.md` grounding baseline, and the policy and projection plugin stubs. Vendored reference copies SHALL be written under `.repos/` as read-only grounding material. Tier defaults SHALL be pre-filled so no model knowledge is required from the engineer. The scaffold SHALL NOT modify any existing source file.

#### Scenario: Scaffold completes all four steps
- **WHEN** the engineer confirms "Run scaffold" on the initialisation screen
- **THEN** the coordinator executes the four scaffold steps in order: agent roster, specification structure, grounding baseline, policy & projection plugins
- **AND** each step is reported in the terminal panel as it completes
- **AND** on completion `.opencode/agents/` contains the six canonical role files, `docs/specifications/` contains `specification.template.md` and `README.md`, `AGENTS.md` exists at the repo root, and at least one projection plugin stub exists

#### Scenario: Tier defaults require no model input
- **WHEN** the initialisation screen is displayed before scaffolding
- **THEN** the tier fields (capable, mid) are pre-filled with sensible defaults from the coordinator registry
- **AND** the engineer can proceed through scaffolding without changing them

#### Scenario: Scaffold is idempotent on re-run
- **WHEN** the scaffold action is run on a folder that already contains some scaffold artefacts
- **THEN** existing artefacts are not overwritten unless they are absent or flagged as stale
- **AND** the terminal panel reports which steps were skipped

#### Scenario: Source files are untouched
- **WHEN** the scaffold runs in a folder that contains application source code
- **THEN** no file outside `.opencode/`, `docs/specifications/`, `AGENTS.md`, and `.repos/` is created, modified, or deleted

### Requirement: Repository analysis and grounding baseline
`capability: grounding` ¬∑ `delta: ADDED (feat/project-setup-and-grounding)`

The system SHALL, after a project folder is opened or scaffolded, use a harness session to analyse the repository and produce or update `AGENTS.md` with a summary of the codebase structure, key entry points, and relevant conventions. The analysis SHALL NOT modify any source file. On completion the client SHALL surface `AGENTS.md`, any existing guides under `docs/`, and relevant code extracts so spec authoring begins grounded.

#### Scenario: Fresh grounding baseline produced after scaffold
- **WHEN** scaffolding completes on a folder that contains existing source code
- **THEN** the coordinator opens a harness session scoped to the researcher role
- **AND** the session analyses the repository and writes an `AGENTS.md` baseline covering module structure, key entry points, and relevant conventions
- **AND** the session closes and its result is surfaced in the authoring cockpit before the first specification is started

#### Scenario: Grounding baseline refreshed on open
- **WHEN** the engineer opens an existing method-ready project where `AGENTS.md` was last updated more than 24 hours ago
- **THEN** the client offers a "Refresh grounding" action
- **AND** if accepted the coordinator runs a fresh researcher session that rewrites the stale sections of `AGENTS.md`

#### Scenario: Grounding surface shown in the cockpit
- **WHEN** the authoring cockpit opens for a specification
- **THEN** a grounding panel shows the current `AGENTS.md`, any guides under `docs/`, and code extracts relevant to the spec under authorship
- **AND** the agent roster and relevant existing files are reachable from that panel without leaving the cockpit

### Requirement: Tier defaults pre-filled, no model knowledge required
`capability: onboarding` ¬∑ `delta: ADDED (feat/project-setup-and-grounding)`

The system SHALL pre-fill logical model tier defaults (capable, mid) on the initialisation screen from the coordinator registry so that the engineer can complete onboarding and scaffolding without knowing any vendor model identifier. The tier labels SHALL be shown alongside a brief description of their intended use.

#### Scenario: Tier fields shown with descriptions
- **WHEN** the initialisation screen is rendered
- **THEN** each logical tier (capable, mid) is shown with a human-readable label, a brief purpose description, and the model name the registry currently resolves for it
- **AND** no vendor model identifier is required to confirm and run the scaffold

#### Scenario: Tier defaults propagate into the roster
- **WHEN** the scaffold creates the agent roster files under `.opencode/agents/`
- **THEN** each roster file carries a `tier:` reference, not a hardcoded model identifier
- **AND** the coordinator registry is the single place vendor model IDs are recorded

## Design

### Architectural decision
The scaffolding action is a coordinator-side command, not a client-side file write ‚Äî so credentials, git access, and file I/O stay on the host (NFR-1). The client sends a `scaffold.run` request over the existing WebSocket; the coordinator executes it step-by-step, emitting progress events that the client projects onto the terminal panel. Grounding analysis is a thin harness session using the `researcher` role; its output is committed to `AGENTS.md` through the harness, keeping the same git-backed audit trail as any other spec write.

### Target architecture

```
Client (Initialisation screen)
  ‚Üï WebSocket (ws://127.0.0.1:4319)
Coordinator
  ‚îú‚îÄ‚îÄ ScaffoldRunner          ‚Äî orchestrates the four scaffold steps; emits scaffold.step events
  ‚îú‚îÄ‚îÄ FolderInspector         ‚Äî detects method-ready / has-code / empty state
  ‚îú‚îÄ‚îÄ HarnessReachabilityProbe‚Äî pings configured endpoints; result ‚Üí snapshot + reachability event
  ‚îî‚îÄ‚îÄ HarnessAdapter          ‚Äî researcher session for grounding analysis (AGENTS.md)
       ‚Üï HTTP / SSE
OpenCode server (opencode serve)
```

**New coordinator modules** (all within `packages/coordinator/src/`):
- `scaffold.ts` ‚Äî `ScaffoldRunner` with four steps: `agents`, `specs`, `grounding`, `plugins`; emits `scaffold.step` and `scaffold.done` events into the trace and push channel.
- `folder-inspector.ts` ‚Äî `FolderInspector.classify(path)` returns `'method-ready' | 'has-code' | 'empty'`.
- `reachability.ts` ‚Äî `HarnessReachabilityProbe.probe(endpoints)` returns per-endpoint reachability; wired into startup.

**New domain events** (additions to `@arke/contracts/src/events.ts`):

```typescript
export const ScaffoldStepEvent = base.extend({
  type: z.literal('scaffold.step'),
  step: z.enum(['agents', 'specs', 'grounding', 'plugins']),
  status: z.enum(['running', 'done', 'skipped', 'error']),
  detail: z.string().optional(),
});

export const ScaffoldDoneEvent = base.extend({
  type: z.literal('scaffold.done'),
  projectPath: z.string(),
  stepsRun: z.array(z.string()),
});

export const HarnessReachabilityEvent = base.extend({
  type: z.literal('harness.reachability'),
  endpoint: z.string(),
  reachable: z.boolean(),
});
```

These join the `DomainEvent` discriminated union.

**Client changes** (`packages/client/src/`):
- `screens/picker-init-library.tsx` ‚Äî `Initialisation`: replace the simulated `run()` with a `scaffold.run` WebSocket message; consume `scaffold.step` / `scaffold.done` events for the terminal panel.
- `screens/picker-init-library.tsx` ‚Äî `Picker`: on mount, check the snapshot's `harnessReachable` flag; if false, render the harness-setup guidance surface in place of the project list.
- New `HarnessGuidance` component ‚Äî install/start instructions per driver; "Retry connection" fires a `harness.probe` request.

### Data model
No persistent database. Scaffold artefacts are files committed by the harness into git (the same trust boundary as spec files). The coordinator's in-memory snapshot gains a `harnessReachable: boolean` and `projectState: 'method-ready' | 'has-code' | 'empty' | null` field, pushed in the `snapshot` frame on each (re)connection.

### Interfaces and contracts

**WebSocket client‚Üícoordinator messages** (additions):

```typescript
// Check harness reachability; coordinator responds with harness.reachability events
{ type: 'harness.probe' }

// Detect the state of a local folder
{ type: 'folder.inspect', path: string }

// Clone a remote repository
{ type: 'repo.clone', url: string, targetPath: string }

// Run the scaffold steps on the current project path
{ type: 'scaffold.run', path: string, tiers?: { capable?: string; mid?: string } }
```

**Scaffold artefacts written by `ScaffoldRunner`:**

| Step | Artefacts created |
|------|-------------------|
| `agents` | `.opencode/agents/{spec-author,architect,reviewer-a,reviewer-b,implementer,researcher}.md` |
| `specs` | `docs/specifications/specification.template.md`, `docs/specifications/README.md` |
| `grounding` | `AGENTS.md` at repo root (stub, then enriched by grounding session) |
| `plugins` | `plugins/policy.ts` (permission policy stub), `plugins/projection.ts` (projection stub) |

Vendored references are written to `.repos/` as git submodules or shallow clones; this step is advisory and is skipped silently if git is not available.

**`FolderInspector.classify` heuristic:**

```
method-ready  ‚Üí .opencode/agents/ exists AND docs/specifications/ exists AND AGENTS.md exists
has-code      ‚Üí any source file exists (*.ts, *.js, *.py, *.go, *.rs, ‚Ä¶) but not method-ready
empty         ‚Üí no source files found
```

### Cross-cutting
- **Design template:** the UI in this feature follows the `arke-design` skill (`.claude/skills/arke-design/`) ‚Äî the shadcn neutral token contract, Geist typography, Lucide icons, and the prototype screens (`app/*.jsx`); see [SKILL.md](../../.claude/skills/arke-design/SKILL.md). Do not re-derive styling.
- **Security (NFR-1):** All file I/O, git clone, and harness session creation run inside the coordinator on the host. The client sends intent; the coordinator acts. No file paths or credentials are exposed to the browser beyond the project name.
- **Observability (NFR-7):** Every scaffold step and grounding session is written to `.arke/trace.ndjson` with the same `Trace.write()` path as domain events. The researcher session's NDJSON trace entry records which version of `AGENTS.md` was produced.
- **Resilience (NFR-4):** If a scaffold step fails, the terminal panel shows the error and offers a "Retry step" action. Partial scaffolds are recoverable because each step is idempotent.
- **Accessibility (NFR-6):** The harness-guidance surface and initialisation screen are keyboard-navigable; the terminal panel uses `role="log"` with `aria-live="polite"`.

## Decision log
| # | Decision | Rationale |
|---|----------|-----------|
| 1 | Scaffold runs in the coordinator, not the client | Keeps file I/O and git access on the host (NFR-1); coordinator already owns the harness trust boundary |
| 2 | Grounding analysis via the researcher harness session, not a static template | The codebase is unknown at scaffold time; only a live analysis can produce an accurate `AGENTS.md` for an existing repo |
| 3 | Tier defaults from the coordinator registry, not hardcoded in the client | Vendor model IDs must live in one place behind the gateway (FR-4, roster doc ¬ß2) |
| 4 | Scaffold steps are idempotent | Engineers re-run scaffolding after pulling updates; idempotency prevents accidental overwrites |
| 5 | `FolderInspector` checks three sentinel paths, not a full tree walk | Fast and deterministic; the full tree walk belongs to the grounding session |

## Open questions
- Should the coordinator support vendored `.repos/` as shallow clones (requiring git on the host) or as copied snapshots bundled with the Arke distribution?
- What is the policy when `AGENTS.md` already exists and the grounding session produces a meaningfully different analysis ‚Äî overwrite silently, produce a diff for review, or append a new section?
- Is the harness-reachability probe a one-shot on launch or should it run on a heartbeat interval so the guidance surface appears if the harness dies mid-session?
