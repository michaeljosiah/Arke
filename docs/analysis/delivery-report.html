01 / Executive summary

The loop's scaffolding is real and runs

Arke is a brand-new public project that turns the specification-centric engineering method into a product (per the PRD). This first pass stands up the whole foundation and proves the moving parts in isolation: the contracts are schema-first and validated at the boundary; the coordinator ingests, normalizes, sequences, persists and pushes events over WebSocket; the OpenCode adapter is implemented against the real, documented HTTP/SSE surface (researched live); and the React client faithfully reproduces the approved design baseline with every v1 screen reachable.

It is deliberately not a live end-to-end loop yet — the client still renders against the prototype's in-memory simulation, and the adapter has not been exercised against a running opencode serve. Those two wires are the immediate next step and are scoped below.

Direction of truth

Every architectural choice holds the PRD's core invariant: the specification is the source of truth, the harness owns execution and credentials, and nothing is acted on without a human decision in between (propose · decide · execute).

02 / What was delivered

A method-ready monorepo

Monorepo & tooling

npm workspaces, shared tsconfig.base.json, .gitignore, .editorconfig, MIT licence, a public README, and a .claude/launch.json so the client and coordinator preview in one command.

@arke/contracts

zod schemas: spec lifecycle & frontmatter, the spec anatomy, the normalized DomainEvent union with a per-connection envelope, computed board columns, and the backend-agnostic HarnessAdapter interface + capability flags.

@arke/coordinator

A thin local WebSocket coordinator: ingest → validate → normalize → fold a read model → persist an append-only NDJSON trace → push ordered, sequenced events. Ships a MockAdapter so the loop runs with no live harness.

@arke/adapter-opencode

A real fetch-based adapter mapped to OpenCode's server: sessions (parent/child), sync + async prompting, todos, diff, permission reply, commands, Basic-auth, and SSE → DomainEvent normalization. Tier → gateway model resolution built in.

@arke/client

Vite + React 19 + TypeScript reproducing the design baseline: app shell + sidebar nav, project picker/init, spec library, authoring cockpit, review panel, generation workspace, live board, session detail, diff review, permission overlay, audit, projections, agent roster, harness/model config, settings, notifications — plus the live event-simulation engine and shadcn "neutral" theme (light/dark, Geist).

apps/desktop

An Electron shell skeleton documenting the embed-the-coordinator packaging plan (Electron recommended, Tauri the lighter alternative).

Method scaffolding

The agent roster under .opencode/agents (product-owner, technical-architect, engineering, implementation, reviewer) in OpenCode's real frontmatter format; the canonical specification.template.md; the inaugural SPEC-001; AGENTS.md grounding (mirrored by CLAUDE.md); and ADR-0001.

Research, vendored

The PRD and the design prototype are vendored into docs/. Two source-grounded guides sit in docs/analysis: the OpenCode integration guide and the T3 Code learnings.

06 / OpenCode integration

Grounded in the live documentation

The adapter was written against a researched, source-cited integration guide (opencode-integration-guide.md). Confirmed primitives now mapped in adapter-opencode: POST /session (with parentID for tasks), POST /session/:id/message and /prompt_async, GET /session/:id/todo and /diff, POST /permission/:id/reply, POST /session/:id/command, and the GET /global/event SSE stream normalized into DomainEvents. Auth is HTTP Basic via OPENCODE_SERVER_PASSWORD, held at the host.

Caveats baked into the plan

  • No Last-Event-ID on the SSE stream — the adapter must re-fetch state via REST on reconnect (handled by design; reconnect logic is a next step).
  • The permission reply returns 200 even for stale IDs; don't treat it as proof.
  • Upstream opencode-ai/opencode was archived; an active fork carries the current docs. Always validate shapes against the live GET /doc OpenAPI spec on the target version.
  • Agents resolve a logical tier to a gateway provider (OpenAI-compatible baseURL), keeping models on controlled infrastructure (NFR-5).
07 / T3 Code learnings applied

Patterns adopted from the reference

The PRD cites T3 Code as the architectural reference. Learnings captured in t3code-learnings.md; the ones this foundation already reflects: a thin local coordinator with no cloud backend on the hot path, schema-first contracts validated at the boundary, normalized provider events folded by a read model, and ordered/sequenced push with snapshot-on-connect. Patterns queued for the live phase: the reconnecting five-state WS transport with queued-outbound-drain, typed runtime-signal receipts over polling, a virtualized streaming transcript for the cockpit, and the approval/elicitation composer panels. Where we deliberately differ is intact: spec-as-first-class-aggregate, propose-decide-execute, deterministic projection as the product, required human reviewer, and audit-as-product.

08 / Identified gaps

What is missing or provisional

  • Client uses simulated data. The cockpit/board render against the prototype's in-memory engine, not the coordinator. No WebSocket transport in the client yet.
  • Adapter unexercised against a live server. Endpoint shapes follow the docs but have not been validated against a running opencode serve; event property casing (dot vs underscore) is handled defensively but needs confirming on /doc.
  • Message/transcript events not modelled. DomainEvent covers spec/session/todo/diff/permission/projection but not streaming chat messages — needed for a real cockpit transcript.
  • Projections are UI-only. The deterministic spec→record projection plugin (FR-7) and the policy/direction-of-truth hook are documented but not implemented.
  • No persistence beyond the NDJSON trace. No SQLite for sessions/snapshots/audit query (T3 uses SQLite); the read model is in-memory.
  • Client is a faithful port, not idiomatic shadcn. Screens use the prototype's createElement + inline-style approach; Tailwind/shadcn copy-in components are not yet introduced. Client tsconfig is lenient by design.
  • No tests yet. Contracts round-trip tests, read-model column-derivation tests and a client smoke render are specified in SPEC-001 but not written.
  • Electron is a skeleton. No packaging, signing, or coordinator embedding.
  • CLAUDE.md is a pointer, not a symlink (portable on Windows); convert to a symlink where supported.
  • Reconnect/resilience (NFR-8) and OpenTelemetry export (NFR-7) are designed but not implemented; only the local NDJSON trace exists.
09 / Suggested next steps

Prioritised path to a live loop

  1. Wire the client transport (SPEC-001 T-7). Add a reconnecting WebSocket client to @arke/client that consumes the coordinator's snapshot + event frames and drives the board from real state. Adopt T3's five-state transport with queued-outbound-drain.
  2. Stand up a live OpenCode host and exercise the adapter (T-8). Run opencode serve, point the coordinator's adapter at it, validate every endpoint/event against GET /doc, and implement REST re-fetch on SSE reconnect.
  3. Extend the domain model for messages. Add streaming message/part events to @arke/contracts and a virtualized transcript to the cockpit and session detail.
  4. Implement deterministic projection (FR-7). Ship an OpenCode plugin under .opencode/plugins that reacts to session.idle/status changes and writes Jira/Azure DevOps via MCP, logging every write with its trigger; add the tool.execute.before policy hook.
  5. Add persistence. Introduce SQLite in the coordinator for sessions, spec snapshots and a queryable audit log alongside the trace.
  6. Author a real spec through the loop. Use the cockpit + review panel end-to-end on a throwaway feature to prove "the card moves because the work moved" (the Phase-1 goal).
  7. Harden the client. Introduce Tailwind/shadcn copy-in components incrementally, tighten the client tsconfig, and add the test suite from SPEC-001 §3.2.
  8. Observability (NFR-7). Add OpenTelemetry spans at every coordinator boundary and OTLP export, keeping the NDJSON trace as the local source of truth.
10 / How to run

From a clean clone

git clone https://github.com/michaeljosiah/Arke
cd Arke
npm install

# the React client (design baseline)
npm run dev                 # → http://localhost:5173

# the local coordinator (mock adapter, scripted event stream)
npm run dev:coordinator     # → ws://127.0.0.1:4319

# quality gates
npm run build
npm run typecheck
