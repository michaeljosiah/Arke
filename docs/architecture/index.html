Architecture · the Specification Orchestrator

How Arke is built

A React cockpit and a thin local coordinator sit on top of a coding-agent harness and make the specification the unit of work — from authoring through delivery, across any agent. This document explains that architecture: first in plain language, then layer by layer.

00

In plain terms

Read this part even if you read nothing else. No jargon.

Modern software is increasingly written by AI coding agents. They are fast, but on their own they are unpredictable — they wander off-brief, leave no record of why something was built, and need a human watching every keystroke. Teams cope by writing better and better instructions until, at the top of the ladder, all the intent lives in one authoritative document: a specification that says what to build and why.

Arke is the place that document lives and does its work. Think of it as a control room — a cockpit — for a person we call a Product Engineer. In that cockpit you do three things, in a loop:

1 · Author

write the spec

You and an AI co-write the specification, grounded in the real codebase. A panel of different AI models critiques it. When you're happy, it is committed to the repository as the single source of truth.

2 · Orchestrate

run the agents

Arke dispatches coding agents to build against the spec — many at once, in parallel — and pauses for your approval at every decision point. You stay the one who decides; the agents do the typing.

3 · Keep the record

watch & prove

A live board shows the real state of delivery — a card moves because the work moved, not because someone dragged it. Every approval is written down, permanently.

The architectural idea, in one breath

Arke deliberately does not do the dangerous, heavy work itself. It does not run the AI agents, hold your passwords, or touch your codebase directly. That job belongs to a harness — a separate engine that owns execution and credentials. Arke is the thin, smart layer on top that realises, visualises and coordinates. The browser you look at never holds a secret and never talks to your live systems.

So the shape of the system is simple to picture: a human at the top who decides, a window that shows the picture, a small local translator in the middle, and a powerful engine underneath that does the building. Everything below is just that idea, drawn precisely.

The substrate, in plain terms The engine underneath can be a single coding agent (we start with one called OpenCode) — or it can be Omnigent, a "meta-engine" that drives fifteen different agents and adds cloud sandboxes, session rescue and more. Because Arke talks to the engine through one neutral plug, swapping a single agent for Omnigent unlocks all of that without changing Arke. That plug — and what Omnigent unlocks through it — is a major theme of this document.

01

The big idea — specification as the unit of work

AI code generation is fast but unreliable without structure. Teams climb a maturity ladder to fix that:

The method works on paper, but it has had no home: engineers drive agents by hand, keep tickets and tests in step with the spec manually, and track delivery in tools built for human-authored cards. Arke is that home. It turns the specification-centric method into a working product, and it does so by enforcing one direction of truth.

The direction of truth

The specification is authoritative. Everything else — tasks, code, tickets, tests, tracking, docs — is a generated projection of it, kept in step with it. When the spec changes, the projections are regenerated. Nothing downstream ever flows back into the spec.

The loop author ──▶ review ──▶ approve ──▶ fan out ──▶ sync   — and a change to the spec re-runs the loop. The engineer owns the spec; Arke keeps everything downstream honest to it.
02

Three non-negotiable principles

Every architectural decision below is downstream of three rules. They are what make Arke safe to point at a real, production — even financial — codebase.

① Spec is the source of truth

Its canonical copy is a markdown file in docs/specifications, versioned and reviewed via pull request. Nothing downstream overrides it; projections are generated from it and never flow back.

② Harness owns execution & credentials

All repo, agent, CLI and MCP access happens inside the harness host. The browser is never in the credential path and never calls a system of record.

③ Propose · decide · execute

No agent output reaches the repository, a system of record, or a teammate without a human decision in between. It is never an autonomous pipeline.

Three further rules fall out of these and shape the implementation:

  • Projections to systems of record are deterministic code, not free-form agent behaviour, and every projection write is logged with its trigger.
  • Every governed action is recorded. Spec history is in git; permission decisions and projections are in the append-only trace (.arke/trace.ndjson).
  • Agents reference logical model tiers (capable, mid, fast), resolved per project to an internal model gateway — never hardcoded vendor model IDs.
What Arke is not Not a coding agent · not a system of record · not a Jira/GitHub replacement · not an autonomous pipeline. It realises, visualises and coordinates — it does not execute, and it holds no authoritative state in the browser.

04

The layers in detail

The client — a thin realisation layer

packages/client is a React app that holds no authoritative state. It renders five surfaces — the authoring cockpit, the multi-model review panel, the delivery board, the generation workspace, and session detail — entirely from state the coordinator pushes it. It cannot reach a harness, a repo, or a system of record directly. Its one job is to present the picture and capture human decisions. When it disconnects, it loses nothing: on reconnect it receives a fresh snapshot and re-converges.

The transport is a five-state reconnecting WebSocket machine (connecting → open → reconnecting → closed → disposed) that queues outbound requests — steering messages, permission decisions — while disconnected and drains them in order, at most once when the link returns. All reconnect logic is isolated in one ArkeTransport class so the board and session views never see a dropped socket.

The desktop shell — packaging the trust boundary

apps/desktop is an Electron shell that embeds the coordinator in one signed application. Because the coordinator is a Node process living inside the credential boundary, wrapping it in Electron is the natural packaging: the user runs one app, and the whole trust boundary ships with it. The browser remains a first-class surface — the same client runs in a plain browser against a locally-running coordinator.

The coordinator — the thin local brain

packages/coordinator runs on the harness host, inside the trust boundary, and does five things and no more:

Ingest & normalise

Pulls provider events through a HarnessAdapter and normalises each into the canonical DomainEvent model, validated with zod at the boundary. A malformed event from any backend is caught, not silently trusted.

Fold a read model

Folds events into a ReadModel whose board columns are computed from real signals (spec status, session status, PR/CI state, needsHuman) — never set by hand.

Persist the trace

Writes every event to an append-only NDJSON Trace (.arke/trace.ndjson) before pushing it onward. The trace is the audit source of truth.

Push, ordered & sequenced

Pushes events to clients over WebSocket with a monotonic, per-connection sequence number, replaying current state via a snapshot on every (re)subscribe. Slow clients are bounded, never block the pump.

Crucially, no cloud backend sits on the agent hot path. The coordinator absorbs every harness's quirks so the client learns no harness fact — and so the trust boundary stays a single, local, auditable process.

The contracts — schema-first, backend-agnostic

packages/contracts is the spine. It is zod-first and shared by client, coordinator and every adapter:

FileDefines
spec.tsSpecification lifecycle, frontmatter, model tiers, the spec anatomy.
events.tsThe normalised DomainEvent union, the per-connection EventEnvelope (with correlationId), and the computed board columns.
adapter.tsThe backend-agnostic HarnessAdapter interface and capability flags. This interface — not any one harness — is what the product is built against.

The adapters — where harness facts are absorbed

An adapter implements HarnessAdapter for one backend and keeps everything backend-specific behind the seam. @arke/adapter-opencode is the lean reference adapter against a live opencode serve: it connects with host-only credentials, probes capabilities at startup from GET /doc (advertising only what the live server exposes), owns a durable session-ownership graph that survives restarts, drives roles at a logical tier with a correlation id, and confirms permissions by event, not HTTP status. Unmappable events are dead-lettered with their reason — never silently dropped. @arke/adapter-omnigent is the substrate spike, covered next.

05

The harness seam & Omnigent — the feature multiplier

The single most important architectural choice in Arke is the neutral adapter seam. Arke is built against the HarnessAdapter interface, not against any harness. That one decision is what lets a project run on a single coding agent today and inherit an entire ecosystem tomorrow — by swapping what sits below the seam.

Two kinds of thing can sit below the seam

There are leaf harnesses — a single coding agent, like OpenCode — and there is one meta-harness: Omnigent, "the open-source meta-harness for all your AI agents." A meta-harness is a backend that itself fronts many harnesses. Architecturally it sits at the same layer as Arke's own coordinator/runner — one level up from a leaf. Plugging it in through the seam is the same act as plugging in OpenCode, but what comes with it is vastly larger.

What Omnigent unlocks for Arke

Because Arke targets the neutral seam, adopting Omnigent as the substrate lets Arke inherit a large platform layer "for free" and spend its own effort on the specification lifecycle — the differentiator. The spike (ADR-0002) drove a real OpenCode turn end-to-end through @arke/adapter-omnigent against a live Omnigent 0.3.0 and exited GREEN.

Harness breadth

~15 coding agents (Claude Code, Codex, Cursor, Goose, Qwen, Copilot, OpenCode…) behind one backend, instead of writing an adapter per agent.

Cloud sandboxing

Modal, E2B, Daytona and Kubernetes execution environments — declarative sandboxes Arke's own image model doesn't yet express.

Session fork & rescue

POST /v1/sessions/{id}/fork — a branch/rescue capability the leaf path lacks, useful when a run goes wrong.

Multi-device & collaboration

Session sync across devices and team collaboration ACLs, surfaced through the same coordinator.

Dynamic sub-agents

Runtime self-authored sub-agents as isolated, inbox-linked sessions in their own git worktrees — more capable than static rosters.

Runner model alignment

Omnigent's server/runner split is the same control-plane/runner architecture Arke adopted (SPEC-018), so the mapping is structural, not forced.

How Arke maps onto Omnigent (substrate mode)

The adapter compiles Arke's concepts down to Omnigent's v1 HTTP API at the edge. Arke keeps its abstraction; Omnigent specs become a compiled artifact.

Two rules that keep the seam honest Substrate exclusivity — a project picks either leaf harnesses or Omnigent, never both; the two are mutually exclusive in one project's registry (exclusivity is per-project, not org-wide). Honest about maturity Omnigent is an alpha (v0.3.0), Python-primary project; its API may churn. Arke's hedge is exactly the neutral seam — it can always fall back to OpenCode-direct. The adapter stays on its spike branch until a hardening spec lands (correlation, event-confirmed approvals, reconnect, version pinning). Omnigent widens the menu; it does not replace the reference path.

How an event flows end-to-end

Everything the user sees is the result of a normalised event making one disciplined trip from the harness to the screen. Follow a single agent turn:

The domain event model

The canonical DomainEvent is a discriminated union in @arke/contracts/events.ts. Among its variants are the streaming-transcript and lifecycle events that make the cockpit and board live:

EventCarriesMeaning
message.partsessionId, messageId, partIndex, delta, role, doneA streaming delta from an in-progress turn (assembled in partIndex order, not arrival order).
message.updatedsessionId, messageId, role, text, toolCalls, isStreamingFull turn-state snapshot once a message completes.
turn.quiescentsessionId, turnIdA typed idle receipt — consumers detect completion from this, never from a timeout or a poll.

Each is wrapped in an EventEnvelope that carries a per-connection seq, ts, harness and an optional correlationId (the harness messageID) so every event can be attributed to the request that triggered it. Resilience guarantees fall out of this model:

  • Snapshot-on-connect. Every new connection — including a reconnect — gets the current CardState[] as the very first frame, so a client converges without replaying the event log.
  • Per-connection sequence. seq is scoped to the connection, not a shared counter, so two clients never race over increments.
  • Crash-safe pump. A parse() failure is caught, dead-lettered to the trace, and the pump continues. A slow client's oldest events are dropped (and traced) rather than blocking everyone.
  • Durable ownership. A session-ownership store (.arke/sessions.ndjson) maps sessionId → {kind, parentSessionId, specId} and is restored on startup, so a coordinator restart never loses which session belongs to which spec.
07

Agents, tiers & model resolution

Arke ships a small, versioned roster of agents — and it never hardcodes a vendor model. An agent references a logical tier (capable, mid, fast); the project registry resolves that tier to a concrete model on a concrete harness instance, behind an internal gateway. Vendor IDs live in exactly one place and never reach the client.

RolePurposeTierWrites
spec-authorCo-authors the spec with the human — Why, requirements, scenarios.capabledocs/specifications/
architectFills the design depth — architecture, data model, interfaces.capabledesign sections
reviewer-aIndependent critique, grounded in source.capable · model Anothing (proposes)
reviewer-bIndependent critique on a different model.capable · model Bnothing (proposes)
implementerExecutes the tasks; writes code on the feature branch.midworkspace (code)
researcherGathers and summarises codebase context to ground authoring.midnothing (read-only)

The roster keeps itself honest: reviewers are read-only and must resolve to two different models for genuinely independent critique; the implementer is deliberately mid-tier (a capable model authored the spec precisely so a cheaper model can implement it accurately); projection is not an agent; and no harness gets its own orchestrator agent — the app is the conductor.

Tier → model → instance

An instance binds a driver to credentials, a host and a working directory, and declares which tiers it can serve. A role asks for a tier; the router picks an instance that serves it and returns a concrete selection. The role never names a harness — so "use a capable model" and "use Claude Code" become the same act.

08

Governance — propose · decide · execute

No agent output reaches the repository, a system of record, or a teammate without a human decision in between. Arke encodes this as a loop with a hard gate, and it couples the gate to the risk of the action.

Projections are deterministic, not agentic

Writing spec state into systems of record (Jira, Azure DevOps, git/PR) is deterministic plugin code reacting to events — never free-form agent behaviour. Every projection write is logged with its trigger. This is why "the board moved" can always be trusted: the integration registry decides, deterministically, what each event projects, and the trace records why.

10

The record — trace & observability

Every governed action is recorded, in two complementary places:

Spec history → git

The specification's canonical copy is a markdown file under docs/specifications, versioned and reviewed via pull request. Its full history — every revision, every approval — is git history.

Runtime → the append-only trace

Permission decisions, projections, and every normalised event are written to .arke/trace.ndjson before they take effect. The trace is the audit source of truth, exportable over OTLP.

The trace is deliberately a plain append-only NDJSON file co-located with the session-ownership store — no database dependency in the local path. It records far more than events: snapshot frames ({kind:"snapshot", cardCount}), inbound client requests, dead-letters (with raw type and reason), and slow-client drops. Whatever reaches the read model is in the trace; nothing that reached the board is invisible.

Single project configuration Everything configurable lives in one tracked file, .arke/config.json: the harness/model registry (instances + roster bindings), coordinator settings (port, concurrency, timeouts, query limits, OTLP endpoint), and integrations config. ARKE_* env vars override individual keys. Everything else under .arke/ — trace.ndjson, sessions.ndjson, scaffold-manifest.json, projection-fallback.ndjson — is runtime state, git-ignored.
