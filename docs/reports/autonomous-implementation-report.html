2 · Review methodology — adversarial, every spec

The brief called for the Codex review-bot loop, with internal agents as the fallback. Codex was rate-limited for the entire run (“You have reached your Codex usage limits for code reviews”), so every spec received a thorough internal multi-angle adversarial review in its place — a dedicated review agent reading the full diff against the real code, hunting correctness bugs, removed invariants, and cross-file breakage. This was not a rubber stamp: it found real, shipped-quality bugs on every single spec, which were fixed with regression tests before merge.

Representative bugs caught & fixed

  • 009 — git ref D/F conflict: feat/x/task-0 can't coexist with branch feat/x; switched to a sibling name. Also: idle is per-turn, not terminal — the queue drained too early.
  • 010 — board rendered only 5 of 7 columns; approved/needs-human cards invisible.
  • 012 — needsHuman cleared by any status event; a ghost gate for an unknown session could pin a card forever.
  • 013 — an invalid ticket was permanently unapprovable (edits couldn't add a sorTarget); a generate() TOCTOU race spawned two live sessions.
  • 014 — retry-projection picked the oldest approval, drifting the idempotency key.
  • 015 — the trace rewrite silently defeated the trace-before-action fail-safe (a governed action could proceed with no audit record).

Quality bar held per spec

  • TypeScript strict typecheck — 0 errors across all six workspaces.
  • Full Node test suite green before every merge (grew 165 → 252 coordinator tests).
  • Production vite build + .NET dotnet build + CLI tests green.

3 · Identified gaps & concerns

Stated plainly so they are not mistaken for done. None block the core governed loop; each is a deliberate, documented deferral or an environment limitation.

Verification gap — interactive browser QA

The brief asked for UI verification by driving the app in Chrome. The Chrome extension MCP bridge was unreachable for the entire run (every tabs_context_mcp call returned “not connected”, though Chrome was open). UI was therefore verified to the strongest available headless degree: the production build compiles every screen, the Vite dev server serves and transforms all client modules with zero errors, and the 18 client unit tests pass. Pixel-level interactive QA (clicking through each new screen) was not performed and should be done once the bridge is restored.

Deferred to a live external system (verified-by-mirror / by tests)

  • 008 — GitHub-API open-PR re-fetch, concurrent-capability conflict detection, branch-protection scaffolding (the coordinator self-approval gate is implemented + tested).
  • 013/014 — actual local file writes & live SoR API calls (GitHub/Jira/Azure) and the OpenCode plugin-load health-check need a real harness + integration; the deterministic logic is unit-tested and the plugin mirrors it.
  • 015 — the full @opentelemetry/sdk-node + OTLP BatchSpanProcessor export to Jaeger/Tempo; local span records + allowlist + per-spec causal chain exist and are exportable.
  • 011/013/014/015 — several client screens have their store-fold + live helpers + CLI wired, but the deep live-mode render (rescue section, elicitation overlay, generation/projections/audit live screens) is not fully swapped from the demo screens.

Concerns worth a follow-up pass

  • Identity is "anonymous" (Phase-1, by spec): audit records can't attribute who approved until a coordinator auth layer lands. The schema is ready for it.
  • Trace is unbounded: append-only with no rotation; query is a full-file scan (capped at 500). Fine for now; needs rotation + an index at scale.
  • Per-project vs global env: the integrations probe reads the global process.env rather than each project's .arke/config.json (SPEC-018 intent). Functional, but not yet per-project.
  • Generation/agent output is trusted to be parseable JSON: robust extraction is in place, but a malformed-but-plausible artefact set is only as safe as the human review gate.

4 · Product suggestions

Make the audit trail the product, not a byproduct

Every governed action now lands a resolvable, causally-linked trace record (intent → approval → projection). Surface a “provenance” view per released change: from a merged PR or a Jira ticket, one click back to the spec approval, the reviewer panel verdict, and the human who decided. That story — “prove what every agent did, and who authorised it” — is a differentiator for change-controlled teams and is mostly already in the data.

Close the loop with a coordinator auth layer

The single highest-leverage follow-up: a lightweight local identity (OS user or session token) so identity stops being "anonymous". It unlocks real attribution in the audit surface, the second-human gate, and per-operator metrics — with no schema change.

Treat deferred “live” edges as a conformance suite

The deterministic cores (flatten, fan-out, projection, idempotency, policy) are unit-tested; the live edges (SoR APIs, OTLP, plugin load) are mirrored. Stand up a recorded-fixture conformance run against a sandbox GitHub/Jira and a local Jaeger so those edges get automated coverage without a manual pass.

Unify the “decision” primitive

Permissions, elicitations, generation approvals, and diff approvals are now four parallel trace-before-relay flows with near-identical shapes. A single Decision abstraction (validate → trace → relay/execute, with proposalId/idempotency) would collapse duplication and make the next governed action a config, not a code path.
