01 / Overview

One specification, authored once, drives the whole delivery

The Specification Orchestrator is a React web application that sits on top of an existing coding-agent harness and turns the specification-centric engineering method into a working product. The specification is co-authored by a human and AI, grounded in the codebase, and becomes the single source of truth. Everything downstream, the tasks, the code, the tickets, the tests, the tracking, is generated from it and kept in step with it.

The application does not replace the coding agent and it does not become a system of record. It is the cockpit a Product Engineer works in to orchestrate three things: the creation of a specification, the agents that work against that specification, and a kanban view of the specification moving across delivery. The harness owns execution. The orchestrator owns realisation, visualisation, and coordination of what the harness is doing.

The first harness is OpenCode, chosen because it runs a headless server with a typed API, exposes a granular event stream, maintains per-session task lists, and is open source and self-hostable. A backend-agnostic adapter keeps the orchestration logic independent of any single harness so that GitHub Copilot, Claude Code, or others can be added later without reworking the product.

Asset management is the first use case, because that is where the method was proven, but the product is general. It is open source for any team practising spec-centric engineering, and its governance and audit features serve anyone who must show how a change was decided and who approved it, not finance alone.

02 / Background & context

From a framework to a single specification

AI code generation is fast but unreliable without structure. The team has moved up a maturity ladder to address this: from vibe coding, to context engineering that grounds the AI in agents.md, guides and prompts, to specification-driven development that adds process on top.

The heavy form of that approach split work across three files: requirements.md, specifications.md and tasks.md, each owned by a different agent role. That framework taught the discipline, but splitting the work across three documents added overhead without adding rigour. The current method collapses them into a single specification that holds the business intent, the technical design, the constraints and the task breakdown in one place.

That single specification is the centre of gravity. It lives in the codebase, typically under docs/specifications, is versioned and reviewed through pull request like any other code, and is co-authored by the human and the AI grounded in the repository. A mature, context-engineered codebase makes the specification richer and more accurate. The most capable model drives its creation so that a mid-tier model can implement it with high accuracy.

As the specification becomes the unit of work, the engineer moves from writing code to deciding what is built and owning that it is correct: closer to a Product Engineer than a coder. This PRD specifies the tool that makes that way of working real.

03 / Problem statement

The method works on paper but has no home

The specification-centric method is currently a practice, not a product. It depends on engineers manually driving agents, hand-maintaining the link between a specification and its downstream artifacts, and tracking delivery in tools that were built for human-authored tickets. This creates four problems.

  1. The specification and its artifacts drift. Jira stories, tests and tracking are maintained by hand alongside the specification, so they fall out of step with it and with the code.
  2. Tracking tools assume human decomposition. A board where people drag cards is a parallel record of work that someone has to keep true. It does not reflect what agents are actually doing in the repository.
  3. Agent work is invisible and serial. There is no shared surface that shows agents authoring a specification or implementing tasks, what they are waiting on, or where a human needs to step in.
  4. Governance is implicit. Approvals, the direction of truth between a specification and its projections, and the audit trail all live in people's heads or in disconnected tools.

The orchestrator addresses these by making the specification authoritative, generating the artifacts from it, and projecting real delivery state, rather than a hand-maintained one, onto a single surface.

04 / Vision & thesis

A Product Engineer owns a feature end to end

The future this product is built for is one where a single engineer owns a feature across its whole lifecycle, from specification through to delivery, because the agent collapses work that used to need a chain of people.

The strong, defensible claim is the end-to-end ownership of the lifecycle, not that the work becomes solitary. A human reviewer remains in the loop for what ships, for the same reason code goes through pull request today: someone other than the author confirms what is released. The reviewer's job changes shape. They review a specification and a generated change rather than hand-written code.

The product thesis follows from that. The orchestrator is the place a Product Engineer authors the specification, governs it, dispatches the agents, watches delivery happen, and steps in at the decision points. It is what a human-oriented tracking tool will become once the specification, not the ticket, is the unit of work.

Direction of truth

The specification is authoritative. Tickets, tests, tracking and docs are generated projections of it. When the specification changes, the projections are regenerated from it. Nothing downstream overrides the specification.

05 / Goals & non-goals

What this release is, and is not

Goals

  • Make the specification the authoritative artifact, authored and governed in one place and persisted in the repository.
  • Orchestrate agent roles to co-author the specification and to implement its tasks.
  • Generate downstream artifacts from the specification and keep them in step with it.
  • Project real delivery state onto a kanban board driven by harness events, not manual updates.
  • Keep a human in control at the permission and approval points.
  • Stay harness-agnostic through a single adapter interface.

Non-goals

  • Becoming a system of record. The repository and the integrated tools hold the truth.
  • Reimplementing a coding agent. The harness does execution.
  • Replacing Jira, GitHub or Azure DevOps. The orchestrator drives them through their own interfaces.
  • A team-wide, multi-user collaboration surface in this release. Scope is single-engineer ownership.
  • Holding credentials in the browser, or making system-of-record calls from the client.
  • An autonomous, no-human pipeline. Approval stays human.
06 / Users & personas

The Product Engineer is the primary user

Primary · The Product Engineer

Owns a feature end to end. Authors and governs the specification with the AI, dispatches implementation agents, reviews diffs, answers permission requests, and shepherds the work to merge. Wants the specification to be the thing they think in, and wants delivery state to be true without maintaining it by hand.

Secondary · The Reviewer

A second engineer or lead who reviews the specification and the generated change before it ships. Needs to see intent, the diff, and the decision log in one place. Does not need to author.

Context · Risk & governance

Not a daily user, but a stakeholder whose requirements shape the product: auditability of who approved what, where credentials live, and evidence that a human reviewed what was released. These are first-class constraints rather than later additions for any team that must answer for what it ships, regulated industries among them.

07 / Solution approach

The orchestrator realises a loop the harness executes

The product is the loop, not a screen. Ground the AI in the codebase, co-author the specification, review and approve it, fan it out into downstream artifacts, and keep it in sync as the code moves. The React application is the surface a Product Engineer drives that loop from.

Three principles shape every decision in this document.

  1. The harness owns execution. All interaction with the repository, the coding agent, the registered CLIs and the registered MCP servers happens inside the harness. The orchestrator dispatches and observes; it does not execute.
  2. The specification stays in the codebase. Its canonical copy is a file in the repository, versioned and reviewed. The orchestrator never holds the authoritative copy.
  3. Projections are governed, not hoped for. The link from a specification to a system of record is deterministic code, not a behaviour an agent is trusted to remember.
08 / Architecture

A thin client over a harness that owns everything beneath it

The React client talks directly to the harness over its HTTP API and event stream, through a capability-flagged adapter. There is no separate backend on the agent hot path. The harness host is a trust and credential boundary: it holds the repository working tree, the agent runtime, the orchestration plugins, and the authorised CLIs and MCP servers that reach external systems. The client sees only the realisation of what happens inside it.

8.1 The harness host

OpenCode runs as a headless server against a checkout of the repository. It owns the agent runtime, the sessions, the working tree, the orchestration plugins, and the registered CLIs and MCP servers. Because it is open source and self-hostable, it can run on infrastructure the firm controls and point at an internal model gateway rather than public endpoints. Authorisation for every external tool is established once at the host.

8.2 The React client and the adapter

The client renders the cockpit, the board, the diff and review surfaces, and the permission prompts. Beneath them sits a backend-agnostic adapter that exposes a small interface and is implemented per harness. Today there is one implementation, for OpenCode. The interface, not any one harness, is what the product is built against.

8.3 Communication and events

Requests go from the client to the harness over HTTP. State comes back as a server-sent event stream. The UI is event-driven rather than polled: work is dispatched asynchronously and the board, the diffs and the prompts update as events arrive. This is what lets many agents run at once and still present a single coherent picture.

8.5 A thin local coordinator

A working reference implementation in the wild (see section 21) clarifies one point we framed too strongly. "No backend on the hot path" should mean no cloud backend, not no process at all. In practice the client talks over a WebSocket to a thin coordinator that runs locally on the harness host, because a browser cannot speak some agents’ native transports and you need one place to normalise heterogeneous provider events, gate startup, order the event stream, and survive reconnects. This stays inside our boundary: the coordinator, the agents and the credentials all run on the host, and the browser only renders.

The coordinator normalises each provider’s native events into one canonical domain-event model. It ingests provider events, persists them, exposes a read model, and pushes ordered, sequenced, schema-validated events to the client. The board reads from that normalised model rather than from raw provider output, which is what lets capability differences between harnesses be absorbed cleanly. For OpenCode, whose server already speaks HTTP and SSE, the coordinator is thin; for an agent that speaks stdio or a bespoke protocol, it does more. Either way the client sees one model.

Several mechanisms from the reference are worth adopting directly: typed runtime receipts for async milestones such as turn quiescence and diff finalisation, so the board reacts to signals rather than polling; automatic git checkpoints captured around agent turns for rescue and audit; and a global runtime mode governing approval posture and sandboxing, a supervised mode that asks for approval and writes only within the workspace, and a full-access mode for trusted flows.

8.6 Delivery surfaces and tooling

The client is a React application. In the browser it talks to the coordinator over WebSocket. As a desktop application it ships the same React interface with the coordinator embedded, so there is nothing separate to install and the engineer runs one signed app.

The coordinator is a Node process, which makes a Node-embedding desktop shell the natural packaging. The recommended shell is Electron: because the coordinator is already Node it runs in Electron's main process with no extra runtime, and Electron is well understood for enterprise code-signing and managed distribution. The lighter alternative is Tauri, a Rust shell that runs the Node coordinator as a sidecar; it yields smaller binaries at the cost of a Rust dependency. The choice is Node-native maturity against footprint, and it does not affect the architecture, since either shell embeds the same coordinator and serves the same React client.

For v1 the delivery surfaces are the desktop application and the browser; a mobile client is out of scope. In an enterprise setting the desktop build also gives IT a signed artifact to distribute and control, which the browser surface alone does not.

On the client stack, the React interface is built on Radix primitives through the shadcn/ui pattern, where components are copied into the repository rather than pulled in as a dependency. That suits an open-source tool: contributors own and extend the components directly, there is no library lock-in, and the Radix primitives carry keyboard, focus and screen-reader behaviour by default, which is most of NFR-6 for free. The default theme is a neutral monochrome, deliberately unbranded, so any team can re-skin it to their own identity. The product ships looking like infrastructure, not like one company’s app.

09 / Integration model

The harness owns the integrations, the client owns the picture

Integrations into third-party systems happen at the harness, through CLIs and MCP servers that are registered and authorised at the host. The agent invokes them as tools inside its session, gated by the permission protocol. The browser is never in the credential path and never makes a call to a system of record. This is the same execution boundary that already applies to the repository, extended evenly to external systems.

9.1 Two places an integration can fire

A useful distinction governs how each integration is wired.

  • Agent-side, as part of the work. Creating a branch and opening a pull request is legitimately the agent's own work. It does this through the GitHub CLI or MCP in-session, gated by a permission prompt.
  • Projection-side, as a governed record. Writing to a system of record, the Jira story status or the tracking entry, is driven deterministically from a specification or status change by a plugin, not by free-form agent behaviour. Its only job is to be a reliable, audited record, so it is code.
Why projection is deterministic

An audit trail is only as trustworthy as the thing that writes it. A non-deterministic agent should not be the author of a system-of-record entry. The agent may propose the update; a plugin reacting to a spec-status or todo.updated event performs it the same way every time, and logs it.

10 / Client surfaces & workflows

Three workflows, one control pattern

The client is organised around the three things a Product Engineer does with a specification: draft it, review it, and generate its artifacts. Each has its own surface, and all three obey one rule: the agent proposes, the human decides, the harness executes.

Propose · decide · execute

No agent output reaches the repository, a system of record, or a teammate without a human decision in between. The agent drafts, the Product Engineer approves or edits, and only then does the harness perform the write.

10.1 Authoring cockpit · draft the specification

Drafting happens in a split surface. On one side, a conversation with the authoring agents, the Product Owner, Technical Architect and Engineering roles, where the Product Engineer directs the work in natural language and each turn names the agent and the model in play. On the other, a live preview of the specification as it takes shape, rendered section by section against the anatomy: frontmatter, then requirements, design and tasks.

The preview is a view of the working file the agent is editing in the repository, not a second copy held by the client. A status bar carries the frontmatter and the current state. When the Product Engineer approves, the specification is committed to docs/specifications and its status advances. Until then it lives on a branch, visible and reviewable but not yet authoritative. At any point the Product Engineer can convene a review panel on the working draft without leaving the cockpit (see 10.2).

10.2 Review panel · critique before finalising

Review can be convened on demand at any point during authoring, not only as a final check before approval. When the Product Engineer is unsure about a section, they call a panel there and then rather than waiting for the end. A panel runs more than one agent, each on a different model, for example one on Opus and one on GPT-5.5, and each grounded in the source code. They critique the same specification independently. Different models have different blind spots, so a cross-model review grounded in the codebase catches more than a single reviewer would, and it stops the authoring model's own blind spots from passing unchallenged.

The panel presents each reviewer's critique side by side, marks where they agree and where they diverge, and attaches issues to the specification sections they concern. The Product Engineer adjudicates: accept a point, which feeds straight back into the live draft, dismiss it, or send a section back for revision. The reviewers propose; the human decides. A panel is expected at least once before approval, and may be run as often as the work needs it. The number of reviewers and the models they run are configurable.

10.3 Generation workspace · produce the artifacts

Once the specification is agreed, a generation workflow produces the downstream artifacts: documentation, Jira tickets, tests and tracking. The agent proposes them, drafting the tickets and the docs from the specification. The Product Engineer reviews the proposal and decides, approving, editing or rejecting. Only then does the system execute the write, through the harness's registered CLIs and MCP servers, and deterministically for systems of record. Nothing is written to Jira or to the docs until the human has approved the preview.

11 / Functional requirements

What the orchestrator must do

must
FR-1
Specification authoring workspace

Provide a sectioned authoring surface reflecting the specification anatomy: frontmatter; requirements (summary, scope, acceptance criteria, open questions); design (architectural decision, target architecture, data model, API contracts, application services, security, performance); and tasks (implementation plan, testing, definition of done, decision log). The specification is co-authored by the human and AI and persisted as a markdown file in docs/specifications via the harness. The client never holds the canonical copy.

must
FR-2
Agent-role orchestration

Route authoring to named agent roles, a Product Owner agent for requirements, a Technical Architect agent for design, an Engineering agent for tasks, and dispatch implementation agents for execution. Each message specifies the agent and the model. The orchestrator coordinates the roles; it does not run them.

must
FR-3
Codebase grounding

Use the harness to analyse the repository and produce or update an AGENTS.md baseline, and surface agents.md, guides and relevant existing code so authoring is grounded. The richer the context-engineered codebase, the richer the specification.

must
FR-4
Model tiering

Drive specification authoring with the most capable model and implementation with a mid-tier model. Model selection is per message or session, with sensible defaults per agent role, and is configurable.

must
FR-5
Specification lifecycle & governance

Carry specification status through draft → in-review → approved, stored in frontmatter and reflected in the UI. Tie transitions to pull-request state. The specification is versioned in git and reviewed through pull request.

must
FR-6
Artifact projections (fan-out)

From an approved specification, generate downstream artifacts: tasks, tickets, test scaffolds, tracking entries and docs. Projections are regenerated from the specification when it changes. The specification always wins; downstream edits do not flow back into it.

must
FR-7
Deterministic projection to systems of record

Drive the system-of-record update from a specification or status change deterministically, via a harness plugin reacting to events, not via free-form agent behaviour. Every projection write is logged with the triggering change.

must
FR-8
Parallel task execution

Fan an approved specification into tasks dispatched as asynchronous child sessions, each in its own git worktree or branch, executed concurrently by implementation agents. The orchestrator must not block while they run.

must
FR-9
Delivery board as a projection

Present a kanban board where a card is a specification or a task and its column is computed from frontmatter status, session status and pull-request / CI state. The board updates from the harness event stream. A card moves because the work moved, not because a person dragged it.

must
FR-10
Human-in-the-loop permissions

Surface agent permission requests in the client, let the Product Engineer approve or deny, send the response back to the harness, and log every decision.

should
FR-11
Diff review & rescue

Retrieve and display the per-session diff for review before a pull request is opened, and support reverting and restoring a run so a human can rescue an agent that has gone wrong.

should
FR-12
Reviewer approval

Allow a second human to review the specification and the generated change and approve before merge, mirroring pull-request review. Approval gates the merge.

could
FR-13
Command palette

Expose harness commands (for example author-spec, generate-tasks) as first-class actions the Product Engineer can invoke against a session.

must
FR-14
Authoring cockpit: chat & live preview

Provide a split surface pairing a conversation with the authoring agents (each turn naming the agent role and model) with a live preview of the specification rendered section by section. The preview is a view of the working file in the repository, never a separate copy.

must
FR-15
Approve & persist

On human approval, commit the specification to docs/specifications and advance its status. Before approval the specification lives on a branch, visible and reviewable but not authoritative.

must
FR-16
Multi-model review of the specification

Let the Product Engineer convene a multi-model review on demand at any point in authoring, and at least once before finalisation. A panel runs two or more agents on different models, each grounded in the source code, critiquing the specification independently. Present the critiques side by side with agreement and divergence marked and issues attached to the relevant sections; accepted points feed back into the draft. The number of reviewers and their models are configurable.

must
FR-17
Generation workflow: propose, decide, execute

After the specification is agreed, let an agent propose the downstream artifacts (docs, tickets, tests, tracking), present them as a preview for the human to approve, edit or reject, and only then execute the write through the harness. Nothing is written to a system of record before human approval.

must
FR-18
Method-ready project scaffolding

Initialise a repository with the agent roster (committed in the harness's convention, for example .opencode/agents), the docs/specifications structure, the grounding baseline and the policy and projection plugins. Agents reference logical model tiers resolved to the internal gateway, and the canonical roster is harness-agnostic, materialised per adapter.

should
FR-19
Multi-harness routing

Maintain a registry of connected harnesses with their capabilities and available models, and route a role or session to a harness by the model it needs rather than by an explicit harness mode. Aggregate sessions across harnesses on one board, labelled by harness and model. Continuity lives on the feature branch in git, so sessions are harness-local and disposable.

12 / Adapter, scaffolding & harnesses

One interface, many harnesses, honest about differences

The orchestration logic targets a backend-agnostic interface so that no harness is load-bearing. The OpenCode adapter is the first and, for a while, the richest implementation. Other harnesses expose different surfaces and will not all provide per-session task lists or permission events in the same shape, so the interface carries capability flags. The board and the cockpit degrade against a backend's real capabilities rather than assuming the full contract. The interface is schema-first and validated at the boundary, so a malformed event from any backend is caught rather than silently corrupting the board. Where a backend speaks the Agent Client Protocol (ACP), the emerging standard that OpenCode supports, the adapter normalises through ACP rather than a bespoke integration, and bespoke adapters become the exception used only where a harness exposes no standard protocol.

12.2 Method-ready project scaffolding

Setting up a repository for this method is a first-class action, not a manual checklist. Initialising a project scaffolds the agent roster, the docs/specifications structure, the grounding baseline (AGENTS.md and the guides), and the policy and projection plugins, so a new repository is method-ready from the first commit.

Among what it scaffolds is the canonical specification template. Every project ships one specification.template.md that fixes the format: a single file per feature, authored on the branch, with requirements written as normative SHALL statements that each carry at least one WHEN/THEN scenario, and changes marked by in-place delta tags that flatten into plain contract on merge. The template carries its own lifecycle rules, so an engineer or an agent starting a spec inherits the method’s rigor without having to remember it. This single file is the unit the cockpit previews, the review panel critiques, and the board projects.

The agents ship into the repository, not onto a machine. In OpenCode they are committed under .opencode/agents as markdown files with frontmatter, so they version with the code and are reviewed through pull request like the specification. This turns the authoring and reviewing roles, and the permissions that constrain them, into governed artifacts rather than per-engineer configuration.

Two rules keep the scaffold portable. Agents reference logical model tiers, a capable tier and a mid tier, resolved per project to the internal model gateway, rather than hardcoded vendor model IDs that go stale and bypass the gateway. And the canonical roster is harness-agnostic: the method owns the role definitions and each adapter materialises them into its harness's format. For v1 the scaffold is written once and the project owns it; a managed sync that offers updates as a reviewable diff is a later addition, alongside the team tier.

The scaffold also seeds grounding for the agents. Authoritative reference implementations are vendored into the repository as read-only material under a .repos convention, and the grounding file instructs the agents to prefer patterns from that vendored source over guesses or web search. A single canonical AGENTS.md, symlinked to CLAUDE.md so both conventions resolve to one file, encodes the completion gates (typecheck and checks must pass), the priorities, and the reference repositories. This is how the coding agent is grounded not only in our method but in proven systems built in the wild.

12.3 Working across more than one harness

An engineer may use more than one harness, for example Claude Code for the Claude models and OpenCode for others, and move between them. The design keeps this simple by refusing to make the harness a mode the engineer lives in. The engineer chooses a role and a model; a routing layer chooses the harness.

This works because continuity lives in the repository, not the harness. The specification and the code are on the feature branch in git; a harness session is ephemeral and disposable, so switching harnesses loses nothing durable. You start a fresh session on the other harness against the same branch. The model resolution already in use, tier to concrete model, simply extends to tier to model to harness, so the router and the model-tier map are one mechanism. The agent asks for a tier and never needs to know which harness served it.

The board aggregates sessions across harnesses into one view, labelling each card with the harness and model executing it, so the picture stays unified rather than split into per-harness silos. Capability flags absorb the differences: a card on a harness that reports fewer signals simply shows fewer. The multi-model review panel benefits directly, since one reviewer can run a Claude model through Claude Code and another a different model through OpenCode.

Three things to hold deliberately. Concurrent work by two harnesses against one working tree can collide, so parallel sessions run in their own worktrees off the feature branch. Credentials and integrations are registered per harness host, so each host carries its own authorised CLIs and MCP servers. And Claude Code's headless surface differs from OpenCode's and should be verified before that adapter is built.

13 / Non-functional requirements

Constraints that traceability and control make mandatory

NFR-1 · Security
No credentials in the client

Service tokens and app credentials are never shipped to the browser or placed in URLs. All credentials are held at the harness host where the CLIs and MCP servers are authorised. System-of-record calls originate from the host, not the client.

NFR-2 · Auditability
Every governed action is recorded

Specification history lives in git. Permission decisions and deterministic projections are logged with their trigger. The result is a traceable line from business intent to released change.

NFR-3 · Performance
Event-driven, never polled

Work is dispatched asynchronously and the UI reacts to the event stream. Multiple sessions run concurrently without freezing the picture.

NFR-4 · Reliability
Agent failure is a first-class state

Runs that fail or loop surface as a board state needing a human, with revert and restore available for rescue. Failure is handled in the interface, not hidden.

NFR-5 · Portability & sovereignty
Self-hosted, internal models

The harness is self-hosted and configured against an internal model gateway, so code and prompts stay on controlled infrastructure.

NFR-6 · Quality floor
Accessible, resilient client

Keyboard-navigable, visible focus, reduced-motion respected, responsive. The orchestrator is a tool used daily and is built to that standard.

NFR-7 · Observability & audit
The trace is the audit trail

The coordinator emits OpenTelemetry spans at every boundary: RPC, orchestration command, provider call, external process, persistence and queue handoff. IDs and paths go on span attributes; metric labels stay low-cardinality. Completed spans are written to a local append-only trace as the persisted source of truth, exportable over OTLP to a standard backend. Where teams must keep an audit trail, this structured record of what every agent did, including approvals and projections, is it.

NFR-8 · Resilient streaming
Survive reconnects and partial streams

Push events are ordered, monotonically sequenced per connection, and schema-validated at the boundary. The client transport is a reconnecting state machine that queues outbound requests while disconnected and replays the latest state on resubscribe. Async milestones are modelled as typed receipts rather than polling, and git checkpoints are captured around agent turns for rescue.

14 / Security, governance & audit

Truth, credentials and accountability each have one home

Three boundaries keep the design safe where governance matters, and they reinforce each other.

14.1 The credential boundary

Because the harness owns execution, it owns authorisation. CLIs are authenticated once on the host and MCP servers are registered in the harness configuration. The agent invokes them as tools. The token never touches client-side code because the client was never in the credential path.

14.2 The truth boundary

The specification is authoritative and lives in the repository. Projections into Jira, tests and tracking are generated from it and never override it. This prevents the drift that a parallel system of record would create, and it is the same principle applied to delivery tooling that is applied to the specification itself.

14.3 The accountability boundary

Holding the credential is not the same as being accountable for the write. The deterministic projection plugin is the accountable author of every system-of-record entry, so each entry is reliable and logged. A human reviewer remains accountable for what ships. Owning a feature alone does not remove the second pair of eyes on a release. And the structured trace the coordinator emits, with spans for every provider turn, approval and projection, is the durable record that makes all of this accountable after the fact (see NFR-7).

16 / Phasing & roadmap

Prove the loop on one repo, then widen

Phase 1 · Prove the loop now

One harness (OpenCode), engineer-local execution, a single repository, single-engineer ownership. The specification authoring loop end to end; the board driven by real pull-request, run and event state; GitHub for branches and PRs and Jira for the deterministic projection; permission gates and diff review. The goal is to prove the claim that the card moves because the work moved.

Phase 2 · Widen integrations & review

Add Azure DevOps for repositories and work items; richer fan-out into tests, docs and tracking; the reviewer approval workflow; and a second, capability-flagged harness adapter that also exercises multi-harness routing, so a role can run on whichever harness serves the model it needs.

Phase 3 · Scale & team

Optional central runners or cloud sandboxes for execution at scale; a read-only team view aggregating delivery, underwritten by git; per-user identity and role-based approval above the harness; enterprise audit export; and a managed sync that updates scaffolded agents against the canonical set as a reviewable diff.

18 / Out of scope

Deliberately not in this release

  • A multi-user, team-wide collaboration surface. A read-only team view is deferred and git underwrites it when it arrives.
  • Per-user identity and SSO above the harness. Added with the team view.
  • Central cloud execution. Phase 1 is engineer-local; runners come later.
  • Replacing the integrated tools. The orchestrator drives them; it does not supersede them.
  • Fully autonomous delivery with no human approval.
19 / Open questions

Decisions still to make

  • Execution topology. Engineer-local host versus central runners, and how the client reaches the host across a locked-down corporate network.
  • Identity above the harness. When the team view arrives, how per-user identity, roles and approval are modelled, given the harness uses a single shared server password.
  • Direction-of-truth enforcement. How strictly downstream edits are prevented or reconciled when a projection is changed by hand in Jira.
  • Internal model gateway. The exact provider configuration to route models through controlled infrastructure.
  • Experimental surfaces. How much to lean on the tools endpoints and the compaction hook before they stabilise.
  • Reviewer model for solo ownership. Who reviews when one engineer owns the feature, and how that is assigned.
  • Remote access pattern. Reaching a harness host across a restricted corporate network: a reference implementation proves SSH, a mesh VPN and a relay as options to evaluate.
  • ACP maturity. How complete Agent Client Protocol support is across the harnesses we target before we rely on it as the primary normalisation path.
  • Desktop shell. Electron versus Tauri: Node-native maturity and distribution familiarity against a smaller footprint.
  • Concurrent harness execution. Worktree isolation when more than one harness works a branch at once, and verifying Claude Code's headless surface before the second adapter.
  • Managed agent sync. Whether and how to update scaffolded agents against a canonical set as a reviewable diff, given it implies a central authority.

21 / Prior art & grounding

Grounded in our method and in a system proven in the wild

This product is not the first web layer over coding agents, and that is useful. A mature open-source reference, T3 Code, is a minimal multi-provider GUI over Codex, Claude Code and OpenCode. It deliberately omits everything that is our reason to exist, the specification as source of truth, governance, audit, and the controls a governed environment needs, which makes it an ideal architectural reference rather than a competitor for the same ground. We build and open-source our own for the tighter control that governed delivery demands, and we ground the build in that reference so the coding agent works from proven patterns rather than guesses.

21.1 What we adopt

PatternWhat we take from it
Thin local coordinatorA local WebSocket coordinator wraps the agents and normalises their events; no cloud backend on the hot path (8.5)
Schema-first contractsThe adapter interface is a validated schema shared by client and coordinator, with no runtime logic
Normalised domain eventsProvider-native events are ingested, persisted and exposed as one canonical model the board reads from
ACP for normalisationSpeak the Agent Client Protocol where a harness supports it; bespoke adapters are the exception
Resilient streamingOrdered, sequenced, validated pushes; a reconnecting transport with replay; typed receipts over polling (NFR-8)
Git checkpointsAutomatic checkpoints around agent turns for rescue and audit
Runtime modesA supervised mode (approval on request, workspace-write sandbox) as the default posture where changes must be governed
Observability as auditOpenTelemetry spans at every boundary, a local trace as persisted source of truth, OTLP export (NFR-7)
Vendored groundingReference repositories vendored read-only into the repo, with the grounding file pointing agents at them (12.2)

21.2 Where we differ

The reference is method-agnostic, single-user and local, with no system of record beyond the chat session. Ours is built around the specification as source of truth, the propose-decide-execute control pattern, deterministic projection into governed systems, a human reviewer for what ships, and an audit trail fit for governed delivery. Those are the layer the reference omits, and they are the product.

21.3 Vendored for grounding

When the build begins, the reference implementation and a small set of supporting repositories are vendored into our repository as read-only material, and the grounding file directs the coding agent to prefer their patterns. The agent is therefore grounded in two things at once: our method, and a system that already works in the wild.
